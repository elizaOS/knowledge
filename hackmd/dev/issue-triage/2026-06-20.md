## Patterns / Themes Suggesting Deeper Architectural Problems
- **Config-driven fragility across environments:** Multiple issues stem from env var drift (staging vs prod), mismatched bindings, and proxy/routing inconsistencies.
- **Migration churn without a single “system contract” test suite:** Neon→Railway, BitRouter→Worker gateway, and proxy changes repeatedly surface regressions that would be caught by a canonical end-to-end acceptance pack.
- **Security hardening is reactive and surface-specific:** SSRF and auth/proxy token propagation fixes appear in bursts; a centralized threat-model checklist per “route type” is needed.
- **Operational reliability depends on implicit state stores:** KV/Redis choice and provisioning job recovery correctness are critical; the system needs clearer “source of truth” boundaries and health assertions.

---

## Process Improvements (to prevent repeats)
1. **Add a Release Readiness Gate Pack**: one workflow that runs “login → provision → chat → resume → billing tick (mock)” across staging and blocks prod promotion if it fails.
2. **Environment Parity Checks**: automated diff of critical env vars/bindings (Worker + daemon) with fail-closed validation at boot (print resolved backend choices).
3. **Security Guardrail Library + Mandatory Usage**: one outbound URL validation module, used everywhere, with a standard test matrix (including empty/edge inputs).
4. **Incident Postmortem to Prevent Regression**: for #8696-like events, require: (a) monitoring added, (b) migration plan for existing resources, (c) rollback plan, before closing the incident class.
