## Patterns / Themes Indicating Deeper Architectural Problems
- **Cross-service contract drift (Steward ↔ Cloud frontend/API):** PKCE requirement and signing expectations changed upstream, but client/proxy contracts weren’t enforced with automated compatibility checks.
- **Configuration as a reliability bottleneck:** Missing/incorrect prod secrets and redirect allowlists can fully brick auth; indicates a need for stronger preflight validation and environment parity.
- **Billing as an emergent system without a single source of truth:** Ledger mismatch suggests multiple balance sources with unclear invariants, risking trust and monetization reliability.
- **Autonomy expansion increases security surface area:** As goal loops + self-spend capabilities expand, execution boundaries must be enforced systematically, not per-feature.

## Process Improvement Recommendations
1. **Auth Contract CI Gates:** Add staging E2E tests for OAuth (Google + email) that run on every auth-related change, with a prod-canary check before full rollout.
2. **Environment Preflight Checks:** Automated deploy-step that verifies required secrets, redirect allowlists, and tenant IDs *before* promoting to production.
3. **Ledger Invariants + Reconciliation:** Define authoritative ledger rules in writing, implement reconciliation + alerting, and add idempotency/audit requirements for spend paths.
4. **Security Review Loop for Autonomy:** Lightweight security checklist + mandatory review for any change that touches: tool permissions, signing/trading, vault access, or autonomous spend.
5. **Incident Runbooks + Postmortems:** For outages like login failures, publish a short postmortem and a “how to debug” runbook to reduce recovery time next time.
