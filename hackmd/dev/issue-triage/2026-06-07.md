# Patterns / Themes Indicating Deeper Architectural Problems
- **Auth coupling across services:** Steward hardening (PKCE + HMAC sensitive-path guards) exposed tight coupling between Steward, cloud-frontend, and cloud-api proxy logic. Small upstream changes can fully break login.
- **Breaking refactors without compatibility rails:** The LifeOps split is strategically good, but removal without shims/migration UX increases upgrade risk and support load.
- **Security posture lagging product direction:** Community interest is trending toward autonomous DeFi agents; security boundaries (prompt injection, transaction-signing policy) need to mature in parallel or risk a major incident.

# Process Improvement Recommendations
1. **Production auth canary + “go/no-go” gate:** Add a CI/CD step that runs a real-browser login canary (OAuth + email) against prod immediately post-deploy, paging maintainers on failure.
2. **Upstream change monitoring for Steward:** Track Steward SDK/service changes with a pinned compatibility matrix; require a “Steward bump” checklist (PKCE, signing, tenant, redirect allowlist).
3. **Compatibility policy for major plugin refactors:** For removals/renames, require (a) migration doc, (b) runtime detection with actionable error, and (c) at least one release cycle of deprecation warnings when feasible.
4. **Security RFC pathway:** Establish a lightweight RFC template for high-risk capabilities (wallet signing, trade execution) with threat modeling and default-deny policies before promoting autonomous workflows.
