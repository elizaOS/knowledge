## Cross-Issue Patterns / Themes (Possible Deeper Problems)

- **Tooling contract drift / planner non-determinism:** Even when Stage 1 identifies the correct action, the planner can ignore it and pick “power tools” (SHELL) that are risky and slow. This suggests the system lacks a strong “hint compliance” contract between Stage 1 and the planner.
- **User-facing leakage boundaries are still porous:** Sub-agent control metadata reaching end users indicates missing sanitization layers and unclear “control vs user text” separation.
- **Operational security is part of product quality:** Discord spam shows that community ops controls (verification, permissions, runbooks) need to be treated like first-class release concerns, not ad-hoc moderation.

## Process Improvements (Prevent Recurrence)

1. **Introduce an “Ops/Security Triage” lane** (even a simple GitHub Project board) to track Discord incidents with owners, checklists, and SLA (e.g., P0 response within hours).
2. **Add planner regression tests for common intents** (“search chat for X”, “find message about Y”) that assert:
   - the correct tool/action is selected,
   - no unbounded shell commands are executed,
   - no internal routing metadata is emitted to connectors.
3. **Define and enforce a “user-facing text contract”**:
   - All internal planner/sub-agent annotations must be tagged and stripped before connector output.
   - Add a CI test that fails if known control markers appear in connector responses.
4. **Make CLI/API failures self-diagnosing**:
   - Standardize 4xx error surfaces to include “missing field” hints and example payloads.
   - Add `--debug` guidance in docs and ensure logs are safe to share (redaction).
