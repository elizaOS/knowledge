## Patterns / Themes Indicating Deeper Issues
- **Silent fallback / missing capability signaling:** Multiple areas (spec-decode disablement, performance cliffs) risk “it runs but is slow/broken,” which is worse than failing loudly.
- **Rapid architectural evolution outpacing docs:** v2 native-app integration + OWNER/AGENT identity model is powerful but creates onboarding gaps.
- **Cross-platform + deployment surface area is now huge:** Desktop/mobile/cloud/local inference/Docker all moving together; regressions often come from “matrix edges,” not core logic.
- **Ecosystem-driven growth (plugins/apps) requires upstream guardrails:** Third-party production apps and plugins amplify brand impact and highlight missing best-practice templates.

## Recommendations (Process Improvements)
1. **Convert Discord action items into GitHub issues within 24 hours** (with owner + acceptance criteria), especially for docs/deployment requests.
2. **Add “fail loud” standards:** any degraded mode (spec-decode disabled, GPU kernel fallback, missing stealth preload equivalents) should surface as explicit UI/API status.
3. **Capability/compatibility gating in recommenders:** require machine-verifiable capability flags before routing users to advanced paths (spec-decode, SWA/M-RoPE, special kernels).
4. **Introduce a lightweight “Production App Review” checklist** for ecosystem launches (security, secrets, auth, rate limiting, injection defenses).
5. **Expand nightly bench coverage by model family + architecture traits** (SWA, M-RoPE, unusual head dims) to catch performance/correctness cliffs early.
