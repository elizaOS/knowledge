## Patterns / Themes Suggesting Deeper Issues
- **Planner/tool-selection brittleness:** Even when Stage 1 provides the correct hint (`candidateActions`), downstream planning can choose expensive/incorrect tools (SHELL), indicating the system lacks a strong “hint adherence” contract.
- **Missing guardrails for costly tool calls:** Unbounded filesystem searches and long timeouts are still reachable. This is both a cost/latency issue and a reliability perception issue.
- **User-visible leakage of internal control planes:** Sub-agent routing metadata surfacing to end users suggests output contracts between orchestrator ↔ connector are not strictly enforced.
- **Ops/security surface is part of product perception:** Discord spam and impersonation degrade trust as much as a code vulnerability, but currently appears under-tracked (not consistently turned into issues with owners/SLA).

## Recommendations (Process Improvements)
1. **Discord → GitHub intake pipeline:** Add a lightweight “ops/security incident” issue template and require that actionable Discord reports become tracked issues within 24 hours (owner + priority + mitigation checklist).
2. **Planner regression suite for “hinted tool routing”:** Maintain a small set of canonical prompts that assert:
   - Stage 1 hint present → planner must prefer that action
   - costly SHELL patterns are rejected or tightly scoped
   - sub-agent outputs never include control metadata
3. **Tool cost budgets & safe defaults:** Centralize constraints (max runtime, max output bytes, forbidden command patterns) so connectors and deployments inherit safety automatically.
4. **Improve “deployment CLI contract” validation:** For commands like uploads, validate required parameters client-side and provide copy-pastable fixes; add “known good” examples in docs.
5. **Contributor onboarding for constrained environments:** Document supported dev modes (containerized, remote, read-only) and add a `doctor` command that detects missing JS runtime and suggests paths forward.
