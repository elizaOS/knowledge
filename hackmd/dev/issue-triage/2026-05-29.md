## Cross-issue patterns / themes (possible deeper architectural problems)
- **Planner/tool routing contract drift:** Stage 1 can hint the correct tool, but downstream planner execution still selects unsafe/expensive fallbacks (SHELL) and produces invalid action shapes. This suggests the planner prompt/selection policy is not sufficiently constrained by structured intent signals.
- **User-facing leakage of internal control channels:** Multiple signs of “internal metadata” appearing in user-visible replies (routing kind, orchestration details). This points to missing hard boundaries between control-plane messages and human-visible content.
- **Release reliability depends on many moving parts:** OS + cloud images show repeated “works after merging fixes” but need systematic gating (artifact completeness tests, canaries, contract checks) rather than reactive patches.
- **Operational surfaces are part of product quality:** Discord spam is not “just community”; it directly impacts trust and perceived security of elizaOS.

## Process improvement recommendations
- **Create “Ops/Security” issue templates** (Discord incidents, phishing, infra regressions) so reports become actionable with consistent fields (timestamps, screenshots, audit log entries, suspected permission vectors).
- **Add contract tests where regressions are recurring:**
  - Planner must honor Stage 1 `candidateActions` when present (or explicitly justify override).
  - Sub-agent outputs must be sanitized before connector post.
  - Cloud agent image must boot with an expected minimum set of core plugins.
- **Adopt “release rehearsal” checklists** for OS and cloud: one dry-run tag per week until consistently green, with artifacts verified and attached (checksums, SBOM, provenance, docs).
- **Instrument truthfulness flags in APIs/health endpoints:** expose whether spec-decode is active, whether key capabilities are loaded, and whether the runtime is in degraded mode—so UI and operators can act on real state rather than assumptions.
