# Patterns / Themes Indicating Deeper Architectural Risk
- **Silent fallback behaviors** (spec-decode disabled, routing mistakes) repeatedly create “it runs but it’s wrong/slow” failures. This is a systemic observability/contract issue between runtime, recommender, and inference backends.
- **Capability/permission model maturation** is a recurring bottleneck (connector capabilities, Satellites permissions, dynamic views). Without strict contracts + tests, feature velocity increases regression and security risk.
- **Cross-platform packaging complexity** (symlinks, build pruning, desktop/mobile paths) suggests tooling needs stronger invariants and fixture-based tests.

# Process Improvement Recommendations
1. **Introduce “Fail Loud” standards** for performance-critical toggles:
   - If a user requests a feature (spec-decode/DFlash/OpenVINO) and it can’t activate, require explicit API/UI surfacing (`active=false`, `reason`).
2. **Add capability- and permission-contract test suites**:
   - Connector capability matrix tests (threads/edit/reactions/typing) + orchestrator lifecycle integration tests.
3. **Pre-merge risk gates for large platform PRs**:
   - Require a short threat model + permission diff for Satellites/dynamic views changes; require “message routing invariants” tests for orchestrator changes.
4. **Community ops SLA + documented escalation**:
   - Ensure Discord help requests (especially migration blockers) have an owner rotation and a single canonical “what to do next” link.
