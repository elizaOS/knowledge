## Patterns / Themes Indicating Deeper Architectural Risk
- **Presentation vs. truth split:** Multiple reports indicate the system often *does the right work* but **fails to present the correct final answer** (or leaks envelopes). This suggests a systemic “reply assembly” fragility across orchestrator/planner/tool layers.
- **CI brittleness from fast-moving UI/shell changes:** Frequent UI refactors and route/shell changes appear to cause **test drift** and **gate instability** (Format Check + client tests).
- **Rising security surface area:** As agents gain stronger tooling (skills, browser eval, wallet/DeFi actions, cloud sandboxes), **execution boundaries and confirmations become product-critical**, not optional hardening.

## Recommendations: Process Improvements
1. **Establish a “CI Red = Stop-the-line” rule for `develop`:** Treat format/test failures as P0; require immediate fix before feature merges continue.
2. **Introduce stable “User-Facing Output Contracts”:**
   - enforce a structured `userFacingText` channel from tools/skills
   - add regression tests ensuring envelopes/tool transcripts never leak to user replies
3. **Add a Security Review Checklist for tool/action PRs:**
   - confirmation semantics, logging/auditing, least-privilege defaults
   - prompt-injection test cases for any new high-power action (wallet, shell, browser eval)
4. **Create a lightweight “Designate Owners” map per subsystem** (Orchestrator, CI, Cloud Auth, UI Shell) so triage can assign quickly and avoid diffused responsibility.
