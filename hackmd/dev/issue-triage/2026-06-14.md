8. **P3:** **DISCORD-2026-06-13-WORLD-MODELS** — World model tooling + migration guides.

## 2) Patterns/themes indicating deeper architectural problems
- **Auth/tenant configuration fragility:** Multiple failures manifest as tenant mismatch, invite-only gating, PKCE contract drift, and environment parity issues (staging vs prod).
- **Go-live readiness gaps for Cloud Apps:** Rapid infra iteration is landing, but “known TODOs” in ingress/scaling/security/billing require a formal launch gate to prevent shipping partially wired systems.
- **Environment parity & “hidden configuration” risk:** Repeated themes of staging routing to prod, missing env vars, and manual SSH steps (now improving) suggest configuration drift as a primary failure mode.
- **Community trust as a product dependency:** Scam/spam and governance narratives directly affect adoption and contributor retention, functioning as a “soft availability” risk.

## 3) Recommendations for process improvements
- **Add release-readiness gates for Cloud:** a checklist-driven go/no-go that must pass E2E tests (signup/login, app deploy reachability, teardown, billing idempotency smoke).
- **Mandatory environment parity tests:** automated probes that assert staging never calls prod endpoints and vice versa (CORS allowlists, tenant IDs, router hosts).
- **Operationalize incident detection:** lightweight synthetic monitoring for auth and signup endpoints; alerting on spikes in 4xx/5xx for `/auth/*`.
- **Document and enforce release channel policy:** clearly define stable vs develop usage; provide upgrade tooling/docs to reduce Discord support load.
- **Harden community surface:** enable Discord AutoMod + a clear promo/scam policy, with metrics and escalation paths.
