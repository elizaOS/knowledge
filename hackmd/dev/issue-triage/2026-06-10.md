# Patterns / Themes Indicating Deeper Issues

- **Auth contract drift across services (Steward ↔ cloud-frontend ↔ cloud-api):** PKCE/redirect allowlists and endpoint expectations can change independently, breaking prod sign-in. This suggests insufficient **cross-service contract tests** and **prod-canary validation**.
- **Go-live readiness relies on audit issues rather than decomposed work items:** #8321 contains many discrete, shippable tasks; without splitting, ownership and completion tracking are harder, increasing launch risk.
- **Scale bottlenecks are primarily “systems defaults” (DB connections, ingress/LB assumptions, idempotency):** indicates the architecture is close, but operational hardening and guardrails need to be elevated to first-class engineering deliverables.
- **Security controls exist but are not enforced by provisioning:** defense-in-depth code that isn’t executed is a recurring risk pattern—controls must be applied automatically and verified continuously.

# Process Improvement Recommendations

1. **Add production auth canaries:** automated checks for OAuth authorize → exchange, magic-link email send/verify, and session refresh on every deploy (and hourly).
2. **Introduce “Launch Blocker” checklists for Cloud Apps:** ingress reachable, DB pooling present, teardown verified, billing idempotent, firewall rules applied; require explicit sign-off before enabling `APPS_DEPLOY_ENABLED`.
3. **Split large audit issues into sub-issues with owners + SLAs:** especially for #8321; ensure each item has testable acceptance criteria and a staging verification step.
4. **Mandate “security control execution” tests:** CI/staging scripts that confirm firewall rules active, ports not publicly exposed, metadata blocked, and egress policies enforced.
5. **Registry governance updates:** document review criteria for security-sensitive plugins (signers, key management), including required disclosures and minimal verification steps.
