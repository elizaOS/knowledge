## Patterns / themes suggesting deeper architectural problems
- **Go-live readiness gaps cluster around “last-mile ops”**: bootstrap automation (cloud-init), multi-node routing, and lifecycle cleanup are recurring sources of near-launch risk.
- **Billing correctness is under-specified as invariants**: both #8321 and the dev log flag problems that typically require explicit DB constraints + idempotency and strong audit trails.
- **Cross-service contract drift** (Cloud ↔ Steward) can break the entire product with small upstream changes (PKCE/signing requirements), indicating a need for stronger contract tests and version pinning strategy.

## Recommendations for process improvements
1. **Introduce a “Go-Live Checklist” gate** for Cloud Apps: ingress reachable, DB pooling enabled, multi-node routing validated, billing invariants enforced, defense-in-depth applied, deprovision works.
2. **Add contract/integration smoke tests** for Steward auth flows (Google, email, passkey) that run on every deploy to staging/prod and page on regressions.
3. **Make billing idempotency a hard requirement**: enforce unique constraints + idempotency keys at the DB layer; require “re-run safe” cron design before production.
4. **Operationalize infra TODOs**: convert `STAN TODO` / “manual fix” items into tracked issues with owners and acceptance tests (fresh-node bootstrap CI).
5. **Community audit intake workflow** (for plugin-discord voice and similar): require repro steps + logs + expected/actual before allocating maintainer time; provide a template and a minimal load-test harness for contributors.
