# ElizaOS Developer Update — 2026-05-21 to 2026-05-27

This week continued the V2 hardening effort (post **v2.0.2**) with major progress on: (1) the **Plugin “mode unification”** work (direct vs remote), (2) the **OWNER+AGENT** connector identity model reaching more integrations, (3) Cloud API expanding into deploy + identity endpoints, and (4) security/reliability fixes across runtime + CI.

---

## 1) Core Framework

### Plugin architecture: unify “Plugin” with `mode: "direct" | "remote"`
The multi-PR stack formalizing “remote plugins” (formerly Satellites/Carrots) as first-class plugins landed in stages:

- **P0 scaffolding & vocabulary**: `Plugin` now carries a `mode` field, and manifests/wire-envelope types are aligned accordingly.  
  PR: [#7844](https://github.com/elizaos/eliza/pull/7844)

- **P1 remote-mode runtime**: introduces the worker bootstrap, host-side bridge parity, and an isolated-process worker runner for remote-mode plugins.  
  PR: [#7846](https://github.com/elizaos/eliza/pull/7846)

- **P2 view-host shims**: standardized view-host shims across platforms (web/electrobun/android/ios), plus service + route surfaces for remote plugins.  
  PR: [#7847](https://github.com/elizaos/eliza/pull/7847)

- **P3 agent-generated plugin install API**: adds `installRemotePlugin` plumbing and a first reference sub-agent remote-runner package.  
  PR: [#7848](https://github.com/elizaos/eliza/pull/7848)

**Implication:** plugin authors should treat “remote execution” as a deployment mode, not a separate abstraction. Expect follow-up tightening around permissions and trust policy enforcement at install-time.

Docs/refs:
- Remote plugin host settings UI is surfaced via the settings section introduced in P0 (see changes in `packages/ui/src/components/settings/RemotePluginHostSection.tsx` in [#7844](https://github.com/elizaos/eliza/pull/7844)).

---

### Electrobun platform: first-party satellites + dynamic views (now aligned with the “remote plugin” direction)
The Electrobun convergence work shipped a full host bridge for renderer↔worker commands/events and a dynamic view registry/session manager.  
PR: [#7779](https://github.com/elizaos/eliza/pull/7779)

Even though terminology is converging toward “remote plugins”, this PR is important for developers building rich desktop tooling:

- `carrotInvokeWorker` / `carrotTailWorkerEvents` establish the typed RPC surface for host-managed workers.
- Dynamic views enable agent/plugin-created UI panels backed by canvas/A2UI sessions.

---

### Cloud lifecycle reliability: agent lifecycle ops moved to job queue
Cloud agent lifecycle endpoints were refactored so Workers enqueue jobs instead of calling control-plane cores inline:

- “All lifecycle ops via job queue”: suspend/restart/logs/snapshot/resume now enqueue and return **202 + jobId** instead of synchronous completion.  
  PR: [#7810](https://github.com/elizaos/eliza/pull/7810)

- Frontend updated to track async job IDs for toasts and avoid premature “success” UI.  
  PR: [#7813](https://github.com/elizaos/eliza/pull/7813)

**Developer impact:** if you have custom tooling calling lifecycle endpoints, ensure your client handles `202 Accepted` and polls job status (or reads the job state if you have internal access).

## 2) New Features

### Cloud deploy API (server-side half of `elizaos deploy`)
Two new Cloud API endpoints complete the server-side portion of the `elizaos deploy` design:

- `POST /api/v1/apps/[id]/deploy`
- `GET  /api/v1/apps/[id]/deploy/status`

PR: [#7804](https://github.com/elizaos/eliza/pull/7804)  
CLI design doc (keel): [#7786](https://github.com/elizaos/eliza/pull/7786) (`packages/elizaos/src/commands/DEPLOY_DESIGN.md`)

Example (HTTP):
```bash
curl -X POST \
  "https://api.elizacloud.ai/api/v1/apps/$APP_ID/deploy" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "dryRun": true
  }'
```

## 3) Bug Fixes (critical / developer-impacting)

### Planner loop runaway validation: canonical failure signatures
The `maxRepeatedFailures` guard could be defeated when the model retried the “same” tool with slightly different invalid arg shapes. Fix introduces a canonical failure signature to stop endless retries.

PR: [#7924](https://github.com/elizaos/eliza/pull/7924)

**Why this matters:** it prevents cost blowups and latency spikes in tool-heavy agent flows (especially with permissive models that “keep trying” malformed JSON/tool args).

---

### Stage 1 message handling: preserve speaker context + prevent phantom action claims
Fixes a class of failures where Stage 1 would fabricate entities or claim tool calls were performed. Tightens prompt-rule contracts and preserves speaker names.

PR: [#7923](https://github.com/elizaos/eliza/pull/7923)  
Follow-up refactor removing regex-list evaluators in favor of prompt-rule contracts:
PR: [#7931](https://github.com/elizaos/eliza/pull/7931)

A change attempted to drop `dm.read`/`dm.write` scopes by default (X rejects OAuth if the app lacks entitlement), but was reverted after confirming the production failure was due to callback URL mismatch rather than DM scopes.

**Action for devs:** if you operate your own X app credentials, verify your callback URL matches the configured Cloud OAuth gateway callback precisely; do not rely on scope trimming to fix “Something went wrong”.

## 7) Breaking Changes / V1 → V2 Migration Warnings

### OWNER+AGENT connector model is now “real” (not just conceptual)
Connector registry entries can now declare `accounts: { owner?, agent? }`, and multiple connectors (X, Google, Slack; plus Linear/Calendly/Shopify role-correctness work earlier) are aligning to this.

- Connector schema + audit: [#7817](https://github.com/elizaos/eliza/pull/7817)
- UI role-aware setup: [#7819](https://github.com/elizaos/eliza/pull/7819)
- X dual-role OAuth: [#7851](https://github.com/elizaos/eliza/pull/7851)
- Slack dual-role OAuth + routing: [#7868](https://github.com/elizaos/eliza/pull/7868), [#7876](https://github.com/elizaos/eliza/pull/7876)

**Migration risk:** if your connector code or stored connector-account records assumed a single “user == agent” identity, you must update:
- OAuth start/complete flows to persist the requested role
- outbound client routing to respect account role (OWNER actions should not accidentally post as the agent bot)

---

### Plugin execution model: remote-mode plugins are now first-class
If you built against the older Satellites/Carrots vocabulary, migrate to:
- `Plugin` with `mode: "remote"`
- worker bootstrap + host bridge APIs (per [#7844](https://github.com/elizaos/eliza/pull/7844) → [#7848](https://github.com/elizaos/eliza/pull/7848))

**Migration risk:** manifests, permissions, and view-host integration points may have changed names/locations; audit any hardcoded paths to Electrobun-only artifacts.

---

### Cloud lifecycle client behavior must become async-aware
If you are calling Cloud lifecycle endpoints directly, you must handle:
- `202 + jobId` and delayed completion (PR [#7810](https://github.com/elizaos/eliza/pull/7810))
- UI logic that assumed immediate success may now be incorrect in custom dashboards.

---

### Open issue to watch: planner quality regression for “search chat history”
The planner is still sometimes choosing `SHELL` with unbounded greps instead of `SEARCH_MESSAGES`, and can leak sub-agent routing metadata.

Issue: [#7935](https://github.com/elizaos/eliza/issues/7935)

If you build connectors/tools around message search, consider guarding expensive shell tooling and validating user-facing outputs until the planner prompt/tool ranking is tightened.
