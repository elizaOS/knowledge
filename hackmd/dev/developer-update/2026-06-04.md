## 1) Core Framework (Architecture / Plugin System / Agent Runtime)

### Agentic view surfaces + view-action affinity (agent/runtime + UI integration)
Merged **“Claude/agentic views surface”**: [elizaos/eliza#8080](https://github.com/elizaos/eliza/pull/8080)

Key architectural changes:
- **Agent-surface framework**: plugin views can expose UI elements/capabilities to the agent via a registry + overlay mechanism (`ViewAgentRegistry`, `AgentSurfaceProvider`, `useAgentElement`, `AgentElementOverlay`).
- **View-action affinity**: new runtime layer biases the planner toward actions relevant to the *currently active view* (reduces “wrong tool” picks and improves UX when navigating view-heavy apps).
- **Lifecycle endpoints for compute**: introduces Cloud “sleep/wake” semantics (see API section) to support deep cold-storage of agent compute.
- **Incremental sandbox backups**: adds diff-based backup plumbing + migration to support incremental backups over time (see API section).

Developer impact:
- Plugin view authors can now design view components with an explicit “agent-control” surface area rather than relying on brittle DOM heuristics or prompt-only guidance.
- Planner behavior can vary based on active view context; test your tool-choice assumptions in multi-view flows.

Related docs/paths:
- Runtime: `packages/agent/src/runtime/view-action-affinity.ts`
- UI registry: `packages/ui/src/agent-surface/registry.ts`

### Core routing & planner-loop reliability improvements
- **Keyless web fetch becomes model-agnostic** (not Anthropic-only): [#8097](https://github.com/elizaos/eliza/pull/8097)  
  Moves “live web info inline” capability into a first-class action (`WEB_FETCH`) so non-Anthropic backends can answer live info requests without provider-specific monkey patches.
- **Coding vs web-search misrouting fixed**: [#8160](https://github.com/elizaos/eliza/pull/8160)  
  If a request “looks like coding” but mentions market/live terms, it now routes to coding instead of web search.
- **Planner output sanitation**: [#8159](https://github.com/elizaos/eliza/pull/8159)  
  Suppresses leaked JSON envelopes from user-facing replies when weaker planners hallucinate internal spawn/action argument blobs.

## 2) New Features (with technical details + examples)

### A) `WEB_FETCH` action (keyless, cross-provider live web retrieval)
Merged: [#8097](https://github.com/elizaos/eliza/pull/8097)

What’s new:
- `WEB_FETCH` is now a core runtime action available regardless of model provider.
- Enables “live info” answers inline even when using non-Anthropic models (e.g., OpenAI-compatible OSS backends).

Example (agent-side action invocation pattern):
```ts
// Pseudocode illustrating action-driven fetch in a custom workflow
const result = await runtime.actions.WEB_FETCH({
  url: "https://example.com/status",
  method: "GET",
});

// Use the fetched content as grounding context for the response
return runtime.reply({
  text: `Fetched status page:\n\n${result.text}`,
});
```

Notes:
- If you previously relied on provider-specific web-search tool wiring, migrate to `WEB_FETCH` to keep behavior consistent across providers.

---

### B) Cloud agent sleep/wake + incremental backups (Cloud V2 capability)
Merged as part of: [#8080](https://github.com/elizaos/eliza/pull/8080)

What’s new:
- **Sleep**: backs up agent sandbox, frees compute (intended for cost control / cold storage).
- **Wake**: re-provisions compute and restores sandbox state.
- **Incremental backups**: `agent-backup-diff.ts` supports delta-based backups, and DB migration adds metadata required to chain full + incremental backups.

Example (Cloud API calls):
```bash
# Put an agent into cold storage (async semantics)
curl -X POST "https://api.elizacloud.ai/v1/eliza/agents/$AGENT_ID/sleep" \
  -H "Authorization: Bearer $TOKEN"

# Restore compute + sandbox state
curl -X POST "https://api.elizacloud.ai/v1/eliza/agents/$AGENT_ID/wake" \
  -H "Authorization: Bearer $TOKEN"
```

DB migration:
- `0136_agent_backup_incremental.sql` (adds `backup_kind`, `parent_backup_id`, `content_hash`) via [#8080](https://github.com/elizaos/eliza/pull/8080)

## 3) Bug Fixes (critical fixes with technical context)

### Cloud staging magic-link sign-in failures (CSP + CORS + missing app-auth context)
Fixes landed across multiple PRs (notably):
- Tenant isolation for staging redirect: [#8176](https://github.com/elizaos/eliza/pull/8176)
- Pin tenant via header/proxy behavior: [#8179](https://github.com/elizaos/eliza/pull/8179)
- Callback page robust to missing app-auth context: [#8194](https://github.com/elizaos/eliza/pull/8194)
- CORS allow staging origin: [#8196](https://github.com/elizaos/eliza/pull/8196)
- CSP connect-src allows staging API: [#8197](https://github.com/elizaos/eliza/pull/8197)

Technical root causes addressed:
- **Steward tenant selection** drifting to prod on staging links.
- **CORS allowlist** missing `staging.elizacloud.ai`.
- **CSP connect-src** blocking calls to `api-staging.elizacloud.ai`.
- Auth callback assumed an app-auth context existed; now handles the “no context” path.

### Cloud reasoning models returning `content: null` while still billing tokens
Merged: [#8189](https://github.com/elizaos/eliza/pull/8189)

Fix:
- Cloud API now guarantees a response-token budget for “reasoning models” so hidden reasoning doesn’t consume the entire `max_tokens` budget and yield an empty answer.

Developer impact:
- If you set very low `max_tokens`, you’ll see more predictable completion behavior (and fewer billed-but-empty responses).

### Agent image boot regression fixed by making heavy plugins optional
Merged: [#8105](https://github.com/elizaos/eliza/pull/8105)

Fix:
- Moves `plugin-google` and `plugin-lifeops` to optional so the default image boots reliably.

Impact:
- Hosted deployments won’t fail boot just because these plugins’ external deps/config aren’t present.

### Local inference: typed error signaling for missing native bindings
Merged: [#8191](https://github.com/elizaos/eliza/pull/8191)

Fix:
- Local-inference text handlers now throw **typed “unavailability”** errors rather than generic `Error`, improving downstream error handling and UI messaging.

### Vault: self-heal stale PGlite lock to avoid “bricked” boot after unclean shutdown
Merged: [#8092](https://github.com/elizaos/eliza/pull/8092)

Fix:
- `@elizaos/vault` now heals stale locks in its PGlite store, aligning resilience with the runtime DB client manager.

### Orchestrator: JSON-RPC error details preserved
Merged: [#8098](https://github.com/elizaos/eliza/pull/8098) (closes [#8101](https://github.com/elizaos/eliza/issues/8101))

Fix:
- Preserves `error.data` in JSON-RPC failures so field-level details (e.g., Zod formatted errors) survive transport and can be surfaced for debugging.

### Orchestrator: completion dedup race fixed (duplicate replies)
Merged: [#8161](https://github.com/elizaos/eliza/pull/8161)

Fix:
- Addresses a TOCTOU race in cross-session completion dedup inside `SubAgentRouter.routeSessionEvent`.

### Workspace connector: invalid partition now returns 400 (not 500)
Merged: [#8087](https://github.com/elizaos/eliza/pull/8087)

Fix:
- Ensures connector account gating runs before user-script security checks so invalid partitions fail as client errors.

## 4) API Changes (developer-facing)

### Cloud API: new agent lifecycle endpoints
Introduced in: [#8080](https://github.com/elizaos/eliza/pull/8080)
- `POST /v1/eliza/agents/:agentId/sleep`
- `POST /v1/eliza/agents/:agentId/wake`

Semantics:
- Sleep returns async acceptance (e.g., `202`) and is idempotent for “already sleeping/in-progress”.

### Cloud API: CORS / tenancy behavior changes for staging auth
- CORS allowlist expanded: [#8196](https://github.com/elizaos/eliza/pull/8196)
- Staging tenant isolation: [#8176](https://github.com/elizaos/eliza/pull/8176), [#8179](https://github.com/elizaos/eliza/pull/8179)

If you run custom frontends:
- Ensure your origin is present in the Cloud API CORS allowlist.
- If embedding Steward flows, verify tenant selection in both build-time and request-time paths.

### Runtime: new/standardized `WEB_FETCH` action surface
Merged: [#8097](https://github.com/elizaos/eliza/pull/8097)

Recommendation:
- Treat `WEB_FETCH` as the canonical “live web retrieval” primitive; avoid provider-specific web-search patches.
