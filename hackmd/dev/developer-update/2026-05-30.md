# ElizaOS Developer Update (2026-05-24 → 2026-05-30)

This week focused on runtime reliability (planner/stage-1 correctness, user-facing leakage prevention), major platform capability expansion (remote-mode plugins + dynamic views/Satellites groundwork), cloud lifecycle hardening (job-queue ops + health heartbeat), and ecosystem growth (new connectors + model providers).

## 1) Core Framework

### Plugin architecture: **Plugin/mode unification** (direct vs remote)
A multi-PR stack introduces a single “Plugin” abstraction with an explicit execution mode:

- **P0: schema + vocabulary** — `mode: "direct" | "remote"` ([#7844](https://github.com/elizaos/eliza/pull/7844))
- **P1: remote runtime + bridge + isolated-process worker** ([#7846](https://github.com/elizaos/eliza/pull/7846))
- **P2: unified view-host shims + service/route surfaces (Android/iOS/Electrobun/Web)** ([#7847](https://github.com/elizaos/eliza/pull/7847))
- **P3: `installRemotePlugin` API + first sub-agent runner** ([#7848](https://github.com/elizaos/eliza/pull/7848))

**What this changes conceptually**
- “Satellites/Carrots” are now treated as **remote-mode plugins** with a standardized manifest + runtime envelope.
- Host platforms expose consistent **view-host** and **worker-host** surfaces so plugins can ship UI + capabilities across environments.

> Docs: `packages/agent/docs/capability-router-remote-plugins.md` (landed as part of the release train; see [#7828](https://github.com/elizaos/eliza/pull/7828)).

### Planner & message runtime hardening (Stage 1 → planner loop)
Several fixes addressed real regressions observed in live deployments:

- **Prevent phantom action claims** and preserve speaker context ([#7923](https://github.com/elizaos/eliza/pull/7923))
- **Bound runaway tool-arg validation retries** using a canonical failure signature so `maxRepeatedFailures` actually trips ([#7924](https://github.com/elizaos/eliza/pull/7924))
- **Honor “candidateActions implies planning”** even when Stage 1 contradicts itself (`simple=true` but suggests a tool) ([#7957](https://github.com/elizaos/eliza/pull/7957))
- **Surface REPLY refusal text** when required-tool requests cannot be satisfied (no silent empty replies) ([#7951](https://github.com/elizaos/eliza/pull/7951))
- **Eliminate “bare ack / empty replyText”** on simple-path; clarify interim-ack contract ([#7953](https://github.com/elizaos/eliza/pull/7953))
- **Forbid training-cutoff disclosures** in user-facing replies ([#7965](https://github.com/elizaos/eliza/pull/7965))

Open investigation (still relevant to builders using chat-history tooling): planner often prefers `SHELL` over `SEARCH_MESSAGES`, can run unbounded greps, and may leak sub-agent routing metadata ([#7935](https://github.com/elizaos/eliza/issues/7935)).

## 2) New Features

### A) Remote plugin installation API (agent-generated plugins)
With P3, the runtime can install remote-mode plugins programmatically (e.g., created by an agent, fetched from a registry, or shipped as a manifest+bundle):

- API added in core types/runtime surface: [#7848](https://github.com/elizaos/eliza/pull/7848)

**Example (conceptual)**
```ts
import type { Runtime } from "@elizaos/core";

async function install(runtime: Runtime) {
  await runtime.installRemotePlugin({
    // exact shape depends on the remote manifest package types landed in P0-P3
    id: "my.remote.plugin",
    source: {
      kind: "url",
      manifestUrl: "https://example.com/my.remote.plugin/plugin.json",
    },
    trust: "prompt", // subject to RemotePluginTrustPolicy
  });
}
```

If you previously built on “Carrot/Satellite” naming, migrate to **remote-mode plugin manifests** (see Breaking Changes).

### B) Connector accounts: **OWNER vs AGENT** dual-identity support
Connectors can now declare and manage two independent identities:

- Registry schema supports `accounts: { owner?, agent? }` ([#7817](https://github.com/elizaos/eliza/pull/7817))
- UI supports role-aware account lists + dual-section setup panels ([#7819](https://github.com/elizaos/eliza/pull/7819))
- OAuth callbacks fixed to stop hardcoding role=OWNER for some connectors ([#7820](https://github.com/elizaos/eliza/pull/7820)), with helper extraction follow-up ([#7841](https://github.com/elizaos/eliza/pull/7841))

This is foundational for “act as the agent” vs “act as the user” flows, especially for social and productivity connectors.

### C) Cloud: agent lifecycle ops moved behind a job queue
Workers no longer call core services directly for lifecycle operations:

- Refactors reduce duplicated enqueue helpers ([#7816](https://github.com/elizaos/eliza/pull/7816))

This affects integrators polling for lifecycle completion: expect **202 + jobId** patterns for some operations.

## 7) Breaking Changes / V1 → V2 Migration Warnings

### A) “Satellites/Carrots” naming and manifests → **remote-mode Plugins**
If you previously relied on prototype “Satellite” concepts, the supported direction is:

- Treat capability+UI bundles as **Plugins** with `mode: "remote"` ([#7844](https://github.com/elizaos/eliza/pull/7844) → [#7848](https://github.com/elizaos/eliza/pull/7848))
- Use the worker runtime envelope + host shims from P1/P2 for cross-platform execution.

**Migration risk:** custom installers / runtime loaders that assume old manifest keys or older “carrot” RPC names will break. Align with `packages/plugin-remote-manifest` examples introduced in P0.

---

### B) Connector setup UI assumes role-aware accounts where declared
Connectors can now declare dual identities; if your connector registry entry IDs or account storage assumed a single account, update:

- Registry schema: `accounts: { owner?, agent? }` ([#7817](https://github.com/elizaos/eliza/pull/7817))
- UI entrypoints and account lists may filter by role ([#7819](https://github.com/elizaos/eliza/pull/7819))

---

### C) Cloud lifecycle callers must handle async job flows
If you had scripts expecting synchronous completion from restart/suspend/snapshot/log operations, update them to consume `jobId` responses introduced by the lifecycle queue migration ([#7810](https://github.com/elizaos/eliza/pull/7810)).
