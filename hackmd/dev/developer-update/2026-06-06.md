# ElizaOS Developer Update (Week of 2026-05-31 → 2026-06-06)

This week concentrated on Cloud V2 hardening (auth, routing, billing correctness), richer multi-agent orchestration surfaces (Odysseus-style UI + workbench parity), and reliability fixes across core runtime + developer tooling.

**Primary repos impacted:** [`elizaos/eliza`](https://github.com/elizaos/eliza) (Cloud + runtime + UI + orchestrator)

## 1) Core Framework

### Agent “view surfaces” become agent-controllable (agent-surface + view-action affinity)
PR: **Claude/agentic views surface** ([#8080](https://github.com/elizaos/eliza/pull/8080))

Key architectural additions:
- **Agent-surface framework**: plugin views can register “elements” that an agent can enumerate and interact with (via `ViewAgentRegistry`, `AgentSurfaceProvider`, `useAgentElement`, and overlay tooling).
- **View-action affinity**: the runtime can bias planning toward actions relevant to the **currently active view** (reduces irrelevant tool selection and improves UI-driven workflows).
- **Lifecycle endpoints** for deep compute management in Cloud: sleep/wake plus backup workflows (see “API Changes”).

Developer implication: plugin authors can start exposing structured UI affordances for agent control instead of relying solely on text/tool prompts.

Related docs/entry points:
- Source: `packages/ui/src/agent-surface/*`, `packages/agent/src/runtime/view-action-affinity.ts` (from [#8080](https://github.com/elizaos/eliza/pull/8080))

---

### Core request routing fixes: coding vs web intent detection
PR: **route a coding request mentioning a market term to coding, not web** ([#8160](https://github.com/elizaos/eliza/pull/8160))

Fixes a subtle planner/router heuristic ordering issue:
- `looksLikeWebSearchRequest` previously matched before `looksLikeCodingWorkRequest`, causing misroutes when prompts contained market/live terms.
- Adds regression tests to lock in coding-first behavior when appropriate.

---

### Cross-model web capability: keyless `WEB_FETCH` action (not Anthropic-only)
PR: **keyless WEB_FETCH action** ([#8097](https://github.com/elizaos/eliza/pull/8097))

Previously, “keyless web search/fetch” behavior was effectively gated behind Anthropic-specific patches. This week:
- `WEB_FETCH` became a first-class runtime action usable by non-Anthropic providers (e.g., Cerebras `gpt-oss-*`).

Example (runtime action usage in custom flows):
```ts
// Pseudocode illustrating invoking WEB_FETCH via runtime actions
const result = await runtime.actions.invoke("WEB_FETCH", {
  url: "https://example.com/status",
  method: "GET",
});
```

### Orchestrator reliability: preserve JSON-RPC error diagnostics
PR: **preserve JSON-RPC error.data on ACP request failures** ([#8098](https://github.com/elizaos/eliza/pull/8098)), Issue: [#8101](https://github.com/elizaos/eliza/issues/8101)

## 3) Bug Fixes (Critical)

### Production incident: “Sign in with Eliza Cloud” OAuth 400 (PKCE mismatch)
Issue: **Steward login broken in prod** ([#8257](https://github.com/elizaos/eliza/issues/8257)) (open)

Discord ops notes (2026-06-05) indicate an active production outage:
- OAuth 400 errors on authorize flow (PKCE required)
- Login failures affecting all “Sign in with Eliza Cloud” integrations
- Coordination needed: deploy signing secrets + update tenant redirect allowlists

Fix PR (landed but ensure deployed/verified in prod):
- **send PKCE on Steward OAuth + sign nonce-exchange** ([#8259](https://github.com/elizaos/eliza/pull/8259))

What changed technically:
- Frontend now generates and sends `code_challenge`/`code_challenge_method=S256`
- Nonce exchange calls are HMAC-signed to satisfy Steward’s sensitive-path guard
- Aligns cloud login implementation with Steward’s hardened auth contract

---

### Cloud inference correctness: guarantee response-token budget for reasoning models
PR: **guarantee response-token budget for all reasoning models** ([#8189](https://github.com/elizaos/eliza/pull/8189))

Bug:
- Some reasoning models returned `content: null` while still billing tokens when `max_tokens` was insufficient.

Fix:
- Cloud routing now ensures a minimum response-token budget, preventing “billed but empty answer” failure mode.

## 7) Breaking Changes / V1 → V2 Migration Warnings

### Cloud Auth: PKCE is now effectively required end-to-end
- Steward hardened OAuth requires PKCE; Cloud login had to be updated accordingly.
- If you implemented custom “Sign in with Eliza Cloud” flows (without SDK), you may now fail with OAuth 400 unless you implement PKCE and the updated exchange/signing behavior.

Track:
- Incident: [#8257](https://github.com/elizaos/eliza/issues/8257)
- Fix: [#8259](https://github.com/elizaos/eliza/pull/8259)

**Action:** prefer using the Cloud SDK sign-in helper ([#8251](https://github.com/elizaos/eliza/pull/8251)) rather than maintaining bespoke auth code.
