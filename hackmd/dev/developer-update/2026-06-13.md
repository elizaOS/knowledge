# ElizaOS Developer Update — Week of 2026-06-07 to 2026-06-13

This week focused on **Eliza Cloud Apps (Product 2)** infra stabilization (control-plane + apps data-plane), plus continued hardening of authentication and routing between the Cloud Worker, provisioned agents, and the Apps ingress layer.

---

## 1) Core Framework (architecture / plugin system / runtime)

### Cloud control-plane + Apps data-plane convergence
Work continued to close remaining “go-live” gaps highlighted in **Eliza Cloud Apps: scaling + ingress gaps before go-live** ([#8321](https://github.com/elizaos/eliza/issues/8321)). Two notable infra steps landed:

- **Apps data-plane ingress now bootstraps correctly on new app nodes** by installing the Caddy “front door” directly in app-node cloud-init:  
  - PR: **feat(infra): install Caddy front door in app-node cloud-init** ([#8394](https://github.com/elizaos/eliza/pull/8394))
- **Control-plane bootstrap now installs Headscale via cloud-init**, removing a manual migration step and reducing drift risk during node replacement:  
  - PR: **fix(infra): install headscale in control-plane cloud-init** ([#8393](https://github.com/elizaos/eliza/pull/8393))

These changes reduce the amount of “hand-fixing” required when bringing up fresh Hetzner nodes and make staging verification closer to a production-shaped apply.

### Runtime performance pattern discussed: “filter-first” gRPC pre-pipeline
In Discord, a performance architecture was shared for high-volume workloads: a local **gRPC pipeline that filters/aggregates in RAM** before invoking LLM tokens, and only escalates to MCP for deeper analysis—achieving sub-millisecond latency under spikes (Discord discussion, 2026-06-11). This pattern maps cleanly onto ElizaOS deployments where you want to:
- minimize token spend on “fast path” decisions
- reserve model calls for high-entropy cases
- keep critical-path operations off the model provider

## 3) Bug Fixes (critical fixes + technical context)

### ElizaCloud signup blocked by tenant invitation enforcement
A user reported being unable to sign up for ElizaCloud with:

> “Tenant ‘elizacloud’ requires an invitation to join”

This is a **hard blocker** for developer onboarding and testing. The issue was acknowledged and escalated for investigation (Discord, 2026-06-12). If you are affected, track status via Discord thread:
- Discussion: https://discord.com/channels/1253563208833433701/1253563209462448241

### Apps data-plane “go-live” blocked by Terraform state-provider mismatch
A major blocker remains open:

- **Apps data-plane go-live stalled due to state-provider mismatch** involving an orphaned `random_password` resource  
  - Tracked under: [#8321](https://github.com/elizaos/eliza/issues/8321) (and referenced in the 2026-06-12 infra update)

The mitigation direction this week was to **automate state cleanup and daemon arming** (PRs [#8400](https://github.com/elizaos/eliza/pull/8400), [#8398](https://github.com/elizaos/eliza/pull/8398), [#8397](https://github.com/elizaos/eliza/pull/8397)) so staging can be re-verified with fewer ad-hoc operator steps.

## 7) Breaking Changes (V1 → V2 migration warnings)

### Cloud Apps + Cloud V2 operational expectations
Even when code is “merged,” **go-live readiness** for Apps depends on closing infrastructure gaps. The canonical warning remains the Apps audit issue:
- Do not treat Apps as production-ready until P0 items are closed and re-verified in staging:  
  - [#8321](https://github.com/elizaos/eliza/issues/8321)

### Cloud onboarding / auth fragility: watch tenant + invite flows
Given the reported signup failure (“tenant requires invitation”) and ongoing tenant isolation work, developers integrating “Sign in with Eliza Cloud” should:
- test both **invite acceptance** and **self-serve signup** paths in the environment you target (staging vs prod)
- avoid hard-coding tenant IDs; prefer environment-driven configuration
- validate your onboarding funnel end-to-end (magic link / OAuth / redirects) before shipping
