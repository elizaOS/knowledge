# ElizaOS Developer Update (2026-06-09 → 2026-06-15)

## 1) Core Framework

### Repository consolidation (organizational + workflow impact)
All active development has been consolidated into the single monorepo **`elizaos/eliza`** (confirmed on Discord). If you were tracking “satellite” repos for core/runtime work, treat them as read-only references and pin your upstreams to:
- Repo: https://github.com/elizaos/eliza

### Cloud Apps “data-plane” hardening continues
This week’s work is primarily focused on stabilizing **Eliza Cloud Apps** ahead of launch (auth, billing correctness, deploy automation, and CI reliability). Key moving pieces to be aware of as a developer integrating with Cloud Apps:

- **Staging → production cutover** is in progress; expect frequent changes around feature flags and env wiring.
- **Apps build pipeline now supports “build-from-repo”** so the platform can build user repositories directly (Vercel-like path). (See PR **#8407**: https://github.com/elizaos/eliza/pull/8407)
- **Deployment gating flags** are being activated incrementally (notably `APPS_DEPLOY_ENABLED` on staging path, and production work in progress).

## 2) New Features

### A) Cloud Apps: build-from-repo support (platform builds your repo)
Cloud Apps now supports building an app image directly from a connected repository rather than requiring you to pre-build and push an image yourself.

- PR: **#8407** https://github.com/elizaos/eliza/pull/8407  
- Related platform status: “Launch Stabilization Update (Jun 15, 2026)” (internal aggregation)

**What this enables**
- A single “deploy” action can: clone → buildx → push → deploy → route traffic.
- This is the recommended path for early testers of Cloud Apps, because it removes a large class of developer-local Docker/push issues.

**Developer-facing behavior (conceptual)**
While API shapes may still evolve behind flags, you should architect your app repos to be buildable in CI:
- Deterministic builds (lockfiles committed)
- No interactive build steps
- Explicit exposed port + health check strategy

### B) Headscale: automated CI/CD for the cloud service
Headscale is moving from “manual deploy runbook” to **first-class automated deployment**.

- PR: **#8423** https://github.com/elizaos/eliza/pull/8423

If you depend on tailnet-based routing or internal service discovery, this reduces drift between staging/prod headscale configuration and makes rollouts auditable.

## 3) Bug Fixes (with technical context)

### A) Steward OAuth production login failures resolved via PKCE standardization
A class of “Sign in with Eliza Cloud” failures was rooted in Steward OAuth hardening requiring **PKCE** (`code_challenge` / `code_verifier`) across all login surfaces.

- Underlying incident issue (now closed): **#8257** https://github.com/elizaos/eliza/issues/8257  
- Fixes landed across related PRs (PKCE + signing + tenant pinning):
  - **#8259** https://github.com/elizaos/eliza/pull/8259 (send PKCE + sign nonce-exchange)
  - **#8350** https://github.com/elizaos/eliza/pull/8350 (PKCE extracted into shared helper for all surfaces)

**Why developers should care**
- If you have custom auth entrypoints or embedded login flows, you must ensure you’re using the shared PKCE helpers (or implementing PKCE correctly). Non-PKCE flows will increasingly fail as upstream auth hardens.

### B) Billing integrity: container billing made idempotent (prevents double-debit)
Daily container billing is now **idempotent per billing period**, preventing double charges on cron retries / overlaps.

- PR: **#8273** https://github.com/elizaos/eliza/pull/8273

**Impact**
- Safer retries and operational recovery.
- If you’re building tooling that replays or triggers billing-related workflows, you should remove any assumptions that “rerun == re-bill”.

### C) Organizations credit default corrected to $0 (removes unintended free-credit footgun)
Organizations created outside the intended signup path were inheriting a default credit balance.

- PR: **#8427** https://github.com/elizaos/eliza/pull/8427

**Impact**
- Integrators relying on “implicit starter credit” should explicitly provision credits (or use the supported signup/provision flow).

### D) Model catalog stability: recommended defaults moved to “healthy Cerebras” presets
To reduce gateway instability, the model catalog now recommends Cerebras-native defaults instead of flaky gateway-routed variants.

- PR: **#8426** https://github.com/elizaos/eliza/pull/8426

## 7) Breaking Changes / V1 → V2 Migration Warnings

### A) Develop vs main: version confusion for updaters (practical breaking behavior)
Discord reports confirm developers getting “stuck” on **v1.x (e.g. 1.7.2)** when installing from `main`. Current active work is on **`develop` (2.x)**.

**Action**
```bash
git checkout develop
bun install
bun run build
```

### B) LifeOps refactor: `plugin-lifeops` removed/renamed and split (hard breaking change)
If you’re upgrading to newer 2.x snapshots, note that the monolithic `@elizaos/plugin-lifeops` has been decomposed and renamed into a set of focused plugins plus `plugin-personal-assistant`.

- PR: **#8277** https://github.com/elizaos/eliza/pull/8277

**Migration implications**
- Imports like `@elizaos/plugin-lifeops` will break.
- The `/lifeops` view was removed; domain plugins now own views (`/calendar`, `/todos`, etc.) depending on the extracted plugin.
- Ensure your agent/plugin pack lists are updated accordingly.

### C) Cloud Apps launch blockers (operational breaking conditions)
Two critical blockers remain before full Cloud Apps rollout:
- **NPM_TOKEN permissions for `@elizaos/*` packages** halting release pipeline: **#8428** (referenced in the stabilization report)
- **Cloud 10-user launch tracking** pending infra + prod flags: **#8434** (referenced in the stabilization report)

Until these clear, expect partial availability and staged rollouts.
