# ElizaOS Developer Update (2026-06-07 → 2026-06-13)

This update summarizes production-facing framework and Cloud work landing (mostly on `develop`) during the week, plus relevant implementation guidance from Discord.

---

## 1) Core Framework

### Agent “view surface” + action affinity (agent-controllable UI)
ElizaOS now exposes plugin views as an agent-controllable surface, enabling the runtime to list/interact with view elements and bias planning toward view-relevant actions (a prerequisite for “operator” experiences and richer autonomous UI flows).

* Shipped in: **“Claude/agentic views surface”** [#8080](https://github.com/elizaos/eliza/pull/8080)

Key pieces:
- `ViewAgentRegistry` + `AgentSurfaceProvider`
- `useAgentElement` hook / overlay
- “view-action affinity” module to upweight actions relevant to the active view

**Impact for plugin authors:** views can be made discoverable/controllable by the agent without inventing a bespoke RPC layer per plugin. If your plugin provides a view with UI elements that should be agent-operable, align with the registry + capability handlers introduced in [#8080](https://github.com/elizaos/eliza/pull/8080).

## 2) New Features

### Cloud: container lifecycle API (deploy + mutate)
Cloud API gained explicit container lifecycle endpoints to support programmatic deployment and runtime management.

* Deploy + read route: [#8258](https://github.com/elizaos/eliza/pull/8258)  
* Mutation verbs (PATCH/DELETE/scale): [#8270](https://github.com/elizaos/eliza/pull/8270)

**Example: deploy a container (Cloud API)**
```bash
curl -X POST "https://api.elizacloud.ai/api/v1/containers" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "image": "ghcr.io/your-org/your-app:latest"
  }'
```

**Example: scale an existing container**
```bash
curl -X PATCH "https://api.elizacloud.ai/api/v1/containers/$CONTAINER_ID" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "scale": 2
  }'
```

(Exact payload fields depend on your deployment’s container control-plane configuration; see the route implementations in [#8258](https://github.com/elizaos/eliza/pull/8258) / [#8270](https://github.com/elizaos/eliza/pull/8270).)

## 3) Bug Fixes (critical fixes with context)

### Cloud Apps: deleted apps leaked isolated Postgres + cluster slots
A critical resource leak was fixed: deleting an app did not drop its isolated Postgres DB/ROLE and did not release the cluster allocation slot.

* Fix: [#8401](https://github.com/elizaos/eliza/pull/8401)  
* Related work (feature teardown + slot release): [#8343](https://github.com/elizaos/eliza/pull/8343)

**Why this was severe:** over time, churn would permanently exhaust tenant DB capacity and incur stranded infrastructure cost.

---

### Cloud Apps: host-port allocation collisions at moderate scale
`allocateHostPort()` previously chose ports randomly without checking in-use ports, leading to `docker create` failures at higher density.

* Fix: [#8328](https://github.com/elizaos/eliza/pull/8328)  
* Origin audit item: [#8321](https://github.com/elizaos/eliza/issues/8321) (item 8)

---

### Cloud Auth: Steward OAuth hardening (PKCE + signed proxy calls)
Steward auth hardening caused production sign-in breakage (see open incident issue), and the Cloud worker/SPAs were updated to match PKCE expectations and signed upstream calls.

* PKCE + signed nonce-exchange: [#8259](https://github.com/elizaos/eliza/pull/8259)  
* Sign Steward proxy requests: [#8256](https://github.com/elizaos/eliza/pull/8256)  
* Tenant pinning via `X-Steward-Tenant`: [#8179](https://github.com/elizaos/eliza/pull/8179)  
* Staging auth isolation + CORS/CSP follow-ups: [#8282](https://github.com/elizaos/eliza/pull/8282), [#8196](https://github.com/elizaos/eliza/pull/8196), [#8197](https://github.com/elizaos/eliza/pull/8197), [#8194](https://github.com/elizaos/eliza/pull/8194)

## 7) Breaking Changes / V1 → V2 Migration Warnings

### A) Branching reality: `develop` is the 2.x line
Discord reports developers getting “stuck” on v1.7.2 when updating. The current working practice is: **use `develop` for 2.x** while `main` may lag behind. (Discussion: `#coders`, 2026-06-13.)

**Upgrade guidance (source users):**
```bash
git fetch origin
git checkout develop
git pull --ff-only
bun install
bun run build
```

If you are pinning tags/commits, ensure you are not expecting `main` to contain 2.x work-in-progress.

---

### B) `@elizaos/plugin-lifeops` removed (no compatibility shim)
The LifeOps monolith is gone and renamed/split. There are **no backwards-compat aliases**.

- Refactor: [#8277](https://github.com/elizaos/eliza/pull/8277)
- Follow-up boot crash fix when old exports were still referenced: [#8314](https://github.com/elizaos/eliza/pull/8314)

**What you must do:**
- Remove `@elizaos/plugin-lifeops` from plugin lists/imports.
- Add the new focused plugins you rely on (e.g., `@elizaos/plugin-calendar`, `@elizaos/plugin-todos`, etc.) and/or `@elizaos/plugin-personal-assistant` if you relied on cross-domain orchestration.

---

### C) Cloud Apps assumptions changed: per-tenant DB is opt-in
If you assumed every deployed app gets an isolated Postgres automatically, that is no longer the case after:

- Opt-in DB mode (`none|isolated`): [#8338](https://github.com/elizaos/eliza/pull/8338)

Review any deploy automation that expects `DATABASE_URL`/`POSTGRES_URL` to exist unconditionally.
