# ElizaOS Developer Update (2026-06-16 → 2026-06-23)

This week focused on **core stability in the agent runtime and Cloud control plane**, with targeted fixes to request routing, cloud job recovery behavior, and web-fetch networking correctness. Discord discussion was largely non-technical, but surfaced actionable needs around **branding consistency** and **community moderation**.

---

## 1) Core Framework

### Direct-reply routing: fallback to uncapped execution path
A new WIP change adds a **fallback mechanism for direct-reply requests** when the “capped” execution path can’t satisfy the request, automatically retrying via the uncapped path.

- PR: **[#9074 — fallback mechanism to uncapped path for direct-reply requests](https://github.com/elizaos/eliza/pull/9074)**

**Why this matters:** direct-reply is the lowest-latency “answer now” path; failures here typically surface as inexplicable no-response or misrouted responses. The fallback makes this path more robust without requiring caller-side retries.

## 3) Bug Fixes (critical / high impact)

### Cloud autoscaler cold-boot: prevent provision flapping
Cold-boot scenarios could cause **stale-job recovery to size too aggressively**, leading to “provision flapping” (rapid scale-up/scale-down oscillation or repeated recovery attempts that destabilize the fleet).

- PR: **[#9072 — adjust stale-job recovery sizing to prevent flapping on cold boot](https://github.com/elizaos/eliza/pull/9072)**

**Technical context:** stale-job recovery is effectively a control loop; when it overestimates required capacity or recovery batch size during cold starts, it can trigger repeated reconciles that fight the autoscaler’s own convergence logic.

---

### WEB_FETCH: pinned dual-stack lookup `{ all: true }` handling to resolve IP errors
A WIP fix corrects agent logic for `{ all: true }` lookups in pinned dual-stack web-fetch configurations, addressing **IP resolution errors** observed in the field.

- PR: **[#9070 — handle `{ all: true }` lookups for pinned dual-stack web-fetch](https://github.com/elizaos/eliza/pull/9070)**

**Impact:** fewer intermittent network-layer failures when web-fetch needs to resolve/choose between IPv4/IPv6 targets under “pinned” networking strategies.

## 4) API Changes

No new public API endpoints were introduced in the tracked work this week, but there are behavior-level changes developers should be aware of:

### Direct-reply behavior becomes more resilient (implicit retry/fallback)
If merged, **[#9074](https://github.com/elizaos/eliza/pull/9074)** will change the runtime’s behavior for direct-reply requests by **retrying via a different internal execution path**.

**Developer consideration:** if you rely on strict token budgeting / caps for direct replies, validate how the uncapped fallback affects your latency/cost expectations (especially in high-throughput connectors).
