# ElizaOS Developer Update (2026-05-17 → 2026-05-23)

This week shipped **ElizaOS v2.0.2** and continued the v2+ platform shift toward **native-app integration**, **role-separated connector identity (OWNER vs AGENT)**, and **remote-mode plugins** that can run outside the main agent runtime (desktop/mobile/cloud).

Key references:
- Release: **v2.0.2 develop → main** ([elizaos/eliza#7828](https://github.com/elizaos/eliza/pull/7828))
- Weekly rollup (May 17–23): [2026-05-17.md](https://github.com/elizaos/eliza/blob/main/2026-05-17.md)

## 1) Core Framework

### Plugin system: “Plugin/mode unification” (direct vs remote)
A major architectural through-line this week is the **unification of “satellites/carrot remotes” into first-class Plugins** with a `mode`:

- **P0 — schema + scaffolding**: `mode: "direct" | "remote"` ([#7844](https://github.com/elizaos/eliza/pull/7844))
- **P1 — runtime + bridge + isolated-process worker**: remote plugins actually execute via a worker bootstrap + host bridge ([#7846](https://github.com/elizaos/eliza/pull/7846))
- **P2 — unified view-host shims**: consistent view hosting on Android/iOS/Electrobun/Web ([#7847](https://github.com/elizaos/eliza/pull/7847))
- **P3 — `installRemotePlugin` API + first sub-agent runner**: enabling agent-generated/installed remote plugins ([#7848](https://github.com/elizaos/eliza/pull/7848))

What this means for developers:
- Treat **everything as a Plugin**; “satellites” become a packaging/runtime detail.
- Remote plugins can provide **actions/providers/events/models** while running out-of-process (or at least out-of-runtime) depending on host constraints.

Docs & code pointers:
- Capability routing + remote plugins: `packages/agent/docs/capability-router-remote-plugins.md` (landed in the v2.0.2 release train via [#7828](https://github.com/elizaos/eliza/pull/7828))
- Core types: `packages/core/src/types/plugin.ts` (changed across [#7844](https://github.com/elizaos/eliza/pull/7844), [#7848](https://github.com/elizaos/eliza/pull/7848))

### Electrobun platform: first-party “Satellites” + dynamic view host bridge
Electrobun absorbed the prototype satellite work and introduced the first **dynamic view registry/session substrate**:

- First-party satellites moved into Electrobun + seeded idempotently ([#7779](https://github.com/elizaos/eliza/pull/7779))
- Renderer↔worker command/event bridges added:
  - `carrotInvokeWorker`
  - `carrotTailWorkerEvents`
- Dynamic view lifecycle APIs introduced (register/list/open/push/close/sessions)

Developer impact:
- A foundation now exists for **agent-created UI** (A2UI/canvas-backed) without hardcoding new dashboard panels.
- Event tailing is currently **bounded buffering**, not a realtime pubsub system.

### Runtime identity/ownership: OWNER+AGENT connector model
v2.0.2 formalized role-separated identity across connectors:

- Connector registry schema supports `accounts: { owner?, agent? }` ([#7817](https://github.com/elizaos/eliza/pull/7817))
- UI now partitions setup + account lists by role ([#7819](https://github.com/elizaos/eliza/pull/7819))
- OAuth callback handlers stopped hardcoding `role: "OWNER"` (Linear/Calendly/Shopify) ([#7820](https://github.com/elizaos/eliza/pull/7820)), plus cleanup helper ([#7841](https://github.com/elizaos/eliza/pull/7841))

Docs:
- Owner vs agent audit: `packages/docs/connectors/owner-vs-agent-audit.md` ([#7817](https://github.com/elizaos/eliza/pull/7817))
- Owner role: `packages/docs/agents/owner-role.md` (shipped in the develop→main bundle, see [#7840](https://github.com/elizaos/eliza/pull/7840))

## 2) New Features

### A) Secrets management: `@elizaos/vault` + Settings UI integration
This week’s v2 line continues to harden secrets handling by moving away from plaintext env-style storage.

- `@elizaos/vault` introduced (cross-platform keychain-backed AES-GCM vault + Settings UI) ([#7197](https://github.com/elizaos/eliza/pull/7197))
- Headless Linux safety fix: skip OS keychain when Secret Service isn’t available to prevent native segfault on boot ([#7230](https://github.com/elizaos/eliza/pull/7230))

Relevant behavior changes:
- Sensitive plugin settings saved via UI are now mirrored into the vault.
- Some flows now persist **references** (e.g. `apiKeyRef`) rather than plaintext values (see migration notes under “Breaking Changes”).

Conceptual usage (from `@elizaos/vault` API surface):
```ts
import { sharedVault } from "@elizaos/vault";

const vault = await sharedVault();
await vault.set("OPENAI_API_KEY", process.env.OPENAI_API_KEY!, { sensitive: true });

const key = await vault.get("OPENAI_API_KEY");
```

### B) Cloud lifecycle ops moved to an explicit job queue (async 202)
Cloud agent lifecycle operations are now decoupled from synchronous worker→core calls:

- All lifecycle ops via job queue ([#7810](https://github.com/elizaos/eliza/pull/7810))
- Frontend updated to track async `jobId` for suspend/snapshot/restart/logs ([#7813](https://github.com/elizaos/eliza/pull/7813))
- Follow-up refactors to consolidate enqueue helpers ([#7816](https://github.com/elizaos/eliza/pull/7816)) and move shutdown logic to enqueued jobs ([#7815](https://github.com/elizaos/eliza/pull/7815))

Developer impact:
- Expect **202 Accepted** + `{ jobId }` for lifecycle endpoints that were previously synchronous.
- Poll job status / reflect async completion in UX (see [#7813](https://github.com/elizaos/eliza/pull/7813)).

### C) Local inference & mobile: voice + sandbox routing
Remote plugin routing now extends further into mobile/desktop platform realities:

- Mobile: local voice + sandbox routing ([#7850](https://github.com/elizaos/eliza/pull/7850))
- Android local-inference activation path fixed (`/api/local-inference/active` triggers in-process loader) ([#7769](https://github.com/elizaos/eliza/pull/7769))
- TalkMode: native Android playback for local-inference TTS ([#7763](https://github.com/elizaos/eliza/pull/7763))
- Local-inference: slot save/restore + prewarm via FFI ([#7796](https://github.com/elizaos/eliza/pull/7796))
- Kokoro TTS mobile uses bundled phonemizer first ([#7768](https://github.com/elizaos/eliza/pull/7768))

### D) CLI deploy keel + Cloud deploy endpoints
A first stable “shape” for deploying monetized apps is being established:

- CLI design keel: `elizaos deploy --dry-run --domain ...` ([#7786](https://github.com/elizaos/eliza/pull/7786))
- Cloud endpoints:
  - `POST /api/v1/apps/[id]/deploy`
  - `GET /api/v1/apps/[id]/deploy/status`
  ([#7804](https://github.com/elizaos/eliza/pull/7804))
- Cloud: monetized container app domains support ([#7376](https://github.com/elizaos/eliza/pull/7376))

Example (CLI surface draft from [#7786](https://github.com/elizaos/eliza/pull/7786)):
```bash
elizaos deploy --dry-run --domain test.example
```

## 3) Bug Fixes (with technical context)

### A) Headless Linux boot segfault via OS keychain access
**Fix:** skip OS keychain on headless Linux when Secret Service isn’t available ([#7230](https://github.com/elizaos/eliza/pull/7230)).

Context:
- `@napi-rs/keyring` can hard-crash (native segfault) when libsecret/D-Bus isn’t reachable.
- The vault + confidant chain now avoids attempting keyring access in those headless contexts, preferring passphrase/env remediation paths.

Why it matters:
- Prevents “process disappears” failures on server boots—this is especially critical for self-hosted agent runtimes and CI.

### B) Cloud Docker boot deadlock (TLA / barrel-import hang)
**Fix:** eliminate top-level await + barrel-import deadlock that caused cloud Docker containers to hang before listening ([#7801](https://github.com/elizaos/eliza/pull/7801)).

Symptom:
- Containers stayed alive but never bound the HTTP port; health checks timed out.
Root cause:
- Import-time execution ordering leading to unresolved async initialization.

### C) Discord reply-context grounding + stale setup prompt suppression
**Fix:** structured `platform_reply_reference` injected into message envelope and rendered into core prompt segments ([#7821](https://github.com/elizaos/eliza/pull/7821)).

Why it matters:
- Reduces “replying to the wrong message” failures in threaded/high-volume channels.
- Mitigates repeated/incorrect setup nags after a connector is already configured.

### D) Tailwind v4 overlay regression (z-index classes not emitted)
**Fix:** inline z-index arbitrary classes so Tailwind v4 includes them, restoring modal overlays and dialog backgrounds ([#7814](https://github.com/elizaos/eliza/pull/7814)).

## 4) API Changes

### A) Cloud agent lifecycle endpoints: now async (job queue)
As noted above, lifecycle operations now commonly return:
- `202 Accepted`
- a payload including `jobId`
- frontend and callers must track job completion

PRs:
- Core change: [#7810](https://github.com/elizaos/eliza/pull/7810)
- Frontend adaptation: [#7813](https://github.com/elizaos/eliza/pull/7813)

### B) Connector registry schema: OWNER+AGENT accounts
Connector entry schema now supports an explicit `accounts` block describing role-separated identities.

PR:
- Schema addition: [#7817](https://github.com/elizaos/eliza/pull/7817)
- UI support: [#7819](https://github.com/elizaos/eliza/pull/7819)

Practical implication:
- Connector UIs and OAuth flows should preserve `role` end-to-end and avoid implicitly “upgrading” everything to OWNER.

### C) Remote plugin runtime surfaces
If you maintain internal tooling around plugins, note the new/expanded surfaces:
- `mode: "direct" | "remote"` (schema-level, [#7844](https://github.com/elizaos/eliza/pull/7844))

## 7) Breaking Changes (V1 → V2 migration warnings)

### A) Connector identity is now explicitly role-separated (OWNER vs AGENT)
If you built custom connectors or assumed a single “connected account”:
- Update connector manifests to declare `accounts.owner` / `accounts.agent` where applicable ([#7817](https://github.com/elizaos/eliza/pull/7817)).
- Ensure OAuth start/complete flows preserve the requested role; older handlers that hardcoded OWNER were corrected in core plugins ([#7820](https://github.com/elizaos/eliza/pull/7820)).

Migration risk:
- Incorrect role assignment can cause actions to run under the wrong identity (e.g., posting as the bot when the user expected OWNER, or vice versa).

### B) Cloud lifecycle calls are no longer synchronous
If you directly call Cloud agent endpoints (restart/suspend/snapshot/logs):
- Expect async semantics (202 + `jobId`) post [#7810](https://github.com/elizaos/eliza/pull/7810).
- Update clients to poll / subscribe to completion rather than assuming immediate effect.

### C) Remote plugins become first-class; “satellite/carrot” terminology is collapsing into Plugin(mode)
If you integrated with the earlier Electrobun satellite prototype:
- Expect renamed concepts and a different install/run pathway as remote-mode plugins standardize ([#7844](https://github.com/elizaos/eliza/pull/7844) → [#7848](https://github.com/elizaos/eliza/pull/7848)).

### D) Secrets storage: vault mirror + references may replace plaintext keys
If your integration depends on reading raw keys from `config.env`:
- The platform is moving toward vault-backed secrets and **stored references** (e.g. `apiKeyRef`) rather than plaintext values ([#7197](https://github.com/elizaos/eliza/pull/7197)).
- On headless Linux, ensure you provide a passphrase/env-based master key path as OS keychain may be skipped ([#7230](https://github.com/elizaos/eliza/pull/7230)).
