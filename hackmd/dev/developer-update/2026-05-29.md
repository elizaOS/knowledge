## 1) Core Framework

### Plugin runtime: “Plugin/mode unification” (direct vs remote)
A major architecture line this week is the unification of “local plugins” and “remote runners” under a single **Plugin** abstraction with `mode: "direct" | "remote"`.

- **P0 (schema + vocabulary):** `Plugin` now carries a `mode` field, with shared manifest + envelope types.  
  PR: [#7844](https://github.com/elizaos/eliza/pull/7844)
- **P1 (remote-mode execution):** remote plugins can now run via an isolated worker bootstrap + host bridge, mirroring the direct plugin surfaces.  
  PR: [#7846](https://github.com/elizaos/eliza/pull/7846)
- **P2 (view-host shims + routes/services):** unified host shims across platforms (web/ios/android/electrobun) + service/route plumbing for remote-mode.  
  PR: [#7847](https://github.com/elizaos/eliza/pull/7847)
- **P3 (agent-generated remote plugins):** adds an `installRemotePlugin` API and ships the first reference sub-agent runner plugin.  
  PR: [#7848](https://github.com/elizaos/eliza/pull/7848)

**Why it matters:** this is the foundation for “agent installs a tool it just wrote” and for shipping capability sandboxes consistently across desktop/mobile/cloud.

#### Example: installing a remote plugin (conceptual)
Exact call sites vary by host, but the new architecture is centered on “remote plugin descriptors” and an install surface exposed in core types (see PR #7848).
```ts
import type { PluginDescriptor } from "@elizaos/core";

// Example descriptor shape (see packages/plugin-worker-runtime and PR #7848)
const plugin: PluginDescriptor = {
  id: "com.example.my-remote-plugin",
  mode: "remote",
  worker: { entry: "worker.mjs" },
  view: { entry: "view/index.html" },
  permissions: ["host:manage-carrots"], // tightening planned (see #7779 WIP notes)
};

// Pseudocode: install via the new runtime install surface
await runtime.plugins.installRemotePlugin(plugin);
```

### Planner/runtime reliability hardening
Multiple changes this week focus on reducing “planner weirdness” and protecting user-facing channels from internal control text:

- **Prevent runaway tool-arg validation loops** by canonicalizing failure signatures so `maxRepeatedFailures` can trip reliably.  
  PR: [#7924](https://github.com/elizaos/eliza/pull/7924)
- **Restore evaluator `messageToUser` precedence** so evaluators can override tool `userFacingText` unless explicitly opted in.  
  PR: [#7897](https://github.com/elizaos/eliza/pull/7897)
- **Reduce meta-leaks** (loopback URLs, internal JSON envelopes, training-cutoff chatter) via prompt rules + evaluator sanitation:  
  PRs: [#7955](https://github.com/elizaos/eliza/pull/7955), [#7956](https://github.com/elizaos/eliza/pull/7956), [#7965](https://github.com/elizaos/eliza/pull/7965), [#7953](https://github.com/elizaos/eliza/pull/7953)

## 2) New Features

### Connector accounts: OWNER vs AGENT identity (schema + UI)
ElizaOS connectors can now declare *two independent identities*:

- **OWNER**: the user’s personal account on the platform
- **AGENT**: the agent’s own bot/account identity

This lands as:
- registry schema support: PR [#7817](https://github.com/elizaos/eliza/pull/7817)
- role-aware UI surfaces: PR [#7819](https://github.com/elizaos/eliza/pull/7819)

Docs/audit: [`packages/docs/connectors/owner-vs-agent-audit.md`](https://github.com/elizaos/eliza/blob/develop/packages/docs/connectors/owner-vs-agent-audit.md)

#### Example: connector registry entry enabling OWNER+AGENT
(Structure is defined in #7817; exact connector ids vary.)
```json
{
  "id": "slack",
  "type": "connector",
  "accounts": {
    "owner": { "enabled": true },
    "agent":  { "enabled": true }
  }
}
```

## 3) Bug Fixes (critical + high-impact)

### Cloud API: `tool_choice: "required"` crash (500)
`POST /v1/chat/completions` returned a 500 when callers used `tool_choice: "required"` due to assuming an object shape. This breaks planner flows that intentionally require tool selection.

- Fix: robust mapping for `"required"`.  
  PR: [#7992](https://github.com/elizaos/eliza/pull/7992) (and cherry-pick to `main`: [#7993](https://github.com/elizaos/eliza/pull/7993))

**Developer impact:** if you previously avoided `"required"` due to server crashes, it should now be safe; you can simplify client-side workarounds.

### Headless Linux: vault/keyring segfault on boot
On headless Linux without a Secret Service / D-Bus, `@napi-rs/keyring` could segfault during boot (process-level crash). The fix skips OS keychain usage in those environments while keeping “fail closed” semantics where appropriate.

- PR: [#7230](https://github.com/elizaos/eliza/pull/7230)

### Cloud API security posture: add baseline security headers
Cloud API Worker shipped without security response headers; this has been addressed via middleware.

- PR: [#7996](https://github.com/elizaos/eliza/pull/7996)

### Local inference: DFlash binary discovery + routing hardening
Local inference now discovers DFlash binaries from both isolated state dirs and the default user install root (`~/.eliza/...`) and tightens runtime router wiring.

- PR: [#7983](https://github.com/elizaos/eliza/pull/7983)

### Desktop first-run voice startup hardening (Electrobun packaged runtime)
Fixes first-run failures where packaged apps didn’t correctly resolve bundled runtime/probers/plugins and where vault bootstrapping could abort too early.

- PR: [#7994](https://github.com/elizaos/eliza/pull/7994)

### Agent image boot chain fixes (remote plugin runtime + server-only mode)
Two issues that caused provisioned cloud agents to crash at boot:
- missing built `plugin-worker-runtime` in the image: PR [#7981](https://github.com/elizaos/eliza/pull/7981)
- terminal TUI loading in server-only mode: PR [#7995](https://github.com/elizaos/eliza/pull/7995)

## 4) API Changes (developer-facing)

### Cloud API: `/v1/chat/completions` tool_choice mapping
- `tool_choice: "required"` is now handled correctly (see [#7992](https://github.com/elizaos/eliza/pull/7992)).
- If you’re building client libraries, remove any special-casing that forced `tool_choice` into `{ type: "function", function: { name } }` when you really wanted “required”.

### Cloud: agent lifecycle operations via job queue (async 202 + jobId)
Lifecycle operations (suspend/restart/snapshot/logs) moved behind the job queue so Workers don’t synchronously call cores.

- PR: [#7810](https://github.com/elizaos/eliza/pull/7810)
- Frontend now tracks returned job IDs for toasts/status: PR [#7813](https://github.com/elizaos/eliza/pull/7813)

**Migration note:** if you call lifecycle endpoints directly, expect:
- `202 Accepted`
- a `jobId` you must poll/track instead of assuming completion on first response.

### Cloud: ERC-8004 identity endpoints
New endpoints for registering and querying agent identity:
- PR: [#7937](https://github.com/elizaos/eliza/pull/7937)

### Self-hosted runtime: CORS + bearer auth (cross-platform)
For remote dashboards/mobile/desktop builds connecting to a hosted agent runtime, the runtime now supports CORS + bearer auth and cross-platform build fixes.

- PR: [#7212](https://github.com/elizaos/eliza/pull/7212)

## 7) Breaking Changes / V1 → V2 Migration Warnings

### Plugin manifests and runtime expectations changed (direct/remote unification)
If you have internal tooling that assumes:
- “remote runner” is a different type than “plugin”
- plugin loading only happens in-process
- views/actions/providers are only available from direct plugins

…you’ll need to adapt to the **single Plugin abstraction** with `mode`:
- See PR chain: [#7844](https://github.com/elizaos/eliza/pull/7844) → [#7846](https://github.com/elizaos/eliza/pull/7846) → [#7847](https://github.com/elizaos/eliza/pull/7847) → [#7848](https://github.com/elizaos/eliza/pull/7848)

**Practical guidance:** treat `mode` as part of the compatibility surface when you:
- validate manifests
- decide whether a plugin can run on a given host (web vs desktop vs mobile vs cloud)
- expose permissions/capabilities to the planner

### Cloud lifecycle endpoints are now async jobs (202 + jobId)
If you built automation around “restart/suspend returns when complete”, that assumption is no longer valid after:
- [#7810](https://github.com/elizaos/eliza/pull/7810)

Update your client to track `jobId` and poll status (or subscribe when available).

### Prompt-rule contracts are tightening user-facing text behavior
Several fixes moved behavior from ad-hoc regex evaluators into prompt-rule contracts and evaluator sanitation (e.g., preventing “training cutoff” disclosures, stripping appended JSON envelopes). If you built tests that snapshot exact assistant phrasing, expect diffs:
- [#7931](https://github.com/elizaos/eliza/pull/7931), [#7965](https://github.com/elizaos/eliza/pull/7965), [#7956](https://github.com/elizaos/eliza/pull/7956)
