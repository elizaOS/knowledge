## 1) Core Framework

### LifeOps architecture refactor: monolith → modular plugin suite (breaking)
The long-standing `@elizaos/plugin-lifeops` monolith was decomposed and **removed** (no compatibility alias) in:
- **PR #8277** — “split plugin-lifeops into focused plugins + plugin-personal-assistant”  
  https://github.com/elizaos/eliza/pull/8277

Key structural impacts:
- `plugin-lifeops` is replaced by:
  - `@elizaos/plugin-personal-assistant` (cross-domain orchestration; no `/lifeops` view)
  - New domain plugins: `plugin-relationships`, `plugin-inbox`, `plugin-goals`, `plugin-blocker`, `plugin-finances`, `plugin-remote-desktop`
  - Expanded domain plugins: `plugin-todos`, `plugin-health`, `plugin-calendar`, `plugin-documents`, `plugin-phone`
- Contract rename: `packages/shared/src/contracts/lifeops.ts` → `personal-assistant.ts`
- UI routing change: `/lifeops` view deleted; per-domain plugin views now own their surfaces.

Follow-up boot fix for apps still referencing removed exports:
- **PR #8314** — remove calls to removed `registerLifeOpsApp` and view exports  
  https://github.com/elizaos/eliza/pull/8314

### “Agentic views surface” and view-action affinity
The agent runtime gained a **first-class view surface registry** and a “view-action affinity” layer that biases planning toward actions relevant to the currently active view:
- **PR #8080** — “Claude/agentic views surface”  
  https://github.com/elizaos/eliza/pull/8080

This improves agent controllability over plugin UI surfaces and makes “what the user is looking at” a runtime signal for planning/prompt optimization.

### Cloud V2 production sync
- **PR #8116** — develop → main promotion (“Cloud V2 sync”)  
  https://github.com/elizaos/eliza/pull/8116

This aligns production with the Cloud V2 stack, including data-plane and auth changes.

Example usage (Cloud SDK):

```ts
import { ElizaCloudClient } from "@elizaos/cloud-sdk";

const client = new ElizaCloudClient({
  baseUrl: "https://api.elizacloud.ai/api/v1",
});

// 1) Web sign-in helper (CLI-login style flow)
const session = await client.signInWithElizaCloud({
  appId: "your-app-id",
  // options vary by integration; see SDK README
});

// 2) STT helper
const stt = await client.transcribeAudio({
  audio: {
    filename: "memo.webm",
    contentType: "audio/webm",
    bytes: await Bun.file("memo.webm").arrayBuffer(),
  },
  model: "whisper-1",
});

console.log(stt.text);
```

## 3) Bug Fixes (critical)

### Cloud auth: Steward OAuth PKCE + signed proxy calls (login unblocked)
Steward hardened OAuth and required **PKCE**, breaking “Sign in with Eliza Cloud” flows until the Cloud frontend/API were updated.

Fixes landed across:
- **PR #8259** — send PKCE on Steward OAuth + sign nonce-exchange upstream call  
  https://github.com/elizaos/eliza/pull/8259
- **PR #8256** — sign Steward proxy requests (HMAC guard on sensitive paths)  
  https://github.com/elizaos/eliza/pull/8256
- **PR #8280 / #8287** — cookie refresh session hydration + signed refresh upstream  
  https://github.com/elizaos/eliza/pull/8280  
  https://github.com/elizaos/eliza/pull/8287
- **PR #8279 / #8278** — stabilize redirect URI + mobile cloud onboarding fixes  
  https://github.com/elizaos/eliza/pull/8279  
  https://github.com/elizaos/eliza/pull/8278

Related tracked incident (still useful for historical context):
- **Issue #8257** — Steward login broken (OAuth 400 + email/send 400)  
  https://github.com/elizaos/eliza/issues/8257

### Cloud agent connectivity: browser auth for WS/SSE
Browsers can’t set `Authorization` headers on WebSocket handshakes or EventSource; fixes added query-param token support:
- **PR #8308** — pass WS auth via query param during handshake  
  https://github.com/elizaos/eliza/pull/8308
- **PR #8309** — accept `?token=` for SSE GETs (EventSource auth)  
  https://github.com/elizaos/eliza/pull/8309

### Cloud inference correctness: reasoning models returning `null` but billed
Reasoning models could consume the token budget internally (hidden reasoning) and return `content: null` while still billing.
- **PR #8189** — guarantee response-token budget for reasoning models  
  https://github.com/elizaos/eliza/pull/8189

### Orchestrator correctness: duplicate replies race (TOCTOU)
A check-then-act race in completion dedup could cause duplicate completions across sessions:
- **PR #8161** — close completion-dedup TOCTOU race  
  https://github.com/elizaos/eliza/pull/8161

### Fresh image boot regression: optionalize heavy/default plugins
Default agent image boot was blocked by plugins that should not be mandatory everywhere:
- **PR #8105** — move `plugin-google` and (then-)`plugin-lifeops` to optional so default image boots  
  https://github.com/elizaos/eliza/pull/8105

## 7) Breaking Changes (V1 → V2 migration warnings)

### 7.1 `@elizaos/plugin-lifeops` removal (no shims)
**Breaking:** `@elizaos/plugin-lifeops` is removed and not aliased.
- Migration PR: **#8277**  
  https://github.com/elizaos/eliza/pull/8277
- Boot fix for stale app references: **#8314**  
  https://github.com/elizaos/eliza/pull/8314

**What to do:**
- Replace LifeOps imports with `@elizaos/plugin-personal-assistant` and/or the new domain plugins.
- Update any UI deep links from `/lifeops` to the new domain views (e.g. `/calendar`, `/todos`, `/inbox`, etc., depending on installed plugins).
- Update contract imports (`lifeops.ts` → `personal-assistant.ts`).

### 7.2 Cloud connectivity: WS/SSE token transport changed
If you embed the agent Web UI or implement custom clients:
- WebSockets now support query-param auth (PR #8308).
- SSE now supports `?token=` for EventSource (PR #8309).

If you previously relied on header-based auth for these channels in-browser, update to the query-param pattern.

### 7.3 Cloud V2 behavioral changes to expect
With Cloud V2 promotions (PR #8116) and Apps lane work:
- staging/prod separation tightened (BitRouter base URLs, R2 bindings, Steward tenant pinning)
- more routes are signed/HMAC-protected upstream to Steward
- container/apps hosting introduces new lifecycle concepts and billing surfaces; review **Issue #8321** for remaining scale constraints before expecting “1000-app” behavior.
