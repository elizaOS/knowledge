## 2) New Features

### (A) Cloud: agent sleep/wake lifecycle + incremental sandbox backups
Cloud now supports a “deep cold storage” lifecycle:
- **Sleep**: backup + release compute
- **Wake**: reprovision + restore

Endpoints (Cloud API):
- `POST /api/v1/eliza/agents/:agentId/sleep` (async, returns `202`)  
  PR: https://github.com/elizaos/eliza/pull/8080
- Backups support **incremental diff** storage to reduce backup payload size and improve schedule cadence.  
  PR: https://github.com/elizaos/eliza/pull/8080

**Example (sleep an agent)**
```bash
curl -X POST \
  "https://api.elizacloud.ai/api/v1/eliza/agents/$AGENT_ID/sleep" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN" \
  -H "Content-Type: application/json"
```

### (B) Cloud: containers API now implemented
The Cloud broker exposed container operations conceptually, but the HTTP route was missing. This week added:
- `POST /api/v1/containers` (deploy)
- `GET /api/v1/containers` / `GET /api/v1/containers/:id` (read/list, per implementation)  
PR: https://github.com/elizaos/eliza/pull/8258

This unblocks skills and automation flows that depend on container deploy being a first-class capability (not just an internal broker method).

### (C) Cloud SDK: web sign-in helper + `X-App-Id` affordance for inference
The SDK now includes:
- a helper for the “Sign in with Eliza Cloud” polling loop (previously hand-rolled by third-party apps)
- first-class support for attaching `X-App-Id` to inference requests for attribution/monetization  
PR: https://github.com/elizaos/eliza/pull/8251  
Docs: https://github.com/elizaos/eliza/tree/main/packages/cloud-sdk

### (D) Runtime actions: keyless `WEB_FETCH` for non-Anthropic models
Historically, “keyless web” behavior existed as an Anthropic-specific patch. This week generalized it so agents running other backends (e.g., Cerebras-hosted `gpt-oss`) can fetch live info inline via a standard runtime action.  
PR: https://github.com/elizaos/eliza/pull/8097

**Example (conceptual action call from a custom runtime action)**
```ts
// inside a custom action handler
const res = await runtime.actions.run("WEB_FETCH", {
  url: "https://example.com/status",
  method: "GET",
});
```

### (E) UI framework: `fullBleed` app-shell pages for tool/workbench UIs
A new optional page registration flag removes surrounding dashboard chrome for views that need full-window control (e.g., orchestrator workbenches).  
PR: https://github.com/elizaos/eliza/pull/8180

```ts
// Example registration shape (conceptual)
registerPage({
  path: "/odysseus",
  title: "Odysseus",
  fullBleed: true,
  component: OdysseusShell,
});
```

## 7) Breaking Changes (V1 → V2 migration warnings)

### **BREAKING: `@elizaos/plugin-lifeops` removed**
- `@elizaos/plugin-lifeops` is **gone** (no compatibility alias/shim).
- Replacement:
  - `@elizaos/plugin-personal-assistant` for orchestration actions (`BRIEF`, `PRIORITIZE`, etc.)
  - domain plugins for views/actions previously bundled in LifeOps (relationships/inbox/goals/blocker/finances/remote-desktop + extended todos/health/calendar/documents/phone)
- `/lifeops` dashboard view was deleted; per-domain plugin views are now the supported UI surface.  
PR: https://github.com/elizaos/eliza/pull/8277

**Migration checklist**
1. Replace imports:
   - `@elizaos/plugin-lifeops` → `@elizaos/plugin-personal-assistant` and/or the new domain plugins.
2. Update app manifests:
   - `apps/lifeops.json` → `apps/personal-assistant.json` (and enable desired domain plugins).
3. Update any contract imports:
   - `packages/shared/src/contracts/lifeops.ts` → `personal-assistant.ts`

### UI shell behavior: `fullBleed` is the new standard for chrome-less pages
If you previously depended on orchestrator-specific header hacks, migrate to `fullBleed?: true` in page registration. This is backwards-compatible for other pages but changes how new workbench-like pages should be integrated.  
PR: https://github.com/elizaos/eliza/pull/8180
