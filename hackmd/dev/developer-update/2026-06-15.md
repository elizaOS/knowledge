## 1) Core Framework

### CI architecture: path-based gating for expensive lanes
PR **#8408** adds a lightweight path classifier to CI so expensive test lanes are skipped when unrelated surfaces are untouched (server/client/plugins/desktop/zero-key/cloud), while **manual/scheduled runs still execute full coverage**.

- Merged PR: https://github.com/elizaos/eliza/pull/8408  
- Follow-up (opened): make the gating *label-driven and more explainable* (**#8414**)  
  https://github.com/elizaos/eliza/pull/8414

**Developer impact**
- Faster PR feedback cycles when you’re changing localized areas (e.g., docs, isolated plugins).
- If you rely on a full-stack signal for a risky change that touches a small path surface, expect to opt into expensive lanes (see #8414 once merged for the exact label semantics).

### Apps (Cloud “Product 2”) staging runtime brought closer to “real deploy”
Several infra/ops changes this week reduce the amount of manual operator work required to stand up and test the Apps data-plane:

- Install the **Caddy front door** directly in app-node cloud-init (removes last “apps are unreachable” gap at the infra layer):  
  https://github.com/elizaos/eliza/pull/8394
- Add an **idempotent apps-daemon arming script** (no more hand-editing env on servers):  
  https://github.com/elizaos/eliza/pull/8397
- Add a **dispatchable GitHub workflow** to arm the apps daemon (no operator SSH needed):  
  https://github.com/elizaos/eliza/pull/8398
- Add a one-shot workflow to remove an **orphaned Terraform state entry** after module splits:  
  https://github.com/elizaos/eliza/pull/8400
- Enable `APPS_DEPLOY_ENABLED` on **staging Worker** (turns on the real deploy trigger for staging only):  
  https://github.com/elizaos/eliza/pull/8406

## 2) New Features

### (Cloud) Apps deploy trigger enabled on staging
PR **#8406** flips the staging-only configuration so staging can run the actual “Apps deploy” trigger end-to-end (Worker → deploy backend → app node ingress).

- PR: https://github.com/elizaos/eliza/pull/8406

**Why it matters**
- You can now validate the Apps deploy flow in staging without patching config locally or manually toggling flags on hosts.
- This is foundational for iterating on container hosting, per-app routing, and billing behavior under realistic conditions.

### (Infra/DevOps) Operator-free apps backend arming
The combination of **#8397** + **#8398** turns “arming the apps backend” into a repeatable operation.

- Script: https://github.com/elizaos/eliza/pull/8397  
- Workflow: https://github.com/elizaos/eliza/pull/8398

**Operational example (what to expect)**
Once your org has the right permissions, arming should look like a **workflow_dispatch** run rather than SSHing and editing `/opt/eliza/cloud/.env.local` by hand. Track the exact inputs/outputs in the workflow file from #8398.

## 3) Bug Fixes (Critical / High Impact)

### (Cloud Apps) Deleted apps now tear down isolated tenant DBs (leak fix)
PR **#8401** fixes a critical resource leak: deleting an app that used an **isolated per-tenant Postgres** previously did *not* DROP the DB/ROLE nor release the tenant-db “slot”, leading to stranded cost and eventual capacity exhaustion.

- PR: https://github.com/elizaos/eliza/pull/8401

**Technical context**
- The Apps data-plane uses a “tenant DB cluster” allocation model; leaking slots reduces effective `max_databases` over time.
- #8401 introduces explicit deprovision job wiring so app deletion triggers DB teardown.

**Developer-facing implications**
- App deletion may now entail additional asynchronous cleanup work; if your tooling assumes “delete == immediate resource disappearance”, update expectations (poll status / handle eventual consistency).

### (CI) Reduced unnecessary test execution while preserving scheduled coverage
PR **#8408** is also a reliability fix: it reduces the noise of unrelated failures (quota throttling, flaky UI e2e, etc.) on PRs that don’t touch those surfaces, while keeping full coverage on scheduled runs.

- PR: https://github.com/elizaos/eliza/pull/8408

## 4) API Changes

### (Cloud Apps) Delete semantics now include DB deprovision (behavioral change)
As a result of **#8401**, the “delete app” path now also orchestrates **tenant database teardown** when database mode is isolated.

- PR: https://github.com/elizaos/eliza/pull/8401

**What to watch for**
- If you have automation around app lifecycle, treat deletion as a multi-step cleanup process.
- If you expose app deletion in your own UI, consider surfacing “cleanup in progress” states.

### (Cloud staging) Apps deploy trigger enabled
**#8406** changes staging behavior by enabling the deploy trigger. If you run staging smoke tests or staging environment tooling, you should now expect deploy hooks to fire (and to fail “for real” if infra is misconfigured rather than silently no-op’ing).

- PR: https://github.com/elizaos/eliza/pull/8406
