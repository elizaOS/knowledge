# ElizaOS Developer Update — 2026-06-04 to 2026-06-10

## 1) Core Framework (runtime/architecture)

### LifeOps monolith removal → modular “Personal Assistant” architecture (V2 direction)
The largest architectural shift affecting core composition this week is the **removal of `@elizaos/plugin-lifeops` as a monolith** and the introduction of a slimmer orchestrator plugin plus domain-focused plugins.

- **Refactor:** `plugin-lifeops` split into focused plugins + `plugin-personal-assistant` (merged)  
  PR: https://github.com/elizaos/eliza/pull/8277

This refactor also scaffolds future runtime primitives (owner-state, registries, scheduled-task types) into `@elizaos/core` / `@elizaos/app-core` as *interfaces + READMEs* to guide upcoming migrations, while leaving canonical implementations temporarily in `plugin-personal-assistant`.

**Developer impact:** plugin composition, imports, and view routing expectations change (see **Breaking Changes**).

## 2) New Features

### Eliza Cloud Apps (Product 2): app ingress, per-tenant DB modes, and lifecycle hardening
This week continues the push toward a production-ready “Apps” data plane: per-app routing, safer provisioning, and cost controls.

Key additions/expansions:

- **Apps ingress (Caddy) “front door” for `<id>.apps.elizacloud.ai` routing**  
  PR: https://github.com/elizaos/eliza/pull/8325  
  Provides live per-app reverse-proxy routes via Caddy’s admin API, wiring the deploy flow so Apps are reachable externally (addresses #8321 P0 ingress gap).

- **Opt-in per-tenant database mode (`none|isolated`)**  
  PR: https://github.com/elizaos/eliza/pull/8338  
  Adds an API surface to control DB provisioning and prevents unnecessary DB allocation for apps that don’t need it.

- **Tenant DB teardown + slot release on app deletion**  
  PR: https://github.com/elizaos/eliza/pull/8343  
  Ensures `DROP DATABASE/ROLE` + cluster slot release occur during cleanup (leak closure referenced by the Apps scaling audit).

- **Dedicated tenant-DB node is now optional (default off) to reduce fixed costs**  
  PR: https://github.com/elizaos/eliza/pull/8344

**Example: selecting an Apps database mode**
If you’re building a deploy flow or an internal tool that creates Apps, you should now treat “isolated DB” as an opt-in:

```ts
// Pseudocode: choose DB mode explicitly during/after app creation
// See PR #8338 for the new route wiring and server-side behavior.

type AppDatabaseMode = "none" | "isolated";

const mode: AppDatabaseMode = "isolated"; // or "none" for stateless apps

await fetch(`/api/v1/apps/${appId}/database`, {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    Authorization: `Bearer ${token}`,
  },
  body: JSON.stringify({ mode }),
});
```

### Cloud SDK: STT transcription helper for third-party apps
- **New:** `ElizaCloudClient.transcribeAudio()` (voice/STT)  
  PR: https://github.com/elizaos/eliza/pull/8266

```ts
import { ElizaCloudClient } from "@elizaos/cloud-sdk";

const client = new ElizaCloudClient({
  baseUrl: "https://www.elizacloud.ai/api/v1",
  apiKey: process.env.ELIZA_CLOUD_API_KEY!,
});

const wav = await Bun.file("./sample.wav").arrayBuffer();

const result = await client.transcribeAudio({
  // exact request fields per SDK types in PR #8266
  audio: new Blob([wav], { type: "audio/wav" }),
  // model: "..." // if your org exposes selectable STT models
});

console.log(result.text);
```

## 3) Bug Fixes (critical/high-signal)

### Apps data plane: collision-safe host port allocation
- **Fixed:** random host-port allocation collisions at scale (few hundred containers/node)  
  PR: https://github.com/elizaos/eliza/pull/8328  
  The allocator now probes used ports (best-effort) instead of blind random selection.

**Why it matters:** avoids intermittent `docker create` failures and reduces “heisenbugs” during bursty deploys.

---

### Apps isolation contract: inject tenant DSN under both `DATABASE_URL` and `POSTGRES_URL`

**Practical guidance:** if you ship an Eliza-based container image, treat both variables as supported, but prefer a single canonical variable internally to reduce drift.

## 7) Breaking Changes (V1 → V2 migration warnings)

### `@elizaos/plugin-lifeops` removed (no backwards-compat shim)
- **Breaking refactor:** PR https://github.com/elizaos/eliza/pull/8277
- `plugin-lifeops` is replaced by:
  - `@elizaos/plugin-personal-assistant` (orchestration layer)
  - multiple domain plugins (calendar, inbox, goals, blocker, finances, etc.)
- `/lifeops` dashboard view is removed; domain plugins now own their own views.

**What you need to change**
1. **Update dependencies/imports**
   - Remove `@elizaos/plugin-lifeops`
   - Add the specific plugins you use (calendar/todos/goals/etc.) plus `plugin-personal-assistant` if you relied on the orchestration actions.

2. **Update view routing assumptions**
   - If your UI automation or documentation references `/lifeops`, migrate users to the new domain views (e.g. `/calendar`, `/goals`, etc., depending on enabled plugins).

3. **Default packs / actions**
   - If you referenced LifeOps actions directly, re-map to the new action owners (many actions moved to domain plugins; cross-domain actions remain under Personal Assistant).

**Related follow-up fix (post-refactor crash):**
- SPA boot crash fix after LifeOps removal (`registerLifeOpsApp` calls removed)  
  PR: https://github.com/elizaos/eliza/pull/8314
