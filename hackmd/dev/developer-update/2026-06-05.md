# ElizaOS Developer Update (2026-05-31 → 2026-06-06)

This week centered on **Cloud V2 production sync**, **agent runtime capability expansion (keyless web fetch, view/action affinity)**, and **orchestrator/workbench reliability + UX parity**. Discord discussions were lighter on implementation details but included important ecosystem/process clarifications (plugins now live in `elizaos/eliza`) and new security-focused contributors offering audits around execution boundaries and prompt-injection defense.

---

## 1) Core Framework

### Cloud V2 production promotion
Production was synced with recent Cloud V2 stack changes via:
- **Release promotion:** develop → main (**merged**)  
  PR: https://github.com/elizaos/eliza/pull/8116

This triggered the standard production deploy chain (Cloudflare Worker + Pages frontend, provisioning worker daemon restart, and stable/latest agent image build).

### Agent runtime: view-aware action routing (“agentic view surfaces”)
The merged **agentic views surface** work introduced a framework for making plugin views *agent-controllable*, plus a runtime “affinity” layer that upweights view-relevant actions while the user is on a given view.

- **Merged PR:** “Claude/agentic views surface”  
  PR: https://github.com/elizaos/eliza/pull/8080

Key architecture pieces (high-level):
- `ViewAgentRegistry` + `AgentSurfaceProvider` + `useAgentElement` overlay plumbing (UI ↔ agent capability dispatch)
- **`view-action-affinity`** in `packages/agent/src/runtime/` that feeds prompt optimization with the active view context
- Cloud sleep/wake lifecycle endpoints + incremental backup support (see API Changes)

### Core plugins: boot stability via optionalization
To prevent default images from failing to boot when certain integrations aren’t configured, two plugins were moved to optional:
- `plugin-google`
- `plugin-lifeops`

- **Merged PR:** https://github.com/elizaos/eliza/pull/8105

This affects how “core plugin set” is resolved at runtime (important if you assumed those plugins are always present in default images).

## 2) New Features

### Keyless `WEB_FETCH` action for non-Anthropic models
ElizaOS now supports a **keyless web fetch** action that works across non-Anthropic backends (e.g. Cerebras-hosted `gpt-oss-*`), removing an Anthropic-only monkeypatch dependency.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8097  
- Related issue (closed): https://github.com/elizaos/eliza/issues/8100

**Developer impact:** if you’re building agents that need live web retrieval but are running on non-Anthropic models, you can now rely on a first-class action rather than provider-specific wiring.

Example (conceptual) usage in an agent prompt/action plan:
```ts
// Pseudocode: planner chooses WEB_FETCH when it needs live info
const result = await runtime.actions.WEB_FETCH({
  url: "https://api.example.com/status",
  format: "text",
});

// Feed into response synthesis
return `Status page says:\n\n${result.text}`;
```

### Cloud: agent sleep/wake + incremental diff backups
Cloud agents can now be placed into deep cold-storage and restored later, backed by a more efficient backup system that supports **incremental (diff) backups**.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8080  
- Adds migration `0136_agent_backup_incremental.sql` (columns: `backup_kind`, `parent_backup_id`, `content_hash`, etc.)

This is foundational for:
- lowering idle compute costs
- faster restores
- more frequent snapshotting (cron-backed)

### Cloud: enable per-agent web UI by default
Provisioned agents now default to having their web UI enabled.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8167

**Security note:** this makes UX better out-of-the-box, but you should still validate your auth / network exposure model for any publicly reachable agents.

### App-core: configurable runtime entry filename
The node entry file name used by the runner is now configurable via `ELIZA_ENTRY_FILE`.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8126

This improves whitelabel and specialized packaging workflows that don’t want to ship `eliza.mjs` as the canonical entrypoint.

Example:
```bash
ELIZA_ENTRY_FILE=my-agent-entry.mjs bun run start
```

## 3) Bug Fixes (critical + technical context)

### Orchestrator: JSON-RPC errors now preserve `error.data`
ACP JSON-RPC errors such as `-32602 Invalid params` often carry actionable diagnostics in `error.data` (e.g. Zod formatted field errors). These were being dropped, making failures opaque.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8098  
- Issue (closed): https://github.com/elizaos/eliza/issues/8101

**Why it matters:** if you build custom ACP tools or sub-agent schemas, you now get the detailed field-level failures back for debugging and telemetry.

### Orchestrator: completion de-dup TOCTOU race (duplicate replies)
A check-then-act race in completion de-dup logic could cause duplicate `task_complete` deliveries across sessions.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8161

**Fix class:** tightened cross-session “first poster” ownership around awaited delivery loops.

### Cloud: guarantee response-token budget for reasoning models
Some reasoning models returned `content: null` (while still billing tokens) when `max_tokens` couldn’t accommodate both hidden reasoning and final response.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8189

**Developer impact:** more consistent completions and fewer “paid-but-empty” responses when running reasoning-class models through Cloud.

### Vault: self-heal stale PGlite locks (boot unbrick)
Unclean shutdowns could leave a stale lock in the vault’s PGlite store and brick startup early in wallet-key hydration.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8092

### Core routing: “market term” no longer steals coding tasks into web-search
Routing heuristics were checking web-search patterns before coding-work patterns. A coding prompt containing market/live terms could get misrouted.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8160

### UI onboarding: self-heal when pinned to loopback agent port
Clients pointing at a raw loopback port (e.g. dev default `127.0.0.1:31337`) could land in a confusing auth state.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8082

### Local inference: typed unavailability signal from text handlers
Local llama.cpp text handlers now throw a *typed* unavailability signal rather than a generic `Error` when bindings/bundles are missing.

- **Merged PR:** https://github.com/elizaos/eliza/pull/8191

This improves upstream error handling (UI and runtime can differentiate “not available” from “failed”).

## 4) API Changes

### New Cloud API endpoints (sleep/backups)
Merged in the agentic views surface PR:
- `POST /v1/eliza/agents/:agentId/sleep` (async-style semantics; returns 202 even if already in progress)
- `GET/POST /v1/eliza/agents/:agentId/backups` (agent backup listing/creation; plus cron-backed backup scheduling)

PR: https://github.com/elizaos/eliza/pull/8080

Example:
```bash
curl -X POST \
  "https://api.elizacloud.ai/v1/eliza/agents/$AGENT_ID/sleep" \
  -H "Authorization: Bearer $TOKEN"
```

### CORS / staging auth integration hardening (Steward)
Multiple fixes landed to stabilize staging magic-link auth flows:
- Pin tenant via `X-Steward-Tenant` header and align staging proxying  
  PR: https://github.com/elizaos/eliza/pull/8179
- Allow `staging.elizacloud.ai` in Cloud API CORS allowlist  
  PR: https://github.com/elizaos/eliza/pull/8196
- Allow `api-staging.elizacloud.ai` in staging CSP `connect-src`  
  PR: https://github.com/elizaos/eliza/pull/8197
- Fix magic-link callback when app-auth context is missing  
  PR: https://github.com/elizaos/eliza/pull/8194
- Isolate Steward tenant for staging redirect correctness  
  PR: https://github.com/elizaos/eliza/pull/8176

**Developer impact:** if you embed or automate auth flows against staging, re-check any hardcoded origins and ensure you send the expected tenant header where required.

## 7) Breaking Changes / V1 → V2 Migration Warnings

### Cloud V2 rollout: verify your deployment assumptions
With the Cloud V2 sync promoted to production:
- ensure your automation is compatible with the current prod deploy workflows and schema level
- validate any custom tooling that assumes older provisioning semantics or endpoints

PR: https://github.com/elizaos/eliza/pull/8116

### Plugin repository/process change (developer workflow)
Per Discord architecture guidance (2026-06-02), **all plugins now live in `elizaos/eliza`** and contributors should fork and target `develop`:

- Discussion reference: https://discord.com/channels/1253563208833433701/1253563209462448241

If you previously developed against separate plugin repos, update your contribution workflow accordingly and expect rapid movement on `develop` (frequent rebasing recommended).

### Default plugin availability changed
Because `plugin-google` and `plugin-lifeops` moved to optional, images/environments that assumed those plugins are always installed may now fail at runtime unless explicitly enabled/installed.

PR: https://github.com/elizaos/eliza/pull/8105
