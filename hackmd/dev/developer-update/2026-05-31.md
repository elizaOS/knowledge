## 1) Core Framework

### Plugin runtime unification: `Plugin` now supports `mode: "direct" | "remote"`
Work continues on collapsing the old “Satellites/Carrots/remote runners” vocabulary into a single plugin model that can run either in-process or out-of-process with a shared contract:

- **P0 (scaffolding + rename)**: `Plugin` manifests gain a `mode` discriminator and shared envelope types  
  PR: https://github.com/elizaos/eliza/pull/7844
- **P1 (remote-mode runtime + isolated-process worker)**: remote plugins can actually run via `plugin-worker-runtime` + host bridge  
  PR: https://github.com/elizaos/eliza/pull/7846
- **P2 (view-host shims + service/route surfaces)**: unified host shims across Android/iOS/Electrobun/Web and worker dispatch plumbing  
  PR: https://github.com/elizaos/eliza/pull/7847
- **P3 (installRemotePlugin API + first sub-agent runner)**: adds the first agent-generated plugin install surface and a reference sub-agent runner  
  PR: https://github.com/elizaos/eliza/pull/7848

**Operational note:** a critical follow-up fixed cloud-agent boots by ensuring `plugin-worker-runtime` is built into the agent image:
- PR: https://github.com/elizaos/eliza/pull/7981

### Planner/evaluator hardening (tool routing + output hygiene)
Multiple fixes tightened the planning loop behavior and prevented user-facing leakage of internal control text:

- Bound runaway tool-arg validation loops using a canonical failure signature  
  PR: https://github.com/elizaos/eliza/pull/7924
- Restore evaluator `messageToUser` precedence over tool `userFacingText` (prevents “tool succeeded so ignore evaluator’s explicit final message”)  
  PR: https://github.com/elizaos/eliza/pull/7897
- Strip trailing JSON envelopes from recovered prose (prevents “answer + raw JSON blob”)  
  PR: https://github.com/elizaos/eliza/pull/7956
- Promote to planning if Stage 1 hints a tool even when Stage 1 self-contradicts with `simple=true`  
  PR: https://github.com/elizaos/eliza/pull/7957
- Forbid empty/bare-ack `replyText` on the simple path; clarify interim-ack contract  
  PR: https://github.com/elizaos/eliza/pull/7953
- Forbid “training cutoff” metadata in `replyText` (cleaner, less meta, more deterministic)  
  PR: https://github.com/elizaos/eliza/pull/7965

Related open issue worth tracking if you build chat-recall/search features:
- Planner sometimes chooses `SHELL`/`grep` over `SEARCH_MESSAGES`, can leak sub-agent routing metadata, and can time out on monorepo greps  
  Issue: https://github.com/elizaos/eliza/issues/7935

## 2) New Features

### A) Connector account model: OWNER vs AGENT identities (multi-account connectors)
Connectors can now declare two independent identities:
- **OWNER**: the user’s personal account on the platform
- **AGENT**: the agent’s own bot/service identity

Core schema + docs/audit:
- PR: https://github.com/elizaos/eliza/pull/7817
- PR (UI): role-aware account list + dual-section setup panel: https://github.com/elizaos/eliza/pull/7819

Example connector entry snippet (registry JSON):
```json
{
  "id": "x",
  "type": "connector",
  "accounts": {
    "owner": { "enabled": true },
    "agent": { "enabled": true }
  }
}
```

This enables patterns like “agent posts as itself” vs “agent posts as the user” with explicit role separation in both storage and UI.

### B) Slack connector brought into the monorepo + OWNER posting support
- Slack plugin migration into `plugins/plugin-slack` (source-of-truth now lives in `elizaos/eliza`)

  PR: https://github.com/elizaos/eliza/pull/7868

## 3) Bug Fixes (critical)

### Cloud API: `tool_choice: "required"` crash
A production-breaking issue in the OpenAI-compatible `chat/completions` route caused a 500 when callers set `tool_choice: "required"`.

- Fix on develop: https://github.com/elizaos/eliza/pull/7992  
- Cherry-pick to main: https://github.com/elizaos/eliza/pull/7993

**Impact:** planners that enforce tool usage (or client libraries that emit `"required"`) can now reliably hit Cloud without hard failures.

### Headless Linux boot: vault/keychain segfault avoidance
Agent runtime could segfault on headless Linux when `libsecret` wasn’t available over D-Bus. This is a process-level crash (not catchable JS).

- Fix: skip OS keychain on headless Linux; keep secure behavior for supported environments  
  PR: https://github.com/elizaos/eliza/pull/7230

### Cloud-agent boot chain fixes (image completeness + server-only mode)
Several issues prevented provisioned cloud agents from starting cleanly:

- Missing `plugin-worker-runtime` dist artifacts in agent image → `ERR_MODULE_NOT_FOUND` at boot  
  PR: https://github.com/elizaos/eliza/pull/7981
- Server-only agents should not load the interactive terminal TUI  
  PR: https://github.com/elizaos/eliza/pull/7995
- Agent image completeness: bundle YAML + build `plugin-edge-tts` so core plugin set loads fully  
  PR: https://github.com/elizaos/eliza/pull/8002

### SQL migrator: `varchar` → `uuid` type changes now emit `USING`
Postgres requires a `USING` clause to cast existing values safely during type changes.

- Fix: https://github.com/elizaos/eliza/pull/8009

## 5) Social Media Integrations (Twitter / Telegram / Discord / Farcaster)

### Discord: reply grounding + reduced stale setup prompts
Discord inbound envelopes now carry a structured `platform_reply_reference` that core renders into a `reply_reference` segment to ground the model’s response in the correct message thread, reducing mis-replies and context drift.

- PR: https://github.com/elizaos/eliza/pull/7821

### X (Twitter): OAuth scope regression + revert
A fix attempted to remove DM scopes by default to avoid OAuth failures when the app lacks DM entitlements:

- PR (drop DM scopes): https://github.com/elizaos/eliza/pull/7929  
- Revert (actual cause was callback URL mismatch, not scopes): https://github.com/elizaos/eliza/pull/7930

**Action for plugin-x users:** if you are seeing “Something went wrong” in OAuth, verify your X developer app callback URLs match the environment (don’t rely solely on scope trimming).

### Slack: now first-class in monorepo
See §2B for OWNER posting and dual-token support.

**Telegram / Farcaster:** no notable PR activity in the provided data window; watch for follow-ups as connector role separation expands across the connector set.

## 7) Breaking Changes / Migration Notes (V1 → V2)

### A) Solana plugin relocation: `plugin-solana` → `plugin-wallet` (Solana chain subdir)
From Discord engineering support, the Solana plugin code was moved into the main repo and reorganized:

- New location (develop): `plugins/plugin-wallet/src/chains/solana`  
  Discord context: https://discord.com/channels/1253563208833433701/1253563209462448241 (2026-05-30)

**Migration impact:**
- If you referenced the old `plugin-solana` package name, update imports/config to the new plugin packaging.
- If you maintained forks or external plugin references, rebase on the new directory structure and ensure your runtime loads `plugin-wallet`.

### B) Plugin “mode” unification may require manifest/schema updates
If you ship plugins that previously targeted “remote runner” or Electrobun Satellites/Carrots directly, begin migrating to the unified plugin contract:

- Start at P0/P1 to understand the new manifest + runtime wiring:  
  https://github.com/elizaos/eliza/pull/7844  
  https://github.com/elizaos/eliza/pull/7846

### C) Connector setup now differentiates OWNER vs AGENT
Connector UI + registry schema now model role separation explicitly:

- Schema PR: https://github.com/elizaos/eliza/pull/7817  
- UI PR: https://github.com/elizaos/eliza/pull/7819

If your connector plugins assumed a single identity, you may need to:
- declare `accounts.owner` / `accounts.agent` in connector entries, and
- ensure OAuth callbacks persist the requested role (see also PR: https://github.com/elizaos/eliza/pull/7820)
