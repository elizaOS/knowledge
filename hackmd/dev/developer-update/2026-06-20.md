# ElizaOS Developer Update — Week of 2026-06-14 to 2026-06-20

This week focused on **production hardening after the Neon → Railway migration**, eliminating shared Postgres pool exhaustion, and continuing the Cloud V2 gateway consolidation (Cloudflare Worker as the model gateway). The headline change is moving **new managed agents** to **in-container PGlite** to prevent “too many clients” incidents.

## 1) Core Framework (runtime / architecture / plugin system)

### New default persistence strategy for **managed Cloud agents**: local PGlite (per-container)
To remove pressure from the shared Railway Postgres pool, **new managed agents now persist state in a local in-container PGlite database** (on persistent container storage), instead of connecting to a shared Postgres pool. This is the root-cause fix for the “too many clients” failure mode seen during scaling tests.

- PR: https://github.com/elizaos/eliza/pull/8754  
- Impact: fewer global DB connections, reduced cross-tenant blast radius, improved provisioning stability.

**Operational implication:** Cloud-side state that previously depended on the shared pool must now assume “agent-local” durability. If your plugin depends on cross-agent querying, you should explicitly route that through Cloud APIs / shared services rather than assuming a central Postgres is available.

### Cloud model routing: Cloudflare Worker is now the primary gateway (BitRouter decommissioned)
This week completes the transition away from the Railway-hosted BitRouter service as the LLM gateway:

- PR: https://github.com/elizaos/eliza/pull/8741  
- Prior foundational work (in week context): Worker-native routing + OpenRouter fallback and billing normalization (see PRs #8738/#8728 referenced in the weekly summary)

**Developer takeaway:** Provider selection and pricing enforcement increasingly happen at the **Worker edge**, not in an internal router service. If you were relying on BitRouter-specific model IDs or behaviors, re-test against Worker routing.

## 2) New Features (with examples)

### LEAN_CHAT_PLUGINS preset for faster dedicated chat agents
Dedicated chat agents were cold-booting with unnecessary core plugins (browser, coding tools, etc.). A new preset reduces loaded plugins to a chat-optimized set, improving boot latency and steady-state performance for pure chat use cases.

- Feature PR: https://github.com/elizaos/eliza/pull/8730  
- Also bundled into the managed-agent hardening in https://github.com/elizaos/eliza/pull/8754

**Example: enabling lean chat plugins**
If you build custom agent images or run self-hosted runtimes, you can explicitly enable the lean preset:

```bash
export LEAN_CHAT_PLUGINS=1
bun run dev
```

(Exact env wiring may differ by host flavor; Cloud-managed “chat” agents will increasingly default to this path.)

### Orchestrator: sub-agent nesting + MCP auto-inheritance
Orchestrator parity work continued this week, notably:
- sub-agent nesting (delegation chains)
- MCP server configuration auto-inheritance into sub-agents

- PR: https://github.com/elizaos/eliza/pull/8740  

**Practical effect:** if you spawn ACP/MCP-capable sub-agents from an orchestrator session, they can inherit MCP config without duplicating per-agent configuration.

## 3) Bug Fixes (critical fixes with context)

### Critical: “too many clients” incidents eliminated for new managed agents
Root cause: each dedicated/managed agent created its own Postgres pool (with non-trivial `min` connections), leading to linear connection growth and pool exhaustion.

Fix: **move new managed agents to local PGlite**, removing reliance on the shared Postgres pool for agent state persistence.

- PR: https://github.com/elizaos/eliza/pull/8754  

### Reliability: BitRouter removal and Worker-first gateway reduces cascading failures
BitRouter removal is also a stability win: fewer moving pieces and fewer failure points between Cloud API and upstream providers.

- PR: https://github.com/elizaos/eliza/pull/8741

## 4) API Changes (developer-facing)

### Cloud agent provisioning / persistence expectations changed (Cloud-managed agents)
While not a single endpoint change, the platform contract for managed agents shifts materially:

- **Before:** agent state persisted in shared Postgres (pool contention possible)
- **Now:** new managed agents persist state in **local PGlite** (per-agent container)

If you maintain Cloud automation that expects to inspect agent DB state centrally, update it to use **Cloud API surfaces** rather than DB-level coupling.

### Related Cloud V2 API surfaces worth noting (landed earlier in June but now “production-real”)
If you’re integrating with Cloud Apps / container hosting, these are the key routes introduced recently and used by current platform workflows:
- `/api/v1/containers` (deploy/read) — PR: https://github.com/elizaos/eliza/pull/8258  
- `/api/v1/containers/:id` (PATCH/DELETE/scale) — PR: https://github.com/elizaos/eliza/pull/8270  
- `PATCH /api/v1/eliza/agents/:id` (rename/edit) — PR: https://github.com/elizaos/eliza/pull/8608  

**Example: rename a cloud agent (PATCH /agents/:id)**
```bash
curl -X PATCH "https://api.elizacloud.ai/api/v1/eliza/agents/$AGENT_ID" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"agentName":"my-renamed-agent"}'
```

## 6) Model Provider Updates (OpenAI, Anthropic, Cerebras, etc.)

### Cerebras key load balancing + gateway hardening
Cloud routing now supports round-robin across multiple Cerebras API keys to spread 429 rate limits (reducing 26–55s tail latencies seen under burst load).

- PR: https://github.com/elizaos/eliza/pull/8735  

### Worker-first routing (post-BitRouter)
With BitRouter decommissioned (https://github.com/elizaos/eliza/pull/8741), provider routing is consolidated:
- direct-to-provider when native keys are available
- OpenRouter as fallback where appropriate (depending on model/provider policy)

**Developer action:** if you hardcode model IDs that previously assumed BitRouter (e.g. suffix behavior), validate against `packages/cloud-shared/src/lib/models/catalog.ts` on the branch you deploy, and prefer “bare” provider-native IDs where supported.

### Vision runtime: TFJS removed in favor of ggml YOLOv8n (build hygiene + runtime correctness)
While slightly earlier than the exact week, it is relevant to current plugin authors: plugin-vision replaced TensorFlow.js with a native ggml YOLOv8n detector to eliminate build warnings and simplify runtime deps.

- PR: https://github.com/elizaos/eliza/pull/8725

## 7) Breaking Changes / V1 → V2 Migration Warnings

### ⚠️ Managed Cloud agent persistence changed (shared Postgres → local PGlite)
This is the biggest “behavioral break” for developers with implicit DB assumptions:
- If you depended on shared Postgres observability for agent state, you must migrate to:
  - Cloud API surfaces (agent export/backup routes where available)
  - explicit external storage plugins/services for cross-agent/stateful workflows

Primary PR: https://github.com/elizaos/eliza/pull/8754

### ⚠️ BitRouter removal: any BitRouter-specific configuration should be removed
If your deployment scripts or env expect `BITROUTER_BASE_URL`/BitRouter service reachability, update them to the Worker gateway approach.

Primary PR: https://github.com/elizaos/eliza/pull/8741

### ⚠️ Prior (still relevant) migration warning: plugin-lifeops removal/split
If you are migrating older agents or templates that still reference `@elizaos/plugin-lifeops`, note that it was removed in favor of `plugin-personal-assistant` + focused domain plugins.

- PR: https://github.com/elizaos/eliza/pull/8277  

**Quick check (repo-wide):**
```bash
rg "@elizaos/plugin-lifeops" .
```

If matches exist in your project, update to the new plugin set and adjust any `/lifeops` view expectations (the monolithic view was removed).
