# ElizaOS Developer Update (2026-05-26 → 2026-06-01)

This update summarizes core runtime + Cloud V2 work shipped through June 1st (merged to `develop`, and promoted to `main` for production sync).

---

## 1) Core Framework

### Agent “view surfaces” become first-class, agent-controllable UI primitives
PR **#8080** introduced an **agent-surface framework** that lets plugins expose interactive UI elements that the agent can enumerate and operate through capabilities, plus runtime **view→action affinity** (planner biasing toward view-scoped actions when a given view is active). This is foundational for “Claude/Codex-style” tool + UI co-control.

Key components (high level):
- `ViewAgentRegistry` + `AgentSurfaceProvider` for registering agent-visible UI elements
- `useAgentElement` hook + overlays to expose element metadata to the agent
- `view-action-affinity` to upweight actions relevant to the active view context in planner prompting

Links:
- PR: https://github.com/elizaos/eliza/pull/8080

### Core plugin boot reliability: optionalizing unstable defaults
A default image boot regression was fixed by making `plugin-google` and `plugin-lifeops` **optional** in the core plugin set (boot gate previously failed when these plugins were unavailable/misconfigured).

Links:
- PR: https://github.com/elizaos/eliza/pull/8105  
- Related boot regression: (referenced in PR) #8081

### Repo/plugin structure: Solana wallet plugin moved into core repo
From Discord maintainer notes, `plugin-solana` has been relocated into the main `eliza` repository as **`plugin-wallet`** under `chains/solana/` (structural change impacting import paths and contribution locations).

Context:
- Discord discussion (May 30): https://discord.com/channels/1253563208833433701/1253563209462448241

## 2) New Features

### `/orchestrator` workbench reaches Claude-Code/Codex-style parity (streaming + diffs + SSE + stop)
PR **#8095** shipped a live coding workbench for `/orchestrator`:
- **Streaming conversation** (coalesced chunks into coherent assistant turns)
- Structured **tool-call cards**
- A real **interleaved line diff viewer** (LCS-based) for edits
- **Stop / Esc interrupt** support to terminate runs quickly
- **SSE transport**: `GET /tasks/:taskId/stream` pushes change events; polling retained as fallback

Links:
- PR: https://github.com/elizaos/eliza/pull/8095

Example: subscribing to orchestrator task stream (SSE)
```ts
// Browser-side pseudo-usage
const source = new EventSource(`/api/orchestrator/tasks/${taskId}/stream`);

source.onmessage = (ev) => {
  const msg = JSON.parse(ev.data);
  if (msg.type === "change") refetchTask();
};

// Recommended hardening (see Greptile note in PR): close on permanent errors.
source.onerror = () => source.close();
```

### Keyless `WEB_FETCH` action: inline live-info for non-Anthropic models
PR **#8097** added a **keyless, SSRF-guarded** web fetch action so providers without Anthropic-specific web-search patches (e.g. Cerebras / gpt-oss) can answer live-info queries inline—without delegating to a coding sub-agent.

Links:
- PR: https://github.com/elizaos/eliza/pull/8097  
- Issue closed: https://github.com/elizaos/eliza/issues/8100

Minimal example: registering/using `WEB_FETCH` in a custom action flow
```ts
import { WEB_FETCH } from "@elizaos/agent/runtime/actions/web-fetch";

// The action is now part of runtime custom actions; ensure it's enabled in your runtime/plugin set.
// Then the planner can select it for "live-info" style prompts, even on non-Anthropic providers.
```

### Cloud: incremental sandbox backups + sleep/wake lifecycle endpoints
PR **#8080** also landed Cloud-side lifecycle improvements:
- **Incremental diff backups** for sandboxes (with DB migration **0136** adding `backup_kind`, `parent_backup_id`, `content_hash`)
- **Sleep/Wake endpoints** for cold-storage of agent compute
- Scheduled backup cron for continuous snapshotting

Links:
- PR: https://github.com/elizaos/eliza/pull/8080

### Cloud API: custom Docker image provisioning (in progress)
PR **#8112** (open at time of this snapshot) threads `dockerImage` through:
- `POST /api/v1/eliza/agents` create payload + persistence
- list/detail DTOs
- sandbox provisioning so API-created agents can run **custom images**
- E2E coverage across API + dashboard

Links:
- PR: https://github.com/elizaos/eliza/pull/8112

Example: creating an agent with a custom image
```bash
curl -X POST "$ELIZA_CLOUD_URL/api/v1/eliza/agents" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "agentName": "my-custom-runtime",
    "dockerImage": "ghcr.io/acme/eliza-agent:sha-abcdef"
  }'
```

### Runtime config: configurable `run-node` entry file
PR **#8126** added `ELIZA_ENTRY_FILE` to override the hardcoded `eliza.mjs` node entrypoint used by `packages/app-core/scripts/run-node.mjs`.

Links:
- PR: https://github.com/elizaos/eliza/pull/8126

Example:
```bash
ELIZA_ENTRY_FILE="custom-entry.mjs" bun run packages/app-core/scripts/run-node.mjs
```

## 3) Bug Fixes (critical)

### ACP JSON-RPC failures become diagnosable: preserve `error.data`
ACP sub-agents returning JSON-RPC `-32602 Invalid params` often embed field-level detail in `error.data` (e.g. Zod formatted errors). The transport previously dropped `error.data`, leaving developers with an undebuggable “Invalid params”.

PR **#8098** preserves and surfaces `error.data`.

Links:
- PR: https://github.com/elizaos/eliza/pull/8098  
- Issue closed: https://github.com/elizaos/eliza/issues/8101

### Planner meta-narration regression fixed (tool output lost)
An intermittent planner behavior produced meta summaries like “Listed files…” instead of returning the actual tool stdout. This was tracked and closed as part of the June 1st planner fixes.

Links:
- Issue closed: https://github.com/elizaos/eliza/issues/7960

### Live-info routing regression: `WEB_FETCH` filtered out of planner action surface
Even after registering keyless `WEB_FETCH`, it could be excluded by context gating and name/simile resolution collisions (e.g. `CONTACT_SEARCH` winning due to `SEARCH` simile). The fix ensured canonical resolution and force-inclusion when the request looks like web search.

Links:
- Issue closed: https://github.com/elizaos/eliza/issues/8108

### Vault boot hardening: self-heal stale PGlite lock
Unclean shutdowns could leave a stale lock in the vault’s PGlite store, bricking early startup during wallet-key hydration. PR **#8092** adds lock self-healing similar to the runtime DB’s resilience strategy.

Links:
- PR: https://github.com/elizaos/eliza/pull/8092

## 7) Breaking Changes / Migration Warnings (V1 → V2)

### Plugin location change: Solana plugin moved into `eliza` as `plugin-wallet`
If you depended on an external `plugin-solana` package/repo, update:
- repository path expectations
- import paths / plugin identifiers
- contribution workflow (PRs now target `elizaos/eliza`)

Context:
- Discord summary (May 30): https://discord.com/channels/1253563208833433701/1253563209462448241

### Default core plugins: `plugin-google` and `plugin-lifeops` are now optional
If your agent assumed these were always present, you must:
- explicitly include them in your plugin set / image build, or
- guard against missing actions/views at runtime

Link:
- PR: https://github.com/elizaos/eliza/pull/8105

### Orchestrator behavior change: auto-spawn + default backend
The orchestrator now auto-spawns a live agent on first message and defaults to the vendored **opencode** backend (replacing an unimplemented `elizaos` default). If you had tooling that assumed tasks remain unspawned until an explicit action, adjust your automation.

Link:
- PR: https://github.com/elizaos/eliza/pull/8095

### Cloud V2 production sync
`develop → main` promotion (**#8116**) synced production with recent Cloud V2 stack changes. If you maintain private deployment workflows pinned to older Cloud SDK contracts, revalidate against `main`.

Link:
- PR: https://github.com/elizaos/eliza/pull/8116
