# ElizaOS Developer Update (2026-05-24 → 2026-05-30)

This week focused on runtime hardening (planner/evaluator correctness, sub-agent UX safety), local inference reliability (DFlash routing + FFI parity), connector account role separation (OWNER vs AGENT), and cloud stability/security (job-queue lifecycle, SIWE/KMS fixes, response headers).

## 1) Core Framework

### Plugin system: “direct | remote” mode unification continues to land
The ongoing architecture shift is to treat **everything as a Plugin** with an explicit execution mode:

- `mode: "direct"`: runs in-process with the agent runtime
- `mode: "remote"`: runs in an isolated worker/process with a host bridge (desktop/mobile/cloud)

Key PRs (context + landing pieces):
- **P0 rename + scaffolding**: https://github.com/elizaos/eliza/pull/7844  
- **P1 runtime + bridge + isolated-process worker**: https://github.com/elizaos/eliza/pull/7846  
- **P2 view-host shims + service/route surfaces**: https://github.com/elizaos/eliza/pull/7847  
- **P3 `installRemotePlugin` API + first sub-agent runner**: https://github.com/elizaos/eliza/pull/7848  

What to watch as a developer:
- Remote-mode plugins now have a **different packaging and deployment contract** (worker bootstrap + view host shims).
- Cloud agent images must build and include `plugin-worker-runtime` (see bug fix #7981 below).

### Planner/evaluator correctness and safety hardening
Several fixes landed to make agent behavior more deterministic and prevent leaking internal metadata into user channels:

- **Restore evaluator `messageToUser` precedence** (tool output no longer overrides explicit evaluator final text):  
  https://github.com/elizaos/eliza/pull/7897

## 2) New Features

### Connector accounts: OWNER + AGENT role separation (schema + UI + plugin support)
Connectors can now declare two independent identities (your personal account vs the bot/agent account), enabling workflows like “agent posts as me” while still keeping an agent-owned identity for automation.

- Registry schema + audit doc: https://github.com/elizaos/eliza/pull/7817  
- Role-aware UI panels: https://github.com/elizaos/eliza/pull/7819  
- Follow-ups for cloud OAuth UX: https://github.com/elizaos/eliza/pull/7866  

**Example: connector entry declaring both roles**

```json
{
  "id": "slack",
  "type": "connector",
  "accounts": {
    "owner": {
      "label": "Your Slack (post/read as you)",
      "requiresOAuth": true
    },
    "agent": {
      "label": "Agent bot identity",
      "requiresOAuth": true
    }
  }
}
```

## 3) Bug Fixes (critical)

### Cloud API: `tool_choice: "required"` crashed `/v1/chat/completions`
**Symptom:** 500 with `Cannot read properties of undefined (reading 'name')` when callers pass `tool_choice: "required"`  
**Fix:** handle the string form safely in the mapper, add regression tests.  
- Fix on develop: https://github.com/elizaos/eliza/pull/7992  
- Cherry-pick to main: https://github.com/elizaos/eliza/pull/7993

**Developer impact:** if you rely on OpenAI-compatible tool calling (planner loop, orchestrators), you can now safely set:

```json
{ "tool_choice": "required" }
```

against `api.elizacloud.ai` without triggering a 500.

---

### Headless Linux crash: keychain access caused a native segfault on boot
**Root cause:** `@napi-rs/keyring` + `libsecret` requires a D-Bus Secret Service; on headless servers that can hard-crash the process.  
**Fix:** skip OS keychain usage on headless Linux and preserve fail-closed behavior for real migrations.  
PR: https://github.com/elizaos/eliza/pull/7230

**Developer impact:** self-hosted agents on servers should now boot reliably without needing GNOME keyring / secret service setup.

## 7) Breaking Changes / Migration Warnings (V1 → V2)

### Plugin manifests now model execution mode explicitly
If you maintain plugins (especially “out-of-process” / capability-style integrations), the move to:

- `Plugin { mode: "direct" | "remote" }`

is the key migration axis. See:
- P0 scaffolding: https://github.com/elizaos/eliza/pull/7844
- P1/P2/P3 runtime + APIs: https://github.com/elizaos/eliza/pull/7846 / https://github.com/elizaos/eliza/pull/7847 / https://github.com/elizaos/eliza/pull/7848

**What can break:**
- Plugins that assumed in-process execution may fail when installed/configured as remote-mode without worker bundle artifacts.
- Cloud deployments must ensure `plugin-worker-runtime` is present (fixed for default images in #7981, but custom images need parity).

### Connector registry schema validation got stricter (accounts role separation)
Connector entries that collide IDs or omit required role wiring can now fail registry loading. Example fix:
- Plugin/app ID collision (polymarket) fixed by renaming plugin ID: https://github.com/elizaos/eliza/pull/7314

### Planner/evaluator outputs are more strictly enforced
If you have downstream prompt forks or custom evaluators that relied on:
- tool `userFacingText` overriding evaluator `messageToUser`, or
- trailing JSON/progress markers being tolerated in user-facing replies,

expect different outputs after #7897 / #7956 / #7955. This is intentional tightening to prevent leakage and conversation-memory poisoning.
