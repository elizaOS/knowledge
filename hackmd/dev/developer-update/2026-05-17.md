## 1) Core Framework

### Secrets management: `@elizaos/vault` integrated end-to-end
Work landed earlier in May but remains the highest-impact framework change affecting plugin configuration flows and runtime credential resolution:

- Added **`@elizaos/vault`** (cross-platform secrets vault) and wired it into **Settings UI + runtime operations**, enabling encryption-at-rest backed by:
  - macOS Keychain
  - Windows Credential Manager
  - Linux Secret Service via `@napi-rs/keyring`
  - Headless fallback via passphrase-derived master key  
  PR: https://github.com/elizaos/eliza/pull/7197

- **Headless Linux hard-crash fix**: skip OS keychain on hosts without a usable Secret Service to prevent native segfault on agent boot.  
  PR: https://github.com/elizaos/eliza/pull/7230

**Developer impact**
- Sensitive plugin params saved through Settings now **mirror into the vault** (encrypted) while maintaining backward compatibility with `config.env.*` (write-through).
- Runtime operations now prefer **`apiKeyRef`** (vault reference) over plaintext keys, with hydrate-time migration.

**Docs**
- Vault package docs: `packages/vault/README.md` (in-repo)  
- PR write-up and test plan: https://github.com/elizaos/eliza/pull/7197

### n8n clarifications: completed “needs clarification” round-trip + UI quick-picks
This week’s developer-facing storyline is the maturation of the automation workflow generation loop:

- Host route now supports clarification round-trips so UI can resolve missing fields and retry generation:
  - PR: https://github.com/elizaos/eliza/pull/7316
- UI quick-pick buttons for `needs_clarification` responses:
  - PR: https://github.com/elizaos/eliza/pull/7341
- Free-form clarifications (no `paramPath`) now supported in resolve flow:
  - PR: https://github.com/elizaos/eliza/pull/7370
- Plugin prompt reliability upgrades (structured `ClarificationRequest`, name→id prompt rules):
  - PR: https://github.com/elizaos/eliza/pull/7373

**Developer flow (high level)**
1) Generate workflow
2) If response indicates `needs_clarification`, collect user input
3) Resolve clarification and retry generation

**Example: clarification-aware generation loop**
```ts
import { client } from "@elizaos/app-core/api"; // illustrative import path

async function generateN8nWorkflow(prompt: string) {
  const result = await client.n8n.generateWorkflow({ prompt });

  if (result.status === "needs_clarification") {
    // Render result.clarifications in UI (buttons + optional free-form text)
    const answers = await collectAnswers(result.clarifications);

    return client.n8n.resolveClarification({
      generationId: result.generationId,
      answers
    });
  }

  return result;
}
```

## 3) Bug Fixes

### Prevented headless Linux agent boot segfault (native keyring/libsecret crash)
Root cause: `@napi-rs/keyring` on Linux can hard-crash if no D-Bus Secret Service is reachable (common on servers/CI).  
Fix: skip OS keychain usage on headless Linux and fall back to passphrase master key pathways.  
PR: https://github.com/elizaos/eliza/pull/7230

**Operational takeaway**
If you run agents in containers/servers, prefer an explicit passphrase master key configuration and ensure your deployment doesn’t assume a desktop secret service is present.

---

### Agent config load hardening: skip vault sentinel values
Fix: `applyConfigEnvToProcessEnv` now skips vault sentinel entries so repeated `loadElizaConfig()` calls don’t pollute `process.env` with non-secret placeholders.  
PR: https://github.com/elizaos/eliza/pull/7368

## 4) API Changes

### Vault + Settings routes (new/updated behavior)
- Plugin save flows (`PUT /api/plugins/:id`) now attempt to **mirror sensitive values to the vault** and return structured mirror failures to the UI (rather than silently swallowing errors).  
PR: https://github.com/elizaos/eliza/pull/7197

- Reveal flows (`POST /api/plugins/:id/reveal`) are now **vault-first**, falling back to legacy env/config paths.  
PR: https://github.com/elizaos/eliza/pull/7197

### Runtime operations: provider credentials now prefer references
- `ProviderSwitchIntent` now uses `apiKeyRef` instead of plaintext `apiKey` across the ops pipeline, with auto-migration on hydrate.  
PR: https://github.com/elizaos/eliza/pull/7197

### Cloud apps API (new endpoints)
- `POST /api/v1/apps/:id/chat`
- Managed domain lifecycle endpoints under `/api/v1/apps/:id/domains/*`  
PR: https://github.com/elizaos/eliza/pull/7376

## 7) Breaking Changes (V1 → V2 migration warnings)

### Monorepo consolidation + ecosystem movement
The repo has undergone large-scale consolidation earlier this month (Cloud + plugins integrated into the monorepo), which affects import paths, workspace expectations, and template assumptions:
- PR: https://github.com/elizaos/eliza/pull/7235

**Migration risks**
- If you depended on standalone plugin repos or non-workspace builds, re-validate:
  - plugin package locations (`plugins/*`)
  - workspace dependency resolution (`workspace:*`)
  - CI scripts referencing old paths

### Credentials storage expectations changed (plaintext → references)
With vault integration, assume that:
- plaintext keys may no longer be returned/round-tripped the same way via Settings
- runtime may persist **references** (`apiKeyRef`) instead of secret material  
PR: https://github.com/elizaos/eliza/pull/7197

**Recommendation**
Treat secrets as write-only from UI and design plugin config schemas to support reference-based resolution (vault and external secret managers).
