# ElizaOS Developer Update (2026-05-09 → 2026-05-15)

This update summarizes core runtime + plugin work landing in `elizaos/eliza` this week, plus notable developer discussions from Discord that impact integrators shipping on ElizaOS v2.

## 1) Core Framework

### Cross-platform secrets vault integrated into runtime + Settings UI
ElizaOS v2’s runtime now supports encrypted-at-rest secret storage via a new workspace package, `@elizaos/vault`, wired end-to-end through the Settings UI and plugin save/reveal flows.

Key technical points:
- AES-256-GCM encryption, master key sourced from OS keychain when available (macOS Keychain / Windows Credential Manager / Linux Secret Service).
- Headless Linux fallback path via passphrase-derived master key (for servers/CI without a secret service).
- Vault “write-through mirror” during plugin configuration saves: sensitive fields are mirrored into the vault; failures surface back to the UI.
- Vault-first reveal: plugin credential reveal uses vault value first, then falls back to legacy `process.env` / `config.env`.

PR: **`feat(vault): @elizaos/vault — cross-platform secrets vault + Settings UI integration`**  
https://github.com/elizaos/eliza/pull/7197

Docs (in-repo):  
- `packages/vault/README.md` (GitHub): https://github.com/elizaos/eliza/tree/develop/packages/vault

### Headless Linux hard-crash fix (keychain/libsecret segfault avoidance)
A critical production issue was fixed where `@napi-rs/keyring` could trigger a native segfault on headless Linux hosts when libsecret/D-Bus isn’t reachable. Runtime now skips OS keychain usage in those environments and uses the passphrase path instead.

PR: **`fix(vault, confidant): skip OS keychain on headless Linux to prevent native segfault on agent boot`**  
https://github.com/elizaos/eliza/pull/7230

### Self-hosted runtime hardening: CORS + bearer auth + build fixes
Self-hosted deployments gained improvements around:
- CORS handling for browser dashboards served over HTTPS/custom domains
- Bearer auth support for remote runtime access
- Cross-platform build fixes for Capacitor/Electrobun targets

PR: **`feat(self-hosted): CORS + bearer auth + cross-platform build fixes`**  
https://github.com/elizaos/eliza/pull/7212

## 3) Bug Fixes (Critical)

### Vault sentinel leakage into process env prevented
A subtle but high-impact bug: repeated calls to `loadElizaConfig()` walk `config.env` and populate `process.env`. With vault integration, sentinel/reference values can exist in `config.env`; these must not be exported into runtime env as if they were plaintext credentials.

Fix: skip vault sentinels during `applyConfigEnvToProcessEnv`.

PR: **`fix(agent): skip vault sentinels in applyConfigEnvToProcessEnv`**  
https://github.com/elizaos/eliza/pull/7368

### Cloud auth + SIWE reliability improvements (post-migration regressions)
Several fixes landed to restore correct auth behavior after the cloud→monorepo migration, including SIWE host/domain correctness and cache gating issues.

Notable PRs:
- **Re-apply auth fixes lost in cloud → eliza migration:** https://github.com/elizaos/eliza/pull/7288  
- **Unblock SIWE by bypassing disabled cache singleton:** https://github.com/elizaos/eliza/pull/7324  
- **Pass worker env to getAppHost/getAppUrl (SIWE domain):** https://github.com/elizaos/eliza/pull/7327  

### Developer-reported Farcaster/Neynar rate-limit & credit burn (discussion)
In Discord, a developer reported unexpectedly high Neynar credit usage (~300 credits/hour vs expected ~4 credits/hour) on `v2/farcaster/notifications` despite overriding the default polling interval. Recommendation from core team: switch from polling to **webhooks** to reduce credit consumption.

Discussion context:
- Discord (coders channel): https://discord.com/channels/1253563208833433701/1300025221834739744

Action item for Farcaster integrators:
- Prefer webhook delivery for notifications whenever possible; treat polling as a fallback only.

## 4) API Changes (Developer-facing)

### Secrets handling: `apiKeyRef` and vault-first reveal
With vault integration, secrets are increasingly passed around as references instead of raw keys.

Implications:
- Provider switch operations now prefer `ProviderSwitchIntent.apiKeyRef` instead of plaintext `apiKey`.
- Legacy ops containing plaintext keys are migrated on hydrate.
- Plugin config “reveal” now resolves from vault first.

PR (vault integration, includes `apiKeyRef` path):  
https://github.com/elizaos/eliza/pull/7197

Example: storing a sensitive value in vault (runtime-side)
```ts
import { sharedVault } from "@elizaos/vault";

await sharedVault().set("OPENAI_API_KEY", process.env.OPENAI_API_KEY!, {
  sensitive: true,
});

// Later:
const apiKey = await sharedVault().get("OPENAI_API_KEY");
```

## 7) Breaking Changes / V1 → V2 Migration Warnings

### Plugin registry submission path is unstable (v2 registry 404s)
Developers attempting to submit third-party plugins to the v2 registry reported 404s on:
- `elizaos-plugins/registry` repository access
- `plugins.elizacloud.ai`

This blocks the intended v2 plugin submission flow (especially for BUSL-1.1 licensed plugins that cannot PR into the monorepo). Core team is investigating and considering a policy change to revert to monorepo PR-based submissions.

Discussion context:
- https://discord.com/channels/1253563208833433701/1253563209462448241 (May 12 summary)

**Migration guidance (until registry is restored):**
- Plan for a temporary “direct install / git URL / local plugin” workflow for private or BUSL-licensed plugins.
- Do not assume registry publishing is currently available in CI pipelines.

### Secrets storage expectations changed (plaintext → references)
If you built tooling around reading raw API keys from `config.env`:
- Expect vault-backed keys and/or reference values (`apiKeyRef`) to appear.
- Update any out-of-band config sync scripts to use the vault APIs or the plugin reveal endpoints rather than scraping `config.env`.

Primary PR: https://github.com/elizaos/eliza/pull/7197

### Monorepo consolidation impacts (repo layout + build expectations)
Large-scale consolidation work earlier this month merged cloud + plugins into `elizaos/eliza` and removed legacy Rust/Python scaffolding in the main repo snapshot. If you have automation relying on prior repo structure, pin revisions or update paths accordingly.

Reference PR: https://github.com/elizaos/eliza/pull/7235
