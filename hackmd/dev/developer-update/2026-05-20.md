## 1) Core Framework

### Connector capability detection (new runtime surface area)
Merged: **“connector capability detection + orchestrator hardening + sub-agent UX redesign”** ([PR #7782](https://github.com/elizaos/eliza/pull/7782))

Core runtime now models per-connector feature support via `MessageConnectorCapability`, and exposes new `IAgentRuntime` methods that *must* be implemented by connectors that claim the capability:

- `editMessageOnTarget`
- `addReactionOnTarget`
- `sendTypingOnTarget` / `stopTypingOnTarget`
- `createThreadOnTarget` / `postToThreadOnTarget`

This provides a typed, capability-gated way for higher-level features (orchestrator, UX layers, agents) to use richer platform semantics without hard-coding connector checks.

**Why it matters:** plugins can now request “threads”, “typing indicators”, “edits”, etc., while gracefully degrading when a connector can’t support them.

---

### Planner-loop correctness improvements (tool call + completion gating)
Also in [PR #7782](https://github.com/elizaos/eliza/pull/7782):

- Fixes a streaming branch regression where `toolCalls` / `finishReason` / `usage` promises could be collapsed into `streamedText`, causing false `required_tool_misses` increments.
- Adds planner support for `completed: false` to explicitly avoid synthesizing a premature FINISH (planner can say “not done yet”).

---

### Secrets architecture hardening on headless Linux
Merged: **skip OS keychain on headless Linux to prevent native segfault on agent boot** ([PR #7230](https://github.com/elizaos/eliza/pull/7230))

`@napi-rs/keyring` + libsecret can segfault when a D-Bus Secret Service isn’t available (typical headless server). The vault/keychain stack now detects this scenario and uses a safe fallback path instead of attempting keyring access.

Related vault feature for context: **@elizaos/vault — cross-platform secrets vault + Settings UI integration** ([PR #7197](https://github.com/elizaos/eliza/pull/7197))

## 2) New Features

### A) Vault-backed secrets + UI integration (recommended default for sensitive config)
Merged earlier but actively impacting v2 adopters: [PR #7197](https://github.com/elizaos/eliza/pull/7197)

Key points developers should internalize:

- Sensitive plugin params saved via the Settings UI are now mirrored into the vault (encrypted at rest) and *revealed vault-first*.
- Runtime operations moved toward storing references (`apiKeyRef`) rather than plaintext keys.

**Practical usage (headless servers):** set a passphrase-derived master key instead of OS keychain.

```bash
# Use a passphrase master key on headless Linux/CI (example env var name from vault docs/PR)
export MILADY_VAULT_PASSPHRASE="correct horse battery staple"
```

**Implication for plugin authors:** treat secret material as *write-only* and rely on reveal routes / runtime resolution rather than expecting plaintext persistence in config files.

---

### B) Self-hosted connectivity: CORS + bearer auth
Merged: **feat(self-hosted): CORS + bearer auth + cross-platform build fixes** ([PR #7212](https://github.com/elizaos/eliza/pull/7212))

The self-hosted runtime can now be safely accessed by browser dashboards and remote clients using:

- Explicit CORS handling
- Bearer auth support across the client/server API surfaces

**Example (client calling a self-hosted agent):**
```bash
curl -H "Authorization: Bearer $ELIZA_BEARER_TOKEN" \
     https://your-agent.example.com/api/health
```

(Exact token env/issuance depends on your deployment path—see the PR for route wiring and tests.)

### C) n8n clarifications: end-to-end “needs_clarification” loop
If you are building workflow-generation UX, this is the current canonical path:

- Host route: **n8n clarification roundtrip** ([PR #7316](https://github.com/elizaos/eliza/pull/7316))
- UI: **clarification quick-pick** ([PR #7341](https://github.com/elizaos/eliza/pull/7341))
- Plugin: **structured `ClarificationRequest` + name→id prompt rules** ([PR #7373](https://github.com/elizaos/eliza/pull/7373))

**What changed technically:**
- Clarifications are now structured (not ad-hoc strings), enabling deterministic UI rendering and resolution flows.
- Prompt rules moved toward ID-based directives to reduce “name drift” failures in multi-connector contexts.

## 3) Bug Fixes (critical + developer-impacting)

### Headless Linux crash on boot (native segfault) — fixed
- Fix: [PR #7230](https://github.com/elizaos/eliza/pull/7230)  
- Root cause: attempting OS keychain access via libsecret without a Secret Service agent could segfault at the process level (not catchable JS error).
- Impact: any server deployment without desktop session / secret service could crash immediately on agent start, especially once vault/keychain features were enabled.

**Action:** if you deploy on Linux servers, update to v2.0.0+ and prefer passphrase fallback where appropriate.

---

### Speculative decoding failure modes documented (closed issues)
Several local-inference/spec-decode issues were closed this week, clarifying current limitations and guiding upcoming kernel/runtime gating:

- M-RoPE position validation breaks cross-drafter batches (Qwen3.5) ([Issue #7631](https://github.com/elizaos/eliza/issues/7631))
- SWA bodies silently disable DFlash spec decode (Qwen3.6 / gemma-4) ([Issue #7635](https://github.com/elizaos/eliza/issues/7635))
- Vulkan kernel-selection performance cliff (Qwen2.5-14B) ([Issue #7638](https://github.com/elizaos/eliza/issues/7638))

**Developer implication:** if you surface spec-decode toggles or recommend Qwen3.5/3.6 tiers, ensure your UX/runtime checks explicitly detect whether spec decode is *actually active* (avoid “it loaded the drafter so it must be working” assumptions).

---

### Grammar rule name sanitization for llama.cpp compatibility
Merged: **emit llama-safe grammar rule names** ([PR #7759](https://github.com/elizaos/eliza/pull/7759))

Underscore-containing generated rule identifiers could break downstream llama grammar parsing. The sanitizer now generates identifiers compatible with llama.cpp rule naming constraints.

## 4) API Changes (developer-facing)

### New `IAgentRuntime` connector methods (capability-gated)
From [PR #7782](https://github.com/elizaos/eliza/pull/7782), connectors/plugins integrating at the runtime layer should expect (and optionally implement) these methods when supporting richer UX:

```ts
// (Names as landed; see PR diff for exact signatures and types)
runtime.sendTypingOnTarget(target);
runtime.stopTypingOnTarget(target);

const thread = await runtime.createThreadOnTarget(target, { name: "Build logs" });
await runtime.postToThreadOnTarget(thread, { text: "Step 1 complete" });

const sent = await runtime.sendMessageToTarget(target, { text: "Working…" });
await runtime.editMessageOnTarget(target, sent.metadata.platformMessageId, { text: "Done." });

await runtime.addReactionOnTarget(target, sent.metadata.platformMessageId, "✅");
```

**Migration note:** connectors that don’t declare these capabilities should not be called through these methods; higher-level features now use capability checks to select thread/edit/reaction vs. fallback send.

---

### Planner schema: `completed?: boolean`
Planner output can now explicitly signal incomplete work without forcing an evaluator FINISH. If you maintain custom planners/prompts, ensure your schema/prompting aligns with this optional flag (see [PR #7782](https://github.com/elizaos/eliza/pull/7782)).

## 5) Social Media Integrations (Twitter/Telegram/Discord/Farcaster)

### Discord: capabilities + threads + typing + edits + reactions
Discord connector updates shipped as part of the capability detection work ([PR #7782](https://github.com/elizaos/eliza/pull/7782)):

- Declares capabilities (e.g., `edit_message`, `react_message`, `create_thread`, `post_to_thread`, `typing_indicator`)
- `handleSendMessage` returns a `Memory` containing `metadata.platformMessageId`, enabling future edits/reactions
- New thread creation and post-to-thread handlers (used heavily by the orchestrator UX changes)

**Practical impact:** Orchestrator/sub-agent narration can now be routed into Discord threads to prevent polluting the main-channel conversation context window.

## 7) Breaking Changes / V1 → V2 Migration Warnings

### v2.0.0 is out; migration support is an active pain point
Discord report: a user missed the migration deadline and could not complete migration without admin help (unresolved) (Discord log: 2026-05-17).

**What to watch for in V1→V2 migration:**

1. **Secrets handling changed materially**
   - If you relied on plaintext `config.env.*` being the single source of truth, v2 introduces vault mirroring and reference-style storage.
   - Headless environments must ensure the vault master-key strategy is valid (keychain vs passphrase), or you risk failed reveals and confusing “auth broken” symptoms.

2. **Connector surface is more strictly capability-driven**
   - Plugins that assumed Discord-like semantics (threads/edits/reactions) must now either:
     - declare/implement the capability contract, or
     - implement robust fallbacks (send-only surfaces).

3. **Monorepo consolidation**
   - Some plugins (e.g., Slack) have moved into the monorepo; if you were pointing to external repos, update import paths and workspace configuration accordingly.

**Docs / release notes**
Community explicitly asked for v2 release notes and feature documentation (Discord, May 19). If you maintain downstream templates or internal docs, plan to annotate the above changes as “v2 baseline assumptions”.
