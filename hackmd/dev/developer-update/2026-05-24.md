## 1) Core Framework

### OWNER+AGENT identity architecture (connectors + runtime)
ElizaOS continues converging on a dual-identity model:
- **OWNER**: the human user’s identity on an external platform (e.g., their Gmail / Slack user)
- **AGENT**: the agent’s own platform identity (e.g., a bot account)

Core pieces landed/stabilized:
- Connector registry schema supports declaring both roles:  
  - PR: https://github.com/elizaos/eliza/pull/7817  
  - UI partitioning: https://github.com/elizaos/eliza/pull/7819  
- OAuth callback handlers now respect the requested role instead of hardcoding `OWNER`:  
  - PR: https://github.com/elizaos/eliza/pull/7820  
  - Follow-up helper extraction: https://github.com/elizaos/eliza/pull/7841  

**Developer impact:** if you maintain a connector, ensure both the manifest and OAuth completion logic preserve the role selected during the “start OAuth” flow (see “Breaking Changes” section for migration notes).

### Cloud agent lifecycle ops moved to a job queue (Workers no longer call cores inline)
Cloud lifecycle endpoints were refactored so Cloudflare Workers enqueue lifecycle jobs rather than performing synchronous control-plane operations.
- PR: https://github.com/elizaos/eliza/pull/7810  
- Supporting frontend behavior (toasts track async job IDs): https://github.com/elizaos/eliza/pull/7813  

**Why it matters:** improves reliability under load (no Worker request-time coupling to SSH / node orchestration) and avoids systemic tail-latency spikes.

### Plugin execution model: “plugin mode unification” groundwork
The codebase is actively converging toward a single plugin abstraction with `mode: "direct" | "remote"` (remote-mode plugins executed via host bridges / isolated processes). This underpins native platforms (Electrobun/Android/iOS) and “satellite” style capabilities.
- P0 rename/scaffolding: https://github.com/elizaos/eliza/pull/7844  
- P1 runtime + bridge + isolated-process worker: https://github.com/elizaos/eliza/pull/7846  
- P2 view-host shims + service/route surfaces: https://github.com/elizaos/eliza/pull/7847  
- P3 installRemotePlugin API + first sub-agent runner: https://github.com/elizaos/eliza/pull/7848  

### Electrobun platform: first-party “satellites” + dynamic view host bridge
A large platform integration merged that folds first-party capability satellites into the Electrobun platform and introduces the first iteration of agent-created dynamic views.
- PR: https://github.com/elizaos/eliza/pull/7779

## 2) New Features

### A) Connector registry now supports role-split accounts (OWNER vs AGENT)
Connector manifests can declare independent role slots. This is foundational for “act-as-user” experiences (OWNER) alongside bot/service identities (AGENT).

Relevant docs:
- OWNER role overview: https://github.com/elizaos/eliza/blob/main/packages/docs/agents/owner-role.md  
- Connector audit notes: https://github.com/elizaos/eliza/blob/main/packages/docs/connectors/owner-vs-agent-audit.md  

**Example: connector entry with OWNER+AGENT slots**
```json
{
  "id": "x",
  "type": "connector",
  "accounts": {
    "owner": { "enabled": true, "label": "Your X account (OWNER)" },
    "agent": { "enabled": true, "label": "Agent X account (AGENT)" }
  }
}
```

**UI behavior update:** the connectors setup panel can now display separate sections for OWNER and AGENT and filter accounts accordingly.
- PR: https://github.com/elizaos/eliza/pull/7819  

### B) OAuth connectors now honor requested role during completion
Previously, some OAuth completion handlers were hardcoded to store the resulting credential as `OWNER` even if an AGENT flow was requested. This breaks dual-identity usage and causes “wrong identity” behavior downstream.
- PR: https://github.com/elizaos/eliza/pull/7820  

**Developer note:** if you added custom OAuth providers, validate:
- role propagation from the OAuth start → callback
- storage schema includes role
- UI filters account lists by role when appropriate

### C) Self-hosted runtime: CORS + bearer auth + cross-platform build fixes
Self-hosted deployments received hardening for browser dashboard connectivity and cross-platform builds (desktop/mobile).
- PR: https://github.com/elizaos/eliza/pull/7212  

This is especially relevant if you run:
- a remote agent runtime behind HTTPS
- a browser-based dashboard on a different origin
- Capacitor or Electrobun builds that must reach a remote runtime

### D) Vault: cross-platform secrets storage and Settings integration (continuing rollout)
While the vault feature landed earlier in May, it’s now a practical assumption for plugin credential UX and runtime behavior:
- Vault package + Settings wiring: https://github.com/elizaos/eliza/pull/7197  
- Headless Linux hard-crash fix (skip OS keychain when unavailable): https://github.com/elizaos/eliza/pull/7230  

If you store credentials in plugin config, prefer vault-backed references over plaintext config where supported (see API Changes).

### E) Production proof-point: waifu.fun built on ElizaOS agents
Community shipped **waifu.fun**, an “agentic launchpad” built using ElizaOS v3 agents, fully open source (announced in Discord).
- Discord context (2026-05-22): `#💬-discussion` summary in aggregation  
- Action item: review their open-source repo as a reference implementation (link shared in Discord announcement; not included in the provided dataset)

## 3) Bug Fixes (critical / developer-relevant)

### Cloud lifecycle UX correctness after async job migration
After lifecycle operations moved to the queue (returning `202 + jobId`), the frontend still acted as though operations completed inline, showing success toasts prematurely and reloading state too early.
- Fix: track async job IDs for suspend/snapshot/restart: https://github.com/elizaos/eliza/pull/7813  

**Impact:** prevents misleading “Restarted” / “Suspended” messages when jobs are merely enqueued.

### Registry ID collision: plugin vs app IDs
A hard failure in the registry loader occurred when an app entry and a plugin entry shared the same `id`.
- Fix: rename plugin polymarket id to avoid collision: https://github.com/elizaos/eliza/pull/7314  

**Impact:** if you ship custom registry entries, ensure IDs are globally unique across all entry types.

### Desktop / platform stabilization (Electrobun + Linux path)
Multiple stability fixes landed around desktop startup and delivery, including work that keeps the Linux local-source path stable.
- Stabilization work (Linux local-source startup): https://github.com/elizaos/eliza/pull/7781  
- Additional Electrobun stabilization: https://github.com/elizaos/eliza/pull/7773  

(These are important if you are embedding ElizaOS in a desktop-native distribution or building custom native shells.)

## 4) API Changes

### A) Cloud lifecycle endpoints now behave asynchronously (job queue semantics)
With the lifecycle refactor:
- `restart/suspend/resume/logs/snapshot` are enqueued rather than executed inline in the Worker request.
- Callers should expect **`202 Accepted`** + **`jobId`** (and poll / stream job status depending on the consuming client).

- PR: https://github.com/elizaos/eliza/pull/7810  
- Frontend adaptation: https://github.com/elizaos/eliza/pull/7813  

**Suggested client pattern (pseudo-code)**
```ts
const res = await fetch(`/api/v1/agents/${agentId}/restart`, { method: "POST" });
if (res.status === 202) {
  const { jobId } = await res.json();
  // poll job status or subscribe to job updates (depending on your integration)
}
```

### B) Connector account schema additions
Connector registry entries can declare:
- `accounts.owner`
- `accounts.agent`

- PR: https://github.com/elizaos/eliza/pull/7817  

If you generate or validate connector manifests, update your schema accordingly.

### C) Secrets handling: vault-backed references (“apiKeyRef” direction)
The runtime and Settings UI increasingly expect secrets to be stored via vault references instead of plaintext `config.env.*`. Even where the legacy write still exists, the “reveal” path becomes vault-first.
- PR: https://github.com/elizaos/eliza/pull/7197  

**Developer impact:** if your plugin expects direct environment values, ensure you’re using the platform’s plugin config resolution APIs rather than reading raw files.

## 7) Breaking Changes / Migration Warnings (V1 → V2)

### V2 identity model requires role-aware connector implementations
**Breaking-in-practice:** connectors and downstream integrations that assume a single “user identity” must be updated to understand:
- multiple accounts per connector
- explicit `role` semantics
- UI flows that configure OWNER and AGENT separately

Migration references:
- Schema: https://github.com/elizaos/eliza/pull/7817  
- UI: https://github.com/elizaos/eliza/pull/7819  
- OAuth role correctness: https://github.com/elizaos/eliza/pull/7820  

**Action for connector maintainers:** audit your connector’s account storage and OAuth callback paths for hardcoded roles.

### Cloud lifecycle calls are no longer synchronous
If you had tooling that assumed `restart/suspend/snapshot/logs` complete inside a single request/response, update it for:
- `202 Accepted`
- async job tracking

- PR: https://github.com/elizaos/eliza/pull/7810  

### Secrets storage: stop depending on plaintext `config.env`
With vault integration maturing, code that reads plaintext secrets from config files is increasingly brittle—particularly across desktop/mobile/headless environments.
- Vault integration: https://github.com/elizaos/eliza/pull/7197  
- Headless Linux keychain crash fix: https://github.com/elizaos/eliza/pull/7230  

**Action:** use ElizaOS-provided config/secret resolution surfaces rather than direct file/env assumptions.
