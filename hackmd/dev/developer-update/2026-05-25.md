## 1) Core Framework (runtime, architecture, plugin system)

### Plugin “mode” unification: `direct | remote` (foundation for agent-generated plugins)
ElizaOS is converging on a single plugin abstraction where **everything is a Plugin**, and execution is selected by `mode`:

- `mode: "direct"` — runs in-process (current “normal” plugin behavior).
- `mode: "remote"` — runs out-of-process via a worker runtime and host shims/bridges.

This landed as a multi-PR stack:
- P0: rename + scaffolding: **unify Plugin with mode** ([#7844](https://github.com/elizaos/eliza/pull/7844))
- P1: remote-mode runtime + bridge + isolated-process worker ([#7846](https://github.com/elizaos/eliza/pull/7846))
- P2: unified view-host shims + service + route surfaces ([#7847](https://github.com/elizaos/eliza/pull/7847))
- P3: `installRemotePlugin` API + first sub-agent runner reference impl ([#7848](https://github.com/elizaos/eliza/pull/7848))

**Why it matters:** this is the enabling layer for (a) trusted capability separation, and (b) “installable” plugins created or selected at runtime, including sub-agent tooling packaged as remote plugins.

> Developer docs: see repo docs added around the capability router + remote plugins, e.g. `packages/agent/docs/capability-router-remote-plugins.md` (introduced as part of release work in [#7828](https://github.com/elizaos/eliza/pull/7828)).

### Planner loop safety + stability improvements (tool failure bounding)
A critical class of “runaway retries” in the planner loop was fixed by making repeated tool-arg validation failures collapse to a **canonical failure signature**, so `maxRepeatedFailures` actually triggers even if the model varies argument shapes between attempts:
- **Bound runaway tool-arg validation loops** ([#7924](https://github.com/elizaos/eliza/pull/7924))

Related prompt/runtime behavior adjustments this week:
- Preserve speaker context; prevent “Stage 1 phantom tool/action claims” ([#7923](https://github.com/elizaos/eliza/pull/7923))
- Drop regex-list response-handler evaluators in favor of prompt-rule contracts ([#7931](https://github.com/elizaos/eliza/pull/7931))

**Known open issue (planner quality):**
- Planner sometimes selects `SHELL` over `SEARCH_MESSAGES`, runs unbounded greps, and leaks sub-agent control metadata ([#7935](https://github.com/elizaos/eliza/issues/7935))

## 2) New Features (with technical detail + examples)

### A) Remote plugin installation API (`installRemotePlugin`)
PR **[#7848](https://github.com/elizaos/eliza/pull/7848)** introduces the API surface that allows the runtime/host to install remote-mode plugins described by a manifest.

While the implementation details vary by host (Electrobun / mobile / server), the architectural contract is:

- A remote manifest declares:
  - worker entry (`worker.mjs` or equivalent)
  - optional view assets
  - permissions needed on the host side
- The host installs + registers it idempotently.
- The runtime can then route actions/providers/events/model calls through the remote bridge.

**Conceptual example (host-side)**
```ts
// Pseudocode illustrating the intent of installRemotePlugin usage.
// Exact import paths/types depend on your host environment + runtime wiring.

const manifestUrl = "https://example.com/my-plugin/carrot.json";

const result = await runtime.plugins.installRemotePlugin({
  url: manifestUrl,
  // optional: pin hashes / enforce trust policy depending on your deployment
});

console.log("installed", result.pluginId);
```

### B) Cloud agent identity endpoints (ERC-8004 IdentityRegistry)
Cloud API gained agent identity management endpoints + a TS client for ERC-8004 IdentityRegistry interactions:
- **ERC-8004 register-identity endpoint + TS client** ([#7937](https://github.com/elizaos/eliza/pull/7937))

Endpoints added under:
- `GET /api/v1/eliza/agents/:agentId/api/identity` (current identity)
- `POST /api/v1/eliza/agents/:agentId/api/identity/register` (register on-chain)
- `POST /api/v1/eliza/agents/:agentId/api/identity/uri` (update URI)
- `GET /api/v1/eliza/agents/:agentId/api/identity/onchain` (on-chain lookup)

**Example: register identity (conceptual)**
```bash

  "https://api.elizacloud.ai/api/v1/eliza/agents/$AGENT_ID/api/identity/register" \
  -H "Authorization: Bearer $STEWARD_TOKEN" \

  -d '{
    "chainId": 56,
    "registryAddress": "0xIdentityRegistry...",
    "uri": "https://example.com/agents/sol.json"
  }'
```

## 3) Bug Fixes (critical fixes + context)

### A) Tool retry storms: `maxRepeatedFailures` guard could be bypassed
Fixed in **[#7924](https://github.com/elizaos/eliza/pull/7924)**.

**Root cause:** repeat detection key was too dependent on the exact invalid args; models could “jitter” the JSON shape and avoid triggering the repeated-failure breaker.

**Impact:** prevented infinite/expensive planner loops (especially visible with sub-agent spawning and schema-invalid tool calls).

### B) Wallet proxy correctness in Cloud API (agentId mapping)
Two fixes for Steward wallet proxying:
- Resolve `steward_agent_id` from `agent_server_wallets` ([#7933](https://github.com/elizaos/eliza/pull/7933))
- Fallback to single org wallet for unmapped wallet proxy ([#7934](https://github.com/elizaos/eliza/pull/7934))

**Impact:** avoids mis-routing sandbox UUIDs to Steward where Steward expects a named agent identity, and prevents production breakage for legacy/unmapped rows.

### C) Dev bootstrap: avoid `sudo apt-get` deadlocks in non-interactive runs
`ensure-vision-deps.mjs` now skips attempting `sudo apt-get install` when stdin is not a TTY and sudo would prompt:
- **Fix non-TTY sudo prompt hang** ([#7941](https://github.com/elizaos/eliza/pull/7941))

### D) Twitter OAuth scope regression + revert
- Attempted fix: drop DM scopes by default ([#7929](https://github.com/elizaos/eliza/pull/7929))
- Reverted after confirming the real root cause was X callback URL mismatch, not scopes ([#7930](https://github.com/elizaos/eliza/pull/7930))

**Takeaway:** if you see X OAuth failing with “Something went wrong”, verify the callback URL(s) configured in the X developer app match the deployed callback route exactly.

## 4) API Changes (developer-facing)

### A) Cloud agent lifecycle routes now enqueue jobs (async) instead of running inline
**All lifecycle operations moved to a job queue** so Cloudflare Workers don’t synchronously call core services:
- **All agent lifecycle ops via job queue** ([#7810](https://github.com/elizaos/eliza/pull/7810))
- Frontend toast handling updated to track async job ids ([#7813](https://github.com/elizaos/eliza/pull/7813))

**Behavior change:** endpoints may now return **`202 Accepted` + `{ jobId }`** instead of completing the operation before responding.

**Example: suspend agent**
```bash
curl -X POST \
  "https://api.elizacloud.ai/api/v1/eliza/agents/$AGENT_ID/suspend" \
  -H "Authorization: Bearer $STEWARD_TOKEN"
# -> 202 { "jobId": "..." }
```

If you built tooling around synchronous completion, update it to poll job status (or subscribe to whatever job status mechanism you’re using internally).

### B) Connector registry schema: OWNER vs AGENT accounts
Connectors can now declare two separate identities in registry entries via:
- `accounts: { owner?, agent? }`
([#7817](https://github.com/elizaos/eliza/pull/7817))

UI and registry loader changes followed in:
- Role-aware connector account list + dual-section setup panel ([#7819](https://github.com/elizaos/eliza/pull/7819))
- Unhardcode `role: "OWNER"` in OAuth callbacks for Linear/Calendly/Shopify ([#7820](https://github.com/elizaos/eliza/pull/7820))
- Extract helper for reading requested role ([#7841](https://github.com/elizaos/eliza/pull/7841))

**Implication for developers:** connector account providers should respect the requested OAuth role and persist accounts accordingly; don’t assume OWNER is always correct.

## 7) Breaking Changes / Migration Notes (V1 → V2)

### A) Plugin execution model: remote-mode introduces new operational assumptions
If you are migrating older plugins or host integrations:

- **V1 assumption:** plugins run in-process; filesystem/network access is implicitly host-bound.
- **V2 direction:** plugins may run **out-of-process** (`mode: "remote"`) and must declare permissions + use bridges/shims.

**Action required:** if your plugin depends on host capabilities (FS, PTY, Git, local model), ensure you’re using the supported capability surfaces rather than direct Node calls, and that your manifests/permissions reflect that. See the remote-plugin stack PRs ([#7844](https://github.com/elizaos/eliza/pull/7844), [#7846](https://github.com/elizaos/eliza/pull/7846), [#7847](https://github.com/elizaos/eliza/pull/7847), [#7848](https://github.com/elizaos/eliza/pull/7848)).

### B) Cloud lifecycle APIs are now async (202 + jobId)
If you wrote automation around lifecycle endpoints, treat this as a breaking behavioral change:
- operations like restart/suspend/snapshot/logs are no longer guaranteed synchronous ([#7810](https://github.com/elizaos/eliza/pull/7810))

### C) Connector identity split (OWNER vs AGENT)
Connectors now have explicit identity roles with UI + schema backing:
- schema support ([#7817](https://github.com/elizaos/eliza/pull/7817))
- UI changes ([#7819](https://github.com/elizaos/eliza/pull/7819))
- OAuth callback role correctness ([#7820](https://github.com/elizaos/eliza/pull/7820))

If your connector code assumed a single “current account”, update it to select by role.
