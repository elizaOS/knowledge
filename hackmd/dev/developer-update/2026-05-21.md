# ElizaOS Developer Update (2026-05-17 → 2026-05-21)

This update summarizes core engineering changes landing in/around **v2.0.2** and highlights key ecosystem developments discussed on Discord during the week.

---

## 1) Core Framework

### v2.0.2 release + monorepo consolidation
- **Release:** `develop → main` promotion for **v2.0.2** landed in [elizaos/eliza#7828](https://github.com/elizaos/eliza/pull/7828). This rollup includes:
  - agent runtime hardening (remote plugin trust policy, view/runtime state widening, evaluator output redaction)
  - Cloud platform stability fixes (auth/billing/error isolation)
  - desktop/mobile packaging fixes and CI stabilization

### OWNER+AGENT identity architecture (connectors + registry + UI)
A major architectural step this week is standardizing how connectors can represent two distinct identities:
- **OWNER**: the human user’s platform account
- **AGENT**: the agent’s own platform account

Key merges:
- Registry schema support: [#7817](https://github.com/elizaos/eliza/pull/7817)
- Role-aware setup UI components: [#7819](https://github.com/elizaos/eliza/pull/7819)
- Fix OAuth callback role propagation (Linear/Calendly/Shopify): [#7820](https://github.com/elizaos/eliza/pull/7820)
- X (Twitter) plugin: OWNER+AGENT end-to-end via Eliza Cloud OAuth: [#7851](https://github.com/elizaos/eliza/pull/7851)
- Docs: `packages/docs/agents/owner-role.md` shipped as part of the develop→main promotions (see [#7840](https://github.com/elizaos/eliza/pull/7840))

**Why it matters:** connectors now declare their supported account roles explicitly; the app can drive dual OAuth flows without conflating agent identity with the owner’s privileges.

### Cloud: agent lifecycle operations moved to a job queue
Cloud lifecycle endpoints no longer synchronously call core services directly. Instead they enqueue work and return a **job id**, improving reliability and preventing worker timeouts.
- Main change: [#7810](https://github.com/elizaos/eliza/pull/7810)
- Frontend toast/status now tracks async `jobId`: [#7813](https://github.com/elizaos/eliza/pull/7813)
- Follow-up refactors/cleanups: [#7815](https://github.com/elizaos/eliza/pull/7815), [#7816](https://github.com/elizaos/eliza/pull/7816), [#7845](https://github.com/elizaos/eliza/pull/7845)

**Developer impact:** if you were assuming “restart/suspend/snapshot/logs completes on request”, update your tooling to poll job status and/or consume job events.

### Electrobun “Satellites” + dynamic view bridge (platform layer)
A substantial platform integration landed to converge first-party “Satellite” capabilities (runtime/fs/pty/git/local-model/surface) and introduce an event/command substrate to support agent-created dynamic UI views:
- Merged: [#7779](https://github.com/elizaos/eliza/pull/7779)

This introduces typed RPC surfaces like:
- `carrotInvokeWorker` (command bridge)
- `carrotTailWorkerEvents` (bounded event tailing)
- dynamic view registry/session primitives (`dynamicViewRegister`, `dynamicViewOpen`, `dynamicViewPush`, …)

## 3) Bug Fixes (critical/high impact)

### A) Headless Linux crash: OS keychain access could segfault on boot
**Symptom:** agent process hard-crashed (native segfault) on hosts without a running Secret Service / DBus session when keyring access was attempted.

**Fix:** skip OS keychain on headless Linux; ensure a safe fallback path is used for secrets.
- [#7230](https://github.com/elizaos/eliza/pull/7230)

**Context:** this is particularly important now that secrets/vault patterns are more widely used; the fix prevents “it works on desktop but crashes on servers” behavior.

### B) Discord reply context grounding + stale prompt suppression
**Symptom:** Discord replies could be incorrectly grounded due to missing structured reply-reference context; setup prompts could remain noisy after configuration.

**Fix:** inbound Discord envelopes now include a structured `platform_reply_reference` that core renders into a `reply_reference` segment.
- [#7821](https://github.com/elizaos/eliza/pull/7821)

### C) Tailwind v4 z-index regression breaking dialogs/overlays
**Symptom:** modals rendered without overlay / transparent bodies due to Tailwind emitting behavior for arbitrary `z-*` classes.

**Fix:** inline z-index arbitrary classes so Tailwind v4 emits them correctly.
- [#7814](https://github.com/elizaos/eliza/pull/7814)

### D) Cloud dashboard render-loop on billing fetch
**Symptom:** repeated billing fetch caused a render loop due to a `useEffect` dependency on a callback that was re-created.

**Fix:** stable effect deps to stop continuous refetch.
- [#7374](https://github.com/elizaos/eliza/pull/7374)

## 4) API Changes (developer-facing)

### A) Cloud agent lifecycle endpoints are now asynchronous (job queue)
After [#7810](https://github.com/elizaos/eliza/pull/7810), lifecycle calls generally return **202 + jobId** instead of “operation completed”.

Typical response shape (illustrative):
```json
{
  "ok": true,
  "jobId": "job_01J…",
  "status": "queued"
}
```

**Action required:** if you maintain internal tooling or external dashboards:
- treat `2xx` as “queued”, not “done”
- poll job status or wait for completion before refreshing state
- update success toasts/UX to reflect async completion (frontend reference: [#7813](https://github.com/elizaos/eliza/pull/7813))

### B) Connector registry schema now supports OWNER+AGENT accounts
Connector entries can declare role-specific accounts. See:
- [#7817](https://github.com/elizaos/eliza/pull/7817)
- [#7819](https://github.com/elizaos/eliza/pull/7819)
- Docs audit: `packages/docs/connectors/owner-vs-agent-audit.md` (added in [#7817](https://github.com/elizaos/eliza/pull/7817))

If you author connectors, expect to update your registry entry to describe:
- whether the connector supports OWNER, AGENT, or both
- how setup flows should be presented in UI

### C) Self-hosted runtime: CORS + bearer authentication hardening
Self-hosted connectivity and cross-platform build fixes landed (notably for browser dashboards + mobile/desktop builds):
- [#7212](https://github.com/elizaos/eliza/pull/7212)

If you embed ElizaOS runtime behind a reverse proxy or custom domain, review your:
- allowed origins
- bearer token flow (especially for remote agent dashboards)

## 7) Breaking Changes / V1 → V2 Migration Warnings

### A) Connector identity: assume dual-role support is the new normal
**Risk:** V1-era connectors often assumed a single identity and conflated “user authorization” with “agent posting identity”.

With **v2.0.x**, connectors may expose both OWNER and AGENT accounts:
- update your connector registry entries and UI expectations ([#7817](https://github.com/elizaos/eliza/pull/7817), [#7819](https://github.com/elizaos/eliza/pull/7819))
- ensure OAuth callbacks preserve the requested role (fix reference: [#7820](https://github.com/elizaos/eliza/pull/7820))

### B) Cloud lifecycle API semantics changed (sync → async)
If your integrations call lifecycle endpoints (restart/suspend/snapshot/logs), you must handle queued jobs:
- core change: [#7810](https://github.com/elizaos/eliza/pull/7810)
- frontend behavioral fix: [#7813](https://github.com/elizaos/eliza/pull/7813)

### C) Desktop platform layer is moving toward Satellites + dynamic views
If you depended on older “direct capability calls” in Electrobun, note the new typed RPC bridges and buffering semantics introduced in:
- [#7779](https://github.com/elizaos/eliza/pull/7779)

Plan for:
- bounded event tails (not a live subscription yet)
- worker/renderer boundaries that may require explicit permissions for host-managed operations
