## 1) Core Framework

### Cross-platform secrets handling becomes first-class: `@elizaos/vault`
Merged **`@elizaos/vault`** into the monorepo and wired it through **Settings UI** and the **agent runtime** so sensitive connector/provider credentials no longer need to live as plaintext in `config.env`. The integration is implemented as a **write-through mirror** on plugin saves plus a **vault-first reveal** path.

- PR: **feat(vault): @elizaos/vault — cross-platform secrets vault + Settings UI integration**  
  https://github.com/elizaos/eliza/pull/7197
- Follow-up bugfix for headless Linux stability (see Bug Fixes):  
  https://github.com/elizaos/eliza/pull/7230

Key architectural notes:
- Vault encrypts secrets at rest (AES-256-GCM) with master key stored in OS keychain when available.
- Headless environments can fall back to passphrase-derived keys (scrypt) instead of requiring Secret Service.
- Runtime operations now prefer storing **references** (e.g. `apiKeyRef`) rather than raw secret values.

Docs:
- Vault package README (in-repo): `packages/vault/README.md` (see PR for full detail)

### Repository consolidation: Cloud + plugins pulled into `elizaos/eliza`
A large consolidation landed, bringing Cloud codepaths and plugin sources under one repo layout and removing legacy Rust/Python packaging from this tree.

- PR: **chore: add cloud and plugins, remove rust and python**  
  https://github.com/elizaos/eliza/pull/7235

Operational impact:
- CI/build pipelines now assume a unified workspace layout; downstream forks/templates that relied on old submodule boundaries may need updates (see Breaking Changes).

## 2) New Features

### A) Vault-backed secret storage + Settings UI integration
Saving provider keys via Settings now mirrors sensitive fields into the vault and surfaces mirror failures explicitly instead of silently proceeding.

What changed:
- `PUT /api/plugins/:id` now mirrors sensitive params into the vault (write-through).
- `POST /api/plugins/:id/reveal` now resolves **vault-first**, then falls back to legacy env/config storage.

PR: https://github.com/elizaos/eliza/pull/7197

**Example: store and later reveal a secret (conceptual flow)**

```bash
# Save plugin config (sensitive fields mirrored to vault)
curl -X PUT "http://localhost:3000/api/plugins/@elizaos/plugin-openai" \
  -H "content-type: application/json" \
  -d '{
    "config": {
      "OPENAI_API_KEY": "sk-***",
      "OPENAI_MODEL": "gpt-4.1-mini"
    }
  }'

# Reveal will return the vault-backed value (vault-first)
curl -X POST "http://localhost:3000/api/plugins/@elizaos/plugin-openai/reveal" \
  -H "content-type: application/json" \
  -d '{ "param": "OPENAI_API_KEY" }'
```

Developer note: the runtime/provider switching path now prefers `ProviderSwitchIntent.apiKeyRef` end-to-end (reference to vault entry) rather than persisting plaintext keys (PR #7197).

## 3) Bug Fixes (critical)

### A) Headless Linux segfault on boot when OS keychain is unavailable
Root cause: `@napi-rs/keyring` on Linux can hard-crash (native segfault) if libsecret cannot connect to D-Bus Secret Service (common on headless servers/CI). This was process-fatal and not catchable in JS.

Fix: skip OS keychain on headless Linux, ensuring vault/confidant selects a safe fallback path.

- PR: **fix(vault, confidant): skip OS keychain on headless Linux to prevent native segfault on agent boot**  
  https://github.com/elizaos/eliza/pull/7230

Impact: agents can now boot reliably in server/container environments without provisioning Secret Service.

---

### B) Vault sentinel values leaking into `process.env`
`applyConfigEnvToProcessEnv` was copying vault sentinel placeholders into environment variables during repeated `loadElizaConfig()` calls (invoked by many services post-boot). This could produce confusing runtime behavior where plugins/providers read sentinel strings as “real” credentials.

- PR: **fix(agent): skip vault sentinels in applyConfigEnvToProcessEnv**  
  https://github.com/elizaos/eliza/pull/7368

---

## 7) Breaking Changes / Migration Warnings (V1 → V2)

### A) Secrets handling: prefer references (`apiKeyRef`) over plaintext keys
With vault integration, internal flows increasingly store secret references instead of raw values. If you maintain custom provider-switch operations or serialize runtime operations, ensure you:
- Persist **references** (e.g. `apiKeyRef`) and resolve via vault at runtime
- Avoid copying vault sentinels into env (fixed in #7368, but custom forks may still have older logic)

Relevant PRs:
- https://github.com/elizaos/eliza/pull/7197
- https://github.com/elizaos/eliza/pull/7368

### B) Monorepo layout changes can break downstream assumptions
The repo consolidation PR moved/added substantial Cloud + plugins content and removed Rust/Python release paths from this tree.

- https://github.com/elizaos/eliza/pull/7235

If your build/deploy scripts assumed:
- standalone plugin repos as source-of-truth
- previous submodule boundaries
- Rust/Python artifacts released from this repository
…you should rebase those assumptions onto the new monorepo structure.

### C) Slack plugin location change
If you were consuming Slack connector code from the standalone repo, update imports and dependency targets to the monorepo package location/versioning.

- https://github.com/elizaos/eliza/pull/7375
