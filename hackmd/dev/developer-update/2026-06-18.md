# ElizaOS Developer Update (2026-06-12 → 2026-06-18)

This update covers merged PRs, active discussions, and notable developer-impacting changes across ElizaOS (monorepo: [`elizaos/eliza`](https://github.com/elizaos/eliza)) for the week.

## 1) Core Framework

### Monorepo consolidation (org/process impact)
Community confirmation that “all development work has been consolidated into the `eliza` repository” (Discord, 2026-06-15). This matters for downstream automation and tooling:
- Update any CI scripts, submodules, or forks that still reference legacy split repos.
- Treat `elizaos/eliza` as the canonical location for runtime, plugins, UI, and infra code going forward.

**Discussion context:** Discord thread (2026-06-15) summarized here:  
- https://discord.com/channels/1253563208833433701/1253563209462448241

### Agent “view surface” + view→action affinity (agent runtime + UI contract)
PR **“Claude/agentic views surface”** introduced an agent-surface framework that makes plugin views agent-controllable, plus a “view-action affinity” layer that biases the planner toward actions relevant to the currently active view. It also added agent sleep/wake endpoints and incremental backups.  
- PR: [#8080](https://github.com/elizaos/eliza/pull/8080)

Key developer implications:
- Plugin views can now be registered into an agent-visible element registry (enabling “agent can click/inspect UI” patterns in a structured way).
- The planner prompt can be dynamically optimized using the active view context (reducing tool-choice thrash).

### Plugin boot reliability: optional core plugins in default agent image
Agent boot was regressing when non-essential plugins were treated as required. This week’s work moved some plugins to optional so the default image boots cleanly:  
- PR: [#8105](https://github.com/elizaos/eliza/pull/8105) (“move plugin-google and plugin-lifeops to optional so the default image boots”)

If you ship custom images/configs:
- Ensure your `core-plugins` list explicitly includes required plugins rather than assuming historical defaults.
- Validate in CI using the repo’s boot gates (agent-image build workflows).

**Code example: creating/managing containers via Cloud API**
The Cloud API gained explicit container deploy/read and lifecycle verbs:
- Deploy + read route: [#8258](https://github.com/elizaos/eliza/pull/8258)
- Lifecycle verbs (PATCH/DELETE/scale): [#8270](https://github.com/elizaos/eliza/pull/8270)

Example (illustrative) request sequence:
```bash
# Create a container (deploy)
curl -X POST "https://api.elizacloud.ai/api/v1/containers" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "image": "ghcr.io/your-org/your-app:sha-...",
    "env": { "NODE_ENV": "production" }
  }'

# Scale or mutate (lifecycle)
curl -X PATCH "https://api.elizacloud.ai/api/v1/containers/$CONTAINER_ID" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "replicas": 2 }'

# Delete
curl -X DELETE "https://api.elizacloud.ai/api/v1/containers/$CONTAINER_ID" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN"
```

### Keyless `WEB_FETCH` action (model-agnostic live web fetch)
ElizaOS added a first-class keyless `WEB_FETCH` runtime action so non-Anthropic backends can fetch live info inline without Anthropic-specific monkey patches.  
- PR: [#8097](https://github.com/elizaos/eliza/pull/8097)

**Minimal usage example (agent action call)**
```ts
// In a plugin action handler or a scripted agent flow:
await runtime.actions.WEB_FETCH({
  url: "https://example.com",
  method: "GET"
});
```

### Cloud SDK: audio transcription helper
The Cloud SDK now ships a higher-level `transcribeAudio()` wrapper around `/api/v1/voice/stt`, so apps no longer need to hand-roll multipart uploads:
- PR: [#8266](https://github.com/elizaos/eliza/pull/8266)

Example:
```ts
import { ElizaCloudClient } from "@elizaos/cloud-sdk";

const client = new ElizaCloudClient({ token: process.env.ELIZA_CLOUD_TOKEN! });

const result = await client.transcribeAudio({
  // file: Blob | File | ReadableStream depending on your runtime
  file,
  model: "whisper-large-v3"
});

console.log(result.text);
```

## 3) Bug Fixes (critical + technical context)

### Cloud login breakage: Steward OAuth PKCE + signing fixes
A production login outage was traced to Steward OAuth hardening requiring PKCE and signed proxy calls. Multiple fixes landed across the week to restore parity across login surfaces and environments:
- Original incident issue: [#8257](https://github.com/elizaos/eliza/issues/8257) (closed)
- PKCE + signed nonce exchange: [#8259](https://github.com/elizaos/eliza/pull/8259)
- Cross-surface PKCE extraction into shared helper: [#8350](https://github.com/elizaos/eliza/pull/8350)
- Steward proxy request signing (HMAC guard): [#8256](https://github.com/elizaos/eliza/pull/8256)
- Refresh session hydration and auth fixes: [#8280](https://github.com/elizaos/eliza/pull/8280), [#8287](https://github.com/elizaos/eliza/pull/8287)

Developer takeaway:
- If you embed “Sign in with Eliza Cloud” flows, verify your PKCE implementation uses a stable `redirect_uri` and stores/verifies `code_verifier` correctly.
- When calling Steward-sensitive routes via cloud-api, ensure the request is signed using the shared signing helper.

### Container billing: idempotency hardening (prevents double-debit)
Daily billing cron could double-charge on retries/overlaps. Fix added idempotency constraints and reconciliation hardening:
- PR: [#8273](https://github.com/elizaos/eliza/pull/8273)

### PGlite stale lock self-heal (prevents startup bricking)
An unclean shutdown could leave a PGlite lock that bricked boot early (vault path). Fix adds resilience similar to plugin-sql’s PGlite manager:
- PR: [#8092](https://github.com/elizaos/eliza/pull/8092)

### Orchestrator: duplicate replies race (TOCTOU) fixed
A check-then-act race in completion dedup could emit duplicate user-visible replies across sessions. Fix closes the TOCTOU window:
- PR: [#8161](https://github.com/elizaos/eliza/pull/8161)

### Cloud inference: reasoning models returning `content: null`
Cloud inference could bill tokens but return null content when `max_tokens` was too small for reasoning models. Fix guarantees a response-token budget:
- PR: [#8189](https://github.com/elizaos/eliza/pull/8189)

## 7) Breaking Changes (V1 → V2 migration warnings)

### `plugin-lifeops` removal / decomposition (hard break)
`@elizaos/plugin-lifeops` was decomposed into multiple focused plugins and renamed to `plugin-personal-assistant` (no backward-compat shims). This is a **breaking change** for imports, plugin lists, and any UI routes relying on `/lifeops`.
- Major refactor PR: [#8277](https://github.com/elizaos/eliza/pull/8277)
- Boot-crash fix removing stale `registerLifeOpsApp` calls: [#8314](https://github.com/elizaos/eliza/pull/8314)

**Migration actions:**
1. Remove `@elizaos/plugin-lifeops` from your dependencies and plugin manifests.
2. Add the new plugins you actually use (e.g., calendar/todos/health/finances/inbox/etc.).
3. If you referenced `/lifeops`, update routing to the per-domain views (e.g., `/calendar`, `/todos`, etc.).

### Cloud auth expectations changed (PKCE required)
If you maintain custom login surfaces or embedded auth flows, PKCE is now effectively mandatory for Steward OAuth. Ensure your auth client supports:
- `code_challenge` + `code_challenge_method=S256` on authorize
- replay of `code_verifier` on exchange  
Relevant fixes: [#8259](https://github.com/elizaos/eliza/pull/8259), [#8350](https://github.com/elizaos/eliza/pull/8350)

### Token migration: scams + “no active migration links”
Discord reports confirm the AI16Z → ELIZAOS migration window is closed and any circulating “migration links” are scams (community support flagged this repeatedly). A user asked about late migration options; no official alternative process was posted in the thread.
- Discussion (unanswered late-migration question): https://discord.com/channels/1253563208833433701/1253563209462448241

If you ship wallet/token-related plugins:
- Consider adding explicit warnings and allowlisted domains for any on-chain migration UX.
- Avoid auto-posting “migration help” links; point users to official docs once published.
