# ElizaOS Developer Update (2026-06-10 → 2026-06-17)

This update summarizes engineering work and community-reported issues across the ElizaOS monorepo (`elizaos/eliza`) for the past week. Development is now consolidated into the single `eliza` repository (per Discord update, 2026-06-15).

---

## 1) Core Framework

### Agent/UI connectivity hardening (browser limitations)
Several core transport fixes landed recently to make browser clients work reliably with authenticated local and cloud-hosted agents:

- **WebSocket auth via query param** (browsers can’t set `Authorization` during the WS handshake): [#8308](https://github.com/elizaos/eliza/pull/8308)  
- **SSE (EventSource) auth via `?token=`** (EventSource can’t set headers): [#8309](https://github.com/elizaos/eliza/pull/8309)

These changes reduce “works in curl, fails in browser” classes of bugs in UI ↔ agent runtime communication.

### Plugin packaging / image boot reliability
- Default agent image boot was unblocked by making historically-required plugins optional (notably `plugin-google` and the former `plugin-lifeops` dependency chain): [#8105](https://github.com/elizaos/eliza/pull/8105)

### App-core runtime entry customization
- `run-node.mjs` now supports a configurable entry filename via `ELIZA_ENTRY_FILE`: [#8126](https://github.com/elizaos/eliza/pull/8126)  
  This is mainly useful for custom packaging or downstream embeddings where the entrypoint is not `eliza.mjs`.

```bash
# Example: run a non-standard entry file
ELIZA_ENTRY_FILE=custom-entry.mjs bun run packages/app-core/scripts/run-node.mjs -- --help
```

## 2) New Features

### Eliza Cloud Apps (Product 2): per-tenant isolated container hosting
Major progress continued on the “Apps data plane” initiative (multi-tenant, isolated apps with optional isolated DBs, per-app ingress, and billing):

- **Per-tenant isolated container hosting** foundation: [#8293](https://github.com/elizaos/eliza/pull/8293)
- **Apps ingress front door (Caddy) + live route management**: [#8325](https://github.com/elizaos/eliza/pull/8325)
- **Install Caddy in app-node cloud-init (removes last “apps lane” TODO)**: [#8394](https://github.com/elizaos/eliza/pull/8394)
- **Collision-safe host port allocation** (fixes audit item #8321-8): [#8328](https://github.com/elizaos/eliza/pull/8328)
- **App URL correctness** (apps domain vs agent/milady domain): [#8313](https://github.com/elizaos/eliza/pull/8313)

**Developer-facing implication:** apps are now reachable by stable `<id>.apps*.elizacloud.ai` routing, rather than “containers exist but have no ingress”.

### Container management API surface (Cloud)
The Cloud API gained the missing endpoints needed by the monetized-app loop and operational tooling:

- **Deploy + read route**: [#8258](https://github.com/elizaos/eliza/pull/8258)
- **Lifecycle verbs (PATCH/DELETE/scale)**: [#8270](https://github.com/elizaos/eliza/pull/8270)

### Apps: per-tenant database is now opt-in
Apps no longer force an isolated DB by default:

- **Database mode `none|isolated`** (provision only when needed): [#8338](https://github.com/elizaos/eliza/pull/8338)
- **DSN injection under both `DATABASE_URL` and `POSTGRES_URL`** to avoid env-var contract mismatches: [#8322](https://github.com/elizaos/eliza/pull/8322)

```ts
// Expect both env vars in app containers when databaseMode === "isolated"
process.env.DATABASE_URL  // primary injection
process.env.POSTGRES_URL  // compatibility for frameworks expecting POSTGRES_URL
```

### Cloud SDK: first-class STT helper
Third-party apps can now call speech-to-text without hand-rolling multipart requests:

- `ElizaCloudClient.transcribeAudio()` added: [#8266](https://github.com/elizaos/eliza/pull/8266)

```ts
import { ElizaCloudClient } from "@elizaos/cloud-sdk";

const client = new ElizaCloudClient({ apiKey: process.env.ELIZA_CLOUD_API_KEY! });

const res = await client.transcribeAudio({
  // file: Blob | File | Uint8Array depending on runtime
  file: myAudioBlob,
  model: "whisper-1",
});

console.log(res.text);
```

### Agent capability: keyless `WEB_FETCH` (non-Anthropic models)
A long-standing limitation where “keyless web search” only worked via Anthropic-specific wiring is addressed:

- **Keyless `WEB_FETCH` action** for broader model compatibility: [#8097](https://github.com/elizaos/eliza/pull/8097)

This enables “live info” tasks on models routed via Cerebras / OSS providers without requiring Anthropic-only monkey patches.

## 3) Bug Fixes (critical + high-impact)

### Cloud Apps: prevent permanent Postgres leakage on app deletion
A critical resource leak was fixed: deleting an app with an isolated tenant DB previously **did not drop the DB/role nor release the slot**, causing cost and capacity leakage.

- **Fix: tear down isolated tenant DB on app delete**: [#8401](https://github.com/elizaos/eliza/pull/8401)  
- Related groundwork / follow-ups: [#8343](https://github.com/elizaos/eliza/pull/8343)

**Why it mattered:** the tenant DB cluster had a bounded `max_databases`, and leaked slots would silently reduce platform capacity over time.

### Billing: idempotency hardening (no double-debit)
The daily container billing cron was hardened to prevent double charges on retries or overlapping cron runs:

- **Idempotent container billing per period**: [#8273](https://github.com/elizaos/eliza/pull/8273)

### Auth: Steward OAuth PKCE parity (cloud login reliability)
Steward OAuth began requiring PKCE, breaking login flows until Eliza Cloud surfaces were updated:

- Root issue: [#8257](https://github.com/elizaos/eliza/issues/8257) (closed)
- Fix: **PKCE for all login surfaces**: [#8350](https://github.com/elizaos/eliza/pull/8350)
- Supporting fixes: signing/proxy and tenant pinning (examples): [#8259](https://github.com/elizaos/eliza/pull/8259), [#8256](https://github.com/elizaos/eliza/pull/8256), [#8179](https://github.com/elizaos/eliza/pull/8179)

### Orchestrator correctness: completion dedup race
A TOCTOU race could emit duplicate completions across sessions:

- Fix: [#8161](https://github.com/elizaos/eliza/pull/8161)

### Windows + local dev stability
- Windows cloud dev stacks now resolve `bun.exe` correctly: [#8248](https://github.com/elizaos/eliza/pull/8248)
- ACP sub-agents on Windows now inherit `Path` correctly (case-insensitive): [#8285](https://github.com/elizaos/eliza/pull/8285)

## 4) API Changes (developer-visible)

### Cloud API: containers endpoints
If you were previously calling internal broker methods for container operations, prefer the now-stable REST surface:

- `POST /api/v1/containers` (deploy)
- `GET /api/v1/containers` / `GET /api/v1/containers/:id` (read)
- `PATCH|DELETE|POST(scale) /api/v1/containers/:id` (lifecycle)

See: [#8258](https://github.com/elizaos/eliza/pull/8258), [#8270](https://github.com/elizaos/eliza/pull/8270)

### Auth transport changes for browser clients
- WS and SSE auth can be supplied by query string token now (in addition to headers where supported): [#8308](https://github.com/elizaos/eliza/pull/8308), [#8309](https://github.com/elizaos/eliza/pull/8309)

**Action:** if you maintain a custom UI client, ensure your WS/SSE constructors append `?token=` (and avoid relying on headers that browsers can’t set).

## 7) Breaking Changes (V1 → V2 migration warnings)

### `plugin-lifeops` removal / decomposition (hard break)
The LifeOps monolith has been removed and split into focused plugins, with a renamed orchestrator plugin:

- **Breaking refactor:** [#8277](https://github.com/elizaos/eliza/pull/8277)  
  - `@elizaos/plugin-lifeops` **is gone** (no alias/shim)
  - New: `@elizaos/plugin-personal-assistant` + domain plugins (`plugin-inbox`, `plugin-goals`, `plugin-finances`, `plugin-relationships`, `plugin-blocker`, `plugin-remote-desktop`, etc.)
  - `/lifeops` view removed; domain plugins own their own views now

Follow-up fixes for white-screen boot crashes caused by stale imports/exports:
- Drop calls to removed `registerLifeOpsApp` exports: [#8314](https://github.com/elizaos/eliza/pull/8314)
- Post-decomposition cleanup: [#8316](https://github.com/elizaos/eliza/pull/8316)

**Migration action for developers:**
- Remove `@elizaos/plugin-lifeops` from your plugin lists.
- Add `@elizaos/plugin-personal-assistant` and whichever domain plugins you need.
- Update any imports from `packages/shared/src/contracts/lifeops.ts` → `personal-assistant.ts` as applicable (see refactor notes in [#8277](https://github.com/elizaos/eliza/pull/8277)).
