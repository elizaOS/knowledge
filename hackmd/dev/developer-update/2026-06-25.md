## 3) Bug Fixes (critical fixes + technical context)

### A) Cloud inference outage: pricing-catalog fetch failures must not hard-500 requests
A production incident class was addressed where failure to fetch a provider pricing catalog caused inference to return HTTP 500s.

- PR: **[#9539](https://github.com/elizaos/eliza/pull/9539)** `fix(cloud): pricing-catalog fetch failure must not 500 inference`

**Technical context**
- The pricing layer is used for billing attribution / cost computation; when upstream pricing endpoints fail (e.g. transient 404/5xx), the request previously failed the entire inference call.
- The fix prevents pricing lookup failures from taking down the chat completions surface.

**Action for app developers**
- If you rely on precise per-model pricing reconciliation, validate that your downstream accounting tolerates `pricing unavailable` paths (e.g., “bill with last-known / fallback” vs “fail closed”). The platform is trending toward “serve the answer, reconcile later” for availability.

### B) Windows compatibility: shell-spawning fixes across agent tools and desktop runtime
Multiple Windows-specific regressions were fixed to align `.cmd` shims and file URL behavior with Node/Bun process spawning semantics.

- PR: **[#9542](https://github.com/elizaos/eliza/pull/9542)** `fix(app-core): build desktop renderer fallback URL with pathToFileURL (Windows)`
- PR: **[#9537](https://github.com/elizaos/eliza/pull/9537)** `fix(vault): launch the Bitwarden CLI (.cmd shim) via shell on Windows`
- PR: **[#9541](https://github.com/elizaos/eliza/pull/9541)** `fix(coding-tools): cross-platform source search/list (PowerShell on Windows)`

**Root causes (recurring pattern)**
- `spawn` / `execFile` without `shell: true` cannot execute `npm.cmd`, `bw.cmd`, etc. on Windows.
- File URL construction via naive `file://${path}` breaks on Windows path semantics; `pathToFileURL()` is the correct primitive.

**Developer guidance**
- When writing plugins that call external CLIs, prefer a platform-aware spawn helper and explicitly support Windows `.cmd` entrypoints.
- Avoid manual `file://` concatenation; use `pathToFileURL()`.

## 4) API Changes

No explicit breaking public API renames were surfaced in the week’s summary, but there are two *behavioral contract* changes developers should note:

1) **Cloud inference availability behavior**: the platform is moving toward *not failing the request* when pricing metadata is temporarily unavailable (see **[#9539](https://github.com/elizaos/eliza/pull/9539)**).  
   - If you were implicitly relying on “pricing must be present” as an enforcement gate, you should add explicit checks in your application logic.

2) **Cross-platform CLI execution expectations**: Windows now requires shell-based spawning for common workflows (vault tools, coding-tools search/list).  
   - If you maintain custom actions that wrap external CLIs, verify they work on Windows runners.

## 7) Breaking Changes / V1 → V2 Migration Warnings

No single “hard breaking” V1→V2 API removal is called out in the provided activity for this exact week window, but there are two migration pressure points to flag:

1) **Benchmark/CI policy tightening (effective break for workflows)**  
   - With **[#9549](https://github.com/elizaos/eliza/pull/9549)**, benchmark additions that aren’t wired will be rejected by CI gates. If you maintain downstream forks with custom benchmark directories, expect merge friction unless you adopt the same wiring conventions.

2) **Cloud Apps (Product 2) security hardening is in flight and may change deployment assumptions**  
   - Two high-priority Cloud Apps PRs were highlighted as blockers (still in progress in the Jun 24 notes):
     - **[#9230](https://github.com/elizaos/eliza/pull/9230)** isolate untrusted app builds from the runtime fleet
     - **[#9229](https://github.com/elizaos/eliza/pull/9229)** bind app host ports to loopback + route ingress via node-local Caddy
   - If you are already integrating with `/apps` deploy flows on staging/prod, expect tightened network and build isolation semantics (which is the correct direction for multi-tenant safety).
