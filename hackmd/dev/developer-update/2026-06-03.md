### Agent runtime: view-aware planning (‚Äúagentic view surfaces‚Äù)
Merged: **‚ÄúClaude/agentic views surface‚Äù** ([PR #8080](https://github.com/elizaos/eliza/pull/8080)).

Key architectural additions:
- **Agent-surface framework** in `@elizaos/ui`: a registry + overlay + hooks that expose UI elements/capabilities to the agent.
- **View-action affinity** in `packages/agent`: when a user is looking at a specific view, the planner prompt is biased toward actions relevant to that view (reduces ‚Äúwrong tool‚Äù selections and improves deterministic UX).

Also included:
- **Sleep/Wake lifecycle** for agents (Cloud): deep cold-storage of compute with restore paths.
- **Incremental diff backups** for sandboxes, plus scheduled backup cron (Cloud).

This is a foundational change for agents that must operate *within* complex UI surfaces (orchestrator workbench, dashboards, admin panels) without relying purely on chat-tool selection.

## 2) New Features

### Keyless `WEB_FETCH` for non-Anthropic models
Merged: [PR #8097](https://github.com/elizaos/eliza/pull/8097)

Previously, ‚Äúkeyless web search‚Äù behavior was effectively an Anthropic-only patch. Now **`WEB_FETCH` is a first-class runtime action** so models like gpt-oss class backends can fetch live info inline without provider-specific monkey patches.

**Developer-facing impact:** if you ship agents on non-Anthropic providers (or mixed-provider setups), you can rely on a consistent action path.

Example (conceptual action call from an agent test / custom runtime action harness):
```ts
// pseudo-code illustrating the now-unified action availability
const result = await runtime.actions.WEB_FETCH.run({
  url: "https://api.coindesk.com/v1/bpi/currentprice.json",
  format: "json",
});

return result.userFacingText ?? result.text;
```

### Cloud: per-agent Web UI enabled by default
Merged: [PR #8167](https://github.com/elizaos/eliza/pull/8167)

Cloud-managed agent configs now default to **`ELIZA_UI_ENABLE=true`**, making the per-agent sandbox UI the standard interaction surface (with prior security concerns addressed via Cloud-side controls).

If you previously depended on a headless-only provisioned agent, explicitly disable:
```bash
export ELIZA_UI_ENABLE=false
```

## 3) Bug Fixes (critical)

### Orchestrator: duplicate replies (TOCTOU race)
Merged: [PR #8161](https://github.com/elizaos/eliza/pull/8161)

Root cause: completion dedup used a check-then-act sequence around an awaited delivery loop, allowing two sessions to both ‚Äúwin‚Äù and post. Fix closes the race by making the ‚Äúfirst poster‚Äù claim atomic across the awaited boundary.

Why it matters: duplicate replies are catastrophic for multi-agent orchestration UX (and can trigger downstream automations twice).

### Core routing: coding tasks misrouted to web search if they contain market terms
Merged: [PR #8160](https://github.com/elizaos/eliza/pull/8160)

Routing logic previously evaluated `looksLikeWebSearchRequest` before `looksLikeCodingWorkRequest`. A request like ‚Äúbuild an app that tracks market X‚Äù could get routed into web. Fix prioritizes coding detection first and adds regression tests.

### Planner-loop: leaked JSON envelopes into user-facing replies
Merged: [PR #8159](https://github.com/elizaos/eliza/pull/8159)

Weak planners sometimes echoed internal `TASKS` spawn-arg JSON into `messageToUser`. This PR suppresses the leaked envelope and adds a regression test.

Related incident class (now closed): ‚ÄúUSE_SKILL output wrapping leaks raw bash invocation JSON‚Äù  
Issue [#7970](https://github.com/elizaos/eliza/issues/7970) (closed 2026-06-02)

### Cloud inference: guarantee response-token budget for reasoning models
Merged: [PR #8189](https://github.com/elizaos/eliza/pull/8189)

Problem: some reasoning models returned `content: null` while still billing tokens when `max_tokens` was too small to cover both hidden reasoning and the final answer. Fix enforces a **minimum response-token budget** for reasoning models and updates pricing/reasoning detection tests.

### Desktop/dev install: fresh-clone reliability
Merged:
- Build private workspace packages in postinstall: [PR #8146](https://github.com/elizaos/eliza/pull/8146) (fixes [#8143](https://github.com/elizaos/eliza/issues/8143))
- Bootstrap fixes unblocking fresh-clone `dev:desktop`: [PR #8168](https://github.com/elizaos/eliza/pull/8168)

### Vault: self-heal stale PGlite lock
Merged: [PR #8092](https://github.com/elizaos/eliza/pull/8092)

Prevents unclean shutdowns from bricking startup due to stale locks in the vault‚Äôs PGlite store.

## 4) API Changes (developer-visible)

### New Cloud endpoints: agent sleep/wake + backups surfaces
Merged in [PR #8080](https://github.com/elizaos/eliza/pull/8080):
- `POST /v1/eliza/agents/:agentId/sleep` (async; returns 202)
- `POST /v1/eliza/agents/:agentId/wake` (via Cloud service layer)
- Backups routes expanded (incremental backup metadata columns via migration 0136)

If you operate your own Cloud API client, expect new lifecycle concepts:
- ‚Äúsleep‚Äù implies **backup then compute release**
- ‚Äúwake‚Äù implies **re-provision + restore**

### Cloud API: Steward staging pin via header
Merged: [PR #8179](https://github.com/elizaos/eliza/pull/8179)

Staging auth flows now rely on `X-Steward-Tenant` pinning to avoid bundler/env substitution pitfalls.

### UI framework: `fullBleed?: boolean` page registration (opt-in)
In-flight (not merged as of this update): **Odysseus-style orchestrator UI** ([PR #8180](https://github.com/elizaos/eliza/pull/8180)) introduces a generalized `fullBleed` mounting mode in `@elizaos/ui` that removes shell chrome for pages that own the full window.

No breaking change expected (flag is opt-in), but if you maintain custom shells or view routers, watch for:
- `AppShellPageRegistration.fullBleed`
- host shell behavior changes for full-bleed routes
