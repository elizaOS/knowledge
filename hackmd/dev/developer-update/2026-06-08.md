# ElizaOS Developer Update (2026-06-01 → 2026-06-07)

This week focused on Cloud V2 hardening (auth, routing, billing correctness), major plugin modularization (LifeOps breakup), and a big step forward in agent-controllable UI surfaces + orchestrator workbench UX.

> Key repos: [`elizaos/eliza`](https://github.com/elizaos/eliza)

## 1) Core Framework

### Agent-controllable “view surfaces” + view-action affinity
PR: **“Claude/agentic views surface”** ([#8080](https://github.com/elizaos/eliza/pull/8080))

* Introduced an **agent-surface framework** in `@elizaos/ui` that makes plugin views discoverable/controllable by the agent runtime via a registry + overlay system.
  * Adds `ViewAgentRegistry`, `AgentSurfaceProvider`, `useAgentElement`, and the overlay plumbing for capability dispatch.
* Added **view-action affinity** in the agent runtime so planners **prefer view-relevant actions** while the user is looking at a given view (reduces “wrong tool” selections and improves UX).
* Cloud operations primitives landed alongside it:
  * **Sleep/Wake** endpoints (deep compute cold-storage).
  * **Incremental diff backups** for sandboxes and a scheduled backups cron.

Technical implications:
* If you ship custom views, you can now register agent-addressable elements and let the planner “see” UI structure.
* Planner prompting now includes “active view context” to bias action selection—useful if you’re building domain-heavy dashboards.

## 2) New Features

### (Cloud) Container hosting routes for “Eliza Cloud Apps”
PRs:
* **Eliza Cloud Apps (per-tenant isolated container hosting)** ([#8293](https://github.com/elizaos/eliza/pull/8293))
* **/api/v1/containers deploy + read** ([#8258](https://github.com/elizaos/eliza/pull/8258))
* **Container lifecycle verbs (PATCH/DELETE/scale)** ([#8270](https://github.com/elizaos/eliza/pull/8270))

New public API surface (Cloud Worker):
* `POST /api/v1/containers` – deploy container
* `GET /api/v1/containers` – list
* `GET /api/v1/containers/:id` – read
* `PATCH /api/v1/containers/:id` – mutate (e.g., scale/config)
* `DELETE /api/v1/containers/:id` – teardown

Example: deploying a container app (server-side / CI)
```bash
curl -X POST "https://www.elizacloud.ai/api/v1/containers" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "image": "ghcr.io/acme/myapp:sha-abcdef",
    "name": "myapp-prod",
    "env": { "NODE_ENV": "production" }
  }'
```

Notes:
* This is designed for per-tenant isolation (networking + DB isolation) and integrates with metered billing already shipped previously.

### (Agent runtime) Keyless `WEB_FETCH` for non-Anthropic models
PR: **Keyless WEB_FETCH action** ([#8097](https://github.com/elizaos/eliza/pull/8097))

Historically the “no-key web lookup” path was Anthropic-specific. This week:
* `WEB_FETCH` is available so **non-Anthropic** deployments (e.g. Cerebras `gpt-oss-*`) can fetch live info inline.

Example: calling `WEB_FETCH` from a custom action (conceptual)
```ts
import { WEB_FETCH } from "@elizaos/agent/runtime/actions/web-fetch";

const result = await runtime.runAction(WEB_FETCH, {
  url: "https://api.example.com/price",
  method: "GET",
});
```

## 3) Bug Fixes (Critical)

### Orchestrator duplicate replies (TOCTOU race)
PR: **Close completion-dedup TOCTOU race** ([#8161](https://github.com/elizaos/eliza/pull/8161))

Issue class:
* Two sessions could “win” the completion posting slot due to a check-then-act race in `SubAgentRouter.routeSessionEvent`.

Fix:
* Makes dedup atomic across awaited delivery loops, preventing duplicate user-facing completions.

---

### Cloud login breakage after Steward PKCE hardening
Issue: **Steward login broken in prod** ([#8257](https://github.com/elizaos/eliza/issues/8257))  
Fix PR: **Send PKCE + sign nonce-exchange** ([#8259](https://github.com/elizaos/eliza/pull/8259))  
Related hardening: **Sign Steward proxy requests (HMAC guard)** ([#8256](https://github.com/elizaos/eliza/pull/8256))

What happened:
* Steward `/authorize` began requiring PKCE + additional request signing, but the Cloud SPA + Worker proxy weren’t aligned, causing widespread 400/403 failures.

What changed:
* Cloud frontend now generates/verifies PKCE in the OAuth flow.
* Cloud API signs upstream Steward requests to satisfy HMAC middleware.

If you maintain a custom Cloud fork:
* Ensure your login UI and Worker proxy are on/after #8259 and #8256, otherwise “Sign in with Eliza Cloud” will fail against hardened Steward deployments.

---

### Cloud reasoning models returning `null` content while billing tokens
PR: **Guarantee response-token budget for reasoning models** ([#8189](https://github.com/elizaos/eliza/pull/8189))

Root cause:
* `max_tokens` could be too small to cover hidden reasoning + final answer, yielding `content: null` while still consuming billable tokens.

Fix:
* Cloud completion route enforces a minimum response budget for reasoning models to guarantee a final answer payload.

**SSE / WebSocket auth compatibility**
PRs:
* **SSE GET accepts `?token=`** ([#8309](https://github.com/elizaos/eliza/pull/8309))
* **WebSocket handshake accepts token via query param** ([#8308](https://github.com/elizaos/eliza/pull/8308))

Why:
* Browsers can’t attach `Authorization` headers to `EventSource` or WS handshake consistently; query-param tokens are now supported.

Example: WS with token in URL (browser)
```ts
const ws = new WebSocket(
  `wss://${location.host}/ws?clientId=${clientId}&token=${encodeURIComponent(apiToken)}`
);
```

## 7) Breaking Changes / V1 → V2 Migration Warnings

### **Breaking: `@elizaos/plugin-lifeops` removed (no compatibility shim)**
PR: **Split LifeOps into focused plugins + plugin-personal-assistant** ([#8277](https://github.com/elizaos/eliza/pull/8277))

What changed:
* `@elizaos/plugin-lifeops` is **gone**, replaced by:
  * `@elizaos/plugin-personal-assistant` (cross-domain orchestration)
  * New domain plugins: `plugin-inbox`, `plugin-goals`, `plugin-finances`, `plugin-relationships`, `plugin-blocker`, `plugin-remote-desktop`
  * Extended domain plugins: `plugin-calendar`, `plugin-todos`, `plugin-health`, `plugin-documents`, `plugin-phone`, etc.
* The `/lifeops` dashboard view was removed in favor of per-domain views.

Required migration actions:
1. **Remove** `@elizaos/plugin-lifeops` from your plugin list / imports.
2. **Add** `@elizaos/plugin-personal-assistant` and whichever domain plugins you actually use.
3. Update any contracts/imports renamed from `lifeops` to `personal-assistant`.

A follow-up boot fix indicates common failure mode:
* App boot could crash with `registerLifeOpsApp is not a function` until callers were updated ([#8314](https://github.com/elizaos/eliza/pull/8314)).
