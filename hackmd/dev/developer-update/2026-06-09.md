# ElizaOS Developer Update (2026-06-02 → 2026-06-08)

This update summarizes developer-facing changes landing across core runtime, cloud, UI, and plugins for the week ending **2026-06-08** (published **2026-06-09**).

---

## 1) Core Framework

### Agent-controllable “View Agent Surface” + view-action affinity
ElizaOS now exposes a first-class *agent surface* abstraction for plugin views, enabling agents to enumerate and interact with structured UI elements, and biases planners toward actions relevant to the currently active view.

- **PR:** “Claude/agentic views surface” [#8080](https://github.com/elizaos/eliza/pull/8080)  
- **What changed (high level):**
  - A `ViewAgentRegistry` / `AgentSurfaceProvider` system was introduced (in `@elizaos/ui`) to register view elements.
  - The agent runtime gained a view-context affinity layer so planners upweight view-scoped runtime actions while the user is on that view.
  - New cloud endpoints to **sleep/wake agents** and **incremental sandbox backups** landed alongside this surface.

**Why it matters:** if you build complex plugin UIs (or “workbenches”), you can now make them agent-operable without writing bespoke tool-calling glue for each component tree.

### Core plugin boot reliability: optionalizing heavyweight plugins in default images
Default agent images were failing to boot when certain plugins were missing or misconfigured. The default core-plugin set now treats some integrations as optional to preserve bootability.

- **PR:** `fix(agent): move plugin-google and plugin-lifeops to optional so the default image boots` [#8105](https://github.com/elizaos/eliza/pull/8105)

### Message routing correctness improvements for coding vs. web/search intent
Several routing heuristics were tightened to avoid misclassifying coding prompts as web searches (especially when prompts mention market terms).

- **PR:** `fix(core): route a coding request mentioning a market term to coding, not web` [#8160](https://github.com/elizaos/eliza/pull/8160)

### Cloud V2 stack promotion to production
A large develop → main promotion synced Cloud V2 infrastructure and runtime changes into production.

- **PR:** `release: promote develop → main (Cloud V2 sync)` [#8116](https://github.com/elizaos/eliza/pull/8116)

## 2) New Features

### Eliza Cloud Apps (Product 2): per-tenant isolated container hosting
A major capability landed: **deploy arbitrary containerized apps** with **per-tenant database isolation**, per-app URLs, and metered hosting billing.

- **Core PR:** `feat(cloud): Eliza Cloud Apps (Product 2) — per-tenant-isolated container hosting` [#8293](https://github.com/elizaos/eliza/pull/8293)
- **Terraform / infra:** apps data-plane module [#8296](https://github.com/elizaos/eliza/pull/8296), CI plan/apply workflow [#8297](https://github.com/elizaos/eliza/pull/8297)

#### Apps ingress routing (Caddy admin API “routing client”)
Apps are now reachable via `<shortid>.apps.elizacloud.ai` using a per-app Caddy route provisioned dynamically.

- **PR:** `feat(apps): ingress — per-app Caddy routing client + deploy-flow wiring` [#8325](https://github.com/elizaos/eliza/pull/8325)

**Operational note:** an audit issue tracked that apps were “unreachable until ingress exists”; this feature closes that P0 gap.  
- **Issue:** “Eliza Cloud Apps: scaling + ingress gaps before go-live (audit, evidenced)” [#8321](https://github.com/elizaos/eliza/issues/8321)

#### Tenant DB isolation contract: DSN injected under both env vars
Some images expect `POSTGRES_URL`, others `DATABASE_URL`. Deployment now injects **both**, preventing silent fallbacks that could break tenant isolation guarantees.

- **PR:** `fix(apps): inject tenant DSN under both DATABASE_URL and POSTGRES_URL` [#8322](https://github.com/elizaos/eliza/pull/8322)

### Cloud API: container management routes (create/list/get + lifecycle verbs)
The Cloud API now exposes a first-class containers surface, including lifecycle operations required by higher-level “deploy/monetize/survive” loops.

- **PR:** `feat(cloud-api): add /api/v1/containers deploy + read route` [#8258](https://github.com/elizaos/eliza/pull/8258)  
- **PR:** `feat(cloud-api): add container lifecycle verbs (PATCH/DELETE/scale)` [#8270](https://github.com/elizaos/eliza/pull/8270)

Example (conceptual) usage from a backend/service:
```bash
# Create / deploy a container (route introduced in #8258)
curl -X POST "https://www.elizacloud.ai/api/v1/containers" \
  -H "Authorization: Bearer $ELIZA_CLOUD_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "image": "ghcr.io/your-org/your-app:latest",
    "name": "my-app",
    "env": { "NODE_ENV": "production" }
  }'
```

### Cloud SDK: audio transcription helper
Third-party apps can now call STT without hand-rolling multipart forms.

- **PR:** `feat(cloud-sdk): add ElizaCloudClient.transcribeAudio() for voice/stt` [#8266](https://github.com/elizaos/eliza/pull/8266)

Example:
```ts
import { ElizaCloudClient } from "@elizaos/cloud-sdk";
import { readFile } from "node:fs/promises";

const client = new ElizaCloudClient({ baseUrl: "https://www.elizacloud.ai" });

const audio = await readFile("./sample.webm");

const result = await client.transcribeAudio({
  // exact field names/types: see packages/cloud-sdk/src/types.ts
  file: new Blob([audio], { type: "audio/webm" }),
  model: "whisper-1",
});

console.log(result.text);
```

## 3) Bug Fixes (critical / high-impact)

### Apps data-plane: collision-safe host-port allocation under density
High-density deployments could fail intermittently due to random port collisions during `docker create`. Port allocation now probes in-use ports to avoid collisions.

- **PR:** `fix(apps): collision-safe host-port allocation` [#8328](https://github.com/elizaos/eliza/pull/8328)  
- **Context:** explicitly tracked as item #8 in [#8321](https://github.com/elizaos/eliza/issues/8321)

### Apps data-plane: DB connectivity from isolated containers (DB “ambassador”)
Apps run on per-app `--internal` Docker networks (no egress). That also blocked connectivity to their own tenant DB. A DB ambassador path now allows containers to reach the tenant DB while keeping the isolation model.

- **PR:** `fix(apps): DB ambassador so isolated app containers can reach their tenant DB` [#8304](https://github.com/elizaos/eliza/pull/8304)

### Eliza Cloud login recovery: Steward OAuth PKCE + signed exchange/proxy calls
Steward auth hardening broke “Sign in with Eliza Cloud” until Eliza Cloud’s frontend and proxy paths were updated to:
- send PKCE parameters
- sign sensitive upstream calls

- **Issue (prod breakage):** “Steward login broken in prod…” [#8257](https://github.com/elizaos/eliza/issues/8257)
- **PR:** `fix(cloud): send PKCE on Steward OAuth + sign the nonce-exchange call` [#8259](https://github.com/elizaos/eliza/pull/8259)
- **PR:** `fix(cloud-api): sign Steward proxy requests to satisfy HMAC guard` [#8256](https://github.com/elizaos/eliza/pull/8256)
- Additional stabilizers:
  - `fix(cloud): stabilize Steward OAuth redirect URI` [#8279](https://github.com/elizaos/eliza/pull/8279)
  - `fix(cloud): hydrate Steward cookie refresh sessions` [#8280](https://github.com/elizaos/eliza/pull/8280)
  - `fix(cloud): handle @stwd/sdk 0.10 auth result types` [#8271](https://github.com/elizaos/eliza/pull/8271)
  - `fix(cloud): bump @stwd/sdk to 0.10.1 (passkey challengeId login fix)` [#8267](https://github.com/elizaos/eliza/pull/8267)

### Cloud inference: ensure response-token budget for reasoning models
A production-grade failure mode was addressed where reasoning models could return `content: null` (while still billing tokens) when `max_tokens` was insufficient for both hidden reasoning + final output.

- **PR:** `fix(cloud): guarantee response-token budget for all reasoning models` [#8189](https://github.com/elizaos/eliza/pull/8189)

### Orchestrator correctness: eliminate duplicate completion replies (TOCTOU)
A check-then-act race in completion deduplication could emit duplicate replies under concurrent sessions.

- **PR:** `fix(orchestrator): close completion-dedup TOCTOU race (duplicate replies)` [#8161](https://github.com/elizaos/eliza/pull/8161)

## 7) Breaking Changes (V1 → V2 migration warnings)

### `@elizaos/plugin-lifeops` removed (no compatibility shim)
The LifeOps monolith was decomposed into multiple focused plugins and renamed. This is a hard break: imports and plugin identifiers must be updated.

- **PR:** `refactor(lifeops): split plugin-lifeops into focused plugins + plugin-personal-assistant` [#8277](https://github.com/elizaos/eliza/pull/8277)

**What breaks:**
- `@elizaos/plugin-lifeops` is **gone** (not aliased).
- `/lifeops` dashboard view was removed; domain plugins now own their views.
- Internal exports renamed (`appLifeOpsPlugin` → `personalAssistantPlugin`, etc.).
- App registry entry renamed (`apps/lifeops.json` → `apps/personal-assistant.json`).

**Migration guidance (developer action items):**
1. Replace dependencies:
   - Remove `@elizaos/plugin-lifeops`
   - Add `@elizaos/plugin-personal-assistant` plus the domain plugins you actually need (e.g. calendar, goals, inbox, todos, finances, etc.).
2. Update any app/plugin loader references to LifeOps registration hooks.
   - A follow-up fix was required to prevent SPA boot crashes:
     - **PR:** `fix(plugin-lifeops): drop calls to removed registerLifeOpsApp + LifeOps view exports` [#8314](https://github.com/elizaos/eliza/pull/8314)
3. Update any UI deep links from `/lifeops` to the per-domain routes provided by the new plugins.

### Cloud V2 auth expectations (PKCE + signed upstream)
If you maintain custom login surfaces or embed Eliza Cloud auth, ensure your implementation supports PKCE and upstream signing patterns consistent with the updated Steward requirements.

- Track: [#8257](https://github.com/elizaos/eliza/issues/8257), implemented in [#8259](https://github.com/elizaos/eliza/pull/8259) / [#8256](https://github.com/elizaos/eliza/pull/8256)
