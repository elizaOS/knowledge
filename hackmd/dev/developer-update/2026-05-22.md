This update covers the week leading into **2026-05-22**, including the **v2.0.2** release train and follow-on platform stabilization work.

## 1) Core Framework (Architecture / Plugin System / Agent Runtime)

### v2.0.2 release cut
- **Release promotion**: `develop → main` for **v2.0.2** landed in [elizaos/eliza#7828](https://github.com/elizaos/eliza/pull/7828). This is a large promotion PR; developers should scan the included items list (Cloud, runtime, mobile, desktop, plugins) before pinning versions in production.

### OWNER+AGENT identity model (connectors & UI)
The most important conceptual shift in v2 is that a connector can represent **two identities**:
- **OWNER**: the human user’s account on the external platform
- **AGENT**: the agent’s own account/bot identity

Key changes:
- **Connector registry schema now supports dual identities** via `accounts: { owner?, agent? }`:
  - [elizaos/eliza#7817](https://github.com/elizaos/eliza/pull/7817) — schema + audit doc
  - [elizaos/eliza#7819](https://github.com/elizaos/eliza/pull/7819) — UI updates: role-aware account lists + dual-section setup panel
- **OAuth callbacks no longer force-role OWNER** for Linear/Calendly/Shopify:
  - [elizaos/eliza#7820](https://github.com/elizaos/eliza/pull/7820)
  - Follow-up helper extraction: [elizaos/eliza#7841](https://github.com/elizaos/eliza/pull/7841)

Why this matters: connector-backed tools (email, calendars, commerce, DMs) can now safely operate with explicit identity separation. This is foundational for “native apps integration” and reduces accidental privilege mixing.

Relevant docs added/updated this week:
- OWNER role doc: `packages/docs/agents/owner-role.md` (landed via develop→main promotion; see [#7840](https://github.com/elizaos/eliza/pull/7840) file list)

### Plugin “mode unification”: direct vs remote
A multi-PR stack formalized the idea that “everything is a Plugin”, with an explicit execution mode:
- **P0**: types + manifest scaffolding — [elizaos/eliza#7844](https://github.com/elizaos/eliza/pull/7844)
- **P1**: remote-mode runtime + isolated-process worker + host bridge — [elizaos/eliza#7846](https://github.com/elizaos/eliza/pull/7846)
- **P2**: unified view-host shims across platforms — [elizaos/eliza#7847](https://github.com/elizaos/eliza/pull/7847)
- **P3**: `installRemotePlugin` API + first sub-agent runner reference implementation — [elizaos/eliza#7848](https://github.com/elizaos/eliza/pull/7848)

This is the backbone for “agent-generated plugins” and for running higher-risk capabilities out-of-process.

### Cloud agent lifecycle operations moved to a job queue
Cloud lifecycle operations (restart/suspend/logs/snapshot/resume, etc.) were refactored so Workers **enqueue jobs** instead of directly calling core services inline:
- [elizaos/eliza#7810](https://github.com/elizaos/eliza/pull/7810) — “all agent lifecycle ops via job queue”
- Follow-on refactors:
  - [elizaos/eliza#7815](https://github.com/elizaos/eliza/pull/7815)
  - [elizaos/eliza#7816](https://github.com/elizaos/eliza/pull/7816)
- Frontend updated to respect async job semantics:
  - [elizaos/eliza#7813](https://github.com/elizaos/eliza/pull/7813)

Impact: API callers must expect `202 Accepted` + `jobId` rather than synchronous completion for lifecycle endpoints.

### Electrobun “Satellites” + dynamic view host bridge
First-party capability “Satellites” are now integrated into the Electrobun platform package, along with a typed command/event bridge and early dynamic-view infrastructure:
- [elizaos/eliza#7779](https://github.com/elizaos/eliza/pull/7779)

This establishes:
- renderer→worker invocation (`carrotInvokeWorker`)
- worker event tailing (`carrotTailWorkerEvents`) with bounded buffers and cursor semantics
- dynamic agent-created canvas/A2UI views (register/list/open/push/close)

- UI: per-role busy state for dual connect buttons:
  - [elizaos/eliza#7866](https://github.com/elizaos/eliza/pull/7866)

**Developer-facing behavior**
A connector can now maintain distinct account records per role. In practice, you should:
- store bot tokens/agent identities under **AGENT**
- store user OAuth tokens under **OWNER**
- route actions (e.g., “send message”, “create calendar event”) through the appropriate role

**Example: selecting a role in connector-facing code**
(Exact APIs vary by connector; conceptually you’ll filter/select accounts by `role`.)

```ts
import type { ConnectorAccountRole } from "@elizaos/core";

// Pseudocode: pick an OWNER account when you need the user's authority
const ownerAccount = accounts.find((a) => a.role === "OWNER" satisfies ConnectorAccountRole);

if (!ownerAccount) {
  throw new Error("No OWNER account connected; prompt user to connect OWNER role.");
}
```

### B) Remote plugin install surface (`installRemotePlugin`)
The plugin stack now includes an explicit API for agent-installed remote plugins:
- [elizaos/eliza#7848](https://github.com/elizaos/eliza/pull/7848)

**Conceptual workflow**
1. Agent obtains a remote plugin manifest (permissions, worker entrypoint, optional view assets)
2. Host validates against trust policy
3. Plugin is installed and executed in remote/isolated worker runtime

Because this touches trust boundaries, review:
- manifest schema/types in [#7844](https://github.com/elizaos/eliza/pull/7844)
- worker runtime bootstrap in [#7846](https://github.com/elizaos/eliza/pull/7846)

## 4) API Changes (Developer-visible)

### A) Cloud agent lifecycle endpoints now async (job queue)
From [#7810](https://github.com/elizaos/eliza/pull/7810) + [#7813](https://github.com/elizaos/eliza/pull/7813):

- Expect lifecycle endpoints to return:
  - `202 Accepted`
  - a `jobId` you must poll/track

**Example: handle 202 + jobId**
```ts
const res = await fetch(`/api/v1/agents/${agentId}/restart`, { method: "POST" });

if (res.status !== 202) throw new Error(`Unexpected status ${res.status}`);
const { jobId } = await res.json();

// Then poll a job status endpoint (exact route depends on your Cloud API surface)
```

If you have custom tooling that assumed synchronous restart/suspend/snapshot, update it to handle async semantics.

### B) Connector registry schema: `accounts` block added
- [#7817](https://github.com/elizaos/eliza/pull/7817)

This is a schema extension, but it affects:
- registry validation
- connector setup UI rendering
- OAuth callbacks that need to preserve requested role

### C) Vault integration introduces `apiKeyRef` flows (secrets handling)
Vault feature originally landed earlier ([#7197](https://github.com/elizaos/eliza/pull/7197)) but remains central to v2-era API expectations:
- provider switching and config hydration increasingly prefer **references** (`apiKeyRef`) rather than plaintext keys

If you’re implementing new providers or connector-like secrets, prefer storing references and resolving through the vault layer rather than persisting secrets in config.

Docs:
- `packages/vault/README.md` (in-repo)

## 7) Breaking Changes / Migration Warnings (V1 → V2)

### A) Connector identity semantics changed (OWNER vs AGENT)
If you are migrating custom connectors/plugins from v1:
- Do not assume “the connected account” is always the agent’s identity.
- Update any OAuth completion handlers to persist the **requested role** rather than hardcoding OWNER (see [#7820](https://github.com/elizaos/eliza/pull/7820)).

**Migration checklist**
- Audit connector manifests for `accounts` support: [#7817](https://github.com/elizaos/eliza/pull/7817)
- Update UI flows if you built custom connector panels (role-aware rendering is now first-class): [#7819](https://github.com/elizaos/eliza/pull/7819)

### B) Cloud lifecycle endpoints: sync → async
If you have automation/scripts that call restart/suspend/snapshot/logs and expect completion on the same request:
- You must update for `202 + jobId` semantics (see [#7810](https://github.com/elizaos/eliza/pull/7810), [#7813](https://github.com/elizaos/eliza/pull/7813)).

### C) Plugin execution model: direct vs remote
If you rely on plugins running in-process:
- Review the new `mode: "direct" | "remote"` types and the remote worker runtime. Your deployment/trust posture must account for:
  - manifest validation
  - permission gating
  - view-host shims across platforms  
  (See [#7844](https://github.com/elizaos/eliza/pull/7844), [#7846](https://github.com/elizaos/eliza/pull/7846), [#7847](https://github.com/elizaos/eliza/pull/7847).)

### D) Dockerization guidance (from Discord)
A community VoiceAI platform dockerization thread reinforced a practical migration warning for self-hosters:
- Avoid a monolithic “single container does everything” migration for real-time voice stacks.
- Prefer multi-container separation (inference, API, DB, queue/worker) and converge later if needed.
