## 1) Core Framework

### Managed agent architecture: per-agent in-container PGlite (scalability + isolation)
A production incident class (“too many clients” / connection exhaustion) was addressed by moving **new managed (Dedicated) cloud agents** off the shared Postgres pool and onto a **local in-container PGlite** store. This reduces shared DB connection pressure and improves cold-start reliability at scale.

* PR: **“new managed agents use local in-container DB + lean chat plugins”** — [#8754](https://github.com/elizaos/eliza/pull/8754)  
* Related incident tracker: [#8696](https://github.com/elizaos/eliza/issues/8696)  
* Context: Discord daily infra note (2026-06-20) also confirms the connection pooling root cause and the PGlite-based fix.

**Implementation notes (what changed):**
- `managed-eliza-config` now provisions PGlite as the default state store for newly created managed agents (Dedicated tier), rather than routing plugin-sql to a shared Postgres by default.
- Dedicated agents also gained a “lean chat” plugin preset (below) to reduce boot time and dependency surface.

### LEAN_CHAT_PLUGINS preset for dedicated chat agents (faster boot, fewer deps)
Dedicated chat agents previously loaded a heavy default plugin set (coding tools, browser, orchestrator, etc.). A new preset reduces cold boot and steady-state overhead for “chat-only” agents.

* PR: [#8730](https://github.com/elizaos/eliza/pull/8730)

### Keyless `WEB_FETCH` action generalized beyond Anthropic-only path
Keyless web fetch/search was previously gated behind Anthropic-specific wiring. This week, ElizaOS generalized a **keyless `WEB_FETCH` runtime action** so agents running non-Anthropic models (e.g., Cerebras-hosted `gpt-oss-*`) can still answer live-info questions inline.

* PR: [#8097](https://github.com/elizaos/eliza/pull/8097)  
* Issue: [#8100](https://github.com/elizaos/eliza/issues/8100)

**Example (invoking `WEB_FETCH` from an action/tool call)**
```ts

await runtime.actions.invoke("WEB_FETCH", {
  url: "https://example.com/status",
  method: "GET",
  // Optionally: headers, extract, etc. depending on the action schema
```

## 3) Bug Fixes (critical fixes with technical context)

### (Cloud) “too many clients” / connection pooling exhaustion
**Symptom:** Dedicated agents at moderate concurrency triggered Postgres connection exhaustion (“too many clients”), causing instability and cascading failures.

**Fix:** Default new managed agents to **in-container PGlite** and integrate lean chat plugin presets.

- PR: [#8754](https://github.com/elizaos/eliza/pull/8754)

### (Wallet) Turn latency spikes (~28s) due to unbounded balance RPC
**Symptom:** Dedicated agents had multi-second reply delays; root cause was balance RPC that could hang long enough to block the response path.

**Fix:** Bound per-turn EVM balance checks so they cannot stall the chat completion path.

- PR: [#8748](https://github.com/elizaos/eliza/pull/8748)

### (Auth) Steward OAuth breakage in prod: PKCE required but missing
**Symptom:** “Sign in with Eliza Cloud” failed due to Steward requiring PKCE (`code_challenge`) while frontend/nonce exchange didn’t send it, causing 400s and blocking login.

**Fix:** Implement PKCE end-to-end and sign nonce-exchange calls.

- PR: [#8259](https://github.com/elizaos/eliza/pull/8259)  
- Multi-surface follow-up (shared helper for PKCE): [#8350](https://github.com/elizaos/eliza/pull/8350)  
- Original incident issue (historical context): [#8257](https://github.com/elizaos/eliza/issues/8257)

### (Core) Chat reply blocked by background “facts” stage
**Symptom:** Dedicated cloud agents took 30–43s/turn even when the main response was ready quickly.

**Fix:** Ensure the background facts/relationships stage does not block the user-facing reply path.

- PR: [#8705](https://github.com/elizaos/eliza/pull/8705)

## 4) API Changes (developer-facing)

### Cloud API: Containers lifecycle routes
Developers integrating app hosting should now treat `/api/v1/containers` as the primary control surface for container apps:

- `POST /api/v1/containers` (deploy/create) — [#8258](https://github.com/elizaos/eliza/pull/8258)

### Cloud API: Agent edit endpoint (rename + in-place config edit)
`PATCH /api/v1/eliza/agents/:id` now supports an edit body that does **not** require an `action` field when renaming or updating config in place.

- PR: [#8608](https://github.com/elizaos/eliza/pull/8608)

### Cloud API CORS + headers
If you’re building third-party web apps:
- Public API origins were expanded (CORS hardening): [#8245](https://github.com/elizaos/eliza/pull/8245)
- Allow `Idempotency-Key` and `X-Affiliate-Code` headers: [#8252](https://github.com/elizaos/eliza/pull/8252)

### SSE / WebSocket auth (Cloud + browser constraints)
Because browsers can’t set `Authorization` headers on SSE `EventSource` or `WebSocket` constructors, the platform standardized support for `?token=` on these upgrade/stream paths.

- SSE `?token=` support: [#8309](https://github.com/elizaos/eliza/pull/8309)  
- WS token forwarding/auth unification: [#8639](https://github.com/elizaos/eliza/pull/8639)

## 7) Breaking Changes / V1 → V2 Migration Warnings

### `@elizaos/plugin-lifeops` removal (no backward-compat shims)
The LifeOps monolith was decomposed; the old package is removed rather than aliased. Any agent configs or imports referencing `@elizaos/plugin-lifeops` must migrate to the new plugin set (e.g., `plugin-personal-assistant`, `plugin-calendar`, etc.).

- PR: [#8277](https://github.com/elizaos/eliza/pull/8277)

**Migration guidance (high level):**
- Replace `plugin-lifeops` with:
  - `@elizaos/plugin-personal-assistant` for orchestration actions (BRIEF/PRIORITIZE/etc.)
  - Domain plugins such as `@elizaos/plugin-calendar`, `@elizaos/plugin-todos`, `@elizaos/plugin-health`, etc.

### Cloud auth + transport changes (token-in-query for WS/SSE)
If you maintain custom UIs or connectors that rely on WS/SSE, ensure you support the platform’s standardized `?token=` approach for browser limitations, and do not assume `Authorization` headers will exist at handshake time.

- SSE auth: [#8309](https://github.com/elizaos/eliza/pull/8309)  
- WS unified auth: [#8639](https://github.com/elizaos/eliza/pull/8639)
