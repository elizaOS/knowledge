### Strategic bet evaluation: `@elizaos/plugin-peaq`

#### Opportunity scorecard
- **User impact (if true):** Very high (distribution, narrative, real-world agents).
- **Technical risk:** High (unknown APIs, security boundaries, edge constraints, SLAs, device heterogeneity).
- **Business risk:** Medium (partner dependency; could distract from stabilizing Cloud core).
- **Dependency risk:** High (requires clear execution environment + task/billing semantics alignment).

#### Suggested path (minimize distraction, maximize learning)
1. **Partner validation (Week 1):**
   - Confirm peaqOS integration surface: messaging, identity, task dispatch, payment rails, device capability model.
   - Confirm realistic “3M devices” accessibility (what fraction are addressable via this integration?).
2. **Define MVP (Week 2):**
   - A “hello device” agent that can receive a task, reason, and emit an action result back to peaq.
   - Security model: key handling, signing, attestation if available.
3. **Go/No-go gate:** Only proceed to full plugin build if MVP can be demonstrated with a real device/workload and a clear developer adoption plan.

**Resource allocation recommendation**
- Assign **1 tech lead (part-time)** + **1 integration engineer** for a **timeboxed discovery** (no more than 2 weeks) while keeping the core team focused on Cloud onboarding + Apps data-plane stabilization.
