## 3) Strategic Prioritization (Impact × Risk × Dependencies)

### Priority 0 (immediate: reduces systemic risk)
**A) Publish token migration transparency pack (policy + numbers)**
- User impact: **Very high** (trust, reduces scams/support load).
- Technical risk: **Low** (mostly data extraction + comms).
- Dependencies: access to migration ledger/allocations; decision on unmigrated tokens.
- Recommendation:
  - Ship a short, canonical post answering:
    - migration window dates (already known: closed Feb 4, 2026),
    - **% migrated** (even if approximate),
    - **explicit policy** for unmigrated tokens (burn, treasury, redistribution, vesting extension, etc.),
    - where to verify (links, hashes, addresses).
  - Add a “**No DMs / no seed phrase**” anti-scam banner to docs + Discord pinned message in relevant channels.

**B) Lock down runtime/plugin initialization determinism (refactor critical path)**
- User impact: **High** (stability for Milady app launch + plugin ecosystem).
- Technical risk: **Medium–High** (breaking changes).
- Dependencies: runtime refactor proposal (deadline referenced), plugin-sql + plugin-personality integration contracts.
- Recommendation:
  - Enforce “infra is explicit dependency” rule:
    - DB adapter passed via constructor / runtime dependency injection, not side-effects.
  - Add automated test:
    - randomized plugin init order to catch race conditions.
  - Provide migration guide for plugin authors (minimal API surface, clear patterns).

### Priority 1 (next: unlocks safe on-chain utility)
**C) x402Guard: standardize a non-custodial transaction policy layer**
- User impact: **High** for DeFi agents; expands “can safely do money things” narrative.
- Technical risk: **Medium** (cross-chain handling, session key management, policy UX).
- Dependencies: early testers + plugin API stability + wallet integration patterns.
- Recommendation:
  - Treat as “reference security plugin” with:
    - default policies (spend caps, allowlists),
    - audit logs/receipts,
    - integration examples for Base + Solana.
  - Run an early tester cohort: target **5–10 DeFi agent teams**; measure:
    - time-to-integrate,
    - prevented-policy-violations,
    - false positives/negatives.

**D) Milady Prediction: define measurable accuracy + execution constraints**
- User impact: **Medium–High** if it becomes dependable; otherwise risks being noise.
- Technical risk: **Medium** (data quality, calibration, market selection, backtesting).
- Dependencies: data aggregation pipeline + threshold decision + on-chain execution guardrails (pairs well with x402Guard-like controls).
- Recommendation:
  - Define 2 KPIs before expanding scope:
    - calibration metric (e.g., Brier score / log loss) over a rolling window,
    - execution rule (e.g., act only when probability > X and edge > fees/slippage).
  - Publish a “do we execute?” policy to prevent reckless on-chain actions.

### Priority 2 (growth: capitalize on traction)
**E) OpenClaw China PMF: convert signal into repeatable distribution**
- User impact: **High** (top-of-funnel + credibility).
- Technical risk: **Low–Medium** (support load, localization, packaging).
- Dependencies: packaging guidance, hardware recommendations, onboarding docs.
- Recommendation:
  - Produce an “official” lightweight deployment guide:
    - recommended Mac Mini specs,
    - dedicated machine rationale,
    - troubleshooting checklist.
  - Instrument installs/active usage where possible to confirm retention.

**F) Workflows-as-a-Service + skill.md: productize discovery → paid execution loop**
- User impact: **Medium** (developer monetization, ecosystem flywheel).
- Technical risk: **Medium** (spec drift, fraud/low-quality agents, billing disputes).
- Dependencies: `skill.md` metadata quality; rating + receipts reliability.
- Recommendation:
  - Promote a minimal `skill.md` schema (versioned) and validation tool.
  - Track funnel metrics:
    - skill.md impressions → agent registrations → paid executions → repeat rate.
