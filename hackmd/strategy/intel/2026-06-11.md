## 3) Strategic Prioritization (Impact × Risk × Dependencies)

### Priority A — Protect Cloud access + trust surfaces (highest user impact)
1) **Cloud authentication reliability** (dependency for every cloud user + third-party app)
- Risk: any auth break blocks onboarding and monetization loops.
- Action: maintain a “**Cloud Auth Health**” dashboard item (status of Steward OAuth/email flows, refresh behavior, callback paths).
- Resourcing: **1 dedicated owner** for auth flows + **1 reviewer** for rapid PR turnaround.

2) **Cloud Apps go-live blockers (scale/ingress)** (revenue + credibility risk)
- The open scaling/ingress gaps documented in repo issues represent a **go-live gating set** (ingress, connection pooling, placement, etc.).
- Recommendation: treat Apps data-plane as a **launch checklist with explicit P0/P1 acceptance tests**, not ad-hoc fixes.

### Priority B — Convert community energy into shipped value (high leverage, low technical risk)
**Problem:** high inbound “I can help” with no structured conversion.
- Action: stand up a **Contribution Intake Board** with 3 lanes:
  - “Help wanted (Discord/bots/community)”
  - “Plugin ecosystem (registry/reviews/examples)”
  - “Product QA (smoke tests, docs fixes, repro capture)”
- Add a Discord command or bot response template:
  - `!contribute` → links, owners, current priority tasks, expectations

### Priority C — Plugin ecosystem governance (medium impact, compounding over time)
- Third-party registry is receiving security-sensitive plugins (e.g., signers handling exchange keys).
- Actions (process-level, minimal code):
  - Define a **security review rubric** for registry acceptance:
    - key custody model, threat assumptions, dependency/license scan, minimal docs requirements
  - Publish expected **review SLA** and escalation path if it slips
- Immediate: ensure **PR #8294** receives review within the stated ~48h window to reinforce contributor trust.
