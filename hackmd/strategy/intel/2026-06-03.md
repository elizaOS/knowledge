#### Priority 2 — DeFi integrations: Kamino plugin vs aggregator strategy
- **Decision needed:** protocol-specific depth (Kamino) vs abstraction convenience (multi-chain aggregator).
- **User impact:** High (drives real adoption among DeFAI builders).
- **Technical risk:** High (lending/borrow/vaults are risk-heavy; developer-experience abstractions are hard).
- **Recommendation (strategy + sequencing):**
  - **Phase 1 (2–4 weeks): “Kamino Core Actions MVP”** as a reference-quality plugin:
    - Read-only + safe write paths first (deposit/withdraw to vaults with strict limits; no leverage by default).
    - Mandatory simulation / preview outputs; structured risk warnings in tool schema.
  - **Phase 2: define an “Eliza DeFi Action Interface”** that other protocols can implement:
    - Minimal common verbs: `getPositions`, `quote`, `buildTx`, `simulate`, `execute`.
    - Keep protocol-specific modules for advanced features (limit orders, special vault strategies).
  - This resolves the debate by **making Kamino the proving ground** for a later abstraction—without blocking go-to-market.

#### Priority 3 — Contributor experience: reduce friction from fast-moving `develop`
- **User impact:** Medium–High (external plugin throughput).
- **Technical risk:** Low.
- **Recommendation (high leverage, low cost):**
  - Add a **“Plugin Dev Quickstart”** with:
    - repo location (`elizaOS/eliza`), target branch (`develop`), rebase cadence expectations,
    - minimal plugin skeleton + test command,
    - a compatibility note about breaking changes and how to pin interfaces.
  - Introduce a **weekly “API churn note”** in release/dev updates for plugin authors.
