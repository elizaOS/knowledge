## 2) User Experience Intelligence

### Feedback themes (categorized by impact)

#### A) Critical UX blockers (P0)
- **Cloud sign-in reliability risk** (open issue: Steward login prod OAuth/email 400s).
  - User impact: blocks **all “Sign in with Eliza Cloud”** entry points (dashboard + third-party + CLI login).
  - Business impact: directly interrupts provisioning, monetization loops, and partner adoption.

#### B) High-impact friction (P1)
- **Newcomer cognitive overload / unclear learning path**
  - Observed behavior: users seek “context” before technical steps; rely on LLMs to navigate complexity.
  - Failure mode: user churn before first successful build/run.

#### C) Ecosystem trust & safety (P1)
- **Scammer presence** + rising interest in DeFi agents
  - Risk: reputational harm; compromised agents; unsafe transaction signing patterns.

#### D) Opportunity & engagement gaps (P2)
- Multiple posts offering help or seeking work (developers/security engineer) with **no closure loop**.
  - Cost: slower integration/testing, missed collaborations, lower retention of skilled contributors.

### Usage patterns vs intended design (notable mismatches)
- Intended: #coders as technical ramp. Observed: newcomers start in general discussion with “life situation + discovery story” and need a **guided funnel**, not a channel redirect.
- Intended: modular plugins improve clarity. Risk: removal of monolithic “LifeOps” view without crisp migration UX can feel like “features disappeared” to existing users.

### Implementation opportunities (UX + community)
- Create a **single “Start Here” decision tree** pinned in Discord and docs:
  - “I want to build a plugin” / “I want to run locally” / “I want Cloud agent + monetization” / “I want orchestration UI”.
- Add **lightweight contribution matchmaking**:
  - a weekly “open help wanted” post + a template for “I can help / I need help” including timezone, stack, and a 1-hour starter task.

### Critical-path dependencies to call out
- **Cloud adoption depends on auth stability**: Steward OAuth/email/passkey/MFA + cookie hydration must be treated as a single “system,” not separate fixes.
- **Modular plugin success depends on migration narrative**:
  - Clear mapping: “Old LifeOps → New Personal Assistant + domain plugins”
  - DB/schema coexistence and view replacement needs explicit upgrade notes and sanity checklist.

### Resource allocation recommendations (next 7–10 days)
1. **Auth Tiger Team (1–2 engineers + QA)**  
   - Goal: eliminate remaining prod login regressions; add regression tests for OAuth/email/passkey + cookie refresh + CLI login.
   - Deliverables:
     - A single “Auth health” dashboard/checklist (prod/staging).
     - E2E tests that simulate: OAuth → callback → token refresh → CLI polling → dashboard session continuity.

2. **Migration + Onboarding (1 engineer + 1 technical writer / DX)**  
   - Goal: reduce cognitive load and prevent “feature disappeared” sentiment.
   - Deliverables:
     - Migration doc: LifeOps → Personal Assistant + domain plugins (with screenshots and view mappings).
     - “Choose your path” onboarding page + pinned Discord post.

3. **Community Ops (part-time owner, can be non-engineering)**  
   - Goal: close loops on recruitment and service offers.
   - Deliverables:
     - Weekly “Buildathon / bounties / hiring” thread; route to issues labeled `good first issue` / `help wanted`.
     - A lightweight intake form for contractors/service providers (security/plugin dev).
