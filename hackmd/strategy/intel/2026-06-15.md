### Usage patterns vs intended design
- Users are treating:
  - **elizaCloud** as the canonical onboarding surface (not “advanced/beta”), so invitation gating reads as a **platform outage**, not a controlled rollout.
  - **Milady** as a configuration gateway (OpenRouter API settings) → failures there become “Cloud is broken,” even if Cloud is fine.
- Users ask about **bridging** as if elizaOS is chain-native. If that’s not intended, this needs explicit positioning.

### Implementation opportunities (high leverage)
1. **“Access status” single source of truth**: a pinned Discord post + status page blurb + Cloud UI error message that explains:
   - why invites are required (if intentional),
   - how to request/obtain one,
   - tenant vs environment confusion (prod vs staging),
   - known incidents + ETA.
2. **Windows dev:ui diagnostic capture**: scripted command that collects:
   - node/bun versions, OS build,
   - Vite/Bun logs,
   - browser console output,
   - last 200 lines of UI server output.
