## 3) Strategic Prioritization (Impact × Risk × Dependencies)

### Priority 0 (this week): Close the clarity + trust gap around waifu.fun and tokens
**Why now:** unanswered strategic questions are compounding into skepticism; this is a growth and reputational choke point even while infra improves.

**Actions (deliverables-first)**
1) **Waifu.fun canonical explainer (1 page)**
   - Audience: community + prospective users + partners.
   - Must answer:
     - Who it’s for (user personas)
     - What “launching trading agents” means (capabilities, constraints, lifecycle)
     - How it relates to ElizaOS / Eliza Cloud Apps / managed agents
     - What success looks like (examples)
   - Owner: PM/Comms + technical reviewer from waifu.fun builder(s).

2) **Token strategy statement (short, explicit)**
   - Clarify:
     - token relationships (if any),
     - why separate token exists (if intended),
     - whether consolidation is planned,
     - how value accrues across ecosystem products.
   - Include a “what we will not do” section to reduce fear/rumor loops.

3) **Partnership intake path**
   - Create a single public contact route (form/email + Discord workflow).
   - Add an auto-response with expected timelines + required info.

**KPIs to start tracking (no new instrumentation required)**
- Median time-to-first-answer for strategic questions in Discord (manual weekly sample).
- % of waifu.fun-related questions resolved with a canonical link.
- Partnership inquiries/week and response SLA adherence.

---

### Priority 1 (2–4 weeks): Make “no-code” real for non-devs (reduce cognitive load)
**Context:** repeated request for “button-based” workflows suggests current no-code either isn’t discoverable, isn’t sufficient, or is framed in developer language.

**Recommended scope**
- **Task templates** (prebuilt agent recipes) rather than generic agent creation:
  - “Social reply assistant”
  - “Content idea generator”
  - “Community moderation helper”
  - (If trading agents are in-scope) “paper-trading only” template with explicit risk labeling
- **Guided onboarding** that ends with a working agent in <10 minutes, no jargon.

**Risk/Dependency**
- Requires product definition alignment with branding (“Eliza OS” consumer narrative) and safety constraints (especially for finance/trading).

---

### Priority 2 (ongoing): Reliability work should be paired with user-visible health reporting
**What changed:** in-container PGlite for managed agents addresses a known scaling failure mode.

**Next steps**
- Add a small “Cloud Agent Health” status surface (internal dashboard or simple metrics):
  - managed agent DB mode distribution (Postgres vs PGlite)
  - connection pool saturation alerts (where Postgres still used)
  - provisioning failures by class
- Convert infra wins into user trust by publishing periodic stability notes (brief, factual).

---

### Priority 3 (risk containment): Address governance/allegation narratives with a standard response playbook
**Problem:** allegations + scams create an environment where silence reads as confirmation.

**Actions**
- Publish a **governance transparency FAQ**:
  - how funds are managed, what entities exist, what’s audited, what isn’t
  - how to report concerns, and what the investigation process is
- Moderation playbook for scam incidents:
  - pinned “how to verify” + “where to report”
  - post-incident minimal disclosure template (what happened, what users should do)
