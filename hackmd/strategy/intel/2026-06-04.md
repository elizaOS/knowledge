## 3) Strategic Prioritization (Impact × Risk × Dependencies)

### Top initiatives to prioritize (next 1–2 weeks)

#### 1) Stabilize “critical path” authentication & pairing flows (Cloud)
- **Why (impact)**: Auth failures are the fastest way to lose users/testers; staging stability directly affects external validation and community onboarding.
- **Current state**: Magic-link staging issues were fixed via CSP/CORS/context changes; **SSO pairing handler** for dashboard popup flow is WIP.
- **Recommendation**
  - Ship pairing handler behind a feature flag + add **end-to-end auth regression tests** (magic-link + SSO + callback with missing app-auth context).
  - Define a single “Auth Health” dashboard metric: *successful login rate* + *callback completion rate* on staging/prod.

#### 2) Decide DeFi integration strategy: “protocol plugin” vs “aggregation layer”
- **Why (impact)**: DeFi actions are a high-leverage demo surface for agent autonomy; Kamino is positioned as a “critical Solana protocol” and a clear integration gap.
- **Risk**: Abstractions for lending/vaults are **risk-sensitive**; incorrect defaults could cause user loss.
- **Recommendation**
  - Adopt a two-tier strategy:
    - **Tier A (Go-to-market)**: ship a **Kamino-first plugin** with explicit risk controls and limited, well-tested actions (deposit/withdraw/health factor checks).
    - **Tier B (Scale)**: evaluate multi-chain abstraction only after Tier A proves demand; otherwise abstraction risks becoming an underused maintenance layer.
  - Dependency: shared **permissioning + user-defined controls** model aligns with NeverBell-style execution constraints—coordinate patterns.

#### 3) Reduce contributor friction from rapid rebase cadence (DevEx)
- **Why (impact)**: fast-moving develop branch is creating implicit cost; slows ecosystem growth and increases maintainer load.
- **Recommendation**
  - Introduce a “Plugin API stability lane”:
    - weekly tagged **`plugin-api-snapshot`** (or documented compatibility window) and a short **migration guide** per breaking change set.
  - Add a lightweight PR bot comment that detects long-lived PRs and posts a **rebase + conflict risk summary** automatically.

### Targeted technical debt / UX defect paydown

#### A) Orchestrator “narration-envelope” deliverable loss (OPEN issue #8162)
- **User impact**: high—correct work produced but **deliverable is dropped/truncated**, creating “it didn’t do it” perception.
- **Technical risk**: medium—touches routing + evaluator logic; but proposal is scoped (metadata carrier similar to verified URLs).
- **Recommendation**
  - Treat as **P0 UX correctness** for orchestrator: preserve deliverable verbatim when no changeset exists; suppress stale retry chatter without breaking respawn triggers.
  - Add regression tests for: printed-value tasks, retry recovery, and multi-backend truncation patterns.

#### B) UI surface expansion (Odysseus-style orchestrator UI; continuous overlay)
- **User impact**: medium-high—premium interaction model; reduces onboarding friction and supports power users.
- **Technical risk**: medium—large diffs and ongoing CI instability can block merge.
- **Recommendation**
  - Enforce “mergeable slices” only: split UI PRs into (1) shell routing primitives (e.g., `fullBleed`), (2) UI components, (3) view wiring.
  - Update smoke tests to reference the canonical composer (overlay vs in-view) to prevent recurring CI failures.

  - keep CI green (format/smoke drift) and
