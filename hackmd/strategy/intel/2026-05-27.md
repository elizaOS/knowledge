## 3) Strategic Prioritization (Impact × Risk × Dependencies)

### Priority stack (next 7–14 days)

#### 1) Discord anti-spam hardening (P0)
**Why now:** Direct user-safety risk + reputational risk; also reduces support noise.  
**Recommended actions**
- Implement/enable: AutoMod rules, stricter permissions for room creation, verification gates, and bot account containment.
- Add a **single pinned “Official Support Process”** message:
  - “No staff will DM you first”
  - “No new ‘ticket rooms’ are official unless created by X bot”
  - Link to official docs + GitHub issues
**Owner allocation (suggested):** 0.5–1 eng day + 0.5 community ops day.

#### 2) Fix flap upload 400 error (“Operations and map are required”) (P1)
**Why now:** This is the clearest post-launch funnel break; leaving it unresolved converts excitement into churn.  
**Hypotheses to validate quickly**
- CLI schema mismatch: required fields not generated when “manually launching on waifu”
- Backend validation error lacks actionable messaging
**Recommended actions**
- Add request/response logging around flap upload validation (redacting secrets).
- Improve error message to include **example payload** or missing flags.
- Publish a minimal reproducer + known-good command in docs/README.
**Owner allocation:** 1–2 eng days + 0.5 docs day.

#### 3) “Project status” clarity + contributor onramp (P1)
**Why now:** Continuity confusion is a repeated blocker; easy to fix with comms + documentation.  
**Recommended actions**
- Add a short “**ElizaOS is active** (open source) / **Labs is defunct** (company)” statement:
  - Discord welcome channel + README + website footer
- Add “Where to contribute this week” (3–5 bite-size issues) to convert job-seekers into PRs.
**Owner allocation:** 0.5 day community ops + 0.5 day maintainer.

### Medium-term (14–45 days)

#### 4) Runtime environment detection & Linux setup guidance (P2)
**Problem:** Users report inability to run JS; likely missing “doctor” tooling and clear runtime requirements.  
**Recommended actions**
- Add `elizaos doctor` (or equivalent) checks:
  - Node/Bun presence, versions, permissions, basic network ports, optional GPU tooling
- Provide a one-page “Linux quickstart: supported runtimes” and common failure modes.
**Risk:** Moderate (cross-platform edge cases), but high ROI for support load reduction.

#### 5) Tooling + support loop instrumentation
**Recommended actions**
- Add a lightweight intake template for Discord issues:
  - “What command? What output? Which OS? Which version? Paste error.”
- Route resolved issues into a living “Known Issues” doc.
