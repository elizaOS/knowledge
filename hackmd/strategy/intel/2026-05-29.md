### Usage patterns vs intended design (from open #7935)
- Intended: when user asks to search chat history, system should prefer **SEARCH_MESSAGES** or a connector-native retrieval path.
- Observed: planner chooses **SHELL** (expensive unbounded grep), times out, may leak orchestration metadata.
- **User-visible failure modes:**
  - empty/no reply after long latency
  - confusing “internal routing” text shown to end user
  - avoidable compute burn (timeouts, repeated tool attempts)

## 3) STRATEGIC PRIORITIZATION (Impact vs Risk, Dependencies, Resource Allocation)

### Priority matrix (next 1–2 weeks)

#### P0 — Trust & Safety: stop Discord fake ticket spam
- **User impact:** very high (phishing, impersonation).
- **Tech risk:** low–medium (mostly Discord config + bot rules).
- **Dependencies:** Community Ops/mod perms; possibly a small bot update.
- **Actions (recommended):**
  1) Lock down channel creation permissions; require higher trust roles for ticket creation.
  2) Add **automod rules** for “ticket/support” keywords + room-creation patterns.
  3) Publish an **official support policy** (where staff will/won’t DM; how to verify).
  4) Add a “Report fake support” quick action and an incident log channel visible to mods.

#### P0 — Core UX: resolve planner/tool misrouting for chat recall (Issue #7935)
- **User impact:** high (directly affects “assistant usefulness” and reliability).
- **Technical risk:** medium (planner prompt + tool ranking + connector boundaries).
- **Critical path dependencies:** ties into orchestration/sub-agent UX work already ongoing.
- **Actions (recommended):**
  1) **Honor Stage 1 `candidateActions` as a strong preference** when present (ranking rule).
  2) Add **hard caps** for SHELL search patterns (deny `grep -R` on repo root; require scoped paths; enforce output/timeout budgets).
  3) Strip or prevent **sub-agent control metadata** from being user-visible (sanitize at framework boundary, not via regex hacks in replies).
  4) Add a regression test suite: “search chat for X” variations (imperative phrasing) across Discord + CLI contexts.

#### P1 — Community channel quality: reduce “Coders” spam; increase technical signal
- **User impact:** medium (retains builders; improves organic support).
- **Tech risk:** low.
- **Actions (recommended):**
  - Create a dedicated **#jobs-and-gigs** / **#showcase** channel and enforce posting rules in #coders.
  - Weekly “Help thread” prompt + office hours to seed real technical Q&A.
  - Lightweight tagging: [Help], [Showcase], [Hiring], [Hackathon].

#### P1 — Onboarding reliability: clarify runtime requirements for Linux/headless users
- **User impact:** medium (unblocks contributors).
- **Tech risk:** low.
- **Actions (recommended):**
  - Single doc page: “Minimum to run ElizaOS locally” (Bun/Node expectations, headless caveats, supported OS).
  - Map common blockers to fixes (permissions, missing JS runtime, package prerequisites).
