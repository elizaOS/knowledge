## 2) User Experience Intelligence

### Feedback themes (categorized by impact)

**P0 — Access blocked / trust impact**
- **Eliza Cloud login failures (OAuth 400 / Sign in with Eliza Cloud broken)**: blocks dashboard + third-party apps via shared SSO flow. This is a **hard stop** for any onboarding, monetization, or partner integrations.

**P1 — Economic UX correctness**
- **Ledger mismatch (app-credits vs org-balance)**: blocks “seamless spending”. This undermines monetization loop reliability and can create billing/support escalations.

**P1 — Onboarding & cognitive load**
- Newcomers experiencing **information overload**, unclear “what to learn first,” and a desire to contribute without an obvious path.
- Users are using community chat for orientation rather than documentation—signals **docs/IA not matching user mental models**.

**P2 — Safety & community trust**
- Scam alerts in coders channel: indicates continued need for **visible security posture** and rapid moderation playbooks.

### Usage patterns vs intended design
- Intended: newcomers funnel into “coders” for technical depth.
- Actual: newcomers ask high-level ecosystem questions in general and do not receive a crisp, actionable “start here” path; “help line” framing implies they expect a **support intake system**, not channel-hopping.

### Implementation opportunities (UX)
- Add a **single canonical onboarding funnel**:
  - “Start Here” doc + pinned Discord message + bot command (e.g., `/start`, `/help`, `/contribute`)
  - guided paths: “Builder”, “Plugin dev”, “Cloud app developer”, “Security reviewer”, “Community ops”
- Add a lightweight **triage loop**:
  - tag unanswered questions in general/coders; assign rotating “front desk” coverage.
- Provide “LLM-assisted learning” templates:
  - curated prompts for reading repos/docs faster (aligns with community advice about using LLMs for overload).

### Sentiment tracking (last 72h qualitative)
- **Market sentiment**: negative/defensive (“survival”).
- **Community tone**: supportive, welcoming.
- **Product trust**: at risk due to prod auth outage; reliability must be visibly prioritized.

## 3) Strategic Prioritization

### Priority stack (impact × risk × dependency)

#### P0: Restore Cloud sign-in reliability (OAuth + email)
**Why now:** Blocks all Cloud onboarding + partner apps; reputational risk.  
**Dependencies:** Steward auth expectations (PKCE), signing secrets, tenant redirect allowlists.

**Actions (next 24–48h)**
1. **Incident owner + war room** (single accountable DRI; comms cadence).
2. **Close the loop on PKCE + signing secrets + redirect allowlists**:
   - verify prod tenant redirect allowlist matches deployed origins
   - ensure **signing secrets** deployed where steward proxy expects them
   - add automated config checks to prevent future drift.
3. **Publish a status update + workaround** (if any) in Discord + status page:
   - expected resolution time
   - whether third-party apps should pause SSO rollouts.

**Engineering guardrails (this week)**
- Add an **auth contract test** in CI that runs a headless OAuth authorize→exchange happy-path against staging with PKCE required (prevents “upstream hardened, we broke” repeats).
- Add runtime telemetry: login error rate, OAuth 400 categorization.

#### P1: Resolve app-credits vs org-balance ledger mismatch (monetization correctness)
**Why now:** Monetized autonomous loops are shipping; ledger inconsistency will produce silent failures and support burden.

**Actions (this week)**
- Define canonical source of truth and reconciliation policy:
  - choose ledger invariants (e.g., org-balance ≥ sum(app-credits reserved) or vice versa)
  - implement reconciliation job + audit logs
- Add “spending blocked” UX message that is explicit:
  - show which balance is failing and how to fix (admin action vs system issue)

**Risk note:** autonomy + self-spend caps amplify billing correctness requirements; treat ledger as **critical path**.

#### P1: Onboarding IA + contribution pathways (reduce overload, increase retention)
**Why now:** Community is attracting “career crisis / AI shift” joiners; capturing them requires fast clarity.

**Actions (this week)**
- Ship a **one-page “ElizaOS Map”**:
  - What is ElizaOS vs Hyperia/Hyperforge (already a recurring confusion)
  - What users can do in 30 minutes / 2 hours / 2 days
  - Where to ask what (general vs coders vs security)
- Create a **“good first contribution” board** (docs fixes, plugin registry entries, test improvements).
- Add a Discord **/help intake** that routes to:
  - Cloud access issues (with status link)
  - plugin dev questions
  - “I want to contribute” (links to tasks)

#### P2: Security posture & partner/channel trust
**Why now:** scam alerts + increasing partner interest (e.g., Intuition Box bounties) raises attacker attention.

**Actions (this month)**
- Formalize a **scam response SOP** (reporting, verification, ban, retro).
- Consider appointing/rotating “security champions” and publish safe collaboration guidelines for bounties/partners.
- Evaluate the community-offered security services (prompt injection, agent draining, malicious signing boundary audits) as:
  - a short paid audit sprint, or
  - a public checklist + reference implementation in core.
