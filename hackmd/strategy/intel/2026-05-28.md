#### P2 — Platform compatibility confusion
- User report: **cannot run JavaScript on Linux host** (unclear whether policy, environment lockdown, missing runtime, or misunderstanding).  
  **Impact**: medium (prevents contribution/usage; signals docs and prereq checks are insufficient).  
  **Opportunity**: add “diagnostic decision tree” and preflight script.

### Usage patterns vs intended design
- Discord usage indicates **community is not using channels for iterative debugging**; instead, it’s broadcasting intros/projects. That implies:
  - The help surface is either **hard to discover**, **slow**, or **not rewarded** (no visible “support loop”).
  - There’s a missing conversion funnel from “intro” → “first PR / first plugin / first successful deploy.”

### Implementation opportunities (high leverage)
- Convert recurring friction points into **self-serve**:
  - Preflight checks (Discord + CLI): validate required fields (“operations/map”), runtime availability, and environment constraints before the user hits opaque 400s.
  - Pin “Known Issues / Common Errors” per tool (flap/waifu/local inference) with copy-pastable fixes and log collection steps.

## 3) Strategic Prioritization

### Priority stack (impact × risk × dependency)

#### 1) Secure the Discord entrypoint (P0)
**Problem**: fake support ticket room spam.  
**User impact**: very high. **Tech risk**: low-medium. **Dependency**: Discord admin/config + bot moderation tooling.  
**Actions (next 24–72h)**:
- Lock down channel creation permissions; add AutoMod rules; require verified roles for ticket creation.
- Add a single **official support flow** (one channel + pinned instructions) and an **automated warning** when new “support-like” channels appear.
- Publish a short incident note + guidance: “ElizaOS staff will never DM for keys; only use #support.”

**Success metric**: spam channel creation attempts/day → near zero; user reports of confusion → zero.

#### 2) Fix/contain the “flap upload 400” publish blocker (P1)
**Problem**: 400 “Operations and map are required”; no resolution path surfaced.  
**User impact**: high for builders trying to ship. **Tech risk**: medium (unknown API/contract mismatch).  
**Actions (this sprint)**:
- Add **client-side schema validation** in the `flap upload` command:
  - Fail fast with explicit missing fields and an example minimal manifest.
- Add a **single canonical example**: “manual launch on waifu → flap upload” with expected directory structure and required flags.
- Add structured error handling: if 400 contains known message, print targeted remediation.

**Success metric**: reduction in repeated retries; time-to-first-successful-upload for new users.

#### 3) Close the “platform prereqs” gap for Linux/dev environments (P2)
**Problem**: “can’t run JS on Linux host” + general setup uncertainty.  
**User impact**: medium but broad. **Tech risk**: low.  
**Actions (this sprint)**:
- Provide a **preflight script**: checks Bun/Node availability, glibc, permissions, network, and prints install commands.
- Update docs: “Minimum supported host environments” + “If you’re in a restricted host (no JS), here are options: container/devbox/remote runner.”

**Success metric**: fewer “can’t run X” posts; higher conversion from new intros to successful local run.

#### 4) Productize MCP “live social ingestion” as an experimental track (P3 / option value)
**Signal**: AgentKey MCP + OpenClaw used successfully for real-time Twitter/Reddit sentiment.  
**User impact**: potentially high for research agents; **risk**: medium (API policy, rate limits, compliance).  
**Actions (exploratory, 1–2 eng-days)**:
- Create an **“Experimental Integrations”** doc section with a minimal reference architecture and compliance notes.
- Decide whether to support as:
  - a plugin template + documented interface, or
  - a curated community recipe (no official support).

**Success metric**: number of reproducible community setups; reduced fragmentation of “one-off” experiments.
