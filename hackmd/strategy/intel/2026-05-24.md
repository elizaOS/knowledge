## 3) Strategic Prioritization (Impact × Risk × Leverage)

### Recommended priorities (next 1–2 weeks)

#### P0 — Publish canonical “Roadmap + Status + Sunsets” (High impact, Low risk)
**Why:** Directly addresses the highest-visibility unanswered question; reduces repeated Discord churn; prevents misinformation around Babylon/migration.
**Deliverable (single page + pinned Discord post):**
- “Now / Next / Later” for 60–90 days
- Version clarity (v2.0 vs v2.0.2 vs v3 agents in production)
- Sunsets: ai16z migration closed; Babylon closed; Feed as successor component
- Links: waifu.fun repo as reference implementation

**Success metric:** roadmap question rate in Discord drops; fewer repeated token/migration/Babylon questions week-over-week.

#### P0 — Security guidance for wallet-enabled agents (High impact, Medium risk)
**Why:** Trust is existential in Web3/agent contexts; community is actively searching for practices.
**Deliverable:** “Wallet Access Security Baseline” doc + reference config snippets:
- Spending caps / per-action allowances
- Human-in-the-loop confirmations for high-risk actions
- Prompt-injection hardening checklist (tool schema validation, allowlists, taint tracking for external text, separate system/tool channels)
- Key custody guidance (vault usage, environment separation, role-based connector accounts OWNER vs AGENT)
- Incident response playbook (rotate keys, revoke sessions, audit logs)

**Optional leverage:** Host an interview session with the researcher and convert outputs into a public writeup.

#### P1 — Onboarding: 2 “first success” demos (Medium-High impact, Low-Medium risk)
**Why:** Repeated demand for YouTube/live demos; reduces time-to-value.
**Two demo targets:**
1. “Build + run an agent with one connector (Discord/X) using OWNER+AGENT”
2. “Local voice loop demo” (even if minimal) with known-good hardware profile

**Success metric:** increased new-user retention signals (Discord follow-up questions shift from “how start” to “how customize”).

#### P1 — Deployment reference: voice platform container blueprint (Medium impact, Medium risk)
**Why:** Real-time voice apps are hard to containerize; a blessed reference reduces builder churn.
**Deliverable:** Compose template:
- Separate inference / API / DB / queue-worker
- GPU passthrough notes
- Persistent storage + networking
- “Single-container: not recommended” rationale + exceptions
