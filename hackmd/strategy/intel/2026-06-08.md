## 3) Strategic Prioritization (Impact × Risk × Dependencies)

### Priority 0 — Restore/guarantee Cloud login reliability (auth)
**Why now:** Direct revenue + adoption gate; blocks third-party apps and internal flows.

- **Primary target:** Close **#8257** (prod Steward login broken).
- **Dependency mapping:**
  - Steward OAuth contract (PKCE requirements) ↔ cloud-frontend URL builder ↔ cloud-api nonce exchange ↔ Steward service config
- **Recommendations (actionable)**
  1. **Add release gate:** automated “prod-like” auth e2e that verifies:
     - `/authorize` returns redirect with PKCE
     - email send/verify succeeds
     - token refresh works (cookie + localStorage hydration if applicable)
  2. **Define an auth compatibility matrix** (Steward SDK version × required params × tenant header rules). Store it in-repo and pin versions intentionally.
  3. **Add runtime alerting:** detect spikes in 400s on auth routes; page the oncall/dev owner.

**Resource ask (1 week):** 1 senior engineer (cloud) + 0.5 frontend engineer.

---

### Priority 1 — Onboarding: reduce newcomer overwhelm, increase activation
**Why now:** Community signals show “where do I start?” is the dominant friction; high velocity makes this worse.

- **Recommendations**
  1. Ship a **“First 60 Minutes with ElizaOS”** path:
     - Install → run local agent → open UI → enable one plugin → complete one task → (optional) cloud sign-in
  2. Create 3 curated tracks:
     - **Agent User** (daily usage, prompts, safety)
     - **Plugin Developer** (hello-plugin, actions/views, testing)
     - **Cloud App Builder** (container deploy loop, billing basics)
  3. Add an **LLM-assisted learning posture** explicitly (since community already uses it): provide “copy/paste prompts to ask your LLM” for debugging/setup.

**Resource ask (1–2 weeks):** 1 DX/doc owner + rotating eng reviewers.

---

### Priority 2 — Ecosystem growth: convert buildathon interest into shipped integrations
**Why now:** External demand for Eliza-experienced devs is a leverage point; without structure it dissipates.

- **Recommendations**
  1. Stand up a **Buildathon Support Kit**:
     - reference agent templates (DeFi portfolio bot, data fetcher, alerting agent)
     - security checklist (keys, scopes, spending caps)
  2. Create a lightweight **“Looking for contributors”** process:
     - Discord channel or pinned thread + minimal vetting + expected response SLA
  3. Track conversion metrics: #requests → #matches → #repos launched → #registry entries.

**Resource ask (ongoing):** 0.25 community lead + 0.25 eng advocate.

---

### Priority 3 — Manage technical debt from rapid surface expansion (prevent “velocity tax”)
**Why now:** Massive refactors and UI expansions increase regression probability; auth regressions already indicate gating gaps.

- **Recommendations**
  1. Institute a **“blast radius” rubric** for merges (auth, onboarding, routing, billing) requiring stronger tests and staged rollout notes.
  2. Add a weekly **stability budget**: allocate fixed % of merges to bugfix/CI hardening.
  3. Formalize “big merge” strategy: break multi-million-line sync merges into smaller, reviewable phases when possible.

**Resource ask:** engineering management + CI owner alignment.
