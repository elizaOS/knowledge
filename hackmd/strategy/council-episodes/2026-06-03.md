## 1) Episode Overview
- **RETRO-2026-04 — Monthly Retro: April 2026 (episode-retro-2026-04)**  
  April showed strong engineering momentum (parallel action execution merged; database refactor underway; early marketplace/payment/memory and safety/spend proposals), but **community confidence hit a low** due to token decline, perceived strategic drift, and repeated **developer onboarding failures** (notably macOS/Bun postinstall). The council aligned that **reliability + developer experience (DX)** must become the immediate priority, and that **security + economic/spend governance** are now inseparable from the core framework narrative.

---

## 2) Key Strategic Themes
- **Reliability and DX as the gating constraint**
  - “If `elizaos create` fails, nothing else matters” became the dominant strategic framing.
  - New capabilities (concurrency, refactors, plugins) don’t register as progress if first-run onboarding is brittle.

- **V2 stability vs. V3 acceleration**
  - V3 work is justified, but **abandoning v2 integrators** is actively fragmenting the ecosystem.
  - Need a **clear v2 support policy** and an explicit **v3 release criteria roadmap** to restore predictability.

- **Economic autonomy requires secure defaults**
  - Identity, spend controls, reputation, and transaction safety are no longer “optional plugins later.”
  - Agents touching real money implies ElizaOS must behave like a **security product** with safe-by-default governance.

- **Security posture as table stakes**
  - Even without incidents, the expectation is formalized processes: disclosure, supply-chain hygiene, automated scans, and dependency discipline.

- **Community trust and incentive alignment**
  - Token collapse created a **contributor retention risk** (forced sell pressure, morale loss).
  - Comms confusion (“core vs side quests”) amplified negative sentiment; cadence and clarity are required to halt the “death spiral” narrative.

---

## 3) Important Decisions / Insights
- **May priorities were explicitly narrowed to three “stop-the-line” initiatives**
  1. **DX/onboarding stabilization (v2 LTS patch set)**  
     - Treat onboarding failures as incidents; make macOS create→run path deterministic and CI-verified.
  2. **Roadmap + support policy clarity**  
     - Publish a one-page **v3 roadmap with exit criteria** (tests, perf targets, migration guide) and a **v2 support window** with defined patch scope.
  3. **Security + spend governance MVP as safe defaults**  
     - Ship **fail-closed** spend controls and a reference implementation developers can safely copy.

- **Strategic deferral**
  - **Broad marketplace ambitions** should be deferred until the core “install → run → deploy” path is stable and trusted.

- **Narrative correction**
  - Reaffirm the North Star but **foreground “boring reliability” and “secure economic autonomy.”**
  - Clarify positioning: **ElizaOS = agent framework + cloud platform**; adjacent experiments should be framed as **downstream adopters**, not competing missions.

- **Trust restoration requires operational transparency**
  - Commit to **weekly ship notes + monthly retros** with measurable status, plus a “human” tone that acknowledges breakages and shows verification paths (dashboards/badges).

- **Compensation and incentives are operational dependencies**
  - Council consensus: without a compensation policy reset (e.g., partial stable/hedged payouts), execution excellence will degrade regardless of architecture quality.

---

## 4) Community Impact (elizaOS Ecosystem)
- **Builders**
  - A stable, CI-verified onboarding path and v2 LTS policy reduces churn, lowers support burden, and restores confidence for integrators shipping on top of v2.
  - Clear v3 criteria reduces fear of endless refactors and ecosystem “abandonware” perception.

- **Users and agent operators**
  - Shipping spend governance and security baselines as defaults makes agent commerce safer, reducing the risk of high-profile failures that would harm the brand.

- **Contributors**
  - Compensation policy updates and a predictable comms cadence directly address retention risk and can stabilize PR throughput.

- **Ecosystem narrative**
  - Moving from “shipping many things” to “shipping the right foundations” aims to reverse sentiment decay and replace rumor-driven churn with measurable progress.

## 5) Action Items
- **Onboarding & DX (v2 LTS)**
  - Fix **macOS/Bun postinstall** create failures; make the “create → run” path reproducible in CI.
  - Launch an **Install Health dashboard** (green/red matrix by OS/runtime) and track weekly.
  - Refresh docs to match reality; eliminate undocumented setup patterns (e.g., Socket.IO usage).

- **Roadmap & Release Governance**
  - Publish a **one-page v3 roadmap** with alpha/beta dates and exit criteria (test coverage, performance targets, migration guide).
  - Publish a **v2 support/LTS policy** with a defined patch scope and support window.

- **Security Baselines**
  - Add **SECURITY.md** and a responsible disclosure process.
  - Add CI automation for **dependency scanning + SBOM**.
  - Ship at least one **fail-closed spend policy preset** (“off by exception”).

- **Spend Governance Reference Implementation**
  - Deliver a reference agent template with:
    - **Allowlisted actions**
    - **Transaction simulation**
    - **Risk checks** (honeypot/rug heuristics)
    - Documented extension points

- **Contributor Compensation & Comms Reset**
  - Announce a compensation policy update (e.g., **partial stablecoin / hedged payouts / milestone conversion**) with a timeline.
  - Commit to **weekly ship notes + monthly retro cadence**, and track contributor retention + PR throughput month-over-month.
