## Episode Overview (2026-06-09)
- **RETRO-2026-04 — Monthly Retro: April 2026 (episode-retro-2026-04)**
  - April showed a sharp divergence: **real engineering momentum** (parallel action execution, DB refactor trajectory, early safety/spend governance work, marketplace/payment/memory plugin proposals, early one-click hosting wins) vs. **community confidence hitting a sentiment floor** driven by token decline, onboarding/DX breakages, and perceived strategic drift.
  - Council conclusion: the project’s next compounding gains depend on making **reliability + developer onboarding “boringly stable”**, while treating **security + agent economics** as inseparable from the core framework narrative.

---

## Key Strategic Themes
- **Reliability & DX as the primary growth gate**
  - Capability gains are being discounted because **first-run experience is fragile** (e.g., macOS Bun postinstall blocking `elizaos create`).
  - Council framed onboarding failures as **“stop-the-line incidents”** that directly erode trust and slow ecosystem building.

- **V2 stability vs. V3 acceleration (avoiding ecosystem abandonment)**
  - V3 exploration is valid, but council highlighted ecosystem risk: **v2 integrators feel stranded**, creating fragmentation.
  - Need an explicit **v2 support policy** (LTS-like minimal patch scope + window) to protect builders.

- **Security + spend governance becoming “core product,” not optional plugins**
  - Autonomous agents interacting with real money require **safe defaults**, not optional complexity.
  - “Fail-closed” spend controls, disclosure processes, and supply-chain hygiene are now table stakes for credibility.

- **Token decline → contributor retention + execution risk**
  - Paying contributors in a rapidly declining token creates **forced sell pressure and morale damage**, increasing attrition risk and slowing throughput—fueling a perceived “death spiral.”

- **Narrative coherence & comms cadence**
  - Confusion around what is “core” vs. “side quests” amplifies mistrust during downturns.
  - Council emphasized that comms must be **measurable, time-bound, and consistent** (weekly ship notes + monthly retros).

---

## Important Decisions / Insights
- **Strategic prioritization for the next month**
  - If limited to a few initiatives, council aligned on:
    1. **DX/onboarding stability first** (especially macOS create→run path).
    2. **Roadmap clarity**: v3 release criteria + dates, plus a **v2 support window**.
    3. **Security + spend governance MVP** as **reference implementation and safe defaults**.
  - Explicit deferral: **broad marketplace ambitions** should wait until the “core path is stable and trusted.”

- **DX determinism as a credibility gate for V3**
  - V3 credibility depends on proving **deterministic onboarding and install success**; otherwise “foundations won’t matter.”

- **Reframing the North Star**
  - Keep ambition, but foreground two requirements as first-class:
    - **“Boring reliability”**
    - **“Secure economic autonomy”**
  - Clarification proposed: **ElizaOS = agent framework + cloud platform**; experiments should be framed as **downstream adopters**, not competing missions.

- **Transparency as a trust mechanic**
  - Council stressed “proof, not claims” via public artifacts:
    - Install Health dashboard
    - Version support matrix
    - SECURITY.md + automated scanning outputs
    - Regular ship notes with metrics

## Community Impact (elizaOS ecosystem)
- **Builders**
  - A stabilized create→run path and clearer v2/v3 compatibility policy reduces wasted setup time, lowers support load, and increases the odds that new developers become long-term contributors.
- **Integrators and plugin authors**
  - A v2 support window + explicit patch scope reduces fragmentation and risk of “abandonware” perception, enabling ecosystem products to plan releases and maintenance confidently.
- **Users of agent commerce / real-money agents**
  - Shipping spend governance as a reference template (allowlists, simulation, risk checks, fail-closed presets) improves safety posture and reduces catastrophic failure risk.
- **Contributors**
  - A compensation policy reset (partial stable/hedged payouts) directly addresses retention and throughput risk, improving delivery reliability.
- **Wider community sentiment**
  - A predictable comms cadence (weekly ship notes + monthly retro) and clearer mission boundaries can counter rumor-driven churn and the “death spiral” narrative.

## Action Items
- **Onboarding & DX stabilization (v2 LTS patch set)**
  - Fix **Bun postinstall** and other macOS first-run blockers; treat `elizaos create` failures as stop-the-line.
  - Add **CI that reproduces macOS (Apple Silicon) create→run**, and publish a weekly **“Install Health” dashboard** (goal: reduce fresh-install failures by ~80%).

- **Roadmap + release discipline**
  - Publish a **one-page v3 roadmap**: alpha/beta target dates, exit criteria (test coverage, performance targets), and a migration guide plan.
  - Publish a **v2 support policy**: support window + defined patch scope (“v2 LTS” concept) to protect integrators.

- **Security baselines + spend governance defaults**
  - Add **SECURITY.md** and responsible disclosure process.
  - Implement automated **dependency scanning/SBOM** in CI.
  - Ship at least one **fail-closed spend policy preset** (off-by-exception), plus a reference agent template:
    - allowlisted actions
    - transaction simulation
    - risk checks (e.g., honeypot/rug heuristics)
    - documented extension points

- **Contributor incentives + communication reset**
  - Announce a **compensation policy update timeline** (e.g., partial stablecoin / hedged / milestone-based conversion).
  - Commit to **weekly ship notes** + **monthly retro cadence**; track contributor retention and PR throughput month-over-month.
  - Publish a public, human-toned status note acknowledging onboarding breakage and specifying how fixes will be verified (dashboard/badge).
