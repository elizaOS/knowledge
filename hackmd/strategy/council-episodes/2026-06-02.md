## 1) Episode Overview
- **RETRO-2026-04 — Monthly Retro: April 2026**
  - April showed **real engineering momentum** (parallel action execution, database refactor progress, early safety/spend governance proposals, marketplace/payment/memory plugin experimentation) but hit a **community trust low** driven by token decline, onboarding/DX breakages, and perceived strategic drift.
  - Council consensus: the ecosystem is entering an era where **reliability, developer experience, security, and economics are inseparable**—especially as agents touch real money.

## 2) Key Strategic Themes
- **Reliability + DX as the adoption gate**
  - Repeated message: *“If `elizaos create` fails, nothing else matters.”*
  - macOS (Apple Silicon) onboarding regressions (notably Bun postinstall behavior) are treated as **stop-the-line incidents**.
- **V2 stability vs V3 acceleration**
  - V3 work is valid, but **leaving v2 integrators stranded** is actively eroding trust and fragmenting the ecosystem.
  - Need a **defined v2 support policy** and explicit v3 release criteria to restore predictability.
- **Security + spend governance as core, not “later plugins”**
  - Identity, spend controls, reputation, and “fail-closed” policies are now prerequisites for credible autonomous agents.
  - Security expectations rising: disclosure process, dependency hygiene, supply-chain hardening.
- **Token collapse → contributor retention and execution risk**
  - Paying contributors in a rapidly declining token creates forced sell pressure and morale damage; threatens throughput regardless of architecture quality.
- **Narrative coherence and comms cadence**
  - Community confusion about what is “core ElizaOS” vs adjacent experiments is compounding negativity; roadmaps and status cadence are now strategic infrastructure.

## 3) Important Decisions / Insights
- **May priority stack (explicitly chosen as the “three slots”)**
  1. **Onboarding/DX stability (v2 LTS patch set)**
     - Fix Bun postinstall/macOS create→run path; add CI reproduction and visibility.
  2. **V3 roadmap + release criteria + v2 support window**
     - Publish a one-page plan with dates, exit criteria (tests/perf/migration guide), and a clearly bounded v2 LTS scope.
  3. **Security baseline + spend governance MVP as safe defaults**
     - Ship at least one **fail-closed** spend policy preset and a reference agent template (allowlists + simulation + risk checks).
- **Strategic deferral**
  - Council explicitly deprioritized **broad marketplace ambitions** until the “core path” is stable and trusted.
- **Reframed North Star**
  - Keep the mission, but foreground: **“boring reliability” + “secure economic autonomy.”**
  - Clarify positioning: **ElizaOS = agent framework + cloud platform;** experiments should be framed as **downstream adopters**, not competing missions.

## 4) Community Impact
- **Builders**
  - A stable first-run path and a v2 LTS policy reduce churn, support load, and “install roulette,” improving contributor conversion and plugin ecosystem health.
- **Integrators and ecosystem projects**
  - A published compatibility/support window prevents fragmentation and protects teams building on v2 while v3 evolves.
- **Security and reputation**
  - Shipping formal security processes and safe-by-default spend controls reduces the probability of a trust-destroying incident as agents handle funds.
- **Contributors**
  - A compensation reset (e.g., partial stable payouts/hedging) is positioned as essential to prevent a “death spiral” of declining token → attrition → slower shipping → worse sentiment.
- **Wider narrative**
  - More predictable comms (weekly ship notes + monthly retros) aims to replace rumor cycles with measurable progress and receipts.
