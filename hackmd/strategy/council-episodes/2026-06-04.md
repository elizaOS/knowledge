## 1. Episode Overview
- **RETRO-2026-04 — Monthly Retro: April 2026 (episode-retro-2026-04)**
  - April showed **real engineering momentum** (parallel action execution merged, ongoing database refactor, early safety/spend governance direction, marketplace/payment/memory plugin work) but also a **community confidence low** driven by token decline, onboarding/DX breakages, and perceived strategic drift.
  - Council conclusion: **reliability + developer experience are still the gating constraints**, and **economic/security primitives are now inseparable from the core framework story**.

## 2. Key Strategic Themes
- **“Boring reliability” as the highest-leverage growth lever**
  - New capabilities are being discounted because first-run experience is fragile; reliability regressions erase perceived progress.
- **V2 stability vs V3 acceleration (and ecosystem protection)**
  - V3 work is justified, but **leaving v2 integrators stranded** creates fragmentation and trust loss; support policy must be explicit.
- **Security + spend governance as defaults, not optional plugins**
  - As agents touch real money, “agent frameworks become security products.” Safe, fail-closed defaults must ship with clear reference patterns.
- **Economic/identity layers becoming core platform primitives**
  - Identity, reputation, and spend controls are prerequisites for autonomous agents; they cannot be postponed as “later plugins.”
- **Contributor incentives and communications as execution infrastructure**
  - Token-denominated compensation volatility threatens retention and throughput; missing clarity/cadence amplifies “death spiral” narratives.

## 3. Important Decisions / Insights
- **Stop-the-line priority: onboarding must work**
  - Council aligned that **`elizaos create` failing (macOS/Bun postinstall)** is a critical incident-level blocker; fix before celebrating architectural wins.
- **Define and publish a v2 support policy (“v2 LTS patch set”)**
  - Commitment direction: protect integrators via a **minimal, scoped patch window** and documented patterns; reduce perception that v2 is abandonware.
- **Publish v3 roadmap with explicit release criteria**
  - One-page roadmap required: **alpha/beta dates + exit criteria** (test coverage, perf targets, migration guide) to prevent “endless refactor” fear.
- **Adopt formal security baselines**
  - Immediate baseline items: **SECURITY.md + responsible disclosure**, automated dependency scanning/SBOM in CI, and at least one **fail-closed spend policy preset**.
- **Ship spend governance as a reference template**
  - Deliver a copyable “safe agent” template: **allowlisted actions, transaction simulation, risk checks (honeypot/rug heuristics), and clear extension points**.
- **Compensation + comms reset to stabilize contributors and narrative**
  - Direction: update compensation policy (e.g., **partial stablecoin/hedged payouts**) and commit to **weekly ship-notes + monthly retros** with measurable progress.

## 4. Community Impact
- **Developer adoption and ecosystem health**
  - Stabilizing install/create/run flows (especially macOS) directly improves: builder retention, plugin adoption, and reduced support burden.
- **Reduced fragmentation risk**
  - A clear **v2 support window + v3 criteria** helps third-party builders plan, reduces version/branch confusion, and prevents ecosystem splits.
- **Higher trust for real-money agent use cases**
  - Shipping **safe defaults (fail-closed spend governance) + disclosure process** raises confidence for deploying autonomous agents with payments.
- **Improved contributor sustainability**
  - A compensation policy that reduces forced selling pressure and uncertainty supports consistent throughput, faster fixes, and healthier governance dynamics.
- **Narrative coherence**
  - Clarifying “ElizaOS = agent framework + cloud platform; adjacent experiments are downstream adopters” reduces confusion and rumor-driven churn.
