## 1) Episode Overview
- **RETRO-2026-04 — Monthly Retro: April 2026 (episode-retro-2026-04)**
  - April showed **real engineering momentum** (parallel action execution, runtime/database refactor work, safety/spend governance proposals, marketplace/payment/memory plugins) but also a **trust and sentiment crisis** driven by token decline, perceived strategic drift, and repeated **developer onboarding/DX failures** (notably macOS/Bun postinstall blocking `elizaos create`).
  - Council consensus: **reliability + “boring” onboarding** must lead; **economic autonomy requires security defaults**; and **comms + contributor incentives** must stabilize to prevent a “death spiral” narrative.

## 2) Key Strategic Themes
- **Reliability & Developer Experience as the primary growth gate**
  - “If `elizaos create` fails, nothing else matters” became the stop-the-line principle.
  - Regressions (macOS install) negate perceived progress and drive rumor/sentiment spirals.

- **V2 stability vs. V3 acceleration (support window + release discipline)**
  - V3 work is justified, but **abandonware perception** for v2 integrators is fragmenting the ecosystem.
  - Need explicit **v2 support policy** (scope + window) and **v3 release criteria** to restore predictability.

- **Security + spend governance are now core, not optional**
  - Agents touching real money changes expectations: the framework must ship with **safe defaults**, not “security as a plugin later.”
  - Formal disclosure, supply-chain hygiene, and automated scanning are table stakes.

- **Economic narrative and security narrative are inseparable**
  - Identity, spend controls, and reputation must be treated as **prerequisites** for autonomous agents operating in production environments.

- **Token collapse → contributor retention and execution risk**
  - Token-denominated compensation creates forced selling and morale damage, directly impacting throughput and quality.
  - Without incentive stability, engineering wins will not compound.

- **Comms cadence and mission clarity**
  - Confusion about “what ElizaOS is” vs adjacent experiments increases mistrust when price is down.
  - Need consistent ship notes + roadmap messaging; frame experiments as “powered by ElizaOS,” not competing priorities.

## 3) Important Decisions / Insights
- **May priorities were explicitly narrowed to three non-negotiables**
  - **(1) DX/onboarding stabilization** (macOS/Bun postinstall fix, CI reproducibility, “known-good” install path, docs that match reality).
  - **(2) Roadmap clarity** (one-page v3 roadmap with alpha/beta dates + exit criteria; v2 support window with LTS-like patch scope).
  - **(3) Security + spend governance MVP** as **safe defaults** (fail-closed presets + reference implementation).

- **Explicit deferral decision**
  - Council agreed to **defer broad marketplace ambitions** until the core “create → run” path is stable and trusted.

- **Adopt measurement and public proof, not just fixes**
  - Create an **“Install Health” dashboard** (green/red matrix by OS/runtime) and CI gates to prove stability improvements weekly.

- **Security program baseline commitments**
  - Add **SECURITY.md**, responsible disclosure process, automated dependency scanning/SBOM in CI, and at least one **fail-closed spend preset**.

- **Spend governance must ship as a copyable reference template**
  - Deliver a “safe default agent template” featuring:
    - allowlisted actions
    - transaction simulation
    - risk checks (e.g., honeypot/rug heuristics)
    - documented extension points

- **Compensation and comms are treated as execution infrastructure**
  - Announce a **compensation policy update** (e.g., partial stablecoin/hedged payouts) and commit to **weekly ship-notes + monthly retros** to counteract “death spiral” dynamics.

## 4) Community Impact
- **For builders/integrators**
  - Clear v2 support scope + improved install determinism reduces fragmentation and lowers “time-to-first-success,” directly improving retention and plugin ecosystem health.

- **For security-conscious users and partners**
  - Formal disclosure + supply-chain hygiene + fail-closed spend defaults reposition ElizaOS as **production-grade agent infrastructure**, not just experimentation tooling.

- **For contributors**
  - More stable, transparent compensation improves retention and reduces execution risk during market downturns.

- **For the broader narrative**
  - A consistent North Star (“agent framework + cloud platform” with “secure economic autonomy” and “boring reliability”) reduces rumor churn and aligns experiments as downstream adopters.
