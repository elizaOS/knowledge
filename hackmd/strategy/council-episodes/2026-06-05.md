## 1) Episode Overview
- **RETRO-2026-04 — Monthly Retro: April 2026 (episode-retro-2026-04)**  
  April showed a split reality: **real engineering momentum** (parallel action execution, database refactor progress, safety/spend governance proposals, marketplace/payment/memory plugin work) alongside a **community confidence low** driven by token decline, onboarding failures, and perceived strategic drift. The council aligned that **reliability + developer experience (DX)** are still the primary adoption gates, and that **economic autonomy + security** must now be treated as inseparable from the core framework narrative.

## 2) Key Strategic Themes
- **“Reliability and DX are the product”**
  - New capabilities don’t translate into adoption if first-run experience fails (e.g., `elizaos create` blockers).
  - Council framing: *fewer public “big ideas” until onboarding is boringly stable*.
- **V2 stability vs. V3 acceleration**
  - Ecosystem builders feel stranded when attention shifts to v3 without a clear support policy.
  - Need explicit **v2 support window** and a minimal **v2 LTS patch scope** to prevent fragmentation.
- **Economic autonomy requires default safety**
  - Identity, spend controls, and reputation are no longer “optional plugins later”; they are prerequisites for agents touching real money.
  - Strategic requirement: ship as **safe defaults** (fail-closed), not optional complexity.
- **Security posture is now table stakes**
  - Even without incidents, stakeholders expect formal disclosure, dependency hygiene, and supply-chain hardening.
- **Community trust and narrative coherence**
  - Confusion about what “ElizaOS” is (core vs. experiments) amplifies negative sentiment during token drawdowns.
  - Clear roadmaps and predictable comms cadence are positioned as trust infrastructure.
- **Contributor retention and compensation risk**
  - Token-denominated compensation during steep decline creates forced sell pressure → morale damage → throughput drop (“death spiral” dynamics).

## 3) Important Decisions/Insights
- **Stop-the-line priority: onboarding**
  - Explicit stance: if `elizaos create` fails, everything else is downstream noise.
  - Proposed measurable goal: **reduce fresh-install failures by 80%**, with macOS (Apple Silicon) success reproducible in CI and visible publicly.
- **Lock three near-term priorities (and defer the rest)**
  - Council consensus for the next cycle:
    1) **DX/onboarding stability (v2 LTS patch set)**  
    2) **Clear v3 roadmap + v2 support policy**  
    3) **Security baseline + spend governance MVP as safe defaults**
  - **Defer** broader marketplace ambitions until the “core path is stable and trusted.”
- **V3 credibility depends on determinism**
  - “DX determinism” was treated as a gating constraint: v3 cannot be credible if v2 entry ramps remain unstable.
- **Security and spend governance must ship as a reference path**
  - Shift from scattered plugins to a **reference agent template** that developers can copy safely:
    - allowlisted actions
    - transaction simulation
    - risk checks (e.g., honeypot/rug heuristics)
    - documented extension points
- **Operational security baseline commitments**
  - Adopt **SECURITY.md**, responsible disclosure, automated dependency scanning/SBOM in CI.
  - Ship at least one **fail-closed spend policy preset** (“off by exception”).
- **Comms reset: measurable, honest, time-bound**
  - Publish a **one-page plan** with metrics and owners; explicitly acknowledge breakages and provide verification methods (e.g., install health dashboard).
- **Compensation policy update is strategic, not HR**
  - Council insight: execution excellence is constrained by incentive stability; commit to a policy direction (partial stable payouts/hedging/milestone conversion) even if exact numbers aren’t public yet.

## 4) Community Impact (elizaOS ecosystem)
- **Builders**
  - Short term: clearer expectations via v2 LTS scope, install-health verification, and a stable “create → run” golden path.
  - Medium term: reduced ecosystem fragmentation once version/support policies and v3 criteria are explicit.
- **Plugin authors and integrators**
  - Benefit from standardization around security + spend governance patterns and safer defaults, reducing ad-hoc “lore-based” integration.
- **Agent deployers (real money / real ops)**
  - Higher trust ceiling if “fail-closed” spend governance, disclosure processes, and dependency hygiene become default.
- **Community sentiment and participation**
  - Predictable ship notes + retros can dampen rumor cycles; visible install-health and security baselines create “proof over promises.”
- **Contributors**
  - Retention risk is directly tied to compensation stability; lack of action here threatens PR throughput and support responsiveness.
