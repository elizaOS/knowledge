# User Feedback Report — 2026-06-18 (elizaOS)

## 1) Pain Point Categorization (Top Recurring 5–7)

> Basis: Discord discussions from 2026-06-15 to 2026-06-17 + top GitHub issues/threads surfaced in the June repo summary. Quantification below is relative to the *distinct substantive feedback threads* observed in this slice (not total chat volume).

### A. Community (Moderation, safety, signal-to-noise)
**Frequency/Severity:** High frequency, high severity for engagement quality  
- **Scam/spam flooding in core channels** (#coders and #discussion): repeated wallet-purchase spam (e.g., “buy old Solana wallets”) and fake airdrop posts. This dominated the #coders channel segment (effectively **~100% of visible activity** in that snippet).
- **Moderation workflow regression:** removal of the dedicated “minimod” channel forces tagging scammers in main channels (om1d_sa, odilitime), increasing disruption and moderator workload.

**Most affected users:** newcomers looking for help in #coders; contributors trying to hold technical discussions.

---

### B. Documentation / Communication (Project direction, commercialization clarity)
**Frequency/Severity:** Medium frequency, high severity (trust/strategy confusion)  
- **Confusion about open-source strategy vs monetization path:** ghss44 questioned whether “fully open source” is viable in 2026 and asked for clarity on direction/resources; ivgamer3467 referenced a commercialization partnership, but details weren’t readily accessible/linked.
- **Expectation gap on “direction + resourcing” post-migration:** concerns referenced “a16z lawsuit threats” and implied funding uncertainty; no canonical explainer was provided in-channel.

**Observed rate:** **2 of 3** major Discord product/meta threads (≈**67%**) were primarily about strategy/commercialization clarity rather than building with the framework.

---

### C. Integration (Auth/SSO dependency alignment across services)
**Frequency/Severity:** Medium frequency, very high severity when it occurs  
- GitHub shows recent critical issues around **Steward OAuth/PKCE alignment**, where cloud login broke in prod and blocked “Sign in with Eliza Cloud” across surfaces (#8257, later fixed). Even though resolved, it highlights a systemic integration risk: multi-surface auth flows drifting out of sync.

**Most affected users:** third-party app developers integrating Eliza Cloud sign-in; all cloud dashboard users during incident windows.

---

### D. Technical Functionality (Cloud Apps go-live readiness gaps)
**Frequency/Severity:** Medium frequency, very high severity (go-live blockers)  
- A detailed audit issue highlights **Apps data-plane ingress/scaling limits** and “unreachable apps until ingress exists,” plus DB connection ceilings and env-var contract risks (e.g., `DATABASE_URL` vs `POSTGRES_URL`) (#8321).
- While many follow-up PRs exist, user-facing impact is straightforward: users expect “deploy app → get URL,” and any ingress/runtime ceiling breaks the core promise.

**Most affected users:** cloud app deployers (Product 2), enterprise evaluators.

---

### E. Documentation / Support Policy (Token migration finality + scam confusion)
**Frequency/Severity:** Medium frequency, high severity for affected holders and for security posture  
- **Late token migration question unanswered:** ryan067828 asked if any appeal/alternative migration exists after missing the AI16Z→ELIZAOS deadline; no authoritative response in-thread on 06-17.
- **Scam surface area:** community had to repeatedly clarify “official window closed; links are scams” (06-15/06-17), implying missing canonical “migration status” page and in-Discord pinned guidance.

**Observed rate:** token migration concerns appeared on **2 of 3 days** (06-15 and 06-17) in this dataset slice.

---

### F. Community / UX (Contribution friction: automod blocking proposals)
**Frequency/Severity:** Lower frequency, medium severity  
- Users attempting to post proposals were blocked by automod and redirected to DMs (06-15). This reduces transparency, discourages participation, and makes proposals harder to review/track.

---

### G. Developer Experience (Build/release pipeline sharp edges)
**Frequency/Severity:** Medium frequency in GitHub, medium-to-high severity for maintainers  
- Examples from June issues include packaging/release workflow failures due to missing required secrets (#7837) and runtime dependency externalization causing `ERR_MODULE_NOT_FOUND` outside Bun’s loader (#8142). These are “paper cuts” for contributors and release managers, and they amplify “project maturity” concerns voiced on Discord.

## 2) Usage Pattern Analysis (Actual vs Intended)

### What users are doing (observed)
- **Using Discord as both support channel and governance/strategy forum**: the open-source vs monetization debate (ghss44, ivgamer3467, ross.ross.ross) shows users treating Discord as the primary place to validate project direction.
- **Treating token migration as a supportable workflow**: late migrators ask for “manual appeal / alternate contract pool,” implying expectations of an ongoing customer-support-like process rather than a one-time on-chain event with strict finality.
- **Using #coders as a default help channel, but it’s currently a high-scam surface**: scammers target “builder” spaces because that’s where high-intent users congregate.

### Intended usage (implied by project direction)
- Developers build and deploy agents/apps via elizaOS + Eliza Cloud, with GitHub issues/PRs as the main engineering feedback loop, and Discord as community support.

### Emerging / unexpected use cases
- **Governance & delegation frameworks**: the Semantic Delegation + ERC-7710 + knowledge graph proposal (06-15) indicates demand for *agent delegation primitives* that extend beyond “chat agent” into “delegated authority systems” (liquid democracy, caveat enforcers).
- **Enterprise commercialization interest**: users ask for details on the Secure Blockchain Development Corp. partnership; this is less “open-source hobby” and more “platform + enterprise packaging.”

### Feature requests that align with actual usage
- Clear demand for:
  1. **Canonical commercialization + roadmap explainer** (users are already debating it).
  2. **Token migration status + scam-resistant guidance** (users are actively at risk).
  3. **Discord moderation tooling/workflows** (spam directly blocks real usage).

## 3) Implementation Opportunities (Concrete Solutions, Prioritized)

Below, each major pain point includes 2–3 implementable solutions, ordered by **(Impact / Difficulty)**.

### 1) Scam/spam flooding + moderation workflow regression (Community)
**High impact / Low–Medium difficulty**
1. **Reintroduce a dedicated moderation intake surface**  
   - Bring back a `#minimod-log` (read-only to most) or `#report-scams` channel with a bot that forwards reports to mods, avoiding main-channel tagging.  
   - Similar approach: many OSS Discords use a single “report” channel + mod-only threads for triage.
2. **Harden Discord AutoMod + channel permissions for high-signal areas**  
   - AutoMod rules for common scam patterns (airdrop keywords, “buy wallets,” external links from new accounts).  
   - Add **minimum account age** and/or **server membership duration** for posting links in #coders.
3. **Add a “Safe Links + Official Notices” pinned post and enforce link hygiene**  
   - A single pinned message: “No active migration links; official domains only; report via /report.”  
   - Pair with a bot that replies when “migration” keyword appears, pointing to the canonical page.

---

### 2) Token migration late-holder confusion + scam susceptibility (Documentation/Support Policy)
**High impact / Low difficulty**
1. **Publish a canonical “Migration Status” page + Discord command** (`/migration`)  
   - Must answer: window closed? any appeals? who to contact? how to verify official comms?  
   - Add a self-serve checklist: “If you see a link, it’s a scam; only trust X domain; never DM keys.”
2. **Add a deterministic on-chain verification tool**  
   - A small web page: paste wallet address → shows whether migrated / eligible / deadline status (read-only).  
   - Similar pattern: major token projects ship an “eligibility checker” to reduce support load and phishing risk.
3. **If appeals are impossible, say so explicitly—and explain why**  
   - Users are asking for a manual process; if it cannot exist, the absence needs a clear, repeated rationale to prevent repeated questions and reduce scam conversion.

---

### 3) Open-source strategy vs monetization direction confusion (Communication gap)
**High impact / Low–Medium difficulty**
1. **One-page “How elizaOS funds itself” + “What is commercialized” explainer**  
   - Include: what remains OSS, what is paid (Cloud hosting, enterprise services), how the Secure Blockchain Development Corp. partnership fits, and how community contributions map to impact.
2. **Monthly roadmap + “decision log” posts** (Discord + GitHub Discussions)  
   - Turn recurring debates into a maintained artifact: decisions, tradeoffs, what changed since last month.
3. **Add “What’s not true” section**  
   - Directly address rumors referenced in chat (“lost funds,” “lawsuit threats”) with either confirmation, denial, or “cannot comment” + what users should assume operationally.

*Comparable approach:* projects like Kubernetes/Helm ecosystems maintain governance + funding pages and short “project updates” to prevent strategy debates from restarting weekly.

---

### 4) Cloud Apps go-live readiness gaps (Technical Functionality / Performance)
**High impact / Medium difficulty**
1. **Ship a public “Apps Beta readiness checklist” tied to tracked issues**  
   - Translate #8321-style audits into user-facing readiness: ingress, DB pooling, multi-node routing, billing idempotency, etc.  
   - Prevents user expectations of “1000-app scale” while in “dozens” territory.
2. **Add guardrails in-product**  
   - If the system’s ceiling is “dozens,” enforce quotas and show explicit warnings in UI: “Beta capacity; may queue deployments; expect limited concurrency.”
3. **Operationalize scaling primitives early (pooling + ingress + placement)**  
   - Prioritize: ingress installation, connection pooling (pgbouncer), and multi-node placement logic—these directly map to “app reachable + stable.”

*Comparable approach:* platforms like Fly.io/Render expose clear resource limits + deployment health checks early to keep beta trust.

---

### 5) Proposal submission friction due to automod (Community / UX)
**Medium impact / Low difficulty**
1. **Create a structured “Proposals” intake (GitHub Discussions template or form)**  
   - Fields: problem, scope, budget, timeline, links, contact.  
   - Avoid DMs as the default: DMs don’t scale and exclude community review.
2. **Tune automod for proposal channels**  
   - Allow longer posts / links for verified users or users with specific roles; route first-time posters through a guided template instead of blocking.

---

### 6) Build/release sharp edges (Technical Functionality / Integration)
**Medium impact / Medium difficulty**
1. **“Release readiness” CI gate that detects missing secrets/config before tag**  
   - If certain publish targets are not configured, auto-skip them rather than failing the entire orchestrator workflow (pattern suggested in #7837).
2. **Standardize runtime dependency bundling rules across packages**  
   - Avoid “works only with Bun loader” surprises by bundling critical transitives or hoisting runtime deps (as discussed in #8142).
