## 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

### A) Speculative decoding incompatibilities (M-RoPE + SWA/DFlash)
**High impact / Medium difficulty**
1. **Fail loudly + expose runtime status**
   - If `--spec-type dflash` is requested but incompatible, **abort load** (or require `--allow-spec-fallback`) and surface:
     - `spec_decode_active: false`
     - `spec_decode_reason: "SWA partial sequence removal unsupported"`  
   *Comparable approach:* many inference servers (e.g., TensorRT-LLM / vLLM-style configs) fail fast when an optimization path can’t be activated, rather than silently falling back.

**High impact / High difficulty**
2. **Implement SWA-compatible partial sequence removal (or alternative rollback strategy)**
   - Port/replicate the SWA support needed for partial sequence rollback (as referenced in #7635 via spiritbuun fork lineage).
   *Comparable approach:* llama.cpp ecosystem often carries per-architecture KV-cache logic; upstreaming a clean capability flag + implementation avoids “silent no-op optimizations.”

**Medium impact / Low difficulty**
3. **Catalog/recommender gating**
   - Add explicit capability requirements per tier (e.g., `mrope-spec-decode`, `swa-dflash-supported`) and prevent recommending large tiers that only make sense with speculation on iGPU-class machines.
   *Comparable approach:* Kubernetes-style “node selectors/taints” pattern—don’t schedule workloads onto nodes lacking required capabilities.

---

### B) Planner tool misrouting + shell timeouts + metadata leakage (#7935)
**High impact / Low-to-medium difficulty**
1. **Planner must treat Stage-1 `candidateActions` as a priority signal**
   - Update planner prompt + scoring so `candidateActions=[SEARCH_MESSAGES]` becomes a strong preference unless explicitly impossible.
   *Comparable approach:* OpenAI function-calling guardrails and LangChain tool routers commonly bias toward a “suggested tool” signal to reduce tool drift.

**High impact / Low difficulty**
2. **Hard safety rails for SHELL**
   - Enforce default constraints when planner selects shell:
     - deny `grep -R` at repo root by default
     - cap runtime + output
     - require scoped paths  
   *Comparable approach:* GitHub Copilot-like “safe command” allowlists; CI bots often implement command sandboxing + timeouts with explicit scoping requirements.

**Medium impact / Medium difficulty**
3. **Strip sub-agent control metadata from user-visible replies**
   - Add a connector-facing output filter or a strict sub-agent response contract (no routing headers).
   *Comparable approach:* AutoGPT-style agent/sub-agent separation typically uses structured channels and prevents control-plane text from entering the user channel.

---

### C) Vulkan performance cliffs (kernel selection / model-shape sensitivity)
**High impact / Medium difficulty**
1. **Add “kernel dispatch telemetry” mode + automated regression benchmark**
   - A standard benchmark script that runs a small matrix (3B/8B/14B) and logs selected kernels and throughput to catch cliffs.
   *Comparable approach:* PyTorch operator benchmarking + MLPerf-style “known-good baselines” for hardware targets.

**Medium impact / Medium difficulty**
2. **Heuristic adjustments for kernel selection**
   - Use debug dispatch diffs (as suggested in #7638) to ensure Qwen2.5-14B hits subgroup/optimized paths when safe.

**Medium impact / Low difficulty**
3. **User-facing hardware/model guidance**
   - If a known cliff model is detected, show “performance warning” and suggest alternatives (e.g., 8B tier) until fixed.
