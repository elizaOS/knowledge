## 1) Pain Point Categorization (Top recurring 5–7)

### 1) Documentation — “How do I get started / where are examples?”
**What users report**

- “Current state / release status / quality” questions after v2.0’s architectural shift (native-apps integration) indicate unclear upgrade guidance and maturity expectations.

**Who it affects most**
- New and returning users evaluating whether to adopt v2/v3; builders trying to choose between cloud vs local vs native deployments.

**Signals (quantified)**
- 1 of 3 Discord day summaries explicitly contains onboarding/demos demand; the “state of the platform” question also appears in that same discussion thread (two distinct onboarding signals in a single day).

---

### 2) Technical Functionality — Speculative decoding compatibility failures (M-RoPE, SWA) and silent fallbacks
**What users report**
- Speculative decoding breaks with Qwen3.5 M-RoPE position validation, collapsing acceptance to ~9.5% and throughput to ~0.33 t/s (#7631).
- DFlash speculative decoding is silently disabled for SWA-based bodies (“target context does not support partial sequence removal”), with no clear API-level signal (#7635).

**Who it affects most**
- Power users pushing large models on iGPU/unified memory; anyone following the “recommended ladder” once Qwen tiers become default.

**Why severe**
- Failures degrade into “it runs but is unusably slow” or “feature looks enabled but isn’t,” which destroys trust and makes benchmarks misleading.

---

### 3) Performance — Hardware/backend cliffs and non-obvious tuning (Vulkan kernels, model shapes)
**What users report**
- Intel Lunar Lake iGPU shows a major performance cliff on a specific model shape (Qwen2.5-14B Q4_K_M) with ~13% bandwidth efficiency vs 37–51% on smaller Llama models (#7638).
- Users need clear guidance on when a backend/kernel path is optimized vs falling back to generic (and slow) kernels.

**Who it affects most**
- Local inference users on integrated GPUs (Intel Arc iGPU, unified memory laptops) trying to step up from ~8B to ~14B.

---

### 4) Integration / Deployment — Containerization and “single-container” expectations for complex voice systems
**What users report**
- Real-time VoiceAI deployment via Docker is hard: GPU access, networking, persistent storage, and latency stability; guidance trends toward multi-container separation (inference/API/db/queue) rather than “all-in-one” (Discord 2026-05-21).
- Strong desire for **single-container** deployment for mass distribution, even when architecture reality suggests otherwise.

**Signals (quantified)**
- 1 documented help interaction: greggblazer + keil0780; the user confirmed progress after moving from monolith to two containers, suggesting the pain point is both common and actionable.

---

### 5) UX/UI — Connector identity complexity (OWNER+AGENT) and unclear mental model
**What users report (implicit, recurring via project changes and questions)**
- The platform is actively introducing OWNER+AGENT identity flows (v2.0.2 era). That’s powerful, but it increases UI/mental-model complexity and raises “what account is acting?” confusion risk.
- Related: users ask for “current state/quality” after major architectural shifts—often a symptom of UI + docs not conveying the new model clearly.

**Who it affects most**
- Builders integrating X/Shopify/Calendly/Slack/Discord where “agent identity” vs “my identity” must be explicitly chosen.

---

### 6) Community / Expectation Management — Roadmap uncertainty (airdrop canceled, hardware chip questions)
**What users report**
- Babylon airdrop canceled due to budget constraints (Discord 2026-05-20) and an unanswered question about whether a physical Eliza E1 chip is being developed.
- These trigger “what is real vs aspirational?” uncertainty, especially for new entrants.

## 2) Usage Pattern Analysis (Actual vs intended)

### How users are actually using elizaOS
1. **Production Web3 agent products**
   - waifu.fun launched as an open-source “agentic launchpad” powered by elizaOS v3 agents (Discord 2026-05-22). This is a concrete productionization pattern: tokenized agents + cloud runtime + revenue share.

2. **Local inference on diverse hardware (incl. emulated RISC‑V)**
   - Community successfully boots Debian RISC‑V 64-bit ISO on QEMU in ~35 seconds and runs Eliza v3 on an emulated RISC‑V chip (“eliza e1”) (Discord 2026-05-20).
   - GitHub issues focus heavily on Intel iGPU performance, speculative decoding, and runtime kernels (Vulkan/OpenVINO).

3. **Voice agent platforms and containerized deployment**
   - Users are dockerizing real-time VoiceAI systems and iterating toward service separation (Discord 2026-05-21).

4. **Plugin ecosystem expansion (DeFi + risk tooling)**
   - New DeFi plugin for Base/Aave solvency scoring and flash-crash simulation published to NPM and requesting testing feedback (Discord 2026-05-20/22).

### Emerging / unexpected use cases
- **“Agents building the product” narrative**: Sol (The Architect) reportedly built waifu.fun itself—users are leaning into self-referential “agent builds agent tools” workflows, which increases demand for reproducible templates, deployment recipes, and observability.

### Feature requests aligned with observed usage
- “Live demos / YouTube” aligns with rapid onboarding for builders shipping real products.
- Strong demand for **deployment simplification** (single-container, turnkey compose, cloud deploy CLI).
- Capability gating/compatibility signaling aligns with advanced local inference usage (users want to know what will actually be active).

## 3) Implementation Opportunities (solutions per major pain point)

### A) Onboarding & demos missing (Documentation)
**High impact / low–medium difficulty**
1. **Publish 3 “golden path” video walkthroughs + matching repos**
   - Tracks: (1) “Hello agent + connector,” (2) “Voice agent local,” (3) “Ship to cloud + custom domain.”
   - Use waifu.fun as a flagship case study: architecture diagram + deployment steps + cost model.

2. **Add a “What version should I use?” decision page**
   - Explicitly compare v2 (native-apps integration) vs v3 agents usage patterns; include stability notes, “best for” checklists, and migration warnings.

3. **Create a “Debugging first hour” doc**
   - Common failures: auth/token placement, missing runtime deps, Docker GPU flags, speculative decoding not active.

*Comparable projects*: Supabase and n8n both rely on “start here” guided tutorials + canonical docker-compose examples to reduce Discord support load.

---

### B) Speculative decoding failure modes + silent fallback (Technical Functionality)
**High impact / medium difficulty**
1. **Fail-loud + API-visible capability flags**
   - If `--spec-type dflash` is incompatible, return a structured flag in `/health` and model endpoints (e.g., `spec_decode_active=false`, `reason="SWA partial sequence removal unsupported"`), not only logs (#7635 suggestion).

2. **Catalog/recommender gating by required capabilities**
   - Extend tier metadata to declare constraints (e.g., `requires_mrope_spec_decode_support`, `supports_partial_sequence_removal`).
   - If unsupported, recommender should avoid surfacing tiers that would be unusable without spec decode.

3. **Compatibility test matrix in CI for “popular modern architectures”**
   - Minimal smoke tests for: M‑RoPE (Qwen3.5), SWA (Qwen3.6/gemma-4), and at least one Llama baseline—ensures future merges don’t reintroduce silent disablement.

*Comparable projects*: llama.cpp surfaces backend/feature support in verbose startup summaries; many inference servers also expose runtime feature flags (e.g., “flash-attn enabled”) via health endpoints to prevent placebo configuration.

---

### C) Intel iGPU performance cliffs & kernel selection opacity (Performance)
**Medium–high impact / medium difficulty**
1. **Add a “kernel path / optimization mode” indicator to local inference UI/log summary**
   - Report which Vulkan kernel variant is selected; warn when falling back to generic kernels.

2. **Publish a hardware tuning guide + model ladder validated by benchmarks**
   - Convert community benchmark tables (like #7638) into a maintained guide: “expected t/s by tier on Lunar Lake / AMD iGPU / Apple Silicon.”

3. **Add a regression benchmark for “shape-sensitive” models**
   - Include at least one Qwen-14B-class model in nightly iGPU perf checks to catch dispatch heuristics regressions.

*Comparable projects*: ComfyUI and Stable Diffusion ecosystems publish “known good combinations” (model + backend + flags) to manage a fast-moving kernel landscape.

---

### D) Dockerization complexity for VoiceAI + “single container” expectation (Integration)
**High impact / medium difficulty**
1. **Official reference docker-compose stacks**
   - Provide two templates:
     - `compose.voice.local-dev` (multi-container: API + inference + queue + db)
     - `compose.voice.single-host` (still multi-service but “one command” with internal networking)
   - Document GPU passthrough for NVIDIA/AMD/Intel.

2. **“Bundle mode” packaging guidance instead of true single container**
   - Provide a scripted installer that pulls images and runs compose, achieving “single artifact UX” without merging db/frontend/backend into one image.

3. **Latency and persistence checklist**
   - Standardize: websocket timeouts, volume mounts, queue sizing, and inference warmup.

*Comparable projects*: Many real-time systems (e.g., streaming stacks) avoid single-image monoliths; they ship a single-command compose wrapper to meet the “easy deploy” need without sacrificing operability.

## 4) Communication Gaps (expectations vs reality)

1. **Speculative decoding expectations**
   - Users can configure DFlash/spec decode but it may be disabled or degraded (SWA/M‑RoPE) without clear UI/API feedback (#7631, #7635). Expectation: “I enabled it, so it’s working.” Reality: “it may be silently off.”

2. **Deployment expectations (single container)**
   - Users want monolithic Docker for mass deployment; reality is that real-time voice + GPU + persistence is operationally safer as multi-container. The gap is solvable by messaging “single-command deploy” rather than “single-container.”

3. **Roadmap clarity**
   - Airdrop cancellation and unanswered hardware questions (physical E1 chip) create uncertainty. The project benefits from explicit “not planned / exploratory / committed” labeling.

4. **Platform maturity after major architectural shifts**
   - “What is the current state/quality?” indicates release notes alone aren’t enough; users want a concise product-state dashboard.
