## User Feedback Analysis — 2026-06-05 (based on data through 2026-06-04)

### 1) Pain Point Categorization (top recurring 5–7)

> Sources: Discord (2026-06-02 to 2026-06-04), GitHub issues/PR notes (notably #8162, #8136, #7991, #7970, #8127), repo activity summaries.

#### A. Technical Functionality — **Sub-agent deliverables lost/garbled in orchestration** (High severity, recurring)
- **Problem:** Orchestrator “narration” frequently drops or truncates the actual deliverable even when the sub-agent did the correct work (GitHub **#8162** open).
  - Specific failure modes reported:
    1) Tool output blocks stripped on “no file change” tasks, deleting the deliverable (stdout) entirely.
    2) Parent model re-renders/truncates values (e.g., `2026-06-02 → 202`, BTC price truncation).
    3) Retry attempts leak (“crashed, retrying…”) alongside final answer.
    4) Excess process chatter relayed to users.
- **Quant:** Reporter ran a **48-prompt sweep** and claims **0 capability-broken defects**; the remaining defect class is *presentation* (“narration-envelope”).

#### B. UX/UI — **Orchestrator/workbench conversation UI parity gaps** (Medium–High severity)
- **Problem:** The workbench UI is perceived as “over-chromed” and noisy vs. Codex/Claude-style targets (GitHub **#8136** closed quickly but indicates active dissatisfaction/iteration pressure).
  - Blue-tint theme conflict (“design rule is no blue”).
  - Redundant tool-call rendering, excessive timestamps, tool names shown raw (“bash” vs verb labels).
  - Output truncation drops the meaningful tail (errors/summaries).
  - Loud CTA styling; weak loading/streaming affordances.
  - Reasoning/thinking not properly rendered/available in UI.

#### C. Technical Functionality / UX — **Internal envelopes leaking into user-visible text** (Medium severity, historically recurring)
- **Problem:** Raw JSON/tool envelopes appear in user responses (e.g., USE_SKILL wrapper leaks bash+JSON) (GitHub **#7970** closed; also related class fixed in **#8159** “TASKS spawn-arg JSON envelope”).
- **Why it matters:** Makes the agent look “broken,” pollutes Discord channels, and reveals implementation details (“I am running bash”).

#### D. Performance / Stability — **Local inference + desktop runtime edge-case crashes & availability ambiguity** (Medium severity)
- **Problem:** Desktop FFI prewarm crash on Windows due to token batch sizing assert (GitHub **#7991** closed). Separate but related: local inference handlers needed a typed “unavailability” signal (**#8191** completed).
- **User impact pattern:** Failures happen early (startup/activation), which feels like the product is unstable even when the root cause is a parameter mismatch.

#### E. Integration — **DeFi/action integrations lag behind user intent (Kamino plugin gap)** (High severity for Web3 users)
- **Problem:** Users want agents to **perform meaningful DeFi actions**, not just fetch info; Kamino plugin proposed as “critical Solana protocol” coverage (Discord **2026-06-02**, n4vnt + anton_0413).
- **Friction:** Designing safe abstractions for lending/vaults/risk operations while keeping developer experience simple.

#### F. Documentation / Onboarding — **Fast-moving repo & plugin process ambiguity** (Medium severity)
- **Problem:** Plugin development moved into `elizaOS/eliza` with rapid changes requiring frequent rebases; this is communicated ad-hoc in Discord (stan0473 guidance on 2026-06-02).
- **User-facing symptom:** Developers risk targeting the wrong repo/branch or getting stuck rebasing without clear best practices.

#### G. Community — **Low response/triage for newcomer posts + scammer reports** (Medium severity)
- **Problem:** Newcomers seeking work (“available as developer”) received no responses (Discord **2026-06-04**, anton_0413). Scammer alert occurred (om1d_sa), acknowledged by a mod but indicates ongoing trust/safety pressure.
- **Impact:** Missed contributor conversion + perceived “cold start” for community support.

### 2) Usage Pattern Analysis (actual vs intended)

#### How users are actually using elizaOS
- **Orchestrator as a “coding agent workbench”:** Heavy emphasis on multi-agent coding workflows, diffs, streaming outputs, and “Claude-Code/Codex parity” UX (reinforced by PRs like **#8095** and UI threads like **#8136** / Odysseus-style work).
- **Agents executing real-world operations (Web3/DeFi + security):**
  - Community interest in agents interacting with financial systems with permissions/controls (Discord: **Neverbell** concept on 2026-06-03).
  - Integration demand: Kamino plugin to enable lending/vault/yield/limit-order actions (2026-06-02).
  - Strong security framing: prompt injection prevention, execution boundary auditing, malicious signing / “agent draining” concerns (marsel_51932 intro 2026-06-04).

#### Mismatch vs intended usage (implied)
- The platform’s capabilities appear to work, but **presentation layers** (narration, envelope stripping, UI rendering) undermine perceived quality.
- Users expect “production-ready connectors” and safe action execution, but current integrations and guardrails are still emerging and community-driven.

#### Emerging use cases / unexpected applications
- **Security services as a community “layer”:** Individuals offering security audits and adversarial testing as a quasi-official ecosystem service (marsel_51932).
- **Bounties/partnership-driven development:** External ecosystems (Intuition Box) using Discord as a recruitment funnel for bounties, but lacking a clear path to post/route them.

#### Feature requests aligned with observed usage
- A **comprehensive Kamino plugin** (actions + abstractions + safety).
- Orchestrator UX improvements: cleaner tool cards, better output tail handling, collapsible reasoning, reduced chatter.
- Reliable “deliverable-carrying” mechanism for sub-agent completions.

### 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

#### Pain Point A: Sub-agent deliverables dropped/garbled (Orchestrator narration-envelope) — **Highest priority**
1) **Implement a “verbatim deliverable carrier” in completion metadata** (High impact, Medium difficulty)  
   - Mirror verified-URL handling: stamp `metadata.subAgentDeliverable` with bounded stdout (≤2KB) and render it verbatim in the evaluator/UI when present (as proposed in **#8162**).
   - Similar pattern: tools like GitHub Actions annotations / LangGraph-style “final output channel” separate from logs.
2) **Terminal-attempt-only reply policy** (High impact, Medium–High difficulty)  
   - Suppress stale failed/blocked attempts when later retries succeed; keep respawn triggers internal-only.
   - Similar approach: retries in Temporal/Step Functions show final result while preserving retry history in an inspection panel.
3) **Output discipline enforced at source + UI separation** (Medium impact, Low difficulty)  
   - Continue strengthening sub-agent manuals/prompts (noted as already helping in #8162 via shipped guidance) and ensure UI distinguishes “Answer” vs “Process”.

#### Pain Point B: Orchestrator/workbench UI parity gaps
1) **Tool-call card redesign with semantic labels + smart truncation** (High impact, Medium difficulty)  
   - Replace raw tool names (“bash”) with verbs (“Run command”), de-duplicate command display, remove repeated timestamps, and implement **head+tail** truncation (tail often contains the error).
   - Similar pattern: GitHub Actions log folding + “view full logs” with concise summary row.
2) **Reasoning/thinking capture + collapsible render** (Medium–High impact, Medium difficulty)  
   - Forward `agent_thought_chunk` to UI rather than dropping it (called out in #8136), render as collapsible “Thinking” cell.
3) **Theme/token cleanup + loading skeletons** (Medium impact, Low difficulty)  
   - Fix the “blue tint” token routing; replace centered loading text with skeleton/stream indicators.

#### Pain Point C: Envelope leakage into user responses
1) **Standardize tool result schema: `userFacingText` vs `diagnostics`** (High impact, Medium difficulty)  
   - Ensure planner prefers `userFacingText` for skills/actions where stdout is the answer (proposed in **#7970**; reinforced by **#8159** class of leaks).
2) **Add a final “response sanitizer” pass** (Medium impact, Low difficulty)  
   - Strip known envelope patterns (`{"cmd":...}`, `{"output":...}`) from user-visible message channel while preserving in trajectory logs.
3) **Regression tests from real probes** (Medium impact, Low difficulty)  
   - Turn the “50-probe sweep” and specific reproductions into fixtures to prevent regressions.

#### Pain Point D: Desktop/local inference stability & clarity
1) **Clamp prewarm batch sizing to `n_batch`** (High impact, Medium difficulty)  
   - Prevent asserts like `GGML_ASSERT(n_tokens_all <= cparams.n_batch)` (root cause in #7991).
2) **Expose “local inference readiness” explicitly in UI** (Medium impact, Low difficulty)  
   - Use typed unavailability errors (as in #8191) to show “Local model not available; falling back to cloud” rather than failing ambiguously.
3) **Add a Windows-CUDA prewarm smoke test** (Medium impact, Medium difficulty)  
   - Catch platform-specific parameter mismatches earlier.

#### Pain Point E: Kamino / DeFi integration gap
1) **Ship a “minimal safe core” Kamino action set first** (High impact, Medium difficulty)  
   - Start with read-only + constrained actions (positions, rates, health factor, deposit/withdraw with caps) before adding leverage/complex vault ops.
   - Similar approach: DeFi SDKs often start with “portfolio + simulate” before “execute.”
2) **Safety rails: permissions + simulation-first execution** (High impact, Medium–High difficulty)  
   - Require simulation/preview + explicit user confirmation for risk-sensitive actions (borrow, leverage changes).
3) **Reference abstractions + templates** (Medium impact, Low difficulty)  
   - Provide “DeFi action design guide” for plugin authors (risk taxonomy, confirmation UX, idempotency, error handling).

#### Pain Point F: Plugin dev process & fast-moving develop friction
1) **Document the “one true workflow”** (High impact, Low difficulty)  
   - Where plugins live now (`elizaOS/eliza`), branch targeting (`develop`), rebase cadence, and how to keep PRs mergeable.
2) **Provide a plugin starter + CI helpers** (Medium impact, Medium difficulty)  
   - A generator that pins compatible package versions and includes a rebase checklist.
3) **Stabilize plugin APIs via versioned interfaces** (High impact, High difficulty)  
   - Reduce churn pressure by introducing compatibility layers for plugin authors.

#### Pain Point G: Community responsiveness & trust/safety
1) **Add a “Looking for work / Hiring” thread + routing** (Medium impact, Low difficulty)  
   - Pin a single channel/thread and add a lightweight template (skills, timezone, availability).
2) **Bounty intake pipeline** (Medium–High impact, Low difficulty)  
   - A form + weekly post that curates external bounties (e.g., Intuition Box) and tags maintainers for approval.
3) **Scam-report SOP + automation** (Medium impact, Medium difficulty)  
   - Standard mod workflow, auto-flag patterns, and a public “how to verify” checklist.

### 4) Communication Gaps (expectations vs reality)

- **“Capabilities work, but output looks broken” gap:** Users interpret missing deliverables / envelope leakage as fundamental model failure, even when the underlying work is correct (explicitly described in **#8162** and **#7970**).
  - Improvement: clearly label “Logs vs Answer” and guarantee an “Answer channel” that never gets stripped.
- **Org/ecosystem confusion:** Repeated need to clarify relationships between **ElizaOS, Hyperia, Hyperforge**, and Shaw’s involvement (Discord 2026-06-03).
  - Improvement: a short canonical “Ecosystem & org map” page, kept current.
- **Plugin contribution path ambiguity:** The fact that “all plugins now live in elizaOS/eliza” was delivered via chat guidance (Discord 2026-06-02), not surfaced as a first-class contributor doc.
- **Security expectations for DeFi agents:** Users are already thinking about prompt injection and malicious signing (2026-06-04 intro), but there’s no clear “official stance” on safe boundaries, permissions, and audit expectations.

Recurring questions/signals indicating onboarding gaps:
- “Where do plugins go / which branch do I target?”
- “What is ElizaCloud vs ElizaOS vs Milady vs Waifu?” (architecture explanation was needed on 2026-06-02)
- “Who do we contact to share bounties / partnerships?” (Intuition Box asked for permission/contact)

### 6) Feedback Collection Improvements

#### Channel effectiveness (current state)
- **Discord:** High for networking and quick clarifications, but low structure; captured periods show minimal technical troubleshooting and at least one unanswered “developer for hire” message.
- **GitHub issues:** High-quality, reproducible, evidence-rich reports (e.g., #8162 includes root-cause hypotheses, proposed patches, and quantified sweeps). This is the best current source of actionable product feedback.

#### Improvements for more structured, actionable feedback
1) **Add “UX defect report” + “Agent output defect” issue templates**  
   - Fields: expected vs actual, backend/model, “deliverable type” (stdout/diff/url), and whether it’s a presentation vs capability failure.
2) **Discord → GitHub bridge bot**  
   - Command to turn a Discord message into a prefilled GitHub issue (with links + channel context).
3) **Monthly “probe sweep” program**  
   - Encourage power users to submit standardized sweeps (like the 48/50-probe reports) with a shared format and fixture repo.

#### Underrepresented segments whose feedback is missing
- **Non-Web3 builders** (most visible demands are DeFi/security).
- **Non-core UI users** (desktop end-users, operators who just want reliability).
- **New plugin authors** beyond highly engaged contributors (we see process warnings, but not broad novice feedback).
