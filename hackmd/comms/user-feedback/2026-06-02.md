## 2) Usage Pattern Analysis (Actual vs intended use)

### How users are actually using elizaOS
1. **DeFAI / onchain execution as a primary driver**
   - Community discussion is strongly oriented around **DeFi agents**: swapping, lending/borrowing, perps, bridges, yield (“swap/earn/perps/bridge/borrow”).
   - Builders (Smeltor, N4vnt, faouzi_122) are treating elizaOS as an **agent runtime + integration bus** for financial actions, not just chat automation.

2. **“Coding workbench” orchestration as an expected core UX**
   - Rapid iteration on `/orchestrator` suggests users expect Claude-Code/Codex-like workflows as a first-class experience, not an experimental plugin.

3. **Natural-language → controlled execution infrastructure**
   - NeverBell’s request for early testers indicates an emerging pattern: agents that accept NL intents but execute via **guarded, signable, non-custodial** operations.

### Emerging / unexpected use cases
- **Payments and metering**: Arsenal Decision Engine using **L402 Lightning** suggests demand for paid agent capabilities and usage-based access.
- **Multi-chain unified calldata generation**: returning **signable calldata** as a productized interface is becoming a shared design language.

### Feature requests aligned with usage patterns
- A standardized **DeFi action interface** (multi-protocol abstraction) that still allows protocol-specific “escape hatches.”
- Better **tool output fidelity** (no truncation, no meta-narration) because users are using agents for operational tasks, not summaries.

   - **Reality:** provider differences created capability gaps until WEB_FETCH changes; routing regressions occurred.

   - “Is the project dead?” and “Does waifu fun have its own token?” → needs a canonical explainer.
   - “Where do I post early tester calls / who is the mod?” → needs a clear community process page.
