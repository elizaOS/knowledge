## 1) Pain Point Categorization (Top recurring 5–7)

### 1) Documentation — **Ecosystem & “what is what” confusion** (High frequency, Medium severity)
**What users report (examples):**
- Confusion about **ElizaOS vs ElizaCloud vs Milady vs Waifu**, prompting a full architecture explanation in Discord (stan0473).
- Repeated identity/token confusion: “Does waifu fun have its own token?” and “Is elizaOS still active?” (odilitime clarified both).
- “Is Waifu similar to Virtuals?” asked and **left unanswered** in the captured summary, indicating an expectation gap.

**Why it hurts:**
- Users can’t reliably decide *where to build* (framework) vs *where to deploy* (cloud) vs *what product they’re evaluating* (Milady/Waifu).
- Increases support load and slows contributor onboarding.

**Quantification (within this small sample):**
- **~3 of ~3 Discord days** included at least one “what is X / is the project active / token?” style clarification → this is the dominant newcomer friction in the captured chat.

### 2) Integration — **Major DeFi protocol gaps (Kamino) & unclear integration strategy** (High severity, Medium frequency)
**What users report (examples):**
- Missing **Kamino** integration called out as a “significant gap” (n4vnt), with demand to cover lending/borrowing/vaults/yield/limit orders.
- Architectural debate: **multi-protocol abstraction layer vs protocol-specific plugins** (Smeltor vs n4vnt), reflecting uncertainty on the “blessed” path.

**Quantification:**

### 3) Technical Functionality — **User-facing output corruption/leaks in orchestrator & tool pipelines** (High severity, High frequency on GitHub)
**What users report (examples):**
- **Deliverables dropped/garbled** when narrating sub-agent completion (“narration-envelope”, issue **#8162**, open): printed outputs can be deleted or truncated (e.g., `2026-06-02 → 202`, `70234 → 705`).
- **Internal tool envelopes leaked** to end users:
  - USE_SKILL wrapper leaks raw bash/JSON envelopes (issue **#7970**, closed).
  - Planner leaks spawn-arg JSON envelope to user reply (issue/PR **#8159**, closed).
- Duplicate replies due to race conditions (fixed per June 2 dev summary; PR **#8161**).

**Why it hurts:**
- These are “trust breakers”: users see internals (“I’m running bash”), missing results, or mangled outputs—perceived as instability even when underlying work is correct.

**Quantification (from surfaced top issues):**
- **3 of the 5 highlighted issues** are directly about **final-message integrity** (drop/truncate/leak), suggesting a concentrated, recurring defect class.

## 2) Usage Pattern Analysis (Actual vs intended)

### Observed real-world usage
1) **Agents as DeFi operators (execution, not just Q&A)**
- Users are explicitly pushing for lending/vault management/limit orders (Kamino plugin) and multi-protocol execution layers (Smeltor’s unified interface returning signable calldata).

2) **Multi-agent orchestration for coding/workbench workflows**
- Significant engineering effort and feedback clusters around orchestrator streaming, diffs, tool cards, and “final answer integrity” → users treat elizaOS as a **coding agent platform**, not only a chatbot framework.

3) **Cloud-hosted “sandbox-first” interaction**
- Enabling per-agent web UI by default (Cloud) aligns with how users want to interact: a **web UI per agent** as the primary surface.

### Emerging / unexpected applications
- **Financial-market execution via natural language** (NeverBell seeking testers).
- **Paid decision oracles + MCP + L402** (Arsenal Decision Engine), indicating a path toward “agents as monetizable services.”

### Feature requests that match usage
- **Kamino plugin** (protocol-specific) and/or a **standard DeFi action abstraction** (swap/earn/borrow/bridge/perps).
- **Reliable “deliverable-first” output contract** for orchestrated tasks (especially printed-value tasks).

## 3) Implementation Opportunities (2–3 per major pain point)

### A) Ecosystem confusion (Docs)
**High impact / Low–Medium difficulty**
1. **Single “What is ElizaOS?” canonical page** with a diagram and 30-second decision tree:
   - “If you want to build agents → ElizaOS”
   - “If you want hosting → ElizaCloud”
   - “If you want consumer UI → Milady”
   - “If you’re evaluating agent economies → Waifu”
2. **Add an “FAQ: tokens & project status”** pinned in Discord + mirrored in docs:
   - Explicitly cover waifu fun vs Solace token naming.
3. **Competitive-positioning blurbs** (e.g., “Waifu vs Virtuals: similarities/differences”) to close the unanswered expectation gap.

**Comparable patterns:**
- Kubernetes’ “What is Kubernetes?” + “Components” diagram; Docker’s “Docker Engine vs Docker Desktop vs Hub” clarity pages.

---

### B) DeFi integration gaps + strategy ambiguity (Integration)
**High impact / Medium difficulty**
1. **Publish an “Eliza DeFi Integration RFC”** (1–2 pages) defining:
   - When to build **protocol-specific plugins** (Kamino)
   - When to build **aggregation adapters** (swap/borrow/bridge interface)
   - Required safety constraints for risk actions (borrow/leverage)
2. **Ship a Kamino plugin MVP with a “safe subset” first**:
   - Read-only positions + supply/withdraw (low risk)
   - Borrow/repay gated behind explicit confirmation + risk checks
3. **Define a common DeFi action schema** (even if implementations differ):
   - `getPositions`, `quote`, `buildTx`, `simulate`, `execute`
   - Return **signable calldata** for non-custodial wallets (aligns with Smeltor’s approach)

**Comparable patterns:**
- Yearn + Beefy vault abstractions; 0x/1inch quote→calldata APIs; Coinbase AgentKit-style “build transaction, let wallet sign”.

---

### C) Output integrity defects (Technical Functionality)
**Highest impact / Medium difficulty**
1. **Introduce a “verbatim deliverable carrier” for orchestrator completions** (aligned with #8162 proposal):
   - Store captured stdout / printed value as `metadata.subAgentDeliverable` (bounded size, e.g., ≤2KB)
   - UI renders it verbatim rather than re-summarizing (prevents truncation)
2. **Standardize tool/action outputs into `userFacingText` vs `debugText`**
   - Enforce that only `userFacingText` is eligible for the final response unless explicitly toggled
3. **Add regression tests for “no-change-set printed-output tasks”**
   - Ensure outputs don’t disappear when there’s no diff artifact
   - Include truncation probes across backends (opencode/codex/claude) since behavior differs

**Comparable patterns:**
- LangChain / OpenAI tool calling: separation of tool results from assistant text; “final answer” channel guarantees.

---

### D) Workbench UI parity gaps (UX/UI)
**Medium–High impact / Medium difficulty**
1. **Tool card slimming + tail-preserving truncation**
   - Show action verbs (“Run command”, “Fetch URL”) instead of raw tool names
   - Default truncation should preserve **head + tail** (errors are often at the end)
2. **Reasoning/thinking disclosure pattern**
   - Collapsible “Thinking” blocks (off by default), plus token counts
3. **Consistent theming tokens for orchestrator surfaces**
   - Fix unintended brand-blue leakage; adopt semantic tokens to allow “no-blue” themes

**Comparable patterns:**
- GitHub Actions log folding; VS Code terminal/task UI; Claude Code/Codex “tool call” cards with collapsible details.

### E) Contribution workflow friction (Community/Docs)
**Medium impact / Low difficulty**
1. **“Contributing a plugin” guide**:
   - Repo location (elizaOS/eliza), branch (develop), expected rebase cadence
   - Plugin skeleton template + checklist (tests, docs, actions)
2. **Automation to reduce rebase pain**
   - Provide a `sync-develop` script / documented workflow (e.g., `git fetch upstream && git rebase upstream/develop`)
   - Encourage smaller PR slices for large plugins (Kamino)

   - `integration:solana`, `defi`, `risk-sensitive`, `needs-design-rfc`

**Comparable patterns:**

### F) Desktop/local inference stability (Performance)
**Medium impact / Medium difficulty**
1. **Prewarm batch clamping as a hard rule**
   - Always clamp prewarm tokens to `n_batch` (prevents GGML asserts class)
2. **First-run diagnostics bundle**
   - On crash, capture model bundle, adapter config, and last N logs into a shareable report
3. **“Known issues by platform” doc**
   - Windows-CUDA notes, recommended defaults, and how to change prewarm settings

**Comparable patterns:**
- Electron apps: first-run hardware survey + safe-mode; llama.cpp GUIs often expose conservative batch defaults on Windows.

## 4) Communication Gaps (expectations vs reality)

1. **“What product am I using?” gap**
   - Users conflate ElizaOS (framework) with ElizaCloud (hosting) and Waifu/Milady (apps). This creates misdirected support questions and incorrect mental models.

2. **“Agents can do DeFi” expectation vs current integrations**
   - Users expect action execution (borrow/lend/vault ops), but current capability is gated by missing protocol plugins and safety abstractions.

3. **“It’s broken” perception caused by presentation bugs**
   - The #8162 class is explicitly a *presentation defect*, but users experience it as capability failure (“it didn’t give me the result”).

4. **Contribution targeting uncertainty**
   - People still ask where plugins live and which branch to target; the change to “all plugins live in elizaOS/eliza” isn’t yet self-evident.

**Concrete doc/onboarding fixes**
- Add an onboarding step in README: “Building a plugin? Start here → repo/branch/path.”
- Add a “Capabilities vs apps” page that explicitly distinguishes framework, hosting, and consumer apps.

## 6) Feedback Collection Improvements

### Current channel effectiveness
- **Discord:** great for rapid clarification, but repeated questions indicate answers aren’t being retained/discoverable.
- **GitHub issues:** strong for deep technical defects (e.g., #8162) with clear repro and proposed fixes; however, it skews toward power-user reports.

### Improvements for more structured, actionable feedback
1. **Add a weekly “Top confusions” Discord form** (3 questions, anonymous):
   - “What confused you this week?”
   - “What did you try to do?”
   - “Where did you get stuck?”
2. **Standardize GitHub issue templates for user-facing integrity bugs**
   - Include: backend, reproduction prompt, expected vs actual, whether output was printed-value/no-change-set, screenshots.
3. **Instrument opt-in UX telemetry in workbench**
   - Track truncation events, “copy deliverable” clicks, retry loops—helps quantify how often #8162-class issues occur.

### Underrepresented segments (missing feedback)
- Non-DeFi builders (general automation, knowledge work) are not visible in this sample.
- Non-technical users of Milady/Waifu UI surfaces: little direct usability feedback captured here.
- Windows desktop newcomers beyond power-user crash reporters (need simpler first-run surveys).
