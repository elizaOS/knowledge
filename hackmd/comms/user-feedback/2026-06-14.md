## 2) Usage Pattern Analysis (Actual vs intended)

### Observed real-world usage patterns
1. **Developers run off `develop` to access “real” v2 features**, even though “main” is the default mental model.  
   - Actual behavior: “Most developers work from develop branch.” (Jun 13)
   - Intended behavior (typical OSS expectation): stable main + tagged releases.

2. **Agents as infrastructure components, not chatbots**
   - carlgatz’s architecture treats LLMs as an expensive stage in a pipeline: **filter → RAM ops → deep analysis only when necessary**.
   - This is closer to “event processing + selective reasoning” than classic conversational usage.

3. **World-model-driven agent building is becoming a priority**
   - Community guidance points to external platforms (Vertex/Genie) now, with native tooling promised soon—suggesting demand is ahead of product readiness.

4. **Emerging/adjacent use cases**
   - **Robotics/IoT at scale**: proposal for a native `@elizaos/plugin-peaq` targeting “3M devices” on peaqOS. (Jun 12)
   - **Gaming/trivia community projects**: Ruby trivia app posting daily challenges and new categories. (Jun 13)

### Feature requests aligning with actual usage
- **World model tooling + migration guides** (explicitly requested/announced).
- **Native plugin integrations** for device ecosystems (peaq).
- **Performance primitives** (official “token budget + filtering” patterns, reference architectures).

## 3) Implementation Opportunities (solutions per major pain point)

### A) Versioning confusion (develop vs main)
**High impact / low-to-medium difficulty**
1. **Add a “Release Channels” doc + CLI warning**
   - On install/run, detect if user is on v1.x and print: “v2 is currently on `develop`; use `bunx degit` or `git checkout develop` instructions.”
2. **Publish a simple compatibility matrix**
   - Table: `main` → stable tag, `develop` → v2 preview, “what breaks / what’s experimental”.
3. **Adopt a “release train” cadence with tags**
   - Similar to **Kubernetes** (regular releases) or **Rust** (stable/beta/nightly): even if `develop` moves fast, provide predictable cut points (`v2.1-beta.N`).

---

### B) Cloud auth & invitation gating
**Very high impact / medium difficulty**
1. **Make Cloud access state explicit in-product**
   - If invite-only, show it clearly on signup UI: “Closed beta; request invite” with a link/button, not an opaque tenant error.
2. **Add automated auth smoke tests for prod parity**
   - Similar to how many SaaS teams gate releases: run Playwright/Cypress flows for OAuth + magic link against staging/prod configs prior to deploy.
3. **Centralize auth configuration and publish a “Known Issues” banner**
   - When sign-in breaks, users need a single source of truth (status page or pinned Discord + dashboard banner).

*Comparable approaches:* Supabase and Vercel both use clear “request access / waitlist” flows and status pages; Auth0-style systems maintain strict staging/prod parity tests for OAuth.

---

### C) Scam/spam in Discord reducing trust
**High impact / low difficulty**
1. **Tighten automated moderation**
   - Enable stricter anti-link rules for new accounts, require account age/verification, add spam keyword heuristics.
2. **Create a single “Promotions/Collabs” channel**
   - Keep #discussion high-signal; redirect promotional content.
3. **Add “Report scam” workflow**
   - One command and a visible mod response SLA.

*Comparable approaches:* Large OSS Discords (e.g., Next.js, Reactiflux) segment promo channels and use aggressive anti-link measures for new users.

---

### D) Governance/transparency concerns
**Very high impact / medium difficulty (organizational)**
1. **Publish an official, timestamped statement + FAQ**
   - Address the specific claims (timeline, amounts, counterparties, decision process) and link to primary documents where possible.
2. **Create a lightweight governance page**
   - Who controls funds, what approvals are required, conflict-of-interest policy.
3. **Host a monthly community call + notes**
   - Provide an accountable venue for questions; publish summaries.

*Comparable approaches:* Many crypto-adjacent OSS projects use transparency reports, multi-sig disclosures, and recurring governance calls to reduce rumor-driven narratives.

---

### E) “Is the project abandoned?” / roadmap visibility
**High impact / low difficulty**
1. **Weekly “What shipped / what’s next” pinned post in Discord**
   - Convert progress-channel noise into a digest. Include: Cloud status, releases, blockers.
2. **Add a public roadmap board**
   - Even a simple GitHub Projects view: “Now / Next / Later”.
3. **Expose lead activity differently**
   - Users equate “no commits by person X” with abandonment. Show team activity metrics and highlight other core contributors.

*Comparable approaches:* Astro and Svelte maintain clear weekly updates; many OSS projects pin “state of the union” summaries.

---

### F) World models usability gap
**Medium-to-high impact / medium difficulty**
1. **Ship a minimal “world model starter” template**
   - One golden path: configuration, provider options (Vertex now; native soon), example agent.
2. **Write the promised migration guide early**
   - Even before tooling release, document concepts and expected APIs to reduce uncertainty.
3. **Add a plugin interface for world-model providers**
   - Make Vertex/Genie one implementation; allow community alternatives.

*Comparable approaches:* LangChain/LlamaIndex provide “cookbook” starters and provider abstraction layers that let advanced users swap backends without rewriting apps.

---

### G) Performance/cost patterns (selective reasoning pipelines)
**Medium impact / medium difficulty**
1. **Official “high-throughput pipeline” reference architecture**
   - Document the pattern carlgatz described: filtering, caching, RAM fast-path, selective MCP.
2. **Add built-in instrumentation**
   - Token/cost dashboards per skill/action; latency histograms.
3. **Offer a “budget mode” runtime profile**
   - Conservative defaults: smaller models for routing, stricter tool invocation, caching enabled.

*Comparable approaches:* OpenTelemetry-style metrics adoption; many agent frameworks now ship cost dashboards and caching primitives as first-class features.

## 4) Communication Gaps (expectation vs reality)

### Recurring mismatches
- **Expectation:** “Updating should get me the newest version.”  
  **Reality:** v2 is effectively “preview” on `develop`, and `main` can lag. This is not obvious without insider knowledge.
- **Expectation:** “I can sign up for Eliza Cloud.”  
  **Reality:** The system may be invite-gated or misconfigured, surfacing a confusing tenant error.
- **Expectation:** “World models are part of ElizaOS now.”  
  **Reality:** Current best path is external (Vertex/Genie), with native tooling “~1 month” out.
- **Expectation:** “Silence from a known dev = abandoned project.”  
  **Reality:** Work is ongoing but fragmented across channels and not summarized where newcomers look.

### Documentation/onboarding improvements
- Add a **Cloud onboarding flowchart**: invite → login → create agent/app → common errors.
- Add a **Branch/release selection** section in installation docs (and in README).
- Add a **World models** landing page: what it is, supported providers, current status, examples.

## 6) Feedback Collection Improvements

### Current channel effectiveness
- **Discord #discussion** is high visibility but prone to spam and off-topic governance debates, reducing signal.
- **GitHub issues** capture deep technical audits well (e.g., Apps scaling/ingress gaps), but many users never file issues and instead ask in Discord.

### Improvements for more structured, actionable feedback
1. **Add an in-product “Send feedback” button** (Cloud + desktop UI)
   - Auto-attach version, branch, environment, logs (opt-in).
2. **Adopt issue templates that mirror real pain points**
   - “Cloud signup/login”, “Version upgrade”, “World models”, “Performance/cost”.
3. **Monthly pulse survey**
   - Short form: “What blocked you this month?” with selectable categories.

### Underrepresented segments
- **Non-Discord users** (enterprise/dev teams) and **silent Cloud evaluators** who churn at signup errors are not well represented.
- **Integrators** (IoT/robotics, device ecosystems) appear via proposals but lack an ongoing structured intake (RFC pipeline).
