## 1) Pain Point Categorization (top recurring friction areas)

> Frequency proxy: concentration in “top issues” + repeated PR workstreams; Severity proxy: user-visible breakage, silent failure modes, timeouts, security risk.

### 1) Technical Functionality — Local inference speculative decoding breaks or silently disables
**Why it’s a top pain point:** 2/5 of the highlighted top issues are speculative decoding failures (#7631, #7635), and both describe **silent or degenerative behavior** (acceptance collapse, feature disabled without clear UI/API signal).

**Specific problems**
- **M-RoPE incompatibility:** Speculative decoding rejects drafted batches on Qwen3.5 with “inconsistent sequence positions” and collapses throughput to **0.33 t/s** (#7631).
- **SWA bodies disable DFlash:** For Qwen3.6 / gemma-4 SWA-based bodies, `--spec-type dflash` is **silently disabled** due to “partial sequence removal” incompatibility (#7635). Users can believe DFlash is active when it isn’t.

**Who it affects most**
- Power users and builders running Qwen3.x / gemma models locally (especially iGPU/unified-memory setups).
- Anyone relying on the **recommender ladder** that may surface tiers where speculation is assumed for usability.

---

### 2) Performance — GPU kernel selection and model-shape cliffs on Intel iGPU (Vulkan)
**Specific problems**
- **Large performance cliff on Qwen2.5-14B Q4_K_M:** measured **1.15 t/s**, only **~13%** bandwidth efficiency vs smaller models on same system (#7638). Suspected suboptimal kernel dispatch path.
- In practice, users are forced into smaller models (e.g., 3B) even when VRAM/unified memory is sufficient, because throughput becomes “unusable” at certain shapes.

**Who it affects most**
- Intel Lunar Lake / Arc iGPU users (and likely other Intel iGPU configurations) trying to step up from 8B → 14B quality.

---

### 3) UX/UI — Planner chooses wrong tools, leaks internal metadata, and times out on “simple” user intents
**Specific problems (from open issue #7935)**
- For “search chat history” requests, Stage 1 correctly hints **SEARCH_MESSAGES**, but the planner frequently chooses **SHELL** and runs unbounded `grep -R ...` over the monorepo, causing **60–120s timeouts**.
- Sub-agent pipelines sometimes leak internal routing/control text into user-visible replies (e.g., `**Routing Kind:** QUESTION_FOR_TASK_CREATOR`).
- Planner emits incorrect action shapes (`TASKS { action: "spawn_agent" }` vs canonical `TASKS_SPAWN_AGENT`), leading to failures and empty final replies.

**Severity**
- High: produces empty responses, long delays, cost spikes, and trust damage (“why is it grepping source code when I asked about chat?”).

**Specific problems**

  - GPU path depends on Intel compute runtime packages that are not trivial/non-sudo environments.
  - NPU is compelling for **Whisper ASR**, but not a win for autoregressive LLM decode today.
- Users need clearer “what this backend is good for” guidance (voice vs chat) and simpler install diagnostics.

## 2) Usage Pattern Analysis (actual vs intended use)

### Observed real usage
1. **Local inference as a first-class experience**, especially on consumer devices (Intel iGPU, unified memory).
   - Evidence: top issues focus on Vulkan kernels, speculative decoding correctness, runtime routing, and model tier recommendations.
2. **Voice agent pipeline is emerging as a “killer app”** (ASR/TTS + small LLM body).
   - Evidence: OpenVINO RFC emphasizes Whisper-on-NPU performance and end-to-end TTFA targets (#7633).
3. **Community uses Discord more for announcements/recruiting than for support.**
   - Evidence: 2026-05-27 introductions; 2026-05-28 hackathon + hiring + portfolios; minimal troubleshooting threads.
4. **Web3/agentic finance & onchain benchmarks** are prominent in community attention (hackathon tracks: AI Trading, RWA, agentic wallets).

### Mismatch vs intended usage (inferred)
- Intended: Discord as collaborative support + development discussion.
- Actual: Discord currently behaves like a broadcast/promotions feed; troubleshooting shifts to GitHub issues (power users) or gets lost.

### Feature requests aligning with real usage
- **Runtime kernel expansion (OpenVINO)** to support modern Intel hardware (esp. ASR on NPU).
- **Planner/tooling reliability improvements** for common “assistant” tasks (search, recall, cross-channel lookups).
- **Better observability/verification** for local inference features (spec-decode active, backend selected, fallback reasons).

## 3) Implementation Opportunities (solutions per major pain point)

### Pain Point A: Speculative decoding incompatibility + silent disablement (Technical Functionality)
**High impact / medium difficulty**
1. **Expose “spec decode active” state in runtime APIs**
   - Add `spec_decode_active: boolean` and `spec_decode_reason: string` to `/health` and `/v1/models/<id>` (mirrors suggestion in #7635).
   - UI can show: “DFlash disabled (SWA body KV cache doesn’t support rollback). Running body-only.”
   - Comparable pattern: llama.cpp-based servers often expose backend/features in a model info endpoint; replicate that transparency.

2. **Fail loudly when user explicitly requests DFlash but it can’t run**
   - If `--spec-type dflash` + `-md` provided and compat check fails, **abort startup** unless an explicit `--allow-spec-fallback` flag is set.
   - Prevents “I thought it was faster” confusion.

**Medium impact / higher difficulty**
3. **Implement SWA-aware partial-sequence removal or port known fixes**
   - Port/align the KV-cache partial removal path referenced from spiritbuun fork lineage (#7635).
   - Add conformance tests: load SWA body + DFlash drafter; assert `n_drafted/n_accept` present and non-zero.

---

### Pain Point B: Intel iGPU Vulkan kernel selection cliffs (Performance)
**High impact / medium difficulty**
1. **Add automated “model-shape regression” microbench suite**
   - CI (nightly) runs a small set of representative shapes (3B/8B/14B equivalents) and flags large efficiency regressions.
   - Similar to how graphics projects maintain “performance baselines” per GPU class.

2. **Improve kernel dispatch logging and user-facing diagnostics**
   - Provide a one-command “collect dispatch logs” mode (wrapping `GGML_VK_DEBUG*`) and attach to issue template.
   - Outcome: reduces back-and-forth and increases reproducibility.

**Medium impact / higher difficulty**
3. **Dispatch heuristic fixes / add optimized path for Qwen2.5-14B-like shapes**
   - Use the diffed dispatch logs approach proposed in #7638 to ensure 14B hits subgroup-optimized kernels where appropriate.

---

### Pain Point C: Planner chooses wrong tools, runs expensive shell commands, leaks control text (UX/UI)
**High impact / low-to-medium difficulty**
1. **Planner “candidateActions is a strong hint” rule**
   - When Stage 1 outputs `candidateActions=[SEARCH_MESSAGES]`, planner must prefer it over generic SHELL.
   - Add a planner contract test: for chat-recall prompts, it must call SEARCH_MESSAGES first.

2. **Hard cap and scope for SHELL searches**
   - Introduce safe wrappers: `GREP` tool with mandatory path allowlist + max files + max runtime.
   - If SHELL must be used, enforce: no `grep -R` at repo root; require explicit directory.

**Medium impact / medium difficulty**
3. **Strip/transform sub-agent control metadata at connector boundary**
   - Ensure user-facing output excludes routing headers (`Routing Kind`, internal task state).
   - Comparable approach: AutoGen/LangGraph-style agent frameworks separate “system traces” from “assistant messages”; enforce a strict output channel.

---

### Pain Point D: Discord spam + fake support tickets + low support signal (Community)
**High impact / low difficulty**
1. **Lock down ticket creation and add verified support entrypoints**
   - Restrict channel creation permissions; use a vetted support bot; pin “Official Support” message.
2. **Auto-moderation for recruitment/promotional posts**
   - Add a dedicated `#jobs` / `#promo` channel; enforce redirects via bot; rate-limit new accounts posting links.

**Medium impact / medium difficulty**
3. **Convert “coders” into a help-first channel**
   - Require question template for help posts; weekly office hours thread; tag helpers.

---

### Pain Point E: Status & expectations confusion (Documentation / Communication)
**High impact / low difficulty**
1. **Single canonical “Project status” doc + pinned Discord message**
   - “Open source active; labs defunct; where releases happen; how to verify official announcements.”
2. **“How to tell if X is working” verification guides**
   - Spec decode verification checklist (what fields to see, what logs indicate active).
   - Local inference backend verification (Vulkan vs CPU vs OpenVINO; expected performance ranges).

**Medium impact**
3. **Rumor-control policy**
   - A lightweight policy: official announcements only via specific channels/accounts; everything else is unverified.

## 4) Communication Gaps (expectations vs reality)

### Gap 1: “Speculative decoding enabled” vs “actually doing speculation”
- Reality: DFlash can load and then be disabled by incompatibility checks (#7635) without API-level confirmation.
- Improvement: make feature activation explicit in UI/API; fail loud when user explicitly requested it.

### Gap 2: “Search chat history” user intent vs actual execution behavior
- Expectation: the assistant searches Discord messages.
- Reality: planner may grep local source code, time out, or leak internal routing metadata (#7935).
- Improvement: clarify what connectors support (SEARCH_MESSAGES availability per connector) and route appropriately.

### Gap 3: “Is this project alive / official?”
- Expectation: company status implies project status.
- Reality: open source continues; company defunct (Discord 2026-05-26).
- Improvement: onboarding and README should explicitly address governance and continuity.

### Gap 4: “Discord = support”
- Observed behavior: announcements and recruitment dominate (0 help threads in sampled days).
- Improvement: redesign channel structure and add a structured support intake.

### Improvements (more structured, actionable)

2. **Standard issue templates for local inference**
