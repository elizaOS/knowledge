## 1) Pain Point Categorization (top recurring 5–7)

> Note on quantification: Discord volume in the provided window is low and skewed toward recruiting. GitHub feedback is richer and highly actionable (audits, reproducible bugs, scaling gaps).

### 1. Cloud Authentication / Login Reliability (Integration + Technical Functionality) — **High severity**
**What users report**
- “Sign in with Eliza Cloud” blocked by prod OAuth/email failures: OAuth `/authorize` returning 400 (“code_challenge is required…”) and email send returning 400 (GitHub #8257).
- Multiple follow-up fixes in PRs suggest this was not a one-off misclick but a systemic contract drift between Steward auth requirements and the Cloud frontend/proxy.

**Why it’s recurring**
- Auth is a cross-cutting dependency for dashboard + CLI login + third-party apps using “Sign in with Eliza Cloud,” so breakage is immediately visible and blocks most workflows.

**Most affected**
- New cloud users, third-party app builders integrating Cloud SSO, anyone provisioning/operating agents via the Cloud UI.

---

### 2. Cloud Apps (Product 2) Go-live Readiness: Ingress + Scaling + Billing correctness (Technical Functionality + Performance) — **High severity**
**What users report**
- Apps unreachable without ingress (Caddy not installed / front door missing) and DB connection ceilings without pooling (max_connections default 100) (GitHub #8321).
- Additional scaling risks: single-node placement bug, docker network exhaustion, multi-node ingress LB gaps.
- Billing correctness risk: “billing leaks” called out in internal dev update + audit concerns (billing cron idempotency / double-bill risk in #8321).

**Why it’s recurring**
- Cloud Apps is actively being brought to production; audits identify “P0 blockers” and “P1 scale blockers” that will re-surface as soon as traffic increases.

**Most affected**
- Early adopters deploying apps, internal operators, partner teams planning production launches.

---

### 3. Orchestrator Output “Narration Envelope” Drops or Garbles Deliverables (UX/UI + Technical Functionality) — **Medium–High severity**
**What users report**
- Sub-agent work is correct, but the user-visible deliverable is dropped/truncated/covered by chatter (GitHub #8162).
- Symptoms include stripping tool output blocks on “no files changed” tasks, truncation by parent model, stale retry messages leaking, and verbose process narration.

**Why it’s recurring**
- It affects common “run command and report output” / “print value” tasks, which are frequent in agentic coding workflows.

**Most affected**
- Power users using the orchestrator workbench, evaluators/benchmarkers doing prompt sweeps, coding-oriented users.

---

### 4. Build/Runtime Packaging Surprises (Bun vs Node resolution, transitive deps) (Technical Functionality + Integration) — **Medium severity**
**What users report**
- `packages/app-core` built output externalizes transitive runtime deps (`jose`, `@node-rs/argon2`), causing `ERR_MODULE_NOT_FOUND` when run outside Bun’s loader (GitHub #8142, downstream QA referenced).

**Why it’s recurring**
- Multi-runtime environments (Bun for dev/build, Node-style execution in certain commands/contexts) make dependency resolution mismatches easy to trigger.

**Most affected**
- Contributors running tooling like `doctor`, downstream integrators, CI environments that invoke built artifacts differently than local Bun workflows.

### 5. Discord Channel Signal-to-Noise: Recruiting/Solicitation Dominates Technical Help (Community + UX/UI for community spaces) — **Medium severity**

   - Users are treating elizaOS as a *character-first, multimodal companion runtime*, not only “agent automation.”

   - Example: Cabal-Hunter as an MCP server with x402 payments ($0.05 USDC/query) integrated via `@hugen/plugin-x402-solana` (Discord 2026-06-07).
   - Example: autonomous DeFi portfolio management agent using x402 for Casper buildathon (Discord 2026-06-06).
   - Users are combining elizaOS with paid APIs and on-chain/payment rails, often as pre-trade risk checks.

3. **Cloud-hosted agent/app infrastructure is being evaluated like a production PaaS**
   - Audit-style issues (GitHub #8321) show users expecting Heroku/Fly.io-like “apps are reachable + scale predictably + billing is correct.”
   - “Go-live readiness” feedback is detailed and evidence-based—indicative of serious deployment intent.

4. **Orchestrator used as a coding workbench**
   - The “deliverable dropped” bug (#8162) reflects real usage: high-volume prompt sweeps, sub-agent spawning, and tool-output reporting as user-facing output.

### Emerging/unexpected use cases
- **Fraud detection tooling as a paid microservice** (Cabal-Hunter) plugged into agents as a guardrail before swaps.
- **Benchmarking harnesses for trading strategy quality** (builderradmin methodology) where elizaOS is part of a broader evaluation workflow.

### Feature requests aligned with usage
- Reliable, user-readable orchestrator completions (verbatim deliverables, less chatter) strongly aligns with “workbench” usage.
- First-class “Discord companion starter kit” aligns with frequent multimodal companion questions.
- Cloud Apps: production ingress + pooling + idempotent billing aligns with “PaaS expectation” usage.

## 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

### A) Cloud Auth Reliability (PKCE + email) — **Impact: Very High | Difficulty: Medium**
1. **Add an auth contract test suite + canary checks for prod**
   - Implement synthetic monitoring (CI + scheduled worker) that runs:
     - OAuth authorize URL build includes PKCE params
     - email send returns 2xx
     - full login flow (headless) against staging + prod
   - Similar approach: Supabase/GitHub-style “auth smoke tests” run continuously to catch upstream provider contract changes.

2. **Version-pin and explicitly validate Steward auth capabilities at runtime**
   - On Cloud frontend boot, fetch Steward “capabilities” metadata (PKCE required? email enabled?) and fail with a clear diagnostic screen + status page link.
   - Similar approach: Firebase Auth SDK feature detection + “service status” surfaced to users.

3. **Operationalize incident comms**
   - Add `status.elizacloud.ai` (or GitHub status issue template) and in-product banner when auth is degraded.
   - Low code complexity, high trust impact.

---

### B) Cloud Apps go-live blockers (Ingress + pooling + billing correctness) — **Impact: Very High | Difficulty: High (but modular)**
1. **Ingress “golden path” + validation script as a release gate**
   - Ship and require a `verify-apps-ingress-routing.sh`-style check (already emerging in work) as a pre-prod gate:
     - deploy an example container
     - verify DNS → TLS → reverse proxy → container health
   - Similar approach: Fly.io “machine checks” / Heroku release phase tests.

2. **DB connection pooling + sane defaults**
   - Install/configure pgbouncer on tenant-db nodes; reduce per-app pool defaults in plugin-sql for cloud app images.
   - Add a capacity model doc: “X apps per node with pooling vs without.”
   - Similar approach: Render/Heroku require pooling at scale; Supabase docs push pgbouncer for serverless.

3. **Billing idempotency + leak prevention**
   - Add unique constraints (container_id, billing_period) and make billing cron resumable (cursor-based pagination + checkpointing).
   - Similar approach: Stripe-like idempotency keys; cloud billing pipelines typically enforce uniqueness at DB level.

---

### C) Orchestrator deliverable loss (“narration envelope”) — **Impact: High | Difficulty: Medium**
1. **Introduce a “verbatim deliverable carrier” field**
   - Implement the proposed `metadata.subAgentDeliverable` path (#8162): capture stdout/tool-output before transcript stripping and relay verbatim when present.
   - Similar approach: LangChain/AutoGen style separation of “tool outputs” vs “assistant response,” with explicit UI blocks.

2. **Terminal-attempt rendering and retry suppression**
   - Only show the final successful attempt; keep intermediate “session_state_lost”/retry signals internal.
   - Similar approach: Temporal/Prefect UIs collapse retries into a single run view, with expandable retry history.

3. **UI-level “Deliverable First” rendering**
   - In completion UI, render a dedicated “Result” panel above narration; keep process chatter collapsible.
   - Low-to-medium effort, reduces perceived flakiness even before deeper router changes.

---

### D) Bun/Node packaging mismatch (transitive deps externalized) — **Impact: Medium | Difficulty: Low–Medium**
1. **Make runtime deps explicit or bundle them**
   - Prefer (b) “declare as direct deps” for stability, or (a) bundle known runtime deps into dist output when shipping Node-executed artifacts.
   - Similar approach: Next.js/tsup/rollup build configs often maintain allowlists for externalization.

2. **Add a “runs under Node” CI lane**
   - For any package that produces dist meant to be executed, run a Node-based smoke test that imports/executes `dist/entry.js`.
   - Prevents regressions that only appear outside Bun’s loader.

---

### E) Discord signal-to-noise (recruiting/solicitation) — **Impact: Medium | Difficulty: Low**
1. **Add a dedicated #jobs / #hiring / #freelance channel + enforce posting rules**
   - Auto-mod: redirect solicitation posts from #coders/#discussion to #jobs.
   - Similar approach: many OSS Discords isolate recruiting to protect support channels.

2. **Structured help intake**
   - Add a “Help” forum-style channel with required fields (version, platform, logs, minimal repro).
   - Similar approach: Rust/Reactiflux-style question templates improve answer rate and searchability.

## 4) Communication Gaps (expectations vs reality)

### Gaps observed
- **“Supports voice/Discord/images” vs “works out-of-the-box”**
  - Users asked if voice and image generation are supported; answer is “yes via plugins/models,” but the setup complexity is not front-loaded (Discord 2026-06-07).
  - Improvement: a single “Companion Stack Guide” with a compatibility matrix (Discord plugin + TTS/STT providers + image providers + required config).

- **Cloud Apps scaling expectations**
  - Audit feedback (#8321) explicitly warns real ceiling is “dozens” without pooling/ingress/LB, while users may expect “hundreds/thousands” due to product positioning (“Apps” platform).
  - Improvement: publish an “Apps readiness & limits” doc (current scale envelope, what’s implemented, what’s TODO).

- **Token migration lifecycle clarity**
  - Migration question indicates users miss deadlines and learn only by asking (Discord 2026-06-07).
  - Improvement: pin a “Token/migration status” post and link it from onboarding/FAQ.

### Recurring questions that indicate doc/onboarding gaps
- “Can I integrate X into my Discord server?” (voice + images + model providers)
- “How do I monetize / charge per query?” (x402, MCP servers, plugins)
- “Is Cloud login working / how do I authenticate?” (implied by severity of #8257)

### Current channel effectiveness
- **Discord**: fast for lightweight Q&A, but in this window it’s dominated by solicitation and has low “resolved help” density (e.g., 2026-06-08 had zero resolved help interactions).
- **GitHub**: extremely effective for high-quality technical feedback (repros, audits, PR-ready proposals).

### Improvements for more structured, actionable feedback

   - A command like `/issue` that posts a draft issue template with logs/repro fields.
2. **Add “Support Intake” forms**
   - Separate forms for: Cloud auth, Cloud Apps deploy, Orchestrator bugs, Plugin integrations.
3. **Telemetry-informed prompts**
   - When failures happen (login 400, provisioning fail), show “Report issue” with prefilled context (request id, env, commit SHA).

### Underrepresented segments
- Non-crypto enterprise automation users are not visible in this sample.
- Non-developer end users (desktop app users) are underrepresented compared to infra/orchestrator power users.
- Suggest targeted outreach: short survey embedded in desktop UI + cloud dashboard (“what are you building?”).
