## 1) Pain Point Categorization (top recurring friction areas)

### 1) Integration — Cloud authentication / Steward SSO reliability (**High severity, high frequency**)
**Observed in:** 1/7 items (≈14%) but **platform-wide impact**.  
**Example:** GitHub #8257 “Steward login broken in prod: OAuth authorize 400 (code_challenge) + email/send 400 — blocks all Sign in with Eliza Cloud” (OPEN).  
**User impact:**
- Complete login failure blocks **dashboard + CLI login + third-party apps** that depend on “Sign in with Eliza Cloud”.
- Shows a brittle coupling between Steward auth hardening (PKCE requirement) and elizaOS Cloud frontend/worker flows.

**Primary problems affecting most users**
- OAuth flows and email magic-link flows can regress with upstream Steward changes.
- Staging/prod tenant isolation and redirect/callback expectations are easy to misconfigure.

---

### 2) UX/UI — Orchestrator “deliverable” presentation defects (correct work, poorly surfaced) (**High severity for trust**)
**Observed in:** 1/7 items (≈14%).  
**Example:** GitHub #8162 “sub-agent completion narration drops/garbles the deliverable” (CLOSED).  
**User impact:**
- Users perceive the agent as failing even when sub-agents produce correct artifacts.
- Output truncation, dropped stdout, and process-chatter reduces confidence in multi-agent flows.

**Primary problems affecting most users**
- Deliverable vs. transcript is not structurally separated; the parent model can summarize/truncate.
- Retries and stale attempts can leak into the user-facing message stream.

---

### 3) Technical Functionality — Build/runtime dependency resolution outside Bun loader (**Medium severity, developer-blocking**)
**Observed in:** 1/7 items (≈14%).  
**Example:** GitHub #8142 “app-core dist/entry.js externalizes un-hoisted transitive deps … ERR_MODULE_NOT_FOUND” (CLOSED).  
**User impact:**
- Downstream tooling (e.g., `doctor` or Node-style runners) fails unexpectedly.
- Creates “works on Bun, breaks elsewhere” confusion for contributors and integrators.

**Primary problems affecting most users**
- Transitive runtime deps not hoisted + bundler externals = non-portable runtime artifacts.
- Poorly signposted runtime assumptions (Bun workspace loader vs Node resolution).

### 4) Performance/Runtime Stability — Local inference prewarm crash (Windows/CUDA batch sizing) (**High severity for local users on Windows**)
**Observed in:** 1/7 items (≈14%).  
**Example:** GitHub #7991 “prewarm violates GGML_ASSERT … on eliza-1-0_8b” (CLOSED).

- Undermines “desktop local-first” promise if model warmup is unstable.

**Primary problems affecting most users**

- Bundle manifests/context recommendations not consistently enforced.

## 2) Usage Pattern Analysis (actual vs intended usage)

### How users are using elizaOS (observed)
1. **Agentic finance / DeFi automation** (Discord 2026-06-06): building an *autonomous DeFi portfolio manager* using **x402** (buildathon context).  
   - Indicates a strong pull toward **high-stakes integrations** where reliability + safety + auditability matter.
2. **Cloud-hosted agents and “apps” as a platform** (GitHub/PR activity): strong momentum around **Eliza Cloud Apps** (per-tenant isolated container hosting) and SSO flows.
3. **Multi-agent orchestrator as a “coding workbench”**: issues and PRs show heavy investment in orchestrator UX (deliverables, inspector/workbench, Odysseus-style UI).
4. **Personal assistant domain decomposition**: large refactors (LifeOps → focused plugins) suggest real usage demands modular “daily life” capabilities.

### Mismatches vs intended usage (implied by feedback)
- Intended: “Cloud SSO is a stable entrypoint.”  
  Actual: auth regressions can block the entire platform (#8257), indicating the SSO path is *more fragile than users expect*.
- Intended: “Orchestrator completes tasks and reports results.”  
  Actual: users experience missing/truncated deliverables even when tasks succeed (#8162), i.e., success is not reliably observable.

### Emerging / unexpected use cases
- **Buildathon-grade autonomous finance agents** (portfolio management) suggests demand for:
  - Spend caps, approvals, audit logs, replayable traces, and deterministic “what happened” outputs.
- **Third-party app developers** using Cloud SDK + CORS + login helpers (seen across multiple Cloud-focused PRs) indicates elizaOS is becoming a **platform** more than a standalone app.

### Feature requests aligned with actual usage patterns (inferred)
- Better **project scaffolding** and “first plugin/app” templates for people arriving via buildathons.
- Stronger **structured output contracts** in orchestrator so results can be consumed by finance dashboards, bots, or CI jobs without UI ambiguity.

## 3) Implementation Opportunities (solutions per major pain point)

### A) Cloud auth / Steward SSO fragility (Integration)
**Opportunity 1 (High impact / Medium difficulty): Add “auth contract tests” against a Steward mock + prod canary**
- Implement Playwright/API-level canaries for:
  - OAuth authorize + exchange (PKCE)
  - magic-link send + verify
  - refresh session behavior
- Run in CI and as scheduled prod canary.
- Similar approach: **NextAuth** and **Supabase** projects commonly ship provider regression tests and environment-specific canaries to catch upstream OAuth changes early.

**Opportunity 2 (High impact / Low–Medium difficulty): Centralize OAuth/PKCE logic in one shared module**
- Single “source of truth” for:
  - `redirect_uri` construction
  - PKCE generation/storage
  - nonce exchange signing
- Reduces drift between frontend and worker routes (root cause pattern in #8257).

**Opportunity 3 (Medium impact / Low difficulty): “Status page” + user-facing error mapping**
- Map 400/401/500 auth failures to actionable UI messages (e.g., “Steward auth regression—retry later; incident link”).
- Reduces Discord support load and prevents users from repeatedly trying broken flows.

---

### B) Orchestrator deliverable loss / truncation (UX/UI)
**Opportunity 1 (High impact / Medium difficulty): Make deliverables first-class, verbatim-carried artifacts**
- When a sub-agent completes, store:
  - `deliverable` (verbatim, size-limited)
  - `transcript` (optional, collapsible)
- Render deliverable directly without model re-summarization.  
- Similar approach: **GitHub Copilot Chat** and **Claude Code** separate “final output” from tool traces; **LangChain** often persists tool outputs as structured fields.

**Opportunity 2 (High impact / Low difficulty): UI-level “Result header” that always appears**
- Add a consistent “Result” block and push chatter into collapsed “Details”.
- Helps users quickly verify success and reduces perceived flakiness.

**Opportunity 3 (Medium impact / Medium difficulty): Retry attempt suppression in user feed**
- Only show terminal attempt unless user expands “Attempts”.
- Preserve internal retry signals without narrating intermediate failures.

---

### C) Bun/Node dependency resolution mismatches (Technical Functionality)
**Opportunity 1 (High impact / Medium difficulty): Enforce runtime portability in build output**
- For `app-core dist/entry.js`, either:
  - bundle runtime deps inline, or
  - declare transitive runtime deps as direct deps so they hoist predictably.
- Similar approach: many TS libraries using **tsup/esbuild** provide a “fully bundled” CJS/ESM artifact to avoid runtime resolution surprises.

**Opportunity 2 (Medium impact / Low difficulty): Add a “supported runtimes” matrix to docs + CLI preflight**
- At runtime, detect “running outside Bun loader” and provide a single actionable error (“use bun run … or install hoisted deps”).

**Opportunity 3 (Medium impact / Low difficulty): Expand `doctor` to validate module resolution**
- A lightweight module-resolution probe for known runtime deps reduces whack-a-mole failures.

---

### D) Local inference prewarm crash / batch sizing (Performance)
**Opportunity 1 (High impact / Medium difficulty): Clamp prewarm tokens to `n_batch` and model constraints**
- Centralize a safe prewarm policy:
  - `prewarmTokens = min(recommended, n_batch, hardCeiling)`
- Similar approach: llama.cpp-based apps (e.g., **LM Studio**-style clients) keep conservative warmup defaults and clamp to runtime settings.

**Opportunity 2 (Medium impact / Low difficulty): Add platform-specific defaults**
- Windows/CUDA profiles may need smaller prewarm batches by default.

**Opportunity 3 (Medium impact / Medium difficulty): Add a “safe mode: skip warmup” toggle**
- If warmup fails, recover by disabling warmup automatically and notifying the user.

---

### E) Newcomer onboarding & information overload (Documentation + Community)
**Opportunity 1 (High impact / Low difficulty): Publish a “3 paths” onboarding page**
- “I want to:”
  1) run desktop locally,  
  2) provision a cloud agent,  
  3) build a plugin/app.
- Include time estimates, prerequisites, and the first successful milestone for each.

**Opportunity 2 (Medium impact / Medium difficulty): In-product contextual help**
- Tooltips/links from the UI to the exact doc section (e.g., orchestrator, plugins, cloud login troubleshooting).

**Opportunity 3 (Medium impact / Low difficulty): Curated “starter prompts” + sample projects**
- Build on the existing prompt suggestions work by adding a “Try these 5 tasks” per path (local/cloud/plugin).

---

### F) Project-to-contributor matchmaking (Community)
**Opportunity 1 (High impact / Low difficulty): Create a “#project-board” channel + pinned template**
- Template fields: repo link, goal, timebox, required skills, paid/unpaid, mentor availability.

**Opportunity 2 (Medium impact / Medium difficulty): Weekly “office hours / pairing hour”**
- 60 minutes where project owners pitch and contributors join breakouts.

**Opportunity 3 (Medium impact / Low difficulty): Tag GitHub issues as “help wanted: buildathon”**
- Converts buildathon energy into trackable issues + PRs.

## 4) Communication Gaps (expectations vs reality)

### Gap 1: “Agent succeeded” vs “user saw success”
- **Signal:** #8162 shows capability correctness but UX presentation failure.
- **Fix:** Document and UI-label the difference between:
  - tool transcript,
  - final deliverable,
  - retries/attempts.

### Gap 2: “Eliza Cloud sign-in is stable” expectation
- **Signal:** #8257 is a full-blocker; users can’t self-diagnose.
- **Fix:** Add a Cloud status indicator and a troubleshooting page (“If you see code_challenge errors…”).

### Gap 3: “Where do I learn the ecosystem?” repeated newcomer question
- **Signal:** Discord user explicitly overwhelmed; directed ad hoc to “coders channel”.
- **Fix:** Replace “ask in Discord” as the primary path with an explicit onboarding doc that Discord mods can link.

## 6) Feedback Collection Improvements

### Current channel effectiveness
- **Discord:** good for discovery and social support, but feedback is **unstructured** and hard to convert into issues (e.g., recruitment requests, onboarding confusion).
- **GitHub:** strong for technical issues, but often lacks **user environment + repro standardization** unless the reporter is experienced.

### Improvements for more structured, actionable feedback
1. **In-app “Send feedback”** (desktop + cloud UI) with:
   - logs snapshot, platform, version, active plugins, last error.
2. **Discord → GitHub bridge workflow**
   - A slash-command like `/issue` that opens a prefilled GitHub issue template.
3. **Monthly UX pulse survey** (5 questions)
   - Track “time-to-first-success”, “login success”, “first orchestrator success”, etc.

### Underrepresented segments
- **Non-developer end users** (people who want assistants, not frameworks): little direct feedback besides “overwhelm”.
- **Mobile-first users**: some evidence of onboarding fixes in PRs, but little qualitative feedback in this snapshot.
- **Third-party app builders**: present via Cloud SDK work, but their pain points are likely underreported unless explicitly solicited.
