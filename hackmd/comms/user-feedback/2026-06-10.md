## User Feedback Analysis — 2026-06-10 (based on 2026-06-07 to 2026-06-09 signals)

### Data notes (for quantification)
This analysis is based on **~9 discrete feedback signals** captured across Discord summaries (plugin registry workflow questions, integration/how-to questions, voice/Discord plugin audit mention, cloud status question) and GitHub issues/PR context referenced in the dataset (notably **#8321, #8257, #8162, #8142**). Percentages below are therefore directional and reflect “share of observed signals,” not a statistically representative user survey.

## 1) Pain Point Categorization (Top recurring 5–7)

### 1) **Integration — Cloud auth & onboarding reliability (high severity)**
**What users are experiencing**
- **Steward / Eliza Cloud login breaks** block all downstream apps using “Sign in with Eliza Cloud” (**GitHub #8257**, open). Users see OAuth 400s (PKCE/code_challenge) and email/send 400s, preventing access entirely.
- Cloud pairing / agent access has required multiple follow-up fixes (e.g., token via query param for SSE, WS auth handshake constraints), indicating friction in browser-based integrations even when core is functioning.

**Frequency (observed):** ~**22%** (2/9 signals) but **highest severity** (blocks usage).

---

### 2) **Technical Functionality — Eliza Cloud Apps scalability + ingress gaps (high severity)**
**What users are experiencing**
- A code-level audit highlights that the Cloud Apps data plane is “sound” but has **pre-launch blockers** and scale ceilings (**GitHub #8321**, open): ingress not installed (apps unreachable), Postgres connection ceilings/no pgbouncer, env-var contract risks, single-node placement, docker network exhaustion, port collisions, teardown leaks, and multi-node routing gaps.
- This manifests as: “can’t reach apps,” “won’t scale beyond dozens,” and “isolation might silently not be active.”

**Frequency (observed):** ~**11%** (1/9) but very high impact (platform viability).

---

### 3) **Documentation — Plugin submission workflow & registry expectations (medium severity, high frequency)**
**What users are experiencing**
- Confusion about **where** to submit third-party plugins and **who reviews**/**how long it takes** (Discord: PR **#8294** registry entry; questions about targeting `elizaOS/eliza`, `develop`, and turnaround time).
- Review ownership is described as “Shaw’s agent,” with an **~48-hour** expectation, but this isn’t surfaced as a predictable, trackable workflow to contributors.

**Frequency (observed):** ~**22%** (2/9), moderate severity (contribution friction).

---

### 4) **Technical Functionality / UX — Orchestrator output presentation (“deliverable dropped/garbled”) (medium severity)**
**What users are experiencing**
- Sub-agent “task_complete” narration can **drop/truncate** the actual deliverable (stdout/printed values) even when underlying work is correct (**GitHub #8162**, closed but important UX signal).
- Symptoms include truncation by the parent model, stale retries leaking, and excess process chatter.

**Frequency (observed):** ~**11%** (1/9), medium severity (trust/clarity).

---

### 5) **Performance / Technical Functionality — Real-time voice + Discord synchronization under load (medium severity)**
**What users are experiencing**
- A community member reported auditing `@elizaos/plugin-discord` voice manager behavior under load (Discord: `voice.ts` + real-time voice event management and text synchronization). This points to perceived fragility/performance risk in voice+Discord deployments.

**Frequency (observed):** ~**11%** (1/9), medium severity, likely to grow as voice companions increase.

---

### 6) **Integration / Packaging — Runtime dependency resolution outside Bun’s loader (low–medium severity)**
**What users are experiencing**
- Built artifacts (`packages/app-core dist/entry.js`) externalize transitive deps leading to `ERR_MODULE_NOT_FOUND` outside Bun’s loader (**GitHub #8142**, closed). This impacts “doctor”/Node-style execution and downstream QA.

**Frequency (observed):** ~**11%** (1/9), medium impact for tooling adoption.

---

### 7) **Community / Expectation-setting — “Is the project currently working?” (low severity, reputational)**
**What users are experiencing**
- A user asked in general chat whether the project is “currently working” (Discord daily JSON summary). This is a classic signal of unclear “known issues,” release readiness, and onboarding confidence.

**Frequency (observed):** ~**11%** (1/9), low immediate severity but reputational risk.

## 2) Usage Pattern Analysis (actual vs intended)

### Observed real-world usage patterns
1. **AI companions on Discord with voice + images**  
   - Users are assembling stacks like **OpenRouter (GLM-5) + ElevenLabs + image generation** and want to integrate into personal Discord servers (Discord: thaoryel).  
   - This suggests elizaOS is being used less as “enterprise automation” and more as **multi-modal social agents**.

2. **Crypto / trading / risk tooling as agents + MCP servers**
   - “Cabal-Hunter” is positioned as a **pay-per-query MCP server** ($0.05 USDC/query) integrated via `@hugen/plugin-x402-solana` and invoked before swaps (`check_cabal_risk(mintAddress)`) (Discord: paulf76).
   - Users are applying elizaOS as an **agent wrapper around specialized APIs** with monetization and safety checks.

3. **Cloud Apps as a hosting substrate (Product 2)**
   - The Cloud Apps data plane work and audit (#8321) shows intended usage: deploy isolated apps with per-tenant DB + domains + billing.  
   - Actual usage is at the “pre-go-live validation” stage: testing ingress, DB isolation, scaling.

### Emerging/unexpected applications
- **Third-party “agent registry” services** (CallTyro) that aim to make agents discoverable by other agents suggests a shift toward **federated discovery** and “agent marketplaces,” adjacent to the project’s own registry model.

### Feature requests that align with usage
- Better **Discord+voice reliability guidance** (reference architectures, load limits).
- First-class **MCP server integration patterns** (templates, tooling, payments hooks).
- A clearer **plugin distribution and discoverability story** (registry + external registries like CallTyro).

## 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

### A) Cloud auth & onboarding reliability (Steward OAuth/email) — **Impact: Very High / Difficulty: Medium**
1. **Add an automated “Auth Contract” CI + canary check**
   - Implement a nightly (and pre-deploy) suite that exercises:
     - OAuth authorize URL generation (PKCE required)
     - email magic link send/verify
     - token refresh
   - Similar approach: **NextAuth** and many SaaS platforms run “login smoke tests” against staging/prod endpoints before promotion.

2. **Publish a public status + “Known Issues” banner in Cloud UI**
   - If auth is degraded, show an explicit banner in `/login` and dashboard with workaround/ETA.
   - Similar approach: **Vercel / GitHub Status** style incident messaging reduces “is it working?” confusion.

3. **Harden backward-compat negotiation with Steward**
   - Version-pin or capability-detect required PKCE behaviors and fail with actionable error copy (“Your deployment requires PKCE; update cloud-frontend ≥ X”).
   - Similar approach: **Stripe API versioning** (explicit contract) and **OAuth feature flags**.

---

### B) Cloud Apps data plane scaling + ingress gaps (#8321) — **Impact: Very High / Difficulty: High**
1. **Ship P0 blockers as a single “Go-live Gate” checklist**
   - Track P0 items (ingress installed, pgbouncer/connection pooling, env-var contract verification) as a merge-blocking milestone.
   - Similar approach: **Kubernetes release criteria** and “production readiness gates.”

2. **Adopt standard ingress + pooling primitives**
   - Ingress: install and manage **Caddy/Traefik/Nginx** with declarative routing + health checks.
   - DB: add **pgbouncer** at tenant-db nodes or sidecar pattern; enforce pool sizes.
   - Similar approach: **Heroku-style routing layer** + pgbouncer in many multi-tenant Postgres systems.

3. **Make isolation verifiable and visible**
   - Add a “DB mode: none/isolated” badge plus a runtime self-test endpoint that confirms the app is using the isolated DSN (not falling back silently).
   - Similar approach: **Supabase** and **Render** surface “connected resource” state and logs to prevent silent misconfiguration.

---

### C) Plugin registry workflow confusion — **Impact: High / Difficulty: Low**
1. **Add a “Submit a plugin” wizard doc + checklist in `packages/registry/README.md`**
   - Include: target repo/branch, file path, JSON schema, CI expectations, and reviewer SLA (“~48 hours”).
   - Provide a copy/paste PR template snippet.

2. **Automate validation + preview**
   - Add a GitHub Action that validates registry entries and posts a comment preview (name, npm link, categories, security notes).
   - Similar approach: **Homebrew formula checks** and **VS Code extension marketplace** validation.

3. **Make review ownership explicit**
   - Use CODEOWNERS for registry entries and show the assigned reviewer bot/user.
   - Similar approach: **Kubernetes OWNERS files** make routing predictable.

---

### D) Orchestrator “deliverable dropped/garbled” — **Impact: Medium / Difficulty: Medium**
1. **Implement a structured “deliverable channel”**
   - Persist stdout/printed result in metadata and render verbatim when present (as proposed in #8162).
   - Similar approach: **Claude Code/Codex** style “Result” panels that separate output from narration.

2. **Add a “final answer only” rendering mode**
   - UI toggle: “Show process chatter” vs “Deliverable only.”
   - Similar approach: IDE assistants often separate “explanation” vs “diff/output”.

3. **Suppress stale retry attempts in user-facing transcript**
   - Keep retries internal unless final attempt fails.
   - Similar approach: job runners (e.g., **Temporal**) show final state while keeping retry history in an inspector.

---

### E) Voice + Discord under load — **Impact: Medium / Difficulty: Medium**
1. **Document a supported concurrency envelope**
   - Provide tested limits: max concurrent speakers, recommended buffering strategy, fallback behavior on desync.
2. **Add load-test harness for `plugin-discord` voice**
   - Synthetic voice event generator + assertions for text sync correctness.
3. **Expose runtime observability**
   - Metrics: queue depth, dropped frames/events, latency; surfaced in a debug view.
   - Similar approach: **Discord.js** community recommends explicit rate/voice state instrumentation.

## 4) Communication Gaps (expectations vs reality)

### Recurring mismatches
- **Expectation:** “I submitted a plugin—what happens next?”  
  **Reality:** Review is manual (“Shaw’s agent”), ~48h typical, but not tracked in a contributor-visible queue.
  - Improvement: publish an explicit SLA + label states (`registry:queued`, `registry:needs-info`, `registry:approved`).

- **Expectation:** “Cloud login should work consistently; it’s foundational.”  
  **Reality:** Steward hardening (PKCE/HMAC) can break flows; users discover via CLI-login and get blocked (#8257).
  - Improvement: add a Cloud “Auth Compatibility” page documenting required client versions and current auth modes.

- **Expectation:** “If an agent ran a command, I’ll see the output.”  
  **Reality:** Orchestrator narration can strip the tool output (#8162).
  - Improvement: product language + UI affordance: “Results are shown in Output blocks; narration may summarize.”

### Questions indicating onboarding/documentation gaps
- “Is the project currently working?” (Discord) suggests missing:
  - a “Start here” health check page
  - a release/stability channel (what’s stable vs experimental)
  - a simple self-test (“run `eliza doctor`” equivalent) with known failure modes.

## 6) Feedback Collection Improvements

### Effectiveness of current channels
- **Discord**: good for quick answers (e.g., registry workflow), but low structure; questions repeat and resolution state isn’t durable.
- **GitHub issues**: high quality for deep technical audits (#8321) and reproducible auth failures (#8257), but can be intimidating for newcomers.

### Improvements for more structured, actionable feedback
1. **Weekly “Top friction” form** (2 minutes)
   - Fields: environment (local/cloud), feature area, blockers, logs, expected behavior.
2. **Discord bot → GitHub issue bridge**
   - For tagged messages like `#bug` or `#docs`, prompt users to auto-generate an issue with context.
3. **Registry submission telemetry**
   - Track: time-to-first-response, time-to-merge, and rejection reasons; publish monthly.

### Underrepresented user segments
- **Non-crypto builders** (education, productivity, customer support) are largely absent in the captured feedback.
- **Non-developer end users** (people running a companion, not building one) are underrepresented—yet companion usage is clearly a key pattern.
- **Enterprise/IT operators** (SSO, compliance) are implied by infra work but not directly represented as “users” providing feedback.
