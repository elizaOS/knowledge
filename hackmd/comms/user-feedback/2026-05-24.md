## User Feedback Analysis — 2026-05-24 (based on community signals through 2026-05-21 to 2026-05-23 + recent GitHub issues/PR context)

### Dataset note
This analysis is based on a small but high-signal sample (≈11 distinct feedback threads/signals) drawn from Discord discussions (token/migration, Babylon access, roadmap, onboarding/demo requests, dockerization for VoiceAI, security research outreach) plus recent GitHub issue themes around local inference/speculative decoding, build/onboarding sharp edges, and hardware/runtime support.

---

## 1) Pain Point Categorization (Top recurring 5–7)

### 1. Documentation (High frequency, High severity)
**What users report**
- **Onboarding gaps / lack of demos:** multiple users asked for “live demonstrations and YouTube examples” to understand capabilities and best practices (Discord 2026-05-21).  
- **Roadmap discoverability:** “What does the next few months look like for ElizaOS?” went unanswered (Discord 2026-05-23), indicating users don’t know where to find authoritative planning info.
- **Lifecycle/deprecation clarity:** confusion around ended processes and shutdown projects (ai16z→ElizaOS migration ended; Babylon shut down / partially repurposed into “Feed”).

**Estimated prevalence (in this sample):** ~36% (4/11 signals) relate to “Where is the doc / what is the current status?”

---

### 2. Technical Functionality (High frequency, High severity)
**What users report**
- **Speculative decoding incompatibilities / silent fallback:** GitHub issues show modern model architectures (e.g., Qwen3.5 M-RoPE; SWA-based bodies) causing speculative decoding to fail or be silently disabled, collapsing throughput (e.g., acceptance ~9% and <1 t/s; DFlash disabled without prominent API/UI signaling).
- **Model/hardware path unpredictability:** performance cliffs on specific hardware/model combinations (Intel Lunar Lake iGPU) suggest kernel selection or backend heuristics can produce “unusable” outcomes in edge cases.

**Estimated prevalence:** ~27% (3/11) are about inference/runtime correctness or “it runs but not as expected.”

---

### 3. Integration (Medium frequency, High severity)
**What users report**
- **Dockerization/deployment complexity for real-time voice agents:** users attempting to containerize VoiceAI systems hit GPU passthrough, networking, persistent storage, latency stability, and service composition complexity (Discord 2026-05-21). There’s explicit interest in “single-container” deployment for mass rollout, which conflicts with best-practice architecture.

**Estimated prevalence:** ~9% (1/11), but high severity because it blocks production deployment.

---

### 4. Community / Support Responsiveness (Medium frequency, Medium severity)
**What users report**
- **Unanswered roadmap question:** indicates expectation of timely direction-setting responses in community spaces.
- **New contributors unsure where to start:** multiple intros from experienced engineers (Web3, full-stack/AI) seeking ways to help, but no clear next step surfaced in-thread.

**Estimated prevalence:** ~18% (2/11).

---

### 5. UX/UI (Lower frequency, Medium severity)
**What users report (implied)**
- Users repeatedly ask “where do I buy tokens” and “can I still migrate,” suggesting key info is not surfaced in a canonical, easy-to-find UI/landing location (even if technically “not product UI,” it’s still user-facing UX).
- For speculative decoding, the pain is partly UX: the system can appear “enabled” while actually falling back (needs better status surfacing in UI/health endpoints).

**Estimated prevalence:** ~9% (1/11), but compounding impact.

---

### 6. Performance (Lower frequency, High severity for affected segment)
**What users report**
- Hardware-specific throughput cliffs (e.g., Qwen2.5-14B Q4_K_M on Lunar Lake iGPU) can drop generation rates to ~1 t/s, effectively blocking use on that class of device.

**Estimated prevalence:** ~9% (1/11) in this sample, but high severity for “local inference on consumer hardware” adopters.

## 2) Usage Pattern Analysis (Actual vs intended usage)

### How users are using elizaOS in practice
1. **Agentic Web3 products in production:** waifu.fun launched as an “agentic launchpad” built on elizaOS v3 agents (Discord 2026-05-22). This indicates real-world adoption in public Web3-facing apps, not just demos.
2. **Voice agents as deployable services:** users are building white-label VoiceAI platforms and focusing on local deployment + GPU acceleration + latency constraints (Discord 2026-05-21).
3. **Wallet-connected agents and security concerns:** community research outreach specifically targets agents with wallet access, emphasizing spending caps and prompt injection defenses (Discord 2026-05-23).
4. **Local inference on heterogeneous hardware:** GitHub feedback centers on Vulkan/iGPU, speculative decoding, and runtime backends—suggesting a meaningful segment is pushing “run locally” rather than cloud-only.

### Emerging / unexpected use cases
- **Security research & audit readiness** as a first-class concern (wallet access, injection protection) is becoming part of how the ecosystem is evaluated.
- **Single-container distribution desire** for complex systems (voice) suggests users want “productized” packaging more than “framework composition.”

### Feature requests aligned with these patterns
- **Clear roadmap + status signals** (especially for deprecations and major shifts like v2 identity architecture).
- **Deployment reference architectures** (voice, GPU, multi-service) and “supported patterns” documentation.
- **Explicit runtime capability/status reporting** (e.g., “spec decode active/inactive”) for reliability in production.

---

### B) Speculative decoding correctness + silent fallback (Technical Functionality + UX)
**Solutions**
1. **(High impact / Medium effort) Add explicit runtime status surfacing**
   - Add `spec_decode_active` + `spec_decode_reason` to `/health` and model info responses (mirroring the GitHub suggestion for DFlash/SWA incompat).
2. **(High impact / Medium effort) Fail loud for incompatible configurations**
   - If user passes `--spec-type dflash` but context is incompatible, return an error (or at minimum a prominent warning in API response headers/fields).
3. **(Medium impact / High effort) Add architecture-aware guards in the recommender**
   - Gate Qwen3.5/3.6 tiers behind “M-RoPE spec-decode supported” and SWA+DFlash compatibility until validated.

**Comparable approaches**
- llama.cpp ecosystem commonly uses explicit runtime flags and verbose “feature active” summaries on startup; extending this into API responses prevents “it looks enabled” failures.
- Ray Serve / Modal: explicit deployment health + capability reporting avoids silent degradation.

---

### C) Deployment friction for VoiceAI dockerization (Integration + Performance)
**Solutions**
1. **(High impact / Medium effort) Publish an official “Voice Agent Reference Deployment”**
   - Compose template: inference (GPU), API gateway, DB, queue/worker, observability; include NVIDIA Container Toolkit notes and latency tuning.
2. **(Medium impact / Low effort) Document supported vs unsupported packaging**
   - Provide a clear stance: “single container is possible but not recommended; here’s when it’s acceptable,” plus a minimal “all-in-one” dev image.
3. **(Medium impact / Medium effort) Provide a “one-command local stack”**
   - `docker compose up` with sensible defaults, health checks, and persistent volumes.

**Comparable approaches**
- OpenAI Whisper deployments and many RAG templates provide compose-based “reference stacks.”
- Supabase examples show how to make multi-service feel “single install” via compose + scripts.

## 6) Feedback Collection Improvements

### Current channels effectiveness
- **Discord:** good for quick Q&A, but important questions (roadmap) can get lost; resolution quality depends on who is online.
- **GitHub issues:** excellent for deep technical feedback (repro steps, benchmarks), but mostly represents power users; less visibility into novice pain.

### Improvements for more structured, actionable feedback
1. **Weekly “Top Questions” digest from Discord → GitHub Discussion**
   - Summarize recurring questions (roadmap, token, migration, demos) and answer once canonically.
2. **Add a lightweight “Deployment/Onboarding” issue template**
   - Capture environment, goal (voice/local/cloud), blockers, and desired docs.
3. **Telemetry-light user survey for installers**
   - After install/onboarding: “what blocked you?” (docs, GPU, connectors, auth, voice).

### Underrepresented segments
- **Non-Discord users** (only filing GitHub issues) who may struggle with token/project status info.
- **Less-technical builders** who need video demos and step-by-step guides rather than architecture guidance.
- **Production operators** (DevOps/SRE) whose pain is packaging, observability, and reliability; only lightly represented by the VoiceAI docker thread.
