## 1) Pain Point Categorization (Top recurring 5–7)

### 2. Community / Communication — **Release expectation drift (V3 “imminent” without a date)**
**Frequency/Severity:** Medium severity, moderate frequency (1 of 4 = **25%**).  
**Examples:**
- **lalanotia** noted “two weeks” since a statement that V3 was imminent; asked for updated status; no concrete answer in the captured discussion.

**Problem affecting most users:**
- Users plan builds/migrations around release claims; lack of date/criteria creates distrust and repeat questions.

---

### 3. Technical Functionality — **Speculative decoding incompatibilities on modern model architectures (M‑RoPE, SWA)**
**Frequency/Severity:** High severity for local-inference power users; appears repeatedly in GitHub issues cluster (3+ linked issues; effectively the dominant technical theme in the provided GitHub top issues).  
**Examples:**
- **#7631:** M‑RoPE position validation rejects cross-drafter Qwen3.5 batches → acceptance collapses to ~9% and **tg < 1 t/s**.
- **#7635:** SWA-based bodies (Qwen3.6, gemma-4) silently disable DFlash spec-decode due to KV cache limitations.

**Problem affecting most users:**
- Users believe “spec decode is on” but it may be **silently disabled** (#7635), producing confusing performance and undermining “recommended tiers” in the local-inference catalog.

---

### 4. Performance — **Hardware/kernel selection causes unusable throughput on some model+device combos**
**Frequency/Severity:** High severity for a subset (Intel iGPU / Vulkan users); recurring within the same reporter’s issues but points to a broader class of problems (kernel heuristics & recommender).  
**Examples:**
- **#7638:** Qwen2.5‑14B Q4_K_M on Lunar Lake iGPU Vulkan runs ~**1.15 t/s**, far below expectations; suggests kernel-selection misrouting.

**Problem affecting most users:**
- Users pick “bigger model” expecting incremental slowdown; instead they hit cliffs that feel like “broken” performance.

---

### 5. Integration / UX — **OpenVINO expectations vs reality (NPU is great for Whisper, not for LLM decode)**
**Frequency/Severity:** Medium severity; shows a mismatch between “NPU support” expectations and what it practically improves.  
**Examples:**
- **#7633 (RFC):** OpenVINO backend works, but NPU **not a win for autoregressive LLM tg**; best use is **Whisper ASR on NPU**.

**Problem affecting most users:**
- “NPU support” can be misunderstood as “faster chat.” Users need clear routing guidance: NPU for ASR, iGPU for LLM decode (on relevant Intel hardware).

---

### 6. Technical Functionality / DX — **Build/runtime sharp edges (missing preload artifacts; CI/Docker failures)**

### 7. Community — **Low activity periods reduce support throughput and newcomer retention**

## 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

### Pain point: Late migration support is blocked + no response path
**High impact / Low–Med difficulty**
1. **Publish a “Late Migration” runbook** (docs + pinned Discord post)  
   - Include: required versions, steps, what’s no longer supported after deadline, and manual recovery paths.
2. **Add a dedicated Discord “migration-support” intake with a simple form** (bot-assisted)  
   - Collect: old version, data location, errors, deadline missed reason; auto-create a GitHub Discussion/Issue if unresolved after 24–48h.
3. **Define an escalation/SLA for admin-contact** (even lightweight)  
   - Example approach: Kubernetes-style deprecation windows + clearly documented “post-window” support level; or Home Assistant “breaking changes” pages with remediation steps.

---

### Pain point: V3 release timeline uncertainty
**High impact / Low difficulty**
1. **Replace “imminent” with a public release criteria checklist**  
   - E.g., “V3 ships when: A, B, C are green,” with links to issues/PRs.
2. **Weekly release heartbeat post** (Discord + GitHub Discussion)  
   - Even “no change” reduces repeated questions.
3. **Adopt a “target + confidence” model**  
   - Example: Rust project-style: “Target week of X; confidence: medium; risks: …”.

---

### Pain point: Speculative decoding silently disabled or incompatible (M‑RoPE, SWA)
**Very high impact / Med difficulty**
1. **Fail loud + surface status in APIs/UI**  
   - If `--spec-type dflash` is requested but incompatible, return a clear error and expose `spec_decode_active=false` via `/health` or `/v1/models/<id>` (mirrors #7635 suggestion).
2. **Capability gating in the recommender/catalog**  
   - Add explicit capability flags (e.g., `mrope-spec-decode`, `swa-partial-seq-removal`) and **prevent recommending tiers** that rely on spec-decode when it cannot function.
3. **Port or implement SWA partial-sequence removal support**  
   - Use upstream/fork reference noted in #7635 (spiritbuun/buun-llama-cpp lineage).  
   - Comparable pattern: llama.cpp backend feature flags + hard gating in frontends (LM Studio/Ollama-like “this backend doesn’t support X” UI messaging).

---

### Pain point: Performance cliffs from kernel selection / routing
**High impact / Med difficulty**
1. **Add “why this tier was chosen” diagnostics in local inference**  
   - Show: selected kernel, quant compatibility, whether optimized kernels applied, and expected tg range.
2. **Automated benchmark sampling for common hardware tiers**  
   - CI-like “microbench suite” on representative devices (or community-run harness upload) to detect regressions/cliffs.
3. **Kernel-selection logging + user-facing “performance troubleshooting” doc**  
   - Similar to how projects like PyTorch/TensorRT advise enabling verbose kernel dispatch logs and provide “known slow paths.”

---

### Pain point: OpenVINO messaging & routing (NPU vs iGPU expectations)
**High impact / Low difficulty**
1. **Document “NPU is for ASR first” prominently**  
   - Put this in the model/runtime picker and in a “Voice agent on Intel” guide.
2. **Split runtime selection by task type** (ASR/TTS/LLM)  
   - Let the system auto-route: Whisper→OpenVINO NPU, LLM→Vulkan iGPU (when available).
3. **Add an “Intel dependencies” one-click checklist**  
   - Detect missing compute runtime components and provide exact install commands; avoid trial-and-error.

---

### Pain point: Build/DX sharp edges (missing stealth preload; CI workflow churn)
**Medium impact / Low difficulty**
1. **Preflight validation script** on `bun run dev`  
   - If a referenced preload/artifact is missing, print actionable instructions and fail early (avoid silent filtering).
2. **“Fresh clone” smoke test in CI**  
   - Ensures dev scripts don’t rely on ungenerated local artifacts.
3. **Consolidate auth setup docs for Anthropic subscription OAuth**  
   - Provide a single canonical path and expected logs for success.

## 4) Communication Gaps (Expectations vs reality)

1. **“Migration deadline” implies no help after cutoff**  
   - Users still expect a supported path; absence of docs + unanswered email creates a trust gap.
2. **“V3 imminent” read as a near-date promise**  
   - Without criteria and a follow-up cadence, users interpret delay as stalled progress.
3. **“Speculative decoding / DFlash” appears enabled when it isn’t**  
   - Silent fallback breaks user mental models; performance issues look like “Eliza is slow” rather than “feature disabled.”
4. **“NPU support” interpreted as “faster chat”**  
   - Actual reality: NPU is best for Whisper/ASR; needs explicit messaging and UI defaults.

Recurring questions indicating onboarding gaps (from the sample):
- “How do I migrate after the deadline?” (unanswered, repeated)
- “When is V3 actually shipping?” (timeline ambiguity)

Suggested documentation/onboarding improvements:
- A single **“Start Here”** page with branching paths: migrate, fresh install, local inference, cloud, connectors.
- A **support escalation guide** (where to ask, expected response time, what data to provide).
