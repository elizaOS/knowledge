## 1) Pain Point Categorization (top recurring friction areas)

### 1) Technical Functionality — Tool routing & planner reliability (high severity)
**What users report**
- The planner often **ignores Stage 1 “candidateActions” hints** and chooses suboptimal or wrong tools, especially for “search chat history” requests.
- Concrete example (GitHub issue **#7935**): asking “look in chat for any mention of xeroflame” leads to repeated **SHELL** calls (including unbounded `grep -R`) that time out, and even emits a wrong action shape (`TASKS { action: "spawn_agent" }` instead of `TASKS_SPAWN_AGENT`). Another repro leaks sub-agent routing metadata into user-facing output (“Routing Kind: QUESTION_FOR_TASK_CREATOR”).

**Who it affects most**
- Users relying on assistants for **chat ops / moderation / recall**, and anyone deploying in Discord where “search history” is a common request.

---

### 2) Performance — Local inference throughput cliffs (high severity)
**What users report**
- Significant performance drop on certain model+backend shapes.
- Example (GitHub issue **#7638**): Qwen2.5-14B Q4_K_M on Lunar Lake iGPU via Vulkan runs at **~1.15 tok/s**, far below bandwidth-based expectations, suggesting **kernel-selection/pathology**.

**Pattern**
- These reports are *hardware- and model-architecture-specific*, but the impact is “model becomes unusable” rather than a small regression.

---

### 3) Technical Functionality — Speculative decoding frequently “doesn’t work” (high severity, high confusion)
**What users report**
- Speculative decoding (DFlash) is **silently disabled** for SWA-based bodies:
  - Example (GitHub issue **#7635**): server logs “speculative decoding not supported” due to “target context does not support partial sequence removal”, then proceeds **body-only** with no API-level signal.
- Spec-decode collapses due to architecture-specific position validation:
  - Example (GitHub issue **#7631**): M-RoPE validation rejects draft batches, acceptance collapses to **~9.5%** and effective decode to **~0.33 tok/s**.

**Why it’s a top pain point**
- Users think they enabled acceleration but actually didn’t; results feel like a “mysterious slowness bug.”

> In the provided top-issues sample, **3/5 (60%)** issues are directly about spec-decode/local-inference acceleration failures or cliffs (#7631, #7635, #7638).

## 2) Usage Pattern Analysis (actual vs intended usage)

### Observed “actual usage”
1. **Local inference performance tuning is a core activity**, not an edge case.
   - Users benchmark kernels, test model tiers, and try to unlock DFlash/spec-decode speedups.
2. **Agents as real-time research tools**:
   - MCP-based setups pulling live social data for sentiment/trend monitoring (AgentKey MCP + OpenClaw).
3. **Agents as “coaches” / applied assistants**:
   - Community members are building applied products like “Resilience Bridge” (predictive stress signals + AI Coach) and looking for similar work.
4. **Discord as the primary entry point**, but not yet a strong support surface:
   - The sampled days show mostly intros/announcements; only a couple of help interactions, including one unresolved blocker.

### Gaps vs intended usage
- Intended: users ask the agent to use platform-native tools (e.g., chat search) and get reliable results.
- Actual: the planner frequently falls back to **SHELL + grep** or spawns sub-agents incorrectly, causing timeouts and weird UX.

### Feature requests aligned with these patterns
- **Explicit “spec decode active” indicators** in local inference endpoints and UI.
- **Stronger, deterministic tool selection** when Stage 1 provides candidate actions (e.g., SEARCH_MESSAGES).
- **First-class MCP onboarding** (supported workflow templates for “live social sentiment agent”).

## 3) Implementation Opportunities (solutions per major pain point)

### Pain Point A: Planner ignores Stage 1 hints; wrong tool choice; timeouts; metadata leaks (Category: Technical Functionality)
**Proposed solutions (prioritized)**
1) **Honor Stage 1 `candidateActions` as a strong preference (high impact / medium difficulty)**
- If Stage 1 outputs `candidateActions=["SEARCH_MESSAGES"]`, planner must select it unless tool is unavailable.
- Add a planner-loop constraint: “select from candidateActions first; only fall back if validation fails.”

2) **Add cost-aware “SHELL guardrails” (high impact / low-medium difficulty)**
- Hard-block dangerous patterns like unbounded `grep -R` at repo root unless user explicitly requests filesystem search.
- Enforce default caps: scope (cwd allowlist), timeout (shorter), output limit, and require a “narrowing question” when scope is large.

3) **Strip or prevent sub-agent control metadata in connector outputs (medium-high impact / low difficulty)**
- Add a connector-facing sanitizer that removes prefixes like “Routing Kind:” and internal routing enums.
- Similar approach: many agent frameworks (e.g., toolformer-style orchestrators and LangChain agent executors) enforce a strict “final answer channel” and never expose tool routing traces unless in debug mode.

---

### Pain Point B: Speculative decoding silently disables or degenerates (Category: Technical Functionality + Performance)
**Proposed solutions (prioritized)**
1) **Fail loud or surface a machine-readable “spec_decode_active=false” (very high impact / low-medium difficulty)**
- Add fields to `/health` and model metadata endpoints indicating whether spec decode is active and why not (mirrors user suggestion in **#7635**).
- If user passes `--spec-type dflash` and it’s incompatible, consider **hard error** instead of warning-only.

2) **Catalog gating: don’t recommend tiers that rely on unsupported spec-decode paths (high impact / medium difficulty)**
- Encode capabilities like `mrope-spec-decode` and `spec-decode-on-swa-body` in tier metadata and prevent routing into broken paths (suggested in **#7631** / **#7635**).
- Similar pattern: projects like llama.cpp frontends and model managers often include “capability matrices” (backend ↔ model architecture ↔ features) to prevent invalid combos.

3) **Upstream/port fix for SWA partial sequence removal (high impact / high difficulty)**
- Investigate/port SWA partial sequence removal logic referenced from `spiritbuun/buun-llama-cpp` (noted in **#7635**).
- This is the “real fix,” but it’s the most engineering-intensive.

---

### Pain Point C: Vulkan kernel selection causes throughput cliffs on specific shapes (Category: Performance)
**Proposed solutions (prioritized)**
1) **Add automated kernel-selection diagnostics bundle (high impact / medium difficulty)**
- One command to run with `GGML_VK_DEBUG*` env vars, collect logs, and attach to issues.
- Similar to how graphics drivers collect “GPU capture bundles” (RenderDoc-style workflows), but minimal.

2) **Expand CI/perf regression matrix for “model shape” coverage (medium-high impact / high difficulty)**
- Add representative Qwen/Llama model shapes to detect cliffs earlier.

3) **Document “known good tiers” per hardware class (medium impact / low difficulty)**
- For example: on Lunar Lake iGPU, recommend smaller Vulkan tiers where kernels are known to be efficient, and clearly label risk tiers.

---

### Pain Point D: Discord spam / fake support tickets (Category: Community + Security)
**Proposed solutions**
1) **Harden Discord permissions & automod rules (high impact / low difficulty)**
- Restrict room creation permissions; require verified roles for ticket workflows.

2) **Add “official support” channel banner + bot DM warning (medium-high impact / low difficulty)**
- Pin a message: staff will not create ad-hoc ticket rooms; link to official help channels.

3) **Telemetry: log and alert on suspicious channel creation patterns (medium impact / medium difficulty)**

---

### Pain Point E: Opaque 400 errors in flap upload (“Operations and map are required”) (Category: Integration / Developer UX)
**Proposed solutions**
1) **Improve error message mapping with actionable remediation (high impact / low difficulty)**
- Detect this 400 response and explain which flags/fields are missing, with a copy-paste example.

2) **Add a “doctor” preflight for flap upload (medium-high impact / medium difficulty)**
- Validate required fields locally before sending request.

3) **Create a single “manual launch → upload” doc path (medium impact / low difficulty)**
- The report specifically involved “after manually launching on waifu”; document supported sequences.

## 4) Communication Gaps (expectations vs reality)

### Gap 1: “Is elizaOS still active?” confusion
- Discord users explicitly asked due to “labs” being defunct; they needed reassurance that **open source continues**.
**Fix**
- Add an “Official Project Status” page (README + docs) that explains governance, maintainers, and where releases happen.

### Gap 2: Spec-decode expectations
- Users expect `--spec-type dflash` to “work” when provided; reality: it can silently disable or degenerate.
**Fix**
- UI/API must explicitly show whether acceleration is active, and why not.

### Gap 3: “Search chat history” expectations
- Users assume chat search is available/works; planner falls back to filesystem grep and fails.
**Fix**
- Document what “SEARCH_MESSAGES” can access (recent window vs full history; connector permissions), and show a “limitations & enablement” checklist.

### Recurring questions indicating onboarding/documentation gaps
- Project continuity (company vs OSS).
- Host prerequisites (JS runtime constraints on Linux).
- Missing invite codes / access gating (Discord: .maxlife asked for invite code).

## 6) Feedback Collection Improvements

### Current channel effectiveness (from observed data)
- **Discord:** good for introductions and announcements, weaker for completing support loops (1 clear unresolved technical blocker observed: flap upload 400).
- **GitHub:** very strong for deep technical issues (repro steps, logs, hypotheses), but skewed toward advanced users.

### Improvements to gather more structured, actionable feedback
1) **Discord → GitHub issue capture bot (low-medium difficulty)**
- When a message contains error text + “help,” prompt the user with a one-click “Create issue” flow including required fields (OS, version, steps).

2) **Standardized templates**
- “Local inference perf/spec-decode” template: model, backend, flags, timings, logs env vars.
- “Connector/planner quality” template: prompt, Stage 1 output, tool calls, user-facing leakage.

3) **Close-the-loop SLA for blockers**
- A lightweight triage rotation ensuring any “blocked install/upload” gets a response within 24 hours.

### Underrepresented segments
- **Non-power users** trying to build “normal apps” (most Discord traffic sampled is intros, not daily usage pain).
- **Windows-first** and **restricted Linux** environments (we have hints, but not systematic feedback).
