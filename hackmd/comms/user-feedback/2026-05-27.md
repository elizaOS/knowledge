## 1) Pain Point Categorization (Top recurring 5–7)

### 1) Technical Functionality — **Planner/tooling chooses wrong actions, timeouts, metadata leaks** (2/9 threads, ~22%; high severity)
**What users hit**
- Planner ignores Stage-1 hints like `SEARCH_MESSAGES` and instead runs **expensive `SHELL` greps** that **timeout** (e.g., `grep -R ...` on the monorepo) and can produce **empty final replies** (GitHub #7935).
- Sub-agent flow leaks internal control text to end-users (e.g., `Routing Kind: QUESTION_FOR_TASK_CREATOR`) (GitHub #7935).
- Action schema mismatch still appears in the wild (`TASKS { action: "spawn_agent" }` vs `TASKS_SPAWN_AGENT`) (GitHub #7935).

**Primary impact**
- Breaks user trust (“agent looks broken”), increases cost/latency, and creates security/UX risk by surfacing internal orchestration metadata.

---

### 2) Performance — **Local inference regressions/edge-case cliffs on modern model architectures & Intel iGPU** (3/9 threads, ~33%; high severity for affected users)
**What users hit**
- Severe **Vulkan performance cliff** for specific model shapes (Qwen2.5-14B on Lunar Lake iGPU) → ~1.15 t/s, effectively unusable (GitHub #7638).
- Speculative decoding incompatibilities cause collapse to **sub-1 t/s** or disablement:
  - M-RoPE position validation rejects cross-drafter Qwen3.5 batches, acceptance collapses (~9%) (GitHub #7631).
  - SWA-based bodies silently disable `--spec-type dflash` (GitHub #7635), yielding body-only throughput with minimal user-visible signaling.

**Primary impact**
- Users investing in “recommended” tiers or modern Qwen/Gemma families encounter hard-to-diagnose slowdowns; erodes confidence in the local inference recommender ladder and “it should be fast on my GPU” expectations.

---

### 3) Integration — **Flap/Waifu upload flow fails with unclear 400 error** (1/9 threads, ~11%; medium-high severity)
**What users hit**
- `flap upload` returns **HTTP 400**: `"Operations and map are required"` after manually launching on waifu; retries don’t help; no resolution in-thread (Discord 2026-05-25).

**Primary impact**
- Blocks publishing/deploying artifacts; suggests missing required parameters, version skew, or an undocumented contract for the upload command.

---

### 4) Community / Trust — **Confusion about project status (“labs defunct” vs OSS alive)** (1/9 threads, ~11%; medium severity)

## 2) Usage Pattern Analysis (actual vs intended)

### Observed real-world usage patterns
1. **Economic agent infrastructure & launchpad workflows**
   - Community announcement: “waifu” as an economic agent launchpad (Discord 2026-05-24).
   - Users attempt to **launch manually** and then **upload via flap** (Discord 2026-05-25).
   - Pattern: elizaOS used as a *deployment + monetization pipeline*, not just an agent runtime.

2. **High-performance local inference tuning on edge hardware**
   - Multiple GitHub issues focused on **Intel Lunar Lake iGPU**, Vulkan kernels, speculative decoding behavior, and runtime kernel options (OpenVINO RFC). This suggests a “power user” segment pushing elizaOS as a **local-first, hardware-aware inference stack**, not merely cloud-hosted.

3. **Connector identity separation (OWNER+AGENT) aligns with how users want to operate**
   - The v2.0.2 cycle emphasizes OWNER+AGENT roles across X/Shopify/Calendly/Linear, matching a real need: “agent has its own identity, but can act on behalf of owner.”

### Emerging / unexpected use cases
- **Voice-agent optimization via heterogeneous compute** (OpenVINO NPU for Whisper + Vulkan for LLM + CPU for TTS) shows elizaOS becoming a *pipeline orchestrator* across devices (GitHub #7633 context).

### Feature requests aligning with usage
- **Expose runtime kernel choices (OpenVINO) in the local inference recommender** for Intel hardware (GitHub #7633).
- **Make spec-decode capability explicit/inspectable** to support Qwen/Gemma tiers safely (GitHub #7631/#7635).
- **Harden chat-history search UX** (planner should prefer `SEARCH_MESSAGES` when asked) (GitHub #7935).

## 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

### A) Planner/tool mis-selection + timeouts + metadata leaks (GitHub #7935)
**High impact / Medium difficulty**
1. **Honor Stage-1 `candidateActions` as a hard preference**
   - If Stage-1 returns `candidateActions=["SEARCH_MESSAGES"]`, planner must try it before `SHELL`.
   - Similar approach: tool-priority contracts used in frameworks like LangChain “tool routing hints” and OpenAI function-calling guardrails.

2. **Default SHELL safety rails: bounded search + denylist patterns**
   - Replace unbounded `grep -R` with a safe wrapper: restrict directories, cap runtime/output, require `--max-count`, enforce timeout ≤ 10–20s.
   - Similar to how GitHub CodeQL/CI scripts and “safe exec” wrappers constrain workspace scans.

**Medium impact / Low difficulty**
3. **Strip sub-agent control metadata at the connector boundary**
   - Enforce an output contract: sub-agents return `{messageToUser, internalNotes}` and connectors only publish `messageToUser`.
   - Similar to how “chain-of-thought” redaction layers separate internal reasoning from user responses.

---

### B) Speculative decoding incompatibility (M-RoPE / SWA) and silent disablement (GitHub #7631, #7635)
**High impact / Medium difficulty**
1. **Fail loudly + surface status in API**
   - If `--spec-type dflash` requested but incompatible, return a structured status in `/health` and model metadata:
     - `spec_decode_requested=true`, `spec_decode_active=false`, `reason="SWA partial sequence removal unsupported"`.
   - Similar to how CUDA capability checks are surfaced in ML runtimes (e.g., PyTorch reports CUDA availability; llama.cpp prints explicit backend availability).

2. **Catalog gating: capability flags per tier**
   - Add a `kernelCaps.required`/`unsupported` flag like `mrope-spec-decode` and `spec-decode-on-swa-body`.
   - Recommender should refuse to route users into spec-decode-dependent tiers unless active.

**High impact / Higher difficulty**
3. **Implement/port SWA-compatible partial sequence removal**
   - Port known-good implementations (e.g., from forks referenced in #7635) into the KV cache backend + spec driver.

---

### C) Vulkan performance cliffs on specific model shapes (GitHub #7638)
**High impact / Medium difficulty**
1. **Add kernel selection diagnostics + “known slow shapes” warnings**
   - Provide a one-command debug mode that logs selected kernels and flags when falling back to generic slow paths.
   - Similar to TensorRT / XLA verbose modes highlighting kernel fusions and fallback ops.

2. **Benchmark-driven kernel heuristics**
   - Expand automated microbench coverage by (model-family × param-shapes) and tune dispatch heuristics accordingly.

**Medium impact / Medium difficulty**
3. **Automatic model recommendation avoidance**
   - If a model family is known to hit a cliff on detected hardware, recommend a smaller/faster tier by default and explain why.

---

### D) Flap/Waifu upload 400: “Operations and map are required” (Discord 2026-05-25)
**High impact / Low difficulty**
1. **Improve error message and CLI validation**
   - Before sending request, validate required flags and print example usage:
     - e.g., `flap upload --operations <file> --map <file>`
   - Similar to Terraform/Kubectl: preflight checks with actionable remediation.

**High impact / Medium difficulty**
2. **Add “doctor” command for upload prerequisites**
   - `flap doctor` verifies auth/invite, required files present, API version compatibility, and prints the exact payload shape.

**Medium impact / Medium difficulty**
3. **Add server-side structured errors**
   - Ensure 400 response includes a machine-readable schema: `{code, missingFields, docsUrl}`.

## 4) Communication Gaps (expectations vs reality)

1. **“Is the project alive?”**
   - Recurring question indicates the community expects a single entity behind elizaOS; reality is OSS continuity without the company.
   - Fix: explicit governance/status docs + pinned Discord message + repo badge linking to “Current maintainers / roadmap”.

2. **Speculative decoding expectations**
   - Users expect `--spec-type dflash` to “work if the drafter loads,” but in SWA cases it silently disables (GitHub #7635).
   - Fix: API-visible `spec_decode_active` + UI warnings + catalog gating.

3. **Chat search expectations**
   - Users asking “search chat” assume a dedicated connector action exists; planner currently “escapes” to shell and fails/timeouts (GitHub #7935).
   - Fix: treat chat search as a first-class capability with deterministic routing.

4. **Upload workflow expectations (waifu/flap)**
   - 400 error suggests undocumented required fields; users expect defaults or guided prompts.
   - Fix: docs + CLI preflight + better server error contracts.
