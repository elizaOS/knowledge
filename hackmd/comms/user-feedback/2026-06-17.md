## 2) Usage Pattern Analysis (Actual vs intended)

### How users are actually using elizaOS (from feedback)
1. **Local developer workflow on Windows**: running `bun run dev:ui` and expecting immediate UI feedback (Discord 2026-06-14).
2. **Cloud as a shared login + hosting substrate**: users treat Eliza Cloud as the “identity layer” for apps (“Sign in with Eliza Cloud”), and as an environment where settings like **OpenRouter keys** must be adjustable (Discord 2026-06-14; GitHub #8257).
3. **Community as a helpdesk + trust oracle** for Web3-related actions (migration legitimacy checks, scam detection) (Discord 2026-06-15).
4. **Builders seeking structured collaboration**: proposal with funded “missions” and integration frameworks (ERC-7710 + knowledge graph) (Discord 2026-06-15).

### Emerging / unexpected use cases
- **Governance + delegation frameworks**: Semantic Delegation with ERC-7710 + knowledge graph suggests users are thinking beyond “agent chat” into *delegated authority systems* and “liquid democracy” style workflows.

### Feature requests aligned with real usage
- **Cross-chain bridging guidance or support** (BNB → Solana) aligns with users associating elizaOS with on-chain agent operations—even if not currently core.
- **Cloud access self-serve** (tenant invites, joining) aligns with cloud onboarding needs (repeated access failures).

## 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

### A) Cloud access/auth + tenant invitation errors
**Impact:** Very high | **Difficulty:** Medium (mostly UX + backend guardrails)

1) **Make “tenant invitation required” actionable**
- Replace generic error with: *which tenant*, *how to request invite*, *link to docs*, *contact channel*, and *self-serve fallback* if allowed.
- Similar approach: **GitHub org invites** and **Slack workspace invites** provide explicit next steps and deep links.

2) **Add an “Auth/Onboarding Status” panel + diagnostics**
- A `/status` style page in cloud frontend showing: auth provider status, steward tenant ID, current org/tenant, invitation state, API reachability.
- Similar approach: **Vercel** and **Supabase** show clear project/org context and actionable errors.

3) **Harden and continuously test login flows**
- Keep a small suite of Playwright “can login” smoke tests in CI against staging (OAuth + email magic link + invite acceptance route).
- Mirror the lesson from #8257: when PKCE requirements changed upstream, it broke all clients. Add **contract tests** for Steward auth expectations.

---

### B) Windows `bun run dev:ui` blank screen
**Impact:** High (blocks contributors) | **Difficulty:** Medium

1) **Add a Windows dev:ui smoke test + crash surfacing**
- CI job on Windows that runs `bun run dev:ui` and asserts non-empty DOM + no console fatal errors.
- Similar approach: **Electron** apps commonly run OS-specific smoke tests to catch Windows-only blank-window regressions.

2) **Improve frontend fatal-error UX**
- Add an error boundary that renders a visible “UI crashed” page and prints the stack trace + common fixes.
- Similar approach: **Create React App / Next.js** dev overlay makes runtime errors impossible to miss.

3) **Add a “known issues” doc entry with logs to collect**
- Minimal checklist: console output, network tab errors, `bun --version`, Windows build tools, and a “paste this diagnostic script output” command.

---

### C) Roadmap/status ambiguity (Milady/Babylon/bridging)
**Impact:** Medium | **Difficulty:** Low–Medium

1) **Maintain a single “Product Status” page**
- For each surface (eliza repo, Cloud, Milady, Babylon): Active / Maintenance / On hold / Deprecated; last update date; where to follow.
- Similar approach: **Kubernetes SIG status pages** and **Rust RFC tracking** reduce repeated “is this alive?” questions.

2) **Add “Feasibility tags” for common asks**
- Example tags: “Not supported”, “Supported via plugin”, “Planned”, “Needs design”.
- Bridging (BNB→Solana) could be explicitly categorized: documentation-only (explain constraints) vs feature roadmap.

3) **Publish a lightweight weekly changelog to Discord**
- The Cron Job posts are helpful, but link rot/dead links and unclear ownership reduce usefulness. Post an official pinned summary with stable URLs.

---

### D) Automod blocks proposals + off-topic noise in coders
**Impact:** Medium | **Difficulty:** Low

1) **Create a structured proposal intake**
- A public `#proposals` channel with a form template, slowmode, and allowlisted keywords; or a GitHub Discussions category.
- Similar approach: **Rust internals / Python PEP** process: clear template + triage labels.

2) **Replace DM-based escalation with a ticket system**
- Use Discord forum posts or a lightweight bot that opens “proposal tickets” routed to community ops.

3) **Enforce channel purpose with routing**
- Coders channel: auto-reply to partnership/wallet solicitations with redirection; moderators can tag and move threads.

---

### E) Cloud Apps scaling readiness gaps (from #8321 audit)
**Impact:** Very high (launch risk) | **Difficulty:** Medium–High

1) **Treat P0 items as a release gate**
- Explicitly gate `APPS_DEPLOY_ENABLED` on: ingress installed, pooling/connection caps, env-var contract verified, basic security rules applied.
- Similar approach: **Kubernetes release criteria**: must-pass conformance + basic operational readiness.

2) **Add load tests + capacity claims grounded in measurement**
- Before “~1000 apps” messaging, run staged load tests (5, 25, 50 apps) and publish observed limits.
- Similar approach: **Linkerd** and **Envoy** publish performance baselines tied to versions.

3) **Operationalize “defense-in-depth” code paths**
- If nftables / allowlists exist but have no callers, either wire them or remove them to avoid security theater. Track with a “security controls: active vs planned” checklist.

## 4) Communication Gaps (expectations vs reality)

### Gap 1: “Cloud join” feels like a standard signup, but actually requires invites/tenant context
- Repeated “tenant invitation requirement” errors indicate users expect self-serve onboarding.
**Fix:** upfront copy on login/join screens: “You need an invite to join an existing tenant, or create a new org.”

### Gap 2: Users assume dormant products are still active (Milady/Babylon)
- Questions about Milady updates and Babylon status show mismatch between perceived activity and actual priorities.
**Fix:** pin status + last-updated timestamps; link to repo activity and roadmap.

### Gap 3: Users assume elizaOS supports cross-chain “bridging”
- “Bridge elizaOS from BNB to Solana” got no response; likely not a defined capability.
**Fix:** add a canonical answer: what “bridging” could mean (agent wallet, state, plugins), and what’s supported today.

### Gap 4: Hiring / ordering humanoid robots
- Users asked about hiring and robot orders; dead link created confusion.
**Fix:** add a short “Not hiring / no robot orders” FAQ + moderation guidance for rumor cleanup.

Recurring questions indicating onboarding gaps (from Discord):
- “Why can’t I access elizaCloud / I need a tenant invitation?”
- “Why is the UI blank on Windows even though the agent runs?”
- “Is X project paused? Where are updates?”
