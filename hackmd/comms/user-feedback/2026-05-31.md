### 4. Technical Functionality / UX — Cloud/Desktop auth confusion (low frequency, high friction when hit)
**Signals**
- Developer deployed a new Docker image on ElizaCloud; agent unexpectedly prompts for **display name + password** despite email-based registration.
- Suggests a mismatch between **auth mode** (email/OAuth) vs **local runtime credential prompt** expectations.

**User-impact**
- Deployment blocks; “what credentials do I enter?” is a hard stop.

---

### 5. Technical Functionality — Planner/tool selection failures & metadata leakage (lower frequency in count, high severity for UX quality)
**Signals (GitHub #7935)**
- When asked to search chat history, planner chooses **SHELL** over **SEARCH_MESSAGES**, runs expensive `grep -R` that times out, returns empty reply.
- Leaks sub-agent control metadata (e.g., “Routing Kind: QUESTION_FOR_TASK_CREATOR”) to user-facing channels.

**User-impact**
- Users perceive the agent as unreliable, slow, and “internally messy”; also creates unnecessary cost/timeouts.

---

### 6. Performance / Technical Functionality — Local inference spec-decode incompatibilities and perf cliffs (power-user reported, potentially broad impact)
**Signals**
- Spec-decoding failures on modern architectures:
  - **M-RoPE validation rejects cross-drafter batches** → acceptance collapses (~9%), tg < 1 t/s (#7631).
  - **SWA bodies silently disable DFlash spec-decode** with minimal surface signaling (#7635).
- Vulkan perf anomaly: Qwen2.5-14B Q4_K_M runs far below expected bandwidth efficiency (#7638).
- OpenVINO runtime RFC (#7633) suggests strong platform opportunity, especially for Whisper/NPU.

**User-impact**
- “Recommended tiers” can be unusable on some hardware; silent fallbacks cause confusion (“why is it slow?”).

---

### 7. Documentation / Policy — Token migration “deadline missed” with no path forward (low frequency, high emotional severity)
**Signals**
- User asked for manual migration after missing Feb 4 deadline; response: closed, no manual migration.

**User-impact**
- High dissatisfaction for affected holders; likely repeats unless clearly documented/automated messaging exists.

## 2) Usage Pattern Analysis (actual vs intended)

### Observed real-world usage patterns
1. **ElizaOS as an orchestration layer for long-running multi-agent ops**, not just chatbots  
   - A developer described a **multi-agent workflow platform** (research → processing → generation → tool execution) with infra stack (Postgres/Redis/K8s/OpenAI).  
   - Pain points they hit: **context management, memory persistence, recovery from failed actions**.

2. **Community building chain-specific plugins (Solana-first), treating core repo as source-of-truth**
   - `plugin-solana` relocation immediately impacted developers.
   - New work: **Kamino integration**, referencing older code and seeking upstream contribution path.

3. **Cloud deployment + container workflows**
   - Users deploy updated Docker images and expect seamless auth/session continuity; unexpected credential prompts are a mismatch.

### Emerging / unexpected use cases
- **Partnership inbound (SKALE Labs asking for contact)**: ecosystem interest beyond current comms funnel.
- **Hardware acceleration experimentation**: Intel iGPU/NPU deep dives (OpenVINO; Vulkan kernel selection) indicating a growing “local-first” power-user segment.

### Feature requests aligning with usage
- Strong alignment between:
  - Better **plugin discovery + contribution tooling** (because people are building plugins now)
  - Better **planner/tool reliability + guardrails** (because multi-step workflows are core)
  - Better **local inference capability gating and surfacing** (because users rely on recommender/runtime defaults)

## 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

### A) Repo reorg discoverability: plugin-solana → plugin-wallet path confusion
**Opportunity 1 (High impact, Low effort): Add permanent “Moved/renamed” stubs**
- Add a small README + deprecated package marker in old locations (or a top-level `MOVED.md` index).
- Include: old path → new path mapping, version/date, rationale, links.

**Opportunity 2 (High impact, Medium effort): Introduce a “plugin index” command**
- `elizaos plugins locate solana` → prints exact monorepo path + docs + examples.
- Similar approach: Rust’s `cargo` subcommands and Kubernetes’ `kubectl explain` pattern for discoverability.

**Opportunity 3 (Medium impact, Low effort): Release notes + Discord pinned FAQ automation**
- Auto-post “breaking structural change” diffs into Discord on merges that move/rename plugin roots.

---

### B) Plugin contribution workflow unclear (PRs, standards, ownership)
**Opportunity 1 (High impact, Medium effort): Add a “Plugin Contribution Quickstart”**
- Minimal steps: repo structure, required files, testing, naming conventions, CI expectations.
- Provide a “reference plugin PR checklist” template.

**Opportunity 2 (High impact, Medium effort): Provide a plugin scaffold generator**
- `elizaos init plugin-kamino` generates correct layout, build config, CI hints.
- Similar approach: Yeoman generators, Next.js `create-next-app`, and Home Assistant integration scaffolds.

**Opportunity 3 (Medium impact, Low effort): Add “seeking maintainers” tags + ownership model**
- Clarify whether contributed plugins are core-maintained, community-maintained, or “best-effort.”
- Reduces expectation mismatch and abandonment anxiety.

---

### C) Project/token status uncertainty (trust + expectations)
**Opportunity 1 (High impact, Low effort): Publish a quarterly “Project Direction & Commitments” note**
- Explicitly: what’s maintained, what’s experimental, what’s deprioritized, and why (e.g., waifu.fun focus).
- Link to active milestones and shipped items (e.g., v2.0.2 release, SOC2 controls, plugin migrations).

**Opportunity 2 (High impact, Medium effort): Create a token-related transparency FAQ**
- Cover: token utility initiatives, migration policy (closed), what users can/can’t expect.
- Put this in docs + pinned Discord answer to reduce repeated cycles.

**Opportunity 3 (Medium impact, Medium effort): Add an “official contacts” endpoint**
- For partnerships (SKALE inbound), route to a form/email backed by an issue label (e.g., `partnership-inquiry`).

*Comparable projects:* many OSS foundations publish “Project Status” pages (e.g., Rust, Next.js RFC cadence) to prevent rumor-driven support churn.

---

### D) ElizaCloud / Docker auth prompt confusion
**Opportunity 1 (High impact, Medium effort): Make auth mode explicit in runtime UX**
- If email/OAuth session exists, do not prompt for password; instead show “You are signed in as <email>” or prompt to open a device-code link.
- If local credential mode is required, explain *why* and how to reset.

**Opportunity 2 (High impact, Medium effort): Add a deterministic “headless login” mechanism**
- E.g., `ELIZA_AUTH_TOKEN` / `ELIZA_SESSION_JWT` injection for containers; or device-code flow printable in logs.
- Similar approach: GitHub CLI `gh auth login` device code; Docker `docker login` via token.

**Opportunity 3 (Medium impact, Low effort): Troubleshooting doc + decision tree**
- “If you see display name/password prompt in ElizaCloud container, do X/Y/Z.”

---

### E) Planner/tool routing failures & metadata leakage (#7935)
**Opportunity 1 (High impact, Medium effort): Enforce Stage-1 “candidateActions” as a strong prior**
- If Stage 1 outputs `candidateActions=[SEARCH_MESSAGES]`, planner should either:
  1) Use it, or
  2) Provide a structured reason it can’t (capability missing), then ask a clarifying question.
- Similar approach: LangChain/agent frameworks often “tool preference” rank or hard-route specific intents.

**Opportunity 2 (High impact, Medium effort): Add hard safety rails for SHELL tool**
- Default constraints:
  - path allowlist (no repo-root grep by default),
  - output caps,
  - timeouts,
  - mandatory scoped queries.
- If unscoped, planner must request user clarification.

**Opportunity 3 (High impact, Low effort): Strip/forbid sub-agent control metadata from user channels**
- Add a post-processor that removes known control headers (routing kind, internal tags) or enforce a strict response schema for user-facing text.
- Comparable: OpenAI function-calling “final answer” gating; many agent runtimes separate “assistant_trace” from “assistant_message”.

---

### F) Local inference spec-decode & capability gating (silent fallbacks, incompatibilities)
**Opportunity 1 (High impact, Medium effort): Surface “spec_decode_active” + compat state in APIs/UI**
- If DFlash/spec-decode is disabled (SWA/M-RoPE), expose it in:
  - `/health`, `/v1/models`, and UI model picker.
- Prevents “it’s slow but I think it’s enabled” confusion.

**Opportunity 2 (High impact, High effort): Add architecture-aware spec-decode support**
- M-RoPE-aware batching, SWA partial-sequence removal support (or explicit hard error).
- If not supported, fail fast with a clear message.

**Opportunity 3 (Medium impact, Medium effort): Recommender/catalog gating**
- Encode capability requirements (e.g., “mrope-spec-decode”, “swa-dflash-compatible”) and refuse to recommend incompatible pairings.

*Comparable projects:* llama.cpp ecosystem commonly uses explicit backend capability flags; projects like Ollama surface “runner/backends” clearly to users.

## 4) Communication Gaps (expectations vs reality)

1. **“Plugin missing” vs “plugin moved/renamed”**
   - Expectation: stable plugin locations.
   - Reality: active reorg; moved into monorepo under new name/path.
   - Fix: migration notes, stubs, and a discoverability tool.

2. **“ElizaOS is dead/abandoned” vs “active but focus shifted”**
   - Expectation: consistent messaging and visible roadmap.
   - Reality: active development continues (releases, infra, plugins) but external perception is shaped by waifu.fun focus and token chatter.
   - Fix: publish explicit status + roadmap + token FAQ.

3. **“I logged in via email” vs “container asks for password”**
   - Expectation: one auth system across UI and deployments.
   - Reality: multiple auth entrypoints; poor runtime messaging when modes diverge.
   - Fix: unify device-code/token login for containers; clearer prompts.

4. **Migration policy confusion**
   - Expectation: late/manual migration path.
   - Reality: closed, no manual migration.
   - Fix: make it unmissable in docs + pinned Discord bot response.

## 6) Feedback Collection Improvements

### Channel effectiveness
- **Discord:** great for fast Q&A, but answers are ephemeral → repeated questions.
- **GitHub issues:** high quality for deep technical problems, but skew toward power users; lacks structured intake for cloud/auth UX problems.

### Improvements
1. **Structured Discord → GitHub capture**
   - A bot command: `/issue` that creates a prefilled GitHub issue with logs/versions.
2. **Standard “environment + version” template for support**
   - Especially for cloud/docker auth and local inference (backend, branch, image tag, login method).
3. **Underrepresented segments to seek out**
   - Non-crypto builders using ElizaOS purely for enterprise workflows (little represented in Discord snippets).
   - Less-technical desktop users (their pain shows up only when UX fails, e.g., auth prompts).
