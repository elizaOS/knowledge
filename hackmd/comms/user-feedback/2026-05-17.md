## 1) Pain Point Categorization (Top recurring issues)

### 1) Technical Functionality — Runtime breakages from “missing/duplicate/unsynced” sources of truth (33% | 3/9)
**What users hit**
- **Missing generated file breaks Anthropic subscription auth**: `dev-ui.mjs` references `./claude-code-stealth.mjs` that doesn’t exist on fresh clone → OAuth/subscription requests return 401 despite valid token (GitHub issue #7210, closed).
- **Schema drift causes fresh DB boots to be unusable**: plugin-sql runtime migrator lacks drizzle `pgTable` defs for tables that the runtime queries → repeated query errors, half-empty state, and user-visible parsing failures (GitHub issue #7222, closed).
- **Duplicate client class breaks UI methods**: two `MiladyClient` definitions → ~155 prototype-augmented methods missing at runtime, causing “is not a function” errors and UI freeze (GitHub issue #7244, closed).

**Why it’s severe**
- These fail “day-0” flows (fresh clone, first boot, first chat), turning onboarding into a debugging session.

---

### 2) Integration — Connector reliability issues (Telegram/Slack/Farcaster) (33% | 3/9)
**What users hit**
- **Telegram silent message loss**: two pollers (plugin + wrapper) consume updates competitively → ~50% dropped messages (GitHub issue #7245, closed).
- **Slack risk flagged on critical inbound path**: missing try/catch around `users.info` can drop events silently if Slack API errors (Greptile P1 note on PR #7375).
- **Farcaster/Neynar credit burn**: notifications endpoint consumed **~300 credits/hour vs expected ~4** even after adjusting polling interval; webhook suggested as mitigation (Discord 2026-05-14).

**Why it’s severe**
- Silent drops (Telegram/Slack) present as “agent is flaky,” which erodes trust faster than explicit errors.

---

### 3) Documentation / Onboarding — Auth detection & setup doesn’t match real-world environments (11% | 1/9)
**What users hit**
- **Codex CLI auth detection outdated**: `hasCodexCliSubscriptionAuth()` only recognizes legacy `~/.codex/auth.json` layout; modern `tokens.access_token` shape misreported as “install required” (GitHub issue #7243, closed).

**Pattern**
- Users increasingly rely on “auto-detect” subscription/auth flows; small upstream format changes cause confusing false negatives.

---

### 4) Community / Communication — Release timeline uncertainty (11% | 1/9)
**What users asked**
- “Two weeks since V3 was ‘imminent’—what’s the updated timeline?” (Discord 2026-05-16, unanswered)

**Why it matters**

### 5) Community / Communication — Partnership announcements lack verifiable details (11% | 1/9)

## 3) Implementation Opportunities (solutions per major pain point)

### Pain Point A: Day-0 runtime breakages (missing files, schema drift, duplicate classes)
**High impact / Medium difficulty**
1) **Add “first-run integrity checks” at boot**
   - Validate that referenced preload artifacts exist (e.g., `claude-code-stealth.mjs`) and **fail loudly** with a targeted fix message.
   - Validate plugin-sql migrations include every table the runtime will query (cross-check abstract schema exports vs drizzle pgTables).
   - Similar approach: **Next.js** and **Rails** both front-load missing migration/config errors with explicit boot diagnostics.

2) **Single source of truth enforcement**
   - Schema: generate drizzle tables from the abstract schemas (or vice versa) to prevent drift.
   - Client: prevent duplicate exported classes via lint rule / CI check (e.g., “one canonical exported class per symbol name in package”).
   - Similar approach: **Prisma** and **TypeORM** push toward generation-driven schemas; **Apollo** uses schema checks in CI to prevent drift.

3) **Golden-path CI: “fresh clone smoke test”**
   - CI job that runs: fresh checkout → install → boot → first chat turn → connector init (mocked) → migration sanity.
   - This would have caught #7210/#7222/#7244 early.

---

### Pain Point B: Connector reliability (Telegram/Slack/Farcaster)
**High impact / Medium difficulty**
1) **Enforce single-consumer ownership for polling connectors**
   - Telegram: ensure mutual exclusion between wrapper poller and plugin poller (one must be disabled automatically).
   - Add runtime warning when two consumers share the same bot token.

2) **Prefer push (webhooks/events) over polling**
   - Neynar: provide a webhook-based reference implementation + docs; make polling explicitly “development only.”
   - Similar approach: **Stripe**, **Slack**, and **GitHub Apps** recommend webhooks for scale/cost reasons.

3) **Connector “delivery guarantees” instrumentation**
   - Add per-connector metrics: received count, processed count, dropped count (with reasons).
   - For Slack: wrap `users.info` and all inbound steps with error boundaries; store an error memory event instead of dropping the message.

---

### Pain Point C: Auth detection / onboarding mismatch (Codex CLI + subscription flows)
**Medium impact / Low difficulty**
1) **Support multiple auth file shapes with explicit detection logs**
   - Codex: recognize `tokens.access_token` format in addition to legacy.
   - Emit a single onboarding log line: “Detected Codex auth via tokens block (vX).”

2) **Add a “Diagnose my setup” screen/command**
   - Checklist: detected providers, connectors, token sources, expected file locations, and “what we tried.”
   - Similar approach: **Docker** (“docker diagnose”), **Supabase CLI** (“supabase status”).

3) **Document “supported auth modes” per provider**
   - A table: API key vs OAuth subscription vs CLI session; known file formats; common failure modes.

---

### Pain Point D: Release & partnership communication ambiguity (V3 timeline + partnership legitimacy questions)
**High impact / Low difficulty**
1) **Publish a living release status page**
   - Include: current milestone, blocked items, and updated estimate ranges.
   - Similar approach: **Kubernetes** release dashboards and **Rust** release trains reduce repeated “when release?” questions.

2) **Partnership announcement standard**
   - Minimum fields: partner legal entity, scope, stage (LOI/signed/POC), deliverables, timeline, and what’s open-source vs proprietary.
   - Prevents community debate driven by missing context.

3) **Pin a weekly “what shipped / what’s next” post**
   - Short, consistent cadence reduces speculation.

## 4) Communication Gaps (expectations vs reality)

### Recurring expectation mismatches
- **“Imminent” release language vs no confirmed date** (V3 question). Users interpret “imminent” as days, not weeks.
- **“Auto-detect” auth/connectors vs brittle real-world setups**
  - Codex CLI detection failed due to upstream format change.
  - Anthropic subscription OAuth depended on a missing preload artifact.
- **Polling interval changes vs actual API credit behavior**
  - Neynar credits were far higher than expected even after override, implying hidden multipliers or multiple calls per loop.

### Specific improvements
- Add an “Operational Reality” section in docs per connector/provider:
  - “Polling costs,” recommended webhook path, rate-limit notes, and how to verify call volume.
- Add “Release language guidelines” internally:
  - Replace “imminent” with a date range + confidence level.

## 6) Feedback Collection Improvements

### Current channel effectiveness
- **Discord**: good for fast questions (release timing, API usage) but tends to produce **untracked** outcomes (“V3 question unanswered”).
- **GitHub issues**: extremely actionable (repros, diffs, impact), but skewed toward advanced users; newcomers may not file.

### Improvements to gather more structured feedback
1) **Weekly triage digest**
   - Convert top Discord questions into GitHub Discussions/Issues with owners and status.
2) **Connector health survey (quarterly)**
   - Ask: which connector, message volume, failure modes, webhook vs polling, cost surprises.
3) **In-product “Report a problem” bundle**
   - Auto-attach anonymized logs: enabled plugins, connector mode (poll/webhook), provider routing, migration status.

### Underrepresented segments
- **Non-technical SMB builders** (implied by “Agentic SME solution” commercialization) are not visibly represented in GitHub issue traffic.
- **Self-hosters at scale** (rate-limit/cost concerns show up once, but likely broader) need targeted outreach.
