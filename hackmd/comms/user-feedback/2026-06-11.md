## 2) Usage Pattern Analysis (actual vs intended)

### How users are actually using elizaOS
1. **Cloud-first, multi-tenant deployment as a core expectation**  
   GitHub focus (e.g., #8321, #8257) shows users treat Eliza Cloud Apps + Cloud SSO as production infrastructure, not an experimental add-on.

2. **Agents as operators for dev workflows (coding workbench + orchestrator UI)**  
   The orchestrator is used for “Claude-Code/Codex-style” flows; the key pain is not agent capability but **how results are rendered** back to the user (#8162).

3. **Plugin ecosystem as a primary extension mechanism**  
   Third-party plugin registry submissions (e.g., PR #8294 for `@usenami/plugin-signer`) indicate that “extend via plugins” is a real adoption path, and review latency/process clarity matters.

4. **Community Discord used as a marketplace + recruiting hub**  
   A large portion of Discord activity is intros, job posts, and service offerings (AI engineers, artists, comms coordinators). This is “usage,” but not product support.

### Emerging / unexpected use cases
- **Secure exchange signing and enclave-backed key management** via third-party plugins (`@usenami/plugin-signer` storing API keys in AWS Nitro Enclaves). This indicates demand for “operational security plugins” beyond typical chat/tooling.

### Feature requests aligning with real usage
- **Reliability guardrails for Cloud Apps** (ingress, pooling, placement, idempotent billing) align with the observed “ship real apps” usage pattern (#8321).
- **Output fidelity guarantees** for orchestrator completion messages align with “agent-as-dev-tool” usage (#8162).

## 3) Implementation Opportunities (by major pain point)

### A) Cloud Apps scale/ingress gaps (#8321)
**1) Close P0 reachability + contract gaps (High impact, Medium difficulty)**
- Ship and enforce an ingress baseline (Caddy/Traefik) as part of app-node bootstrap; add a deploy-time smoke check that verifies `https://<id>.apps...` routes to container.
- Add a “contract test” in CI for env vars: ensure both `DATABASE_URL` and `POSTGRES_URL` are injected and actively used (detect silent fallback like PGlite).
- Similar projects: **Fly.io / Render** run post-deploy health checks and block promotion if routing/health endpoints fail.

**2) Add connection pooling and sane defaults (High impact, Medium difficulty)**
- Introduce **pgbouncer** (or equivalent) on tenant-db nodes; lower per-app pool max or make it adaptive to tier.
- Preconfigure `max_connections` and pool sizing based on expected concurrency; expose as env-config with documented defaults.
- Similar projects: **Supabase** and **Heroku Postgres** rely on poolers to avoid connection exhaustion with many small apps.

**3) Multi-node readiness hardening (High impact, High difficulty)**
- Fix placement logic (don’t always pick `seed[0]`); implement node-aware scheduling and a load balancer front door (Hetzner LB / Cloudflare) for app nodes.
- Configure Docker address pools to avoid network exhaustion per node.
- Similar projects: **Kubernetes** solves placement/LB/networking systematically; if staying on raw Docker, borrow minimal patterns: deterministic port allocation + registry of allocations + LB in front.

---

### B) Cloud login / Steward PKCE regressions (#8257)
**1) “Auth contract” integration tests (High impact, Low–Medium difficulty)**
- Add an end-to-end CI test that exercises:
  - OAuth authorize URL contains PKCE parameters
  - code exchange succeeds
  - email send/verify succeeds (where applicable)
- Gate production deploys on these checks for both staging and prod endpoints.

**2) Versioned auth flow + explicit compatibility matrix (Medium impact, Low difficulty)**
- Document “PKCE required as of Steward version X” and encode it as a versioned contract in code (e.g., feature flag or capability discovery endpoint).

**3) Reduce blast radius with graceful degradation (Medium impact, Medium difficulty)**
- If Cloud SSO is down, provide a fallback path for local/dev (e.g., local token login, or temporary session issuance) with clear UI messaging.
- Similar projects: many SaaS products implement “status + retry later” UX and keep local/dev modes separate from production auth.

---

### C) Orchestrator deliverable presentation defects (#8162)
**1) Structural “deliverable carrier” (High impact, Low–Medium difficulty)**
- Implement the proposed `metadata.subAgentDeliverable` path and consume it verbatim on completion when there’s no change set and no verified URLs.
- This avoids re-summarization truncation and preserves “printed value” tasks.

**2) Terminal-only reply filtering (Medium impact, Medium difficulty)**
- Suppress stale attempts and tool chatter; show only the final successful attempt’s deliverable (while still using internal signals to trigger respawn).
- Similar projects: **OpenHands** and **Cursor-like agent loops** often separate “internal trace” from “user-facing answer.”

**3) UX affordance: “Show trace” toggle (Medium impact, Low difficulty)**
- Default to clean deliverable; provide an expandable “execution trace” panel for power users.
