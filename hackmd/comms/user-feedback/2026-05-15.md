# User Feedback Analysis — 2026-05-15 (elizaOS)

## 1) Pain Point Categorization (Top recurring 5–7)

**Sample size:** 12 distinct feedback signals (Discord threads + GitHub issues/reviews in provided data). Percentages below are of these 12 items.

### A. Integration (most frequent; **6/12 = 50%**)
**What’s impacting users most**
- **Unexpected third-party API cost/rate behavior (Neynar):** user reported **~300 credits/hour vs expected ~4 credits/hour** on `v2/farcaster/notifications`, even after overriding the default polling interval; asked for documentation on credit consumption and how to control it.
- **Connector reliability failures causing silent loss / broken flows:**
  - Telegram: dual long-poll consumers racing on the same token causes **silent message loss** (“every other message” pattern) (GitHub issue #7245).
  - Auth/provider connectors: missing “Claude Code stealth” preload file breaks Anthropic OAuth subscription auth with persistent **401** (issue #7210); Codex CLI auth file format drift causes “install required” false negative (issue #7243).
- **Ecosystem integration demand is trending on-chain:** requests/proposals for **agent escrow (Base)** and **GODL mining protocol (Solana)** integrations, both emphasizing SDK/websocket hooks and agent wallets.

### B. Documentation & Product Direction (high severity for onboarding; **3/12 = 25%**)
**What’s impacting users most**
- **Roadmap ambiguity for Eliza v3:** repeated questions on whether v3 is “infra stabilization” vs “public release of full autonomous workflow stack,” plus unclear **AgentID/identity system** status (Discord, 2026-05-13).
- **Missing operational docs for integrations:** Neynar credit accounting expectations and recommended patterns (polling vs webhooks) are not documented where users look first.
- **Plugin submission process unclear during registry outage:** users blocked by registry 404s and unsure how BUSL-1.1 plugins should be submitted (Discord, 2026-05-12).

### C. Technical Functionality / Fresh-install breakages (high severity; **2/12 = 17%**)
**What’s impacting users most**
- **Database/migration drift:** `plugin-sql` runtime-migrator missing drizzle table defs leads to repeated query errors and chat becoming effectively unusable on fresh PGLite boot (issue #7222).
- **Client/runtime API mismatch breaks UI flows:** duplicate `MiladyClient` classes means prototype augmentation doesn’t land, causing widespread `is not a function` UI failures (issue #7244).

### D. Performance & Model Capability (emerging but important; **1/12 = 8%**)
**What’s impacting users most**
- **Action calling + planning quality:** active work reported on Qwen-based stack with planned fine-tuning pipeline specifically to improve **action calling and planning** (Discord, 2026-05-14). This signals real-world use is pushing beyond “chat” into reliable tool execution.

### E. Community / Trust & Safety (high risk; **1/12 = 8%**)
**What’s impacting users most**
- **Discord security + spam/scams:** reports of compromised admin accounts and recurring scammer activity; spam investment solicitations observed and moderated (Discord, 2026-05-12 to 2026-05-14).

## 2) Usage Pattern Analysis (actual vs intended)

### How users are actually using elizaOS
- **Agent-as-integrator for external systems (dominant):** most issues center on connectors and third-party services (Telegram, Farcaster/Neynar, Anthropic/Codex auth). Users treat elizaOS as an *integration runtime* more than a standalone agent demo.
- **Agents interacting with on-chain protocols (growing):**
  - Base Mainnet escrow protocol with “Pay or Burn” dispute resolution for agent-to-agent agreements (Discord, 2026-05-14).
  - Solana “GODL” gamified mining/staking with websocket streams + automation hooks + `skill.md` integration format (Discord, 2026-05-13).
- **Multi-hardware deployment expectations:** Eliza-1 model series spans **mobile offline (0.6B)** to **million-token context (27B-1M)**, indicating users want agents to run everywhere—from phones to servers (Discord, 2026-05-13).

### Emerging / unexpected use cases
- **“Zero UI” protocol components:** escrow protocol intentionally ships raw contracts + SDKs only. This suggests an ecosystem segment wants elizaOS to be a *headless automation substrate* and will accept minimal UI if SDK ergonomics are strong.

### Feature requests aligning with real usage
- **Webhook-first patterns** for event streams (explicit suggestion for Neynar).
- **Standardized integration descriptors** (`skill.md` acceptance confirmed) and likely demand for a broader “integration contract” across protocols/services.
- **Agent wallet + on-chain automation primitives** (requested in GODL integration proposal).

## 3) Implementation Opportunities (solutions per major pain point)

### 1) Third-party API cost surprises (Neynar credits) — Integration + Docs
**Observed:** 300 credits/hour actual vs 4 expected on notifications endpoint; user uncertain how billing maps to polling behavior.
- **High impact / Low–Med effort**
  1) **Add an “API credit estimator” + diagnostics panel** per connector: log request frequency, endpoint, response codes, and an *estimated credits/hour* counter surfaced in Settings and logs (e.g., “Farcaster: 150 req/hr → est 300 credits/hr”).
     - Similar pattern: Stripe and Twilio dashboards emphasize usage metrics and request counts; Sentry-style “breadcrumbs” help correlate spikes.
  2) **Provide webhook-first connector mode + migration guide** (polling fallback only): ship a canonical Neynar webhook handler + local tunneling instructions.
     - Similar: GitHub Apps recommend webhooks over polling; Slack event subscriptions are webhook-centric with retry semantics.
- **Medium impact / Low effort**
  3) **Ship safe defaults:** enforce minimum poll interval, jitter + exponential backoff on errors, and a hard cap (“stop polling if credits/hour exceeds threshold unless user confirms”).

### 2) Connector silent failures (Telegram message loss; auth drift) — Integration + Technical Functionality
**Observed:** dual pollers cause silent drops; missing preload breaks Anthropic OAuth; Codex auth detection outdated.
- **High impact / Medium effort**
  1) **“Single owner per token” enforcement:** at runtime boot, detect multiple consumers for the same Telegram bot token and hard-error with an actionable message (“Disable plugin X or wrapper Y”).
     - Similar: Kubernetes controllers enforce leader election; Docker prevents two daemons from binding the same socket.
  2) **Compatibility tests for auth file formats + connector boot:** add fixture-based tests for Codex legacy + modern `tokens.access_token` shapes; add a “preload existence” test that fails CI if stealth mode is enabled but file is missing.
     - Similar: Homebrew formula CI checks prevent missing artifacts; VS Code extensions run activation tests.
- **Medium impact / Low effort**
  3) **Make silent drops impossible:** standardize a connector error contract where every inbound message either becomes a stored memory + ack, or produces a visible error log + metric increment.

### 3) Plugin registry outage + submission policy confusion — Integration + Documentation
**Observed:** registry repo and `plugins.elizacloud.ai` returning 404; BUSL-1.1 plugins can’t PR monorepo; uncertainty about the “right” path.
- **High impact / Low–Med effort**
  1) **Publish a “Submission Status Page”** (even a GitHub Pages doc) that states current registry availability, supported licenses, and the temporary submission workaround (with examples).
     - Similar: Rust crates.io and PyPI status pages; Kubernetes release docs provide “known issues + workarounds.”
  2) **Add CI checks for registry endpoints + repo visibility** (simple uptime/404 smoke) and alert maintainers on breakage.
- **Medium impact / Medium effort**
  3) **Dual-lane submission model:** “Open-source plugins via PR” + “restricted-license plugins via registry publish” with a documented review pipeline and metadata schema validator (JSON schema + preflight CLI).

### 4) Fresh install breakages (missing DB tables; duplicate client class) — Technical Functionality + UX/UI
**Observed:** missing tables cause cascading provider/evaluator failures; UI crashes due to missing client methods.
- **High impact / Medium effort**
  1) **Boot-time sanity checks:** on startup, validate required DB tables exist (and required client prototype methods exist) and fail fast with a guided remediation.
     - Similar: Rails checks pending migrations; Prisma warns on schema mismatch; Next.js shows actionable compile/runtime overlays.
  2) **Single source of truth for schemas:** generate drizzle defs from the abstract schemas (or vice versa) and add a “schema drift” unit test that asserts parity.
- **Medium impact / Low effort**
  3) **Guardrails in the UI:** if a client method is undefined, show a non-blocking banner with “API mismatch—please update/rebuild” rather than freezing key surfaces like Chat.

### 5) Roadmap/expectation mismatch (v3 focus; AgentID status) — Documentation + Community
**Observed:** repeated unanswered questions about v3 priorities and identity integration.
- **High impact / Low effort**
  1) **Maintain a 1-page “Now / Next / Later” v3 roadmap** pinned in Discord + repo, explicitly calling out AgentID status (“in design”, “in implementation”, “deferred”) and what “public release” means.
     - Similar: Linear/Notion roadmap snapshots; Rust edition guides clearly define “what changes when.”
  2) **Monthly “Architecture office hours” thread**: collect v3 questions and answer them asynchronously with links to PRs/issues.

### 6) Trust & safety issues (Discord scams / compromised accounts) — Community
- **High impact / Medium effort**
  1) **Security hardening playbook:** enforce 2FA for admins/mods, rotate tokens, audit bot permissions, and publish a “How to spot scams” pinned post.
     - Similar: OpenSSF community guidelines; major OSS Discords enforce mandatory 2FA for elevated roles.
  2) **Auto-moderation + reporting funnel:** add keyword/URL filters, rate limits for new accounts, and a `/report` workflow that opens a mod ticket.

## 4) Communication Gaps (expectations vs reality)

### Where expectations don’t match reality
- **“Override polling interval” ≠ predictable billing:** users expect “120s polling” to map linearly to credits, but actual consumption suggests hidden factors (server-side paging, per-notification costs, or multiple calls).
- **“There is a plugin registry” ≠ “it’s reachable and stable”:** 404s create uncertainty about whether the ecosystem path is official or abandoned.
- **“Eliza v3 is coming” ≠ clarity on scope:** users can’t tell if they should build on v2/v3 now, or wait.

### Recurring questions indicating onboarding/doc gaps
- “Is there documentation on Neynar credits consumption?”
- “How do I submit a plugin to the v2 registry, especially with BUSL-1.1?”
- “Is Eliza v3 stabilizing infra or nearing full autonomous workflow release?”
- “What is the integration status of AgentID/identity?”

### Specific improvements to align expectations
- Add an “Integration Costs & Limits” doc section per connector (Neynar/Farcaster, Telegram, Slack, etc.) with: request patterns, webhook alternatives, backoff defaults, and a cost troubleshooting checklist.
- Add “Known Issues / Current Outages” page (docs + Discord pinned message) covering registry availability and current workaround.

## 6) Feedback Collection Improvements

### Effectiveness of current channels
- **Discord:** good for rapid triage (e.g., webhook suggestion), but issues remain unresolved/uncertain (“resolution not confirmed” on Neynar credits; unanswered roadmap questions).
- **GitHub issues:** excellent for deep technical reports (repro steps, diffs), but may miss lighter UX confusion until it becomes a bug report.

### More structured, actionable feedback collection
1) **Add a standard “Integration Incident” issue template** capturing: provider, endpoint, expected vs actual cost/behavior, polling/webhook config, logs snippet, and reproduction.
2) **Discord → GitHub escalation bot/workflow:** allow helpers/mods to convert a Discord thread into a prefilled issue with links and key fields.
3) **Monthly connector health survey** (short): “Which connectors are you using? Any silent drops? Unexpected costs? Setup time?”

### Underrepresented user segments (missing feedback)
- **Non-coders / UI-first users**: current feedback is highly technical; likely many abandon silently when onboarding breaks.
- **Production operators**: limited structured feedback on monitoring, SLAs, cost controls, and incident response beyond individual bug reports.
