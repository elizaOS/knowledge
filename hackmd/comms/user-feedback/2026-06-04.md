# User Feedback Analysis — 2026-06-04 (elizaOS)

## 1) Pain Point Categorization (Top recurring 5–7)

**Method note:** This analysis is based on the last 3 days of Discord summaries (2026-06-01 to 2026-06-03) plus high-signal GitHub issues/PR discussions surfaced in the aggregated feed (notably issues #8162, #7991, #7970, #8127, #8136). In this dataset, **~60% (6/10) of distinct feedback items are “Technical Functionality” defects**, **~20% (2/10) Integration**, and the remainder split between **Documentation/Community expectation-setting**.

### A) Technical Functionality
**Recurring problems affecting most users**
1) **Orchestrator “deliverable” corruption in sub-agent completions** (GitHub #8162, open)  
   - Deliverables dropped/truncated/garbled when planner re-narrates sub-agent output (especially “print the value” / no-change-set tasks).  
   - Stale retry attempts and “process chatter” leak into user-visible final responses.

2) **Tool/skill envelope leakage into user replies** (GitHub #7970, closed but high UX impact)  
   - Raw JSON/bash invocation wrappers appear in Discord responses when using `USE_SKILL`.

3) **Local inference runtime sharp edges** (GitHub #7991 closed; ongoing related work noted in 6/3 dev summary + PR #8191)  
   - Windows/CUDA prewarm token batching assert; missing native binding errors needed typed signaling.

4) **URL verification / freshness gating false negatives** (GitHub #8127, closed)  
   - Served HTTP 200 builds marked “stale/dead” due to mtime freshness gate → causes retries and diff-withholding regressions.

**Severity:** High. These issues make the system appear unreliable even when underlying agent work is correct (a “trust gap” problem).

### B) Integration
**Recurring problems affecting most users**
1) **DeFi actions beyond “info retrieval” need first-class integration surfaces** (Discord 2026-06-02 Kamino plugin request; 2026-06-01 multi-chain aggregator debate)  
   - Users want agents to execute meaningful DeFi operations: lending/borrowing/vault management/limit orders (Kamino), plus multi-chain swap/earn/perps/bridge/borrow.

2) **Abstraction vs protocol-specific integrations is unclear** (Discord 2026-06-01)  
   - Builders are split: unified interface (less boilerplate) vs protocol-native control (risk-sensitive operations need precision).

**Severity:** Medium–High for the current user segment (agent+crypto builders). Blocks adoption for “agent-as-operator” finance use cases.

## 2) Usage Pattern Analysis (Actual vs intended)

### Observed actual usage
1) **Agents as operational executors for finance/DeFi**, not just chat assistants  
   - Users are designing systems that return **signable calldata** for non-custodial wallets (Discord 2026-06-01), and requesting deep protocol coverage (Kamino).

2) **Multi-agent coding/orchestration is a core workflow**, not a niche  
   - The intensity of orchestrator bugs (#8162) and UI parity efforts indicates users rely on sub-agent execution and want “Claude Code/Codex-like” surfaces.

3) **Cloud auth + hosted agents are central**  
   - 6/3 development focus on magic-link sign-in, CSP/CORS, and SSO pairing handler implies many users begin in Cloud flows rather than purely local dev.

### Emerging / unexpected use cases
- **Agent-controlled financial permissions & testing** (Neverbell testers request, Discord 2026-06-01/06-03): users want “natural language → controlled execution” with explicit permissioning.

### Feature requests aligned with usage
- **Comprehensive Kamino plugin** (Discord 2026-06-02): lending/borrowing/vaults/yield/limit orders.  
- **DeFi integration strategy**: either (a) an aggregator interface layer or (b) a reference “gold-standard” protocol plugin with safe abstractions.

## 3) Implementation Opportunities (by major pain point)

### Pain Point 1: Orchestrator deliverable drops/garbling (#8162) — *High impact, Medium difficulty*
**Opportunities**
1) **Implement a “verbatim deliverable carrier” in completion metadata** (High impact, Medium difficulty)  
   - Mirror the existing “verified URLs” carrier approach: store `metadata.subAgentDeliverable` before narration stripping; render verbatim in completion evaluator.  
   - Similar approach: *OpenHands/CodeAct-style* “artifact” channels separated from assistant prose; *LangGraph* separates tool outputs from messages.

2) **Terminal-only reply selection + retry suppression** (High impact, Medium–High difficulty)  
   - Only display final successful attempt; keep failure events internal for respawn triggers.  
   - Similar approach: *Temporal/Conductor* UIs show attempt history in an inspector panel but keep primary output clean.

3) **Strict tool-output hygiene contract for sub-agents** (Medium impact, Low difficulty)  
   - Enforce/validate “deliverable-first” formatting and cap chatter; treat “process” as collapsible.  
   - Similar approach: *Claude Code* and *Cursor* keep tool logs separate from answer.

---

### Pain Point 2: Tool/skill envelope leakage into user replies (#7970) — *High impact, Low difficulty*
**Opportunities**
1) **Standardize `userFacingText` across tool results** (High impact, Low difficulty)  
   - Ensure `USE_SKILL` sets `verifiedUserFacing: true` and populates `userFacingText` with clean output.

2) **Output sanitizer pass before final message assembly** (Medium impact, Low difficulty)  
   - Strip known envelopes (`{"cmd":...}`, `{"output":...}`) at the planner-loop boundary.

3) **Regression tests for “no internal envelopes”** (Medium impact, Low difficulty)  
   - Similar to existing planner-loop envelope suppression work (e.g., fixes like suppressing leaked TASKS JSON).

---

### Pain Point 3: Local inference reliability (prewarm/bindings) — *Medium–High impact, Medium difficulty*
**Opportunities**
1) **Clamp prewarm batch sizing to runtime `n_batch`** (High impact, Medium difficulty)  
   - Avoid Windows/CUDA asserts; use device-specific defaults.

2) **Typed capability detection surfaced to UI** (Medium impact, Medium difficulty)  
   - Continue the “typed error signaling” direction so users see actionable “binding missing / bundle missing” guidance.

3) **Hardware matrix + “known-good bundle presets”** (Medium impact, Medium difficulty)  
   - Similar approach: *llama.cpp ecosystem* publishes recommended configs by GPU/VRAM.

---

### Pain Point 4: DeFi integrations (Kamino + abstraction debate) — *High impact for target segment, High difficulty*
**Opportunities**
1) **Ship a reference “Protocol Gold Plugin” (Kamino) with safe primitives** (High impact, High difficulty)  
   - Actions: open/close lending positions, adjust collateral, deposit/withdraw vaults, place/cancel limit orders.  
   - Include guardrails: slippage caps, health factor thresholds, simulation/dry-run mode.

2) **Define a minimal “DeFi Action Interface” + optional aggregator layer** (Medium–High impact, Medium difficulty)  
   - Separate *interface standardization* from *routing/aggregation*.  
   - Similar approach: *0x/1inch routing APIs* standardize quoting/execution while allowing protocol-native integrations.

3) **Security/permissions framework for financial actions** (High impact, High difficulty)  
   - Align with emerging “user-defined controls and permissions” signals (Neverbell).  
   - Similar approach: “spending limits” + “allowlist actions” like smart-wallet policies (e.g., Safe modules).

## 4) Communication Gaps (expectations vs reality)

1) **Expectation:** “If the agent did the work, I will see the deliverable.”  
   **Reality:** Deliverables can be dropped/truncated by narration layers (#8162).  
   **Improve:** Document the separation between *tool transcript*, *artifact/deliverable*, and *assistant narration*; ensure UI renders deliverables as first-class artifacts.

2) **Expectation:** “Plugins live where open-source plugins usually live (separate repos) and stable branches exist.”  
   **Reality:** Plugins consolidated into `elizaOS/eliza`, rapid `develop` churn, frequent rebasing required (Discord 2026-06-02).  
   **Improve:** Add a short “Plugin Contribution Workflow” doc: branch strategy, rebase cadence, how to avoid conflicts, and recommended PR sizing.

3) **Expectation:** “Project relationships are obvious.”  
   **Reality:** Users ask repeatedly about Hyperia/Hyperforge vs ElizaOS and token/project focus (Discord 2026-06-03/06-01).  
   **Improve:** A public-facing “Project map” and “Not affiliated with” section.

Recurring newcomer questions indicating onboarding gaps:
- “Where do plugins live now / which branch do I target?”  
- “What is ElizaCloud vs ElizaOS?”  
- “Is the project still active / what is the roadmap focus?”
