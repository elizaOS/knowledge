## 1) Pain Point Categorization (Top recurring friction areas)

### 1) Technical Functionality — “Works on my machine” breaks on fresh clone / first boot
**Frequency/Severity:** High severity; repeatedly causes “can’t chat / unusable” outcomes on first run.  
**Evidence/examples:**
- **Missing runtime-migrator tables** in `plugin-sql` caused continuous query failures (`entity_identities`, etc.), leading to broken state composition and downstream parsing errors (Issue **#7222**).  
- **Missing preload artifact** for Anthropic subscription OAuth (“stealth” interceptor) leads to persistent 401s even with valid credentials (Issue **#7210**).  

**What users experience:**
- Boot “looks fine,” but the first real interaction fails (chat replies degrade into retries + parse errors).
- Silent fallbacks make debugging harder (feature appears supported, but is effectively non-functional).

---

### 2) Integration — Messaging connectors silently dropping messages (Telegram/Slack)
**Frequency/Severity:** Very high severity; message loss destroys trust.  
**Evidence/examples:**
- **Telegram dual polling race**: milady wrapper + `@elizaos/plugin-telegram` both poll `getUpdates`, causing ~50% silent message loss (Issue **#7245**).  
- **Slack plugin risk (review finding):** unguarded `getUser()` calls can throw inside event handlers → dropped messages with no memory/reply (PR **#7375** Greptile P1).  

**What users experience:**
- “Bot sometimes replies, sometimes ignores me” with minimal actionable logs.
- Hard to diagnose without deep runtime knowledge.

### 3) UX/UI — Dashboard breaks when client methods are missing; errors cascade into frozen UI
**Frequency/Severity:** High severity; prevents basic usage.  
**Evidence/examples:**
- Duplicate `MiladyClient` class meant augmenters patched a never-instantiated prototype → many UI calls are `undefined`, causing React error boundary → chat UI becomes unresponsive (Issue **#7244**).

**What users experience:**
- The app loads, but core views (Chat/Apps) crash and disable input.

## 2) Usage Pattern Analysis (actual vs intended)

### Observed real-world usage patterns
1) **Connector-first deployments** (Telegram/Discord/Slack) are a primary path to value.
- Users judge reliability by “did my message get a reply every time?”
- This makes silent drops and polling races disproportionately damaging.

2) **Agents as autonomous operators for on-chain protocols** is accelerating.
- GODL integration proposal (Solana) uses `skill.md` + SDK/websockets/automation hooks.
- Multiple escrow/payment discussions suggest agents performing paid work with settlement.

3) **Model experimentation for planning/action-calling** is active among power users.
- Qwen optimizations + fine-tuning pipeline aimed specifically at “action calling and planning” (Discord 2026-05-14).
- Release of Eliza-1 Qwen3.5 variants (mobile offline → million-token context) suggests broad hardware targeting.

### Emerging / unexpected applications
- **“No-UI protocols”** (raw smart contracts + SDKs) designed to be consumed directly by agents.
- **Mobile/offline agent models** (0.6B mobile-optimized) implying edge deployment interest, not just server/cloud.

### Feature requests aligning with usage
- Webhook-based ingestion patterns (vs polling) to reduce costs and improve reliability (Neynar).
- Stronger “integration contracts” for skills (`skill.md`) and connector event handling to avoid message loss.

## 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

### A) Fresh clone / first boot failures (schemas, missing artifacts, silent fallbacks)
**High impact, moderate difficulty**
1) **Add “boot invariants” + fail-fast checks**
   - At runtime start, verify:
     - required SQL tables exist (compare abstract schemas vs drizzle tables; detect drift like #7222).
     - required build artifacts exist (e.g., preload `.mjs` for Anthropic stealth path; #7210).
   - If missing, surface a single actionable error panel in UI + CLI output with exact fix steps.

2) **Make migrations + schema sources-of-truth single-path**
   - Consolidate schema definitions so runtime-migrator cannot drift from app code.
   - Similar approach: **Prisma** and **Django** enforce a primary migration pipeline; Home Assistant uses strict startup checks for integration registration.

3) **Replace “silent filter-out” behavior with explicit warnings**
   - Example: when stealth imports are enabled but missing, log a red warning and link to docs; do not silently proceed.

---

### B) Connector message loss (Telegram/Slack) and reliability guarantees
**Very high impact, moderate difficulty**
1) **Enforce “single consumer per bot token”**
   - If wrapper exists, forcibly disable upstream plugin polling (or vice versa).
   - Add a runtime guard: detect duplicate pollers and refuse to start the second.
   - аналог: Kubernetes controllers use leader election to prevent dual reconciliation.

2) **Standardize connector ingestion on webhooks where possible**
   - Telegram: prefer webhook mode in hosted contexts; for self-hosted polling, ensure exactly one poller.
   - Neynar: move from polling to webhooks to prevent credit spikes.

3) **Harden event handlers against upstream API failures**
   - Slack: wrap `users.info` and other upstream calls in try/catch; on failure, continue with minimal identity and store memory anyway (PR #7375 P1).
   - Similar practice: **Slack Bolt** community patterns recommend “never throw from event handlers.”

---

### C) UX/UI crash cascades from missing API client methods
**High impact, low-to-moderate difficulty**
1) **Eliminate prototype augmentation side-effects**
   - Replace augmenter pattern with explicit composition/imports, or enforce via build-time assertion that augmenters executed.
   - Add a unit test asserting `client.listAppRuns` etc. exist (regression test for #7244).

2) **UI resilience: degrade gracefully instead of freezing**
   - If a client method is missing, show a recoverable error toast and keep input enabled.
   - Similar approach: VS Code extensions isolate failures and keep editor responsive.

---

### D) Roadmap / identity (AgentID) / “eliza OZ” expectation management
**High impact, low difficulty**
1) **Publish a weekly “What’s stable / what’s changing” post**
   - Include: v3 milestone focus, AgentID status, known breaking areas (connectors, auth flows), and expected timelines at a coarse granularity.

2) **Create a single canonical “Versioning & Stability” page**
   - Define: experimental vs beta vs stable APIs (connectors, skills format, identity primitives).
   - Similar to **Kubernetes API stability levels** and **Rust’s feature gates**.

3) **Discord automation: FAQ bot that answers recurring roadmap questions**
   - Pin: v3 scope, AgentID, OZ definition, partnership links.

---

### E) Cloud monetization/app-scoped chat correctness (auth + billing edge cases)
**Very high impact, moderate-to-high difficulty**  
(From PR #7376 review findings)
1) **Fix auth error classification to return 401/403, not 500**
   - Ensure route-level auth exceptions are handled outside `Promise.all` or are mapped explicitly.

2) **Make credit reconcile robust to partial failures**
   - Streaming: once content delivered, never full-refund on reconcile failure; queue reconcile job instead.
   - Non-streaming: if provider response succeeded, return response even if reconcile fails; reconcile asynchronously.
   - Similar patterns: **Stripe** uses idempotent events + async reconciliation; many SaaS billing systems decouple serving responses from final ledger settlement.

## 4) Communication Gaps (expectations vs reality)

### Where expectations don’t match reality
- **“Connector enabled” ≠ “reliable delivery.”** Users assume 1:1 message→reply; current failure modes include silent drops and dual pollers.
- **“Subscription OAuth supported” ≠ “works out of the box.”** Missing stealth preload made Anthropic subscription flow appear broken (#7210).
- **“v3 is coming” but unclear whether it’s stabilization or imminent public stack release.** This blocks adoption planning and community contribution targeting.

### Recurring questions indicating doc/onboarding gaps
- “What is the status of eliza OZ?” (Discord 2026-05-15)
- “Is v3 stabilizing infra or nearing full autonomous workflow release?” (Discord 2026-05-13)
- “Is AgentID integrated yet?” (Discord 2026-05-13)
- “How do I avoid API credit burn?” (Discord 2026-05-14 Neynar)

### Specific documentation improvements
- Add a **Connector Reliability Guide**:
  - polling vs webhooks, recommended defaults, exactly-once/at-least-once semantics, and how to detect duplicate consumers.
- Add an **Auth & Credentials Matrix**:
  - Codex CLI formats (legacy vs modern), Anthropic OAuth subscription caveats, vault vs env var precedence.
- Add a **Roadmap + “Breaking changes this month”** changelog.
