## User Feedback Analysis — 2026-05-21 (covering Discord 2026-05-18 to 2026-05-20 + recent GitHub activity)

### 1) Pain Point Categorization (Top recurring friction areas)

**Overall signal:** Feedback clusters into (a) expectation-setting / project comms (tokens, roadmap, “what is shipping”), and (b) local-inference + hardware/runtime edge cases. Discord skews toward roadmap/token questions; GitHub skews toward deep performance/runtime correctness.

#### A. Documentation (high frequency, medium-high severity)
**Recurring problems**
- **Missing/unclear release notes for v2.0.0**: users explicitly asked what shipped and what changed (Discord 2026-05-19).
- **“How do I…?” integration docs absent**: e.g., “connect Claude AI agent to AgentKey” asked with **no response** (Discord 2026-05-18).
- **Unclear “v3 is close” messaging vs concrete feature lists** (Discord 2026-05-19).

**Who it affects most:** newcomers and evaluators trying to decide whether to build now vs wait.

#### B. Community (high frequency, medium severity)
**Recurring problems**
- **Questions going unanswered**: In the sampled Discord logs, there were **7 distinct user questions** (release notes, AgentKey+Claude, physical chip, token contract/CA, degenai update, airdrop status, Milady buyback chain). **4/7 (~57%) received no answer in the captured period** (physical chip, CA correctness, degenai update, AgentKey integration).
- Heavy self-promotion/networking content dilutes support bandwidth (Discord 2026-05-18/19).

**Who it affects most:** users evaluating legitimacy/clarity (token CA, roadmap) and integrators stuck on first steps.

#### C. Technical Functionality (medium frequency, high severity)
**Recurring problems (GitHub issues)**
- **Speculative decoding silently disabled or non-functional** on modern model architectures:
  - SWA bodies + DFlash: “target context does not support partial sequence removal” disables spec-decode with weak user-facing signaling (#7635).
  - M‑RoPE position validation rejects draft batches (Qwen3.5) collapsing acceptance to ~9% and throughput below 1 t/s (#7631).
- **Build/runtime footguns**: missing preload artifact breaks Anthropic subscription OAuth flow on fresh clone (#7210).

**Who it affects most:** power users self-hosting local inference; anyone following recommended tiers that assume spec-decode works.

#### D. Performance (medium frequency, high severity)
**Recurring problems**
- **Model-specific perf cliffs on Intel iGPU**: Qwen2.5-14B Q4_K_M observed at ~1.15 t/s vs expected bandwidth curve (#7638). Even if “n=1”, the cliff is large enough to block practical use.
- **Spec-decode degenerate performance** when validation rejects batches (#7631).

**Who it affects most:** local inference users on integrated GPUs (Intel Lunar Lake class), which elizaOS is actively optimizing for.

#### E. Integration (medium frequency, medium-high severity)
**Recurring problems**
- **External SDK/plugin integration uncertainty**:
  - based.bid wants to integrate their SDK; elizaOS mentions trading bots but no clear integration path (Discord 2026-05-19).
  - New DeFi plugin (@centinel-risk-protocol/eliza-plugin-centinel) requests testing, but no structured testing guidance or feedback loop is provided (Discord 2026-05-20).

**Who it affects most:** third-party builders (plugins/connectors) who need stable interfaces, examples, and compatibility matrices.

#### F. UX/UI (low frequency in Discord, medium severity in shipped product)
**Observed indirectly via merged PR notes**
- Ongoing UI papercuts like modal overlays/z-index issues (Tailwind v4 emission fix) and onboarding friction (voice onboarding made skippable) appear as reactive fixes, suggesting these are recurring user-impacting issues even if not directly raised in this Discord slice.

#### G. Community/Project Comms (tokenomics & roadmap) (high frequency, high severity for trust)
**Recurring problems**
- **Airdrop expectation mismatch**: “airdrop mentioned long ago” → clarified as canceled due to budget (Discord 2026-05-20).
- **Token detail confusion**: which Milady token is bought back (BNB vs Solana) needed explicit clarification (Discord 2026-05-20).
- **Governance/utility ambiguity**: multiple questions about token utility, buyback %, governance structure; clarified as “early, focus on product + profitability first” (Discord 2026-05-19).
- **Hardware roadmap ambiguity**: “Is there a physical Eliza E1 chip?” asked and unanswered (Discord 2026-05-20).

  - DeFi risk/scoring + flash-crash simulation plugin shipped to npm and seeking testers (Discord 2026-05-20).
  - External SDK integration interest for trading bots (Discord 2026-05-19).

**Emerging / unexpected use cases**

- **DeFi agent risk tooling**: solvency scoring + crash simulation implies users want agents that do *continuous monitoring + scenario stress testing*, not just chat.

**Feature requests aligned with observed usage**
- Better **plugin testing harnesses** and “how to validate a plugin” guidance (implied by the DeFi plugin’s open call for testing).
- Clearer **local inference capability gating** (spec-decode availability, model tier suitability) so recommender doesn’t route users into broken performance paths (explicitly requested in GitHub issues).

### 3) Implementation Opportunities (concrete solutions per major pain point)

#### Pain Point 1: Unanswered questions + support visibility gaps (Community / Documentation)
**Solutions (prioritized by impact vs difficulty)**
1) **Add a “Support Triage” bot + workflow** in Discord that:
   - auto-tags unanswered questions after 2–6 hours,
   - routes to an “On-call helper” role,
   - converts messages with a `?` + keywords into lightweight tickets.  
   *Impact: High | Difficulty: Medium*  
   **Comparable approach:** many OSS Discords (e.g., Home Assistant-style community triage) use helper rotations + tagging to prevent silent drops.
2) **Create a single pinned “Known Answers / Current Status” post** updated weekly:
   - airdrop status, token buybacks chain, governance current state, v2/v3 roadmap, hardware E1 meaning.  
   *Impact: High | Difficulty: Low*
3) **Define “response SLAs” for public questions** (even “we don’t know yet”):
   - target: acknowledge within 24h for roadmap/token questions.  
   *Impact: Medium | Difficulty: Low*

#### Pain Point 2: Release notes + “what shipped” ambiguity (Documentation)
1) **Automate release notes generation** (PR labels → changelog sections) and publish links in Discord release announcements.  
   *Impact: High | Difficulty: Medium*  
   **Comparable approach:** Kubernetes/Next.js-style categorized release notes from labels.
2) **Add “What changed for users?” section** per release (not just dev-facing diffs).  
   *Impact: High | Difficulty: Low*
3) **Versioned migration notes** for common setups (self-hosted, local inference, plugin dev).  
   *Impact: Medium | Difficulty: Medium*

#### Pain Point 3: Speculative decoding failures are silent / confusing (Technical Functionality / Performance)
1) **Fail loud + surface status in API/UI**:
   - If `--spec-type dflash` is incompatible, return explicit API field like `spec_decode_active=false` + reason (mirrors #7635 suggestions).  
   *Impact: Very High | Difficulty: Medium*
2) **Capability gating in recommender/catalog**:
   - Mark tiers requiring SWA-compatible rollback or M‑RoPE-aware spec-decode; don’t recommend spec-decode paths unless runtime reports support.  
   *Impact: High | Difficulty: Medium*  
   **Comparable approach:** Ollama/llama.cpp frontends often gate features by backend capability and expose it in `/api/tags`-style metadata.
3) **Add a “spec-decode diagnostic self-test” command**:
   - runs a short draft/acceptance probe and prints acceptance %, drafted/accepted counts, and whether validation errors occurred.  
   *Impact: High | Difficulty: Medium*

#### Pain Point 4: Hardware performance cliffs lack user-facing guidance (Performance / Documentation)
1) **Publish “known-good model ladder per hardware class”** with measured expectations:
   - e.g., Intel iGPU tiers, when to avoid certain Qwen 14B shapes until kernel selection is fixed.  
   *Impact: High | Difficulty: Medium*
2) **Add kernel-selection logging toggles exposed in UI** (“Performance debug mode”).  
   *Impact: Medium | Difficulty: Medium*
3) **Regression benchmarks in CI for representative shapes** (where feasible):
   - at least token/sec sanity thresholds for a small model set.  
   *Impact: Medium | Difficulty: High*

#### Pain Point 5: Token + roadmap expectation mismatch (Community / Communication)
1) **Add a “Token & Governance” canonical doc** and keep it updated:
   - explicit: “no governance yet”, “profitability first”, “no airdrop”, “buyback chain = BNB token”, “no new tokens planned”.  
   *Impact: High | Difficulty: Low*
2) **Create a “rumor control” FAQ section** in Discord pinned posts:
   - chip rumors, degenai status, CA links.  
   *Impact: Medium | Difficulty: Low*
3) **Provide authoritative links for contract addresses (CA)**:
   - verified sources and how to validate; address the explicit “Is Coingecko CA correct?” question.  
   *Impact: High | Difficulty: Low*

#### Pain Point 6: Third-party plugin testing is unstructured (Integration / Community)
1) **Introduce a “Plugin Testing Template”** (GitHub issue form + Discord thread format):
   - environment, chain/network, expected outputs, logs, minimal repro.  
   *Impact: High | Difficulty: Low*
2) **Add a lightweight compatibility badge** for plugins:
   - “tested on v2.0.2”, “requires Base/Aave credentials”, etc.  
   *Impact: Medium | Difficulty: Medium*
3) **Office-hours for plugin authors** (monthly):

### 4) Communication Gaps (expectations vs reality)

- **“Eliza E1 chip” ambiguity**: Users interpret the emulated RISC‑V target as potential physical hardware; the unanswered question creates speculation risk.

**Targeted improvements**

### 6) Feedback Collection Improvements

**Effectiveness of current channels**
- **Discord** is effective for announcements and quick clarifications but weak at ensuring closure (high unanswered rate in this slice).
- **GitHub** is effective for high-signal technical issues, but those insights don’t appear to propagate back into user-facing guidance quickly.

**Improvements**
1) **Monthly structured pulse survey** (5 questions):
   - setup success, biggest blocker, platform (OS/hardware), top desired plugin, doc gaps.  
2) **Discord → GitHub escalation rule**:
   - any unanswered technical question after 24h becomes a GitHub discussion/issue stub.
3) **Add issue forms for**:
   - “Local inference performance regression”
   - “Spec-decode not active / expected”
   - “Plugin integration request”

**Underrepresented segments**
- Non-technical end users (no direct UX complaints captured here, likely because they churn silently).
- Windows-focused self-hosters (few direct reports in this slice despite significant Windows-related fixes in the repo).
