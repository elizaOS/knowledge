## 3) Implementation Opportunities (2–3 concrete solutions per major pain point)

### Pain Point A: Cloud auth regressions (PKCE, redirects, cookies, MFA) block sign-in
**High impact / Medium difficulty**
1. **Add an automated “Auth Contract” test suite for prod-critical flows**
   - Playwright + API-level checks for: Google OAuth authorize → exchange, email magic-link, passkey, MFA-required branch handling.
   - Gate deploys on these checks (similar to how many SaaS projects run “smoke auth journeys” on each release).
2. **Introduce versioned “Steward compatibility matrix”**
   - Pin supported `@stwd/sdk` versions and document required server settings (PKCE required? signing/HMAC required?) in one place.
   - Similar pattern: Firebase/Auth0 publish explicit SDK/server version compatibility notes to prevent silent breaking changes.
3. **Add runtime “auth diagnostics” page for users**
   - A single screen that shows: tenant id, redirect_uri, PKCE enabled, cookie/session status, last error code, and suggested fix.
   - Reduces support load and shortens time-to-resolution when auth breaks.

---

### Pain Point B: Orchestrator narration envelope drops deliverables
**High impact / Medium difficulty**
1. **Implement a “verbatim deliverable” channel separate from narration**
   - Adopt the proposed pattern in #8162: capture stdout / tool-output before stripping and pass as `metadata.subAgentDeliverable`, then render verbatim.
   - Similar approach: GitHub Actions separates “step summary” from “raw logs”; many agent UIs separate “artifact/output” from “thought/process.”
2. **Terminal-attempt rendering (suppress stale retries)**
   - Only render the last successful attempt, while keeping retry signals internal for respawn logic.
   - Comparable to task runners that collapse retries and show the final status unless expanded.
3. **Add a UI affordance: “Show artifact / Show tool output” toggle**
   - Default: clean final answer + artifact block; optional expansion for transcripts.
   - This preserves power-user debugging without harming normal UX.

---

### Pain Point C: Install/boot failures due to dependency resolution and platform-specific runtime issues
**High impact / Medium difficulty**
1. **Make `app-core` dist self-contained or correctly hoist runtime deps**
   - Choose one: bundle transitive deps into dist (preferred for portability) or declare them as direct deps.
   - Similar to how VS Code extensions bundle critical runtime deps to avoid consumer environment differences.
2. **Add a cross-platform “first-run doctor” that matches real launch paths**
   - Validate: Node vs Bun loader expectations, Windows bun.exe resolution, GPU/local-inference prewarm sizing, presence of required assets.
3. **Harden local-inference warmup sizing**
   - Clamp prewarm tokens to `n_batch` and expose recommended values in bundle manifests; fail gracefully with actionable guidance instead of asserting.

---

### Pain Point D: Documentation mismatch + newcomer information overload
**High impact / Low-to-medium difficulty**
1. **Create a “Golden Paths” onboarding map**
   - 3 tracks: (a) build a plugin, (b) build a Cloud app (SSO + credits), (c) orchestrator multi-agent workflow.
   - Keep each under 30 minutes to first success; link to exact commands and expected outputs.
2. **Docs drift CI**
   - For skills like “build monetized app,” add a CI check that validates referenced routes/headers exist (e.g., `X-App-Id`, `/api/v1/containers`).
   - Similar to Stripe/OpenAPI doc linting against live spec.
3. **Discord “Start Here” bot command**
   - `/start` returns the 3 tracks and the best channel to ask questions; reduces repeated “where do I begin?” posts.

---

### Pain Point E: Community matching (recruitment + contribution pipeline)
**Medium impact / Low difficulty**
1. **Create a weekly “Help Wanted / Hiring / Pairing” thread with tags**
   - Tags: `paid`, `bounty`, `open-source`, `short-task`, `good-first-issue`, `buildathon`.
2. **Convert “looking for work” into structured profiles**
   - Simple form: skills, time zone, preferred area (cloud/auth, plugins, UI, security).
3. **Maintain a “starter task queue”**
   - 10–20 well-scoped issues with setup notes and a reviewer assigned; similar to Rust’s “E-easy” or Kubernetes “good first issue” discipline.

## 4) Communication Gaps (expectations vs reality)

1. **“Agent can do it” vs “UI shows it did it”**
   - #8162 highlights that capability exists but the UI output loses the deliverable; users interpret this as failure.
   - Improvement: explicitly separate “artifact” from “narration” and label it (“Result”, “Command output”, “Diff”).
2. **“Sign in with Eliza Cloud” implied reliability vs frequent auth drift**
   - Users expect SSO to be stable; breaking changes (PKCE requirements, signing/HMAC guards) undermine trust quickly.
   - Improvement: status page + change log for auth provider changes; proactive Discord announcements when Steward updates roll out.
3. **New users unsure where to start**
   - Discord evidence (ross.ross.ross) indicates macro-level confusion and information overload.
   - Improvement: a single canonical onboarding entry point, and clearer “what elizaOS is for” positioning per persona (builder vs operator vs contributor).

Recurring questions indicating onboarding gaps (observed qualitatively)
- “Where do I go to learn the ecosystem?” → directed to coders channel, but no single pinned “start here.”
- “Are there developer opportunities / how do I join a project?” → multiple asks, low conversion.
