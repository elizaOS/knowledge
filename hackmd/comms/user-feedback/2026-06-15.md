## User Feedback Analysis — 2026-06-15 (based on 2026-06-12 to 2026-06-14 community + repo signals)

### Data notes / quantification basis
This period contains **~8 distinct user-reported feedback items** across Discord (#discussion, #coders) plus linked GitHub context referenced by users (Cloud auth/access regressions, orchestrator/UI stability). Percentages below are calculated against these distinct items (not message count), so they should be treated as **directional**.

## 1) Pain Point Categorization (Top recurring 5–7)

### 1. Technical Functionality — **Cloud access blocked / inconsistent entrypoints** (High severity, high frequency)
**Observed in ~38% (3/8) of items**
- Users cannot sign up / join due to: **“Tenant ‘elizacloud’ requires an invitation to join.”** (reported across multiple days).
- Users cannot access elizaCloud via **Milady**, and cannot change **OpenRouter API settings** through Milady.
- Onboarding “not working for multiple applications” (Babylon/Milady) reported as a connected symptom.

**Who it impacts most:** newcomers and returning users trying to use hosted features (Cloud, Milady), and anyone coming through invite links.

---

### 2. UX/UI — **Windows dev UI renders blank/white screen despite backend OK** (High severity, medium frequency)
**Observed in ~13% (1/8) items**, but called out as “multiple users” in the Discord summary (suggesting broader impact than the raw excerpt).
- Command: `bun run dev:ui` on **Windows**
- Agent starts, skills sync, but browser shows **blank screen** across browsers (Chrome/Firefox), cache clear/hard refresh does not help.

**Who it impacts most:** Windows developers evaluating elizaOS locally; likely blocks first impressions and onboarding.

---

### 3. Documentation — **Versioning/release workflow confusion (main vs develop)** (Medium severity, medium frequency)
**Observed in ~13% (1/8) items**
- Users stuck on **v1.7.2** after reinstall, unaware that **2.x lives on `develop`** and `main` lags.
- “Most developers work from develop” is effectively tribal knowledge.

**Who it impacts most:** new contributors, users who expect semantic releases on `main`, anyone following “stable” docs.

---

### 4. Community / Communication — **Perceived project silence / unclear roadmap (Milady, Babylon)** (Medium–high severity, medium frequency)
**Observed in ~25% (2/8) items**
- “Any Milady update? GitHub is silent”
- Babylon status answered: **on hold**, “revive social media component later,” but **no timeline**.
- Broader concern: “Has the team abandoned the project?”

**Who it impacts most:** builders depending on Milady/Babylon, partners, and prospective adopters assessing momentum.

---

### 5. Integration — **Cross-chain / bridging expectation mismatch (BNB → Solana)** (Medium severity, low frequency but high confusion)
**Observed in ~13% (1/8) items**
- User asked “best way to bridge elizaOS from BNB to Solana” and received **no guidance**.

**Who it impacts most:** web3-adjacent users treating elizaOS as a tokenized/networked asset rather than a software framework (important expectation signal).

---

### 6. Community — **Spam/off-topic content in technical channels** (Medium severity, medium frequency)
**Observed in ~13% (1/8) items**, but multiple spam/off-topic messages recorded (partnership asks, wallet solicitations).
- #coders had non-technical posts, including **crypto wallet purchase solicitation**.

---

### 7. Technical Functionality + Documentation — **World models implementation complexity / unclear “official path”** (Medium severity, medium frequency)
**Observed in ~13% (1/8) items**
- Users need guidance on “world models”; recommended workaround: **Google Vertex + Genie** now, with “new tooling in ~1 month.”
- The gap is not the idea, but the **operational recipe**, migration guide, and “what’s supported today.”

**Who it impacts most:** advanced users building embodied/simulation agents; power users who want reproducible pipelines.

## 2) Usage Pattern Analysis (actual vs intended)

### How users are actually using elizaOS
- **Cloud-first evaluation through apps** (Milady/Babylon) rather than starting from the core framework docs. Access failures here become “elizaOS is broken,” even if core is fine.
- **Local developer onboarding via `bun run dev:ui`** on Windows as an entrypoint (not just Mac/Linux).
- **Living on `develop`** as the de facto “current” channel (users treat it as release, not pre-release).
- **Agent capability expansion via orchestration** is emerging strongly in repo work (ack progress modes, sub-agent routing fixes), aligning with user expectation of multi-agent reliability.

### Emerging / unexpected use cases
- **Hardware/IoT agent reasoning**: proposal for a native `@elizaos/plugin-peaq` to reach **3M devices** on peaq network indicates community interest in “agents as embedded decision layers,” not just chat.
- **Web3 bridging mental model**: users ask about “bridging elizaOS” across chains, implying they associate the project with chain assets or deployments; this is an expectation management hotspot.

### Feature requests that align with observed usage
- **Cloud onboarding resiliency**: invitation flows, tenant joining, and app-based entrypoints need to be first-class.
- **Windows dev UX parity**: blank screen issues block the most common “try it now” workflow.
- **World model tooling + migration guides**: matches power-user demand and upcoming planned tooling.

## 3) Implementation Opportunities (solutions per major pain point)

### A) Cloud access blocked (tenant invitation errors; Milady can’t connect; OpenRouter settings blocked)
**Impact:** Very high | **Difficulty:** Medium (mix of product + auth/tenant plumbing)

1) **Ship a single canonical “Access State” diagnostic page** (High impact, low–medium difficulty)
- In Cloud frontend, show:
  - current tenant id / environment
  - whether invite token is present/valid
  - which endpoint returned the “invitation required” error
  - next action (“Request invite”, “Switch tenant”, “Accept invite”)
- Similar pattern: **Supabase** shows clear project/org context and permission gating; **Vercel** shows “You don’t have access to this team” with explicit remediation.

2) **Harden invite link compatibility + unify routes** (High impact, low difficulty)
- Ensure both `/accept-invitation` and `/invite/accept` work everywhere, and add tests to prevent regressions.
- Similar approach: many SaaS apps keep legacy route aliases indefinitely to avoid broken onboarding links.

3) **Add a “Milady ↔ Cloud connectivity checklist” with automated checks** (Medium impact, medium difficulty)
- A small CLI/script or in-app wizard:
  - validates API base URL
  - validates OpenRouter key presence/update permission
  - verifies tenant membership
- Similar pattern: **Firebase** and **Netlify** provide guided connection diagnostics.

---

### B) Windows blank screen on `bun run dev:ui`
**Impact:** High | **Difficulty:** Medium (debug + guardrails)

1) **Add a first-load “UI health” overlay when app fails to mount** (High impact, low difficulty)
- If React root fails or critical routes error, show a fallback page that prints:
  - API base URL, websocket URL
  - last startup phase
  - console error excerpt
- Similar: **Create React App** error overlay; **Next.js** dev error overlay.

2) **Publish a Windows-specific dev checklist and known issues section** (Medium impact, low difficulty)
- Include: Bun version, PATH casing, WS auth limitations, port conflicts, antivirus notes.
- Tie it to reproducible commands and “expected logs.”

3) **Add CI smoke coverage for Windows UI mount** (High impact, medium difficulty)
- A minimal Playwright test on Windows runner that asserts non-blank body + key selector present after `dev:ui` boot.

---

### C) Versioning confusion (stuck on v1.7.2; develop is “real” 2.x)
**Impact:** Medium | **Difficulty:** Low

1) **Make branch expectations explicit in install docs + CLI output** (High impact, low difficulty)
- On install/run, print: “You’re on main (stable) / develop (current).”
- Add `elizaos doctor` output that reports git branch + recommended branch for 2.x.

2) **Adopt a “release” branch or nightly channel** (Medium impact, medium difficulty)
- E.g., `main` = stable, `develop` = dev, `nightly` = auto-cut from develop with artifacts.
- Similar: **Homebrew** formulae, **Rust** channels, many OSS projects with nightly builds.

3) **Pin docs to versions** (Medium impact, medium difficulty)
- Docusaurus-style versioned docs: `v1`, `v2 (current)`, `develop/nightly`.

---

### D) Roadmap/status uncertainty (Milady silence; Babylon on hold)
**Impact:** Medium–high | **Difficulty:** Low–medium

1) **Create a public “Status & Roadmap” page with three tiers** (High impact, low difficulty)
- “Active”, “Maintenance”, “On hold”
- Add owners and “next checkpoint date” (even if tentative).

2) **Auto-post weekly digest to Discord from GitHub activity** (Medium impact, low difficulty)
- Summarize merged PRs affecting Milady/Babylon/Cloud.
- Similar: many OSS communities use GitHub Actions + Discord webhook “release notes” bots.

3) **Label repos/components with “supported entrypoints”** (Medium impact, medium difficulty)
- Make it explicit which apps are recommended (Cloud dashboard vs Milady vs Babylon).

---

### E) Spam/off-topic in technical channels
**Impact:** Medium | **Difficulty:** Low

1) **Tighten channel gating + add “Support template” bot** (High impact, low difficulty)
- Auto-prompt: OS, command run, logs, screenshot.
- Auto-detect wallet/scam keywords → quarantine.

2) **Create a dedicated “partnerships” / “bizdev” channel** (Medium impact, low difficulty)
- Redirects legitimate partnership queries away from #coders.

3) **Add a “unanswered support” triage rotation** (Medium impact, medium difficulty)

### F) World models: implementation complexity
**Impact:** Medium (for advanced segment) | **Difficulty:** Medium

## 4) Communication Gaps (expectations vs reality)

### Recurring mismatches
- **“Cloud signup should work” vs “invite-only tenant gating”**  
  Users interpret the tenant-invite error as a broken product, not an access policy.
- **“main is latest” vs “develop is latest”**  
  Users reinstall expecting updates, but remain on older stable releases.
- **“Milady/Babylon are active products” vs “Babylon on hold, Milady unclear”**  
  Lack of visible commits/updates drives abandonment narratives.
- **“elizaOS can be bridged cross-chain” vs “elizaOS is software, not a chain asset”**  
  The bridging question going unanswered reinforces confusion.

### Concrete doc/onboarding fixes
- Add a **Cloud Access FAQ**: invite policy, how to request access, how to accept invites, common errors.
- Add a **Branch/Release FAQ**: “Which branch should I use?” + “How to update from 1.x to 2.x?”
- Add a **Product status banner** inside Milady/Babylon entrypoints: “On hold / Deprecated / Experimental” + link to roadmap.

## 6) Feedback Collection Improvements

### Effectiveness of current channels
- **Discord is capturing issues**, but the period shows **low closure rate** (“no successful help interactions”), meaning feedback is collected but not resolved in-channel.
- **GitHub has deep technical tracking**, but users in Discord often don’t get routed to the relevant issue/PR (or don’t know it exists).

### Improvements for more structured, actionable feedback
1) **Discord → GitHub issue bridge for “Unanswered after 2h”**  
   Bot prompts reporter to file an issue with a prefilled template (OS, command, logs, screenshot).
2) **Pinned “Known Issues” per platform (Windows/Cloud)**  
   Update weekly; reduces repeat questions.
3) **Add a Cloud onboarding telemetry/error code dictionary**  
   Make tenant/invite failures map to stable codes that support can search.

### Underrepresented segments
- **Non-Discord users**: little visibility into feedback from independent adopters (npm, issues created outside Discord).
- **Windows developers**: surfaced sharply via blank screen; likely more incidents than reported.
- **Enterprise/self-host operators**: no explicit feedback in this slice, despite heavy infra work—consider targeted outreach.
