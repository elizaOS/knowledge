# ElizaOS Weekly Newsletter (2026-05-31 → 2026-06-06)

## 1) Executive Summary

This week was all about **shipping Cloud V2 maturity**, **tightening reliability across the agent runtime**, and **making the orchestrator experience feel more “real product” than “dev tool.”** Three highlights:

1. **Cloud V2 reached a major sync milestone (develop → main promotion).** PR **#8116** promoted `develop` to `main`, triggering production deploy workflows across the backend, frontend, provisioning worker, and agent images—bringing recent Cloud V2 stack changes fully into prod.

2. **Agents got more autonomous and more predictable.** We saw meaningful progress in how agents plan and execute tasks: improved routing (coding vs. web), keyless web fetch capability for non-Anthropic models, and multiple fixes to prevent user-facing “tool envelope” leaks.

3. **The orchestrator/workbench UX continued its rapid climb.** The Claude/Codex-style workbench work (streaming + diffs + stop) is in, and the next-generation “Odysseus-style” UI is actively being built and reviewed—setting the stage for a more premium multi-agent operator experience.

## 2) Development Updates (Core + Cloud + Desktop)

### Cloud & Infrastructure
- **Production sync + deploy automation:**  
  - **#8116** (merged) promoted `develop → main` to align production with Cloud V2 changes and auto-triggered production deploy workflows.
- **Native agent sandbox stability:**  
  - **#8104** (merged) fixed Cloud agent sandbox startup/routing issues and added broader test coverage around local routing and autoscaling defaults.
- **Staging auth & magic-link fixes (Steward tenant isolation):**  
  - **#8176** (merged) isolated the Steward tenant so staging magic links don’t redirect to production.  
  - Follow-on hardening: **#8179**, **#8194**, **#8196**, **#8197** (merged) addressed tenant pinning, callback flow gaps, CORS, and CSP `connect-src` for staging domains.
- **Reasoning-model billing correctness:**  
  - **#8189** (merged) ensured a minimum response-token budget for reasoning models to prevent “billed but empty” completions when `max_tokens` is too small.

### Agent Runtime & Orchestration
- **Keyless live web retrieval across models:**  
  - **#8097** (merged) introduced a **keyless `WEB_FETCH` action** so non-Anthropic models can still answer live-info prompts inline.
- **Orchestrator workbench improvements:**  
  - **#8095** (merged) delivered a **Claude-Code-style coding workbench** experience (streaming, diffs, stop controls, SSE).
- **User-facing transcript cleanliness:**  
  - **#8159** (merged) suppressed leaked `TASKS` spawn-arg JSON envelopes that some weaker planners would accidentally paste into replies.
  - **#8160** (merged) fixed routing so coding requests that mention “market/live” terms go to **coding** rather than being misclassified as web searches.
- **Error reporting & reliability fixes:**  
  - **#8098** (merged) preserved JSON-RPC `error.data` details on ACP failures—critical for debugging schema/param issues.
  - **#8161** (merged) closed a completion de-dup TOCTOU race that could cause duplicate replies.

### Desktop & Local Inference
- **Faster startup and better “first minute” experience:**  
  - **#8148** (merged) delivered **tray-first + decoupled fast startup** (opt-in).  
  - **#8151** (merged) added background warmup for embedding and post-ready warmup for voice models.
- **Local inference robustness:**  
  - **#8191** (merged) improved error signaling when llama.cpp handlers are unavailable (typed unavailability instead of generic errors).
- **Vault resilience:**  
  - **#8092** (merged) added self-healing for stale PGlite locks so unclean shutdowns don’t brick startup.

## 3) Community Spotlight (Discord Highlights)

Even with a quieter technical chat volume, a few threads stood out for ecosystem direction and security posture:

- **Security & plugin craftsmanship (marsel_51932):**  
  Marsel introduced themselves as an AI Engineer/Security Researcher offering **custom production-ready plugins**, **multi-agent setups**, and **agentic/on-chain security auditing**—with a strong emphasis on **execution boundary audits** to prevent “agent draining” and malicious transaction signing, plus **prompt injection defenses**. This is exactly the kind of expertise the ecosystem benefits from as agents become more financially capable.

- **DeFi integration momentum: Kamino plugin proposal:**  
  In discussion on June 2, **n4vnt** proposed building a comprehensive **Kamino (Solana) plugin** covering lending/borrowing, vaults, yield, and limit orders—closing a major gap for agents that need to *do* DeFi, not just *describe* it. **stan0473** clarified process: **all plugins now live in `elizaos/eliza`**, and contributors should fork and target the `develop` branch (with frequent rebasing due to fast-moving core changes).

- **Ecosystem opportunities & moderation:**  
  - **Intuition Box (saulodigital)** flagged upcoming bounties and asked how to share them with the ElizaOS dev community.  
  - **om1d_sa** issued a scammer alert in `#coders`, acknowledged by **odilitime**—a reminder to keep security hygiene high as the community grows.  
  - In partners, **DEMIAN (DAPPCRAFT)** reached out about a potential collaboration with an AI DePIN project (noted as ~0.5B FDV) and requested a call with Shaw.

- **Org clarity & new ideas:**  
  June 3 discussion clarified that **Hyperia is separate from ElizaOS**, and **Hyperforge** is Hyperia’s world-building component. Also spotted: **Neverbell**, an early-stage concept exploring agents interacting with financial systems through user-defined controls/permissions.
