# ElizaOS Weekly Newsletter (2026-06-07 → 2026-06-13)

## 1) Executive Summary

This week was all about **hardening Eliza Cloud Apps infrastructure** while the community explored **real-world scale integrations** and **cost/performance optimization**:

1. **Eliza Cloud Apps ingress + control-plane bootstrapping moved forward.** Staging app-node ingress was enabled by baking a Caddy “front door” into cloud-init (**PR #8394**), and production migrations got smoother with automated Headscale installation in the control-plane cloud-init (**PR #8393**).
2. **Apps data-plane go-live is unblocked step-by-step, with automation replacing manual SSH.** Work is in progress on workflows and scripts to clear orphaned Terraform state and “arm” the apps daemon without SSH (PRs **#8400**, **#8398**, **#8397** in progress), directly addressing the operational friction highlighted in the Apps scaling/ingress gap audit (**Issue #8321**).
3. **Community proposed a major device ecosystem expansion path.** A Discord proposal suggested building a native **`@elizaos/plugin-peaq`** to integrate elizaOS agents with **peaqOS** and its reported **3M+ device network**, positioning elizaOS for robotics/IoT “machine economy” workloads.

## 2) Development Updates (Technical)

### Eliza Cloud Apps: infrastructure stabilization
- **Staging ingress enabled (Caddy front door)**: The staging app node now has ingress support via Caddy configured during app-node cloud-init (**PR #8394**). This is a key prerequisite for making `<app>.apps*.elizacloud.ai` routes actually reachable.
- **Control-plane migration automation**: Headscale installation was automated in the control-plane cloud-init (**PR #8393**), reducing manual steps during control-plane provisioning and migrations.

### Current blocker: apps-data-plane Terraform state mismatch
- The **apps-data-plane go-live remains blocked** by a **state-provider mismatch** involving an orphaned `random_password` resource (**Issue #8321** referenced in internal updates).
- To reduce manual intervention and make recovery repeatable, the team has several “ops automation” PRs in flight:
  - **PR #8400 (in progress):** One-shot workflow to remove orphaned apps-data-plane state entries
  - **PR #8398 (in progress):** Dispatchable workflow to arm the apps daemon (no manual SSH)
  - **PR #8397 (in progress):** Idempotent script for arming the apps-daemon

### Dependency + plugin improvements
- **esbuild bump**: 0.28.0 → 0.28.1 (**PR #8399**, in progress)
- **pypdf bump**: 6.10.2 → 6.12.0 (**PR #8396**, in progress)
- **EvoLink support added to `plugin-openai`** (**PR #8395**, in progress), expanding compatibility for downstream integrations that depend on EvoLink’s interface expectations.

### ElizaCloud sign-up friction surfaced (and escalated)
A user reported being blocked from ElizaCloud sign-up with:
> “Tenant `elizacloud` requires an invitation to join.”

**odilitime** confirmed it’s **already escalated to the team** and under investigation, pending confirmation of the fix. This is exactly the kind of “paper cut” that disproportionately impacts community builders—thanks to everyone who reports these quickly and with exact error text.

### “Spend less, go faster”: sub-millisecond token pipeline idea
**carlgatz** shared a performance-oriented architecture: a **local gRPC pipeline** that **filters and preprocesses high-volume data in RAM**, only calling into deeper MCP/LLM analysis when necessary. The headline claims:
- **Sub-millisecond latency**
- Reduced AI token spend during spikes by doing “cheap” steps locally first

They’re looking for **~5 developers to stress test** and offered to share details. If you’re building high-throughput agent systems (feeds, trading signals, telemetry, robotics), this discussion is worth revisiting.
