# ElizaOS Weekly Newsletter (May 17–23, 2026)

## 1) Executive Summary

This week was a strong “shipping + real-world validation” stretch for ElizaOS:

- **ElizaOS v2.0.2 landed (develop → main):** The release consolidates a large batch of work across core runtime, Cloud, desktop/mobile packaging, and UI stability—setting a cleaner baseline for builders and plugin authors.  
- **OWNER + AGENT identity became real across connectors:** The new dual-identity model moved beyond an idea into concrete registry + UI + OAuth plumbing, making it practical to distinguish *your account* (OWNER) from *your agent’s account* (AGENT) on platforms like X and more.  
- **Production-grade ecosystem proof: waifu.fun launched on v3 agents:** A community-led launchpad built with **elizaOS v3 agents** went live, fully open source—demonstrating that agentic products can ship end-to-end on the stack, not just as demos.

## 2) Development Updates (Core, Cloud, OS, Plugins)

### Core platform: identity and operational reliability
- **v2.0.2 release (big integration merge):** The develop branch was promoted to main as **v2.0.2**, bundling substantial runtime + Cloud + UI fixes and packaging hardening. This is the “everyone should update” moment for teams who want fewer edge-case regressions across platforms.  
  - PR: **chore(release): develop → main (v2.0.2)** (#7828)

- **OWNER + AGENT model rollout:** The “two identities, one connector” approach took major steps forward:
  - Registry schema extended so connectors can declare separate OWNER/AGENT account types (#7817).
  - UI updated to be role-aware and to present a dual setup experience (#7819).
  - OAuth callback logic for several connectors was corrected to avoid hardcoding OWNER during completion flows (#7820).
  - PRs: #7817, #7819, #7820

### Cloud: agent lifecycle moves to a job queue
- **Agent lifecycle operations decoupled via a queue:** Actions like restart/suspend/logs/snapshot were migrated away from synchronous worker-to-core calls into a job-queue model. This improves responsiveness and reduces bottlenecks (especially under load or partial outages).  
  - PR: **feat(cloud): all agent lifecycle ops via job queue** (#7810)

### OS & hardware: cross-architecture momentum
- **RISC-V milestone (community validation):** The community successfully booted an ElizaOS Debian RISC‑V 64-bit ISO under QEMU in ~35 seconds, and confirmed **Eliza v3 running on elizaOS (Linux) on an emulated “eliza e1” RISC‑V chip**. This is an important proof point for hardware abstraction and portability.  
- **elizaOS Live USB hardening:** Continued work on manifests, update architecture, health checks, and packaging of runtime plugins improves reproducibility for “bootable ElizaOS” demos and field testing.  
  - Notable PRs referenced in the weekly summary include #7762 and related OS hardening work.

### Training & local inference: lowering the barrier
- **Low-VRAM training smoke preset:** A new `--low-vram-smoke` preset enables end-to-end SFT validation on **12GB consumer GPUs** (RTX 3060/4070-class), lowering the entry cost for experimentation.  
  - PR: **feat(training): low-VRAM smoke config** (#7805)

### New ecosystem plugin: DeFi risk + solvency tooling
- **Centinel DeFi plugin published:** A new Eliza plugin for DeFi agents on **Base/Aave** shipped to NPM with:
  - real-time solvency scoring
  - flash-crash simulations  
  Package: `@centinel-risk-protocol/eliza-plugin-centinel` (community is actively seeking testers/feedback)

Shaw announced the launch of **waifu.fun**, an **open source agentic launchpad** powered by **elizaOS v3 agents**. The platform supports *tokenized agents* running on Eliza Cloud, with app revenue that can be shared with holders. The first agent, **Sol (The Architect)**, reportedly built waifu.fun itself—an “agent building the product” narrative that’s very on-brand for where ElizaOS is headed.

- Code: https://github.com/waifufun/waifu.fun  
- Sol: https://github.com/0xSolace

### Practical deployment discussion: Dockerizing real-time VoiceAI
A useful technical thread covered how to containerize a complex real-time **VoiceAI white-label** platform. The core guidance (from keil0780) was pragmatic: **avoid a monolith container early** and split into distinct services (inference, API, DB, queue/worker), because GPU access, networking, persistent storage, and latency stability become unmanageable otherwise. The original poster reported progress to “two working containers,” which is a great example of community help translating into measurable progress.

No formal on-chain parameter changes were posted in the provided updates this week, but several *ecosystem-relevant* token notes surfaced:
