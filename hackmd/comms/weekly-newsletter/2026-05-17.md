## 2) Development Updates

While the Discord itself was relatively quiet, a few technically meaningful threads emerged—mostly around reliability and scaling patterns that show up once agents leave the demo phase.

### Model + Planning Improvements
- **Qwen-based stack + optimizations** remain in active use.
- A **fine-tuning pipeline** is being built to improve:
  - **Action calling correctness** (selecting the right tool/action with the right parameters)
  - **Planning** (multi-step sequencing and task decomposition)

Why it matters: the difference between “agent as chat” and “agent as operator” is usually action selection and follow-through. Improving those layers has outsized impact across *every* connector and workflow engine.

### API Usage & Rate Limiting (Neynar)
A practical issue was raised: **Neynar API credit usage** on `v2/farcaster/notifications` was far higher than expected (**~300 credits/hour vs ~4**), even after overriding a default polling interval. The main community recommendation:

- **Switch to webhooks** (push) instead of polling (pull), where possible.

Takeaway: as more agents integrate social and messaging platforms, *event-driven architectures* (webhooks/streams) will often be the difference between sustainable ops and runaway costs.

### Agent-to-Agent Escrow (Base Mainnet)
A new escrow protocol concept was shared:
- **Base Mainnet deployment**
- **Pay or Burn dispute resolution**: if agents can’t agree on an outcome, escrowed funds are burned (game-theoretic deterrent)
- **Pull-payment design** to avoid receiver revert issues
- **“Zero UI”**: contracts + SDKs (Python/C++) only, seeking architectural feedback

Even if you’re not building on-chain: this is a useful design reference for trust-minimized agent collaboration and automated settlement.
