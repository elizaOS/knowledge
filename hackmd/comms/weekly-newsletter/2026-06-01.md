# ElizaOS Weekly Newsletter (2026-05-25 → 2026-05-31)

## 1) Executive Summary

This week was all about **hardening the platform while expanding what builders can ship on top of ElizaOS**:

1. **Security & release reliability took a big step forward.** The core repo continued tightening SOC2-oriented controls and added additional protections around destructive actions and MCP/web behaviors—aimed at making “agent ops” safer in production environments.
2. **Cloud + deployment capabilities accelerated.** Work landed (and continues) on containerized agent provisioning, spend tracking, webhooks, and a blue/green-style fleet upgrade reconciler—laying groundwork for smoother upgrades and less downtime.
3. **Ecosystem modularity kept expanding.** The community pushed new integrations (e.g., a Suede plugin) and clarified major repo structure changes—most notably the **Solana plugin relocation into the main repo** as `plugin-wallet` under `chains/solana`.

## 2) Development Updates (Core + Infra + Plugins)

### Core stability, safety, and “don’t surprise the operator”
A consistent theme in GitHub activity was **reducing footguns** and improving runtime correctness:

- **Security hardening across sensitive operations:** `odilitime` merged a large security-focused change set (**PR #8076**) to better secure destructive actions and tighten MCP spawning/web functionality. This is the kind of work that tends to be invisible when it’s done right—but it’s critical for teams running agents with real permissions.
- **CI/release contract & reliability fixes:** `NubsCarson` merged changes focused on structural logic and release/CI contracts (**PRs #8071, #8072**), while multiple contributors continued to chip away at “works on my machine” drift.
- **Global install smoke tests:** `cyphercodes` implemented smoke testing for global package installs (**PR #8075**), directly targeting a common pain point for CLI users and first-time contributors.

### Cloud & deployment: toward safer upgrades and cleaner provisioning
Several parallel efforts point toward a more mature operational story:

- **Containerized agent provisioning + spend tracking + webhooks:** `0xSolace` drove a cluster of cloud-facing improvements—containerized provisioning (**PR #8073**) plus wiring for webhook and spend-tracking capabilities (**PRs #8094, #8089**). Net effect: more predictable lifecycle management and better accounting/observability.
- **Blue/green style upgrades:** `standujar` opened a major feature PR for a **fleet-upgrade reconciler** supporting blue/green deployment patterns (**PR #8049**). While still in-flight, it signals a move toward upgrade strategies that minimize downtime and reduce rollout risk.
- **Routing & sandbox fixes (in progress):** `SYMBaiEX` opened **PR #8104** to address native agent sandbox routing—large surface-area work that’s likely to improve isolation and correctness once fully integrated.

### Developer experience and platform surfaces
A lot of effort went into polishing edge cases across UI, installers, and workspace connectors:

- **Out-of-box whitelabeling merged:** `Dexploarer` merged **PR #8074** to implement whitelabeling support—useful for teams embedding ElizaOS into their own branded experiences.
- **UI testing + connector error handling (in progress):** `lalalune` opened several PRs including UI testing and workspace connector error-handling improvements (**PRs #8088, #8087, #8084, #8080**).
- **Registry helper for logic nodes:** `cmdenney` opened **PR #8022** introducing `requireLogicNodes()`—a small but meaningful improvement for correctness in agent/logic registration.

### Plugin ecosystem: Solana restructure + new integrations
From Discord, the biggest “practical” update for builders was a repo layout change:

- **Solana plugin moved into core repo:** `plugin-solana` has been relocated and integrated into `elizaos/eliza` as **`plugin-wallet` under `chains/solana`**. If you had scripts, docs, or imports pointing at the old location, you’ll want to update them.
- **Kamino integration momentum:** community members discussed ongoing work toward a **`plugin-kamino`** integration, including the availability of prior reference code with **read functionality for v1.x** to accelerate development.
- **New plugin proposal:** `JasonColapietro` opened **PR #8016** to introduce `@suedeai/plugin-suede`, expanding third-party integrations.
