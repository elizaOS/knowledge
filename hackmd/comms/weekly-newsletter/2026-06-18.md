# ElizaOS Weekly Newsletter (2026-06-12 → 2026-06-18)

## 1) Executive Summary

**A big week for Cloud Apps (Product 2) getting “real internet reachable.”** The team shipped the missing pieces that turn isolated container hosting into something users can actually hit from a browser: an apps ingress front door and tooling to manage it safely and repeatably. The headline: **per-app routing via Caddy + deploy wiring** landed, unblocking `<app>.apps.elizacloud.ai` style endpoints and reducing the amount of “operator SSH” required to stand things up.

**Staging reliability got a serious boost (especially around auth + infra automation).** Multiple fixes tightened up staging/prod isolation (routing, CORS/CSP, token signing, timeouts), while workflows moved closer to “IaC-first.” This reduces footguns like staging silently calling production services.

**Community ops made progress on repo organization and moderation.** On Discord, **all dev work being consolidated into the `eliza` repo** was confirmed, and moderators added capacity to respond to an uptick in spam/scam posts—especially those impersonating token migration support.

---

## 2) Development Updates (Technical)

### Eliza Cloud Apps (Product 2): ingress + deploy activation
If you’ve been following the Apps data-plane effort, this week delivered several of the “P0 blockers” identified in the scaling/ingress audit thread.

Key improvements:
- **Apps ingress front door shipped**: Caddy-based routing now reverse-proxies per-app traffic to the correct container, finally making apps reachable by hostname.  
  - PR: **#8325** “apps ingress — per-app Caddy routing client + deploy-flow wiring”
  - Follow-on hardening: **#8328** “collision-safe host-port allocation”
- **Cloud-init now installs Caddy by default on app nodes**, removing the last “TODO” from the infra template and making new nodes come up ready to serve.  
  - PR: **#8394** “install Caddy front door in app-node cloud-init”
- **Staging deploy trigger turned on**, so staging can run the real deploy path end-to-end (with guardrails to avoid impacting Product 1).  
  - PR: **#8406** “enable APPS_DEPLOY_ENABLED on staging”
- **“Arming” the apps daemon got automated** with idempotent scripts/workflows (no more hand-editing `/opt/eliza/cloud/.env.local`).  
  - PRs: **#8397**, **#8398**, plus permission and safety follow-ups **#8404**, **#8405**, **#8422**
- **Build-from-repo (Vercel-like) path armed on the daemon**, enabling platform-side builds instead of requiring users to pre-push images.  
  - PR: **#8407**

### Tenant DB + cost controls
- **Per-tenant DB mode became opt-in** (e.g., `none|isolated`) to avoid provisioning Postgres for apps that don’t need it.  
  - PR: **#8338**
- **DB teardown and slot release were implemented**, addressing real leak risks when apps are deleted.  
  - PRs: **#8343**, **#8401**

### Cloud auth and platform correctness
Several fixes tightened Steward-based auth flows and staging isolation:
- PKCE/signing hardening across login surfaces (and downstream routes) continued through the month (e.g., **#8259**, **#8350**), and staging-specific corrections kept landing (CORS/CSP/tenant pinning).  
- Staging routing was repeatedly corrected to avoid “staging calls prod” failure modes (e.g., staging Worker hitting production control plane or auth endpoints).

### CI/CD signal-to-noise improvements
- **Expensive test lanes are now path-gated**, reducing PR cycle time when changes don’t touch high-cost surfaces.  
  - PR: **#8408** (and ongoing refinement in **#8414**)

## 3) Community Spotlight (Discord)

### Open source strategy debate: “fully OSS vs hybrid monetization”
A thoughtful thread surfaced concerns about competitiveness in 2026’s AI landscape. One community member argued for a **hybrid model** (monetize newest features, open-source older versions), while others defended the current approach: open source as an acceleration engine (contributors, bug fixes, ecosystem gravity), with commercialization happening *around* the core.

Notable detail raised: a **commercialization partnership between Secure Blockchain Development Corp. and the Eliza Foundation** to bring enterprise-grade agents to market on top of the open-source framework. Community members asked for clearer public documentation on what that partnership includes and how it fits the roadmap.

- Discussion link: https://discord.com/channels/1253563208833433701/1253563209462448241

### Repo consolidation confirmed
Odilitime confirmed that **all development work has been consolidated into the `eliza` repository**, which should simplify contributor onboarding and reduce “where does this live?” confusion.

### Moderation: scam/spam surge and response
Multiple scam patterns hit #discussion and #coders (fake airdrops, wallet-purchase solicitations, and migration impersonation). Moderators responded quickly—**minimod powers were granted** to increase response capacity now that the old minimod channel no longer exists.

## 4) Token Economics (AI16Z, ELIZAOS, auto.fun)

### Token migration status: window closed, scams active
Discord saw renewed questions from users who **missed the AI16Z → ELIZAOS migration deadline** and were looking for a late path (manual appeal, alternative contract pool, etc.). At the same time, the community also identified **active scam attempts** circulating “migration links.”

What’s clear from community responses:
- **The official migration window is closed**, and **any new “migration” links are highly likely to be scams**.
- A user question about a legitimate late-migration process was **not answered definitively** in the thread—this is a good candidate for an official FAQ update.

### auto.fun
No concrete engineering notes or changelog items were surfaced in the provided data this week. If there are auto.fun releases in flight, the community would benefit from a short “state of” post (even if it’s just: what’s shipping next, and what’s being stabilized).
