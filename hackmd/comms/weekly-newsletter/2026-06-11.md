# ElizaOS Weekly Newsletter (2026-06-05 → 2026-06-11)

## 1) Executive Summary

This week was a strong “platform hardening + ecosystem expansion” sprint across Eliza Cloud and the plugin ecosystem:

1. **Eliza Cloud Apps (Product 2) accelerated toward usability at scale.** The community audit and subsequent fixes made the path to “apps that are actually reachable” much clearer—especially around ingress, DB isolation contracts, and operational guardrails.
2. **LifeOps graduated from a monolith into a modular plugin suite.** The large refactor that split `plugin-lifeops` into focused plugins (plus `plugin-personal-assistant`) landed, setting the codebase up for cleaner ownership, better composability, and faster iteration.
3. **Third-party ecosystem momentum continued via registry submissions.** A new security-oriented signer plugin was proposed for the third-party registry, and the Discord community discussed the correct submission workflow and review expectations.

## 2) Development Updates (Technical)

### Cloud & Infrastructure

- **Apps data-plane: key blockers documented and prioritized.** Issue **#8321** (“scaling + ingress gaps before go-live”) is an unusually detailed audit that calls out specific P0/P1/P2 items—most notably:
  - *P0 ingress not installed* (apps unreachable without a front door),
  - *Postgres connection ceiling / lack of pooling*,
  - *env var contract mismatches* that could silently disable tenant isolation.
  If you’re tracking Eliza Cloud Apps readiness, this issue is the canonical checklist.  
- **Steward authentication remained a hotspot.** An open issue, **#8257**, reports production login failures (OAuth PKCE mismatch + email send 400) that can block “Sign in with Eliza Cloud.” Several related fixes landed earlier in the month (PKCE support, signing, and redirect handling), and the remaining work appears to be aligning prod Steward behavior with the Cloud frontend’s assumptions.

### Core platform & Orchestrator UX

- **Orchestrator UI is evolving toward a richer “operator workbench.”** There’s continued momentum behind an Odysseus-inspired orchestrator experience (notably PR **#8180**, still unmerged). Even without merging, the work signals where the UX is heading: better session navigation, richer panels (memory/skills/notes/settings), and a more “multi-tool” operator posture.
- **Reliability improvements continued around cloud routing and model/provider behavior.** Across the month’s merged work, there were multiple fixes aimed at avoiding silent failures and surprise billing behaviors (e.g., ensuring reasoning models have enough response-token budget, and addressing pricing/catalog mismatches).

### Plugin architecture & modularity

- **Major refactor: `plugin-lifeops` split into focused plugins** (merged PR **#8277**). This is a structural change with real downstream implications:
  - `plugin-lifeops` is gone (no backward-compat shim).
  - Functionality moved into domain plugins like inbox/goals/finances/relationships/calendar/todos/health/documents/phone, while `plugin-personal-assistant` retains cross-domain orchestration (briefing, prioritization, scheduled tasks).
  - Expect follow-up work to migrate remaining implementations into the new “future homes” scaffolded in `@elizaos/core` and `@elizaos/app-core`.
