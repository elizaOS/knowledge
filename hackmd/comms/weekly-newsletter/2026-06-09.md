# ElizaOS Weekly Newsletter (2026-06-09)  
**Week:** 2026-06-02 → 2026-06-08

## 1) Executive Summary

**1) Eliza Cloud Apps took a big step toward “real hosting.”**  
The team closed multiple P0 blockers in the Apps data plane: apps can now be reverse-proxied via a per-app Caddy routing client, tenant database isolation was tightened (DSN injection), and host-port allocation was made collision-safe for denser deployments. Net: the core deploy path is starting to look production-shaped—while still surfacing a few “must-fix-before-go-live” risks (see below).

**2) LifeOps is now modular: the monolith is officially split.**  
`plugin-lifeops` was decomposed into a slimmer orchestrator plugin (`plugin-personal-assistant`) plus focused domain plugins (inbox, goals, finances, relationships, etc.). This is a meaningful milestone for maintainability and for teams who want to adopt only the slices they need.

**3) Community conversations leaned “builders + safety.”**  
Discord saw practical discussions on building AI companions (Discord + voice + images), plus a strong thread on fraud detection for crypto trading (wallet-funding lineage analysis) and how to benchmark trading agents without overfitting to luck.

## 2) Development Updates

### Eliza Cloud Apps (data plane) stabilization
Work this week focused on getting the Apps hosting plane reachable, safer under load, and closer to production readiness:

- **Ingress/routing unblocked:** a new **per-app Caddy routing client** enables `<app-id>.apps.elizacloud.ai` style reverse-proxying so deployed apps are actually reachable.  
- **Safer high-density deployments:** **collision-safe host-port allocation** reduces flakiness when many containers are scheduled to the same node.  
- **Stronger tenant isolation:** the per-tenant DB DSN is now injected in a more compatible way (notably ensuring apps see the DB connection via expected env vars), reducing the risk of “silent fallback” to non-isolated storage.  
- **Infra/CI stability improvements:** Terraform apply errors were resolved and staging stability improved; runtime image dependency issues were addressed by hoisting missing modules (notably wallet-related deps) into the runtime image build path.  
- **Pricing and gateway fixes:** BitRouter/AI pricing issues were mitigated, including **HTTP 402 / 5xx** classes of failures by moving toward **BYOK API-key mode** and forcing pricing coverage for OpenAI text-embedding models.

### Known risks (still open)
Two themes emerged as “don’t ship yet” items:

- **Billing leaks identified (critical):** four distinct billing leaks in Apps hosting + per-tenant DB need investigation before go-live. If you’re running staging deployments, assume billing correctness is still under active validation.  
- **Production deployment delay recommended:** some infra-level provisioning remains unresolved (cloud-init for Caddy, pgbouncer, lifecycle management). In plain terms: the platform can deploy apps, but the “operators’ checklist” isn’t fully satisfied yet.

### Plugin architecture and modularity
- The **LifeOps refactor** is a structural win: domain plugins now own their own views/actions, while `plugin-personal-assistant` keeps cross-domain orchestration. Expect some downstream adjustments if you had hard dependencies on the old `plugin-lifeops` shape.

## 3) Community Spotlight (Discord)

### AI companion builders: Discord + voice + images
A helpful thread confirmed ElizaOS is a strong fit for companionship/emotional-support style agents (not just “business automation”). Community members discussed integrating:
- **Discord** as the primary interface  
- **Voice** via plugins (e.g., pairing with ElevenLabs for TTS)  
- **Image generation** (and even video, as demonstrated via agents in Cozy’s arena when models are available)

If you’re building a companion, this is a good week’s takeaway: ElizaOS supports the full “multimodal companion” stack, but you’ll want to plan your model/runtime packaging early (voice and image dependencies can be the sharp edges).

### Safer trading agents: benchmarking + cabal detection
Two standout contributions:

- **Trading agent benchmarking methodology:** a testing-harness approach with a single `decide()` function and consistent fills, forward-tested on hidden periods, then re-tested on additional hidden periods to separate skill from luck. A key empirical result: **simple risk-off strategies beat complex leveraged momentum** during real down days.  
- **Cabal-Hunter (fraud detection):** a tool that traces **wallet funding lineage** (not just contract code) to detect coordinated “cabal” wallets—catching rugs that pass normal checkers. It’s presented as a pay-per-query MCP server ($0.05 USDC/query) and can be integrated into ElizaOS flows (e.g., call `check_cabal_risk(mintAddress)` before swaps).
