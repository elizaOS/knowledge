# ElizaOS Weekly Newsletter (2026-06-08 → 2026-06-14)

## 1) Executive Summary

This week was a mix of **shipping real engineering improvements** and **surfacing important user-facing pain points**:

1. **Faster CI with smarter test gating:** The team merged **path-based gating for expensive test lanes** (PR **#8408**), cutting down unnecessary CI load while keeping scheduled/manual runs full-strength. This should speed up iteration for contributors and reduce “CI queue tax.”
2. **Agent orchestration reliability work accelerated:** Several targeted changes landed or were opened around the **agent orchestrator**—including spawn acknowledgment deduplication, better sub-agent routing, and guardrails to keep noisy internal URLs/route-prefixes out of user deliverables.
3. **Cloud access issues became the top community blocker:** Multiple community members reported being unable to join **elizaCloud** due to a “tenant requires an invitation” error, plus related onboarding and settings issues affecting Babylon/Milady entry points.

## 2) Development Updates (Technical)

### CI/CD & Infra
- **CI: expensive lane gating by changed paths (merged)**  
  PR **#8408** introduced path classifiers for the main test workflows, skipping heavy server/client/plugin/desktop/cloud lanes when unaffected. This is a quality-of-life improvement for everyone opening PRs—especially smaller fixes that used to pay the full CI cost anyway.
- **CI follow-up: label-gated “expensive lanes” (opened)**  
  A follow-up PR **#8414** was opened to make expensive lanes **label-gated and more explainable**, which should help maintainers and contributors understand *why* something ran (or didn’t), and how to force the full matrix when needed.

### Orchestrator & Core Runtime
Work this week focused on making multi-agent execution **more predictable and user-aligned**:
- **Acknowledgment progress mode + spawn-ack deduplication**  
  This aims to reduce duplicate or confusing “spawn acknowledged” behaviors during sub-agent orchestration.
- **Sub-agent routing fix: route by originating user request**  
  A key behavioral fix: app-build (and similar) spawns should be routed based on the **user’s original request**, not the planner’s terse internal task summary—improving correctness and reducing “why did it do that?” moments.
- **Cleaner deliverables: suppress route-prefix / data-source URLs**  
  Additional protections were opened to prevent internal route-prefixes or datasource URLs from leaking into sub-agent deliverables.
- **Provider/core correctness fixes**  
  Opened work also includes bounding relationships provider output and preserving model-chosen coding delegations (instead of downgrading them into generic direct replies).

### DevEx / Tooling Notes (from community + workflow)
- **Versioning reality check:** The Discord discussion reinforced that **`develop` is where 2.x lives**, while `main` may lag. If you’re “stuck on 1.7.2,” the practical fix shared was to track `develop` (with the usual expectation of more churn than stable releases).
