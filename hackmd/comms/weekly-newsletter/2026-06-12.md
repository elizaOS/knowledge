# ElizaOS Weekly Newsletter (2026-06-06 → 2026-06-12)

## 1) Executive Summary

This week was all about **making Eliza Cloud more “real-world durable”**—both in the core runtime and in the surrounding ecosystem.

- **Autonomy + monetization loops took a big step forward.** The agent orchestrator now supports **autonomous sub-agent dispatching via the `USE_SKILL` broker**, designed to enable “monetized-app loops” where agents can execute scoped economic workflows more independently (with guardrails).
- **Cloud infrastructure reliability continued to harden.** Improvements landed around **Terraform-managed networking**, **control-plane DNS decoupling**, and multiple fixes across provisioning and environment-variable passthrough that collectively reduce “it works on staging but not prod” surprises.
- **The plugin ecosystem expanded with security-focused integrations.** Notably, a third-party registry PR added **`@usenami/plugin-signer`**, a keyless CEX/DEX signer using **AWS Nitro Enclaves** for safer API key handling—plus additional new registry entries surfaced in project summaries.

## 2) Development Updates (Tech Highlights)

### Orchestrator + Agent Runtime
- **Autonomous sub-agent dispatching (`USE_SKILL` broker).** This unlocks a cleaner path for sub-agents to trigger higher-level skills (especially relevant for apps that incorporate billing/credits and automated workflows).
- **Session timeout and handshake robustness.** A fix shipped to ensure the orchestrator correctly honors the configured session timeout during ACP initialization (important for long-running or slow-start sub-agents).
- **Windows environment forwarding improvements.** Path-casing quirks on Windows can break sub-agent execution; this week included a fix to ensure environment variables are forwarded reliably in Windows setups.

### Cloud Infrastructure + Provisioning
- **Terraform: data-plane private networks now managed as code.** Instead of hand-managed infra, the team is moving foundational networking into reproducible Terraform workflows, reducing drift.
- **Control-plane DNS decoupled from VM creation.** This lowers the risk of DNS flipping to a not-yet-ready machine during apply—one of those changes you only appreciate after a painful incident.
- **Environment variable passthrough fixes.** Several fixes focused on ensuring “frontend → worker → agent” configurations actually propagate as intended (and don’t silently fall back to defaults).

### Billing, Attribution, and Cost Controls
- **Affiliate revenue-share tracking fixed for embeddings + voice inference.** Attribution now applies beyond chat completions, aligning incentive mechanics across more of the API surface (embeddings/STT/TTS are often the cost center in production apps).
- **CLI stability: global install crash fixes.** The project summary notes improvements to prevent global installation crashes for `elizaos` and the CLI by fixing import/shim issues and removing incompatible dependencies.

### Plugin Ecosystem
- **Third-party registry growth continues.** The big community-facing addition discussed this week is PR **#8294** (details below), and project summaries also mention new registrations such as *neynar-search* and *coinrailz*.

## 3) Community Spotlight (Discord)

### Performance Optimization: gRPC pipeline for token-cost control
In Discord, **carlgatz** shared a detailed architecture for handling **high-volume spikes** (especially relevant for volatile, high-throughput data sources like onchain feeds):

- A **local gRPC pipeline** filters and preprocesses data *before* it hits AI models.
- “Fast path” operations stay **in RAM**, avoiding unnecessary token spend.
- **MCP** is reserved for deeper analysis only when needed.
- Reported outcome: **sub-millisecond latency** during spikes.

They’re looking for **5 developers to help stress test** and offered to share more of the architecture. If you’ve been thinking about cost-aware agent designs, this is a great thread to jump into.
