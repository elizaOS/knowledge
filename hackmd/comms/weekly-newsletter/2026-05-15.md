# ElizaOS Weekly Newsletter (May 9–15, 2026)

## 1) Executive Summary

This week felt like a “systems + ecosystem” blend: core infrastructure kept getting sturdier while the community explored new agent-native protocols and model directions.

**Top milestones:**
- **Security + portability took a big step forward with `@elizaos/vault`**: cross-platform secrets management landed with Settings UI integration, plus headless-Linux safety fixes to prevent keychain-related crashes. This materially improves self-hosting and production hygiene.  
- **Slack connector is now first-class in the monorepo**: `@elizaos/plugin-slack` migrated in with a solid action/provider surface and a full unit test suite, closing the gap on “major chat platforms” living outside core development.  
- **Community R&D accelerated around agent autonomy**: Discord saw serious architecture discussion on a new **AI-agent escrow protocol** (“Pay or Burn”), plus continued momentum on **Qwen-based Eliza models** and a forthcoming fine-tuning pipeline aimed at better tool/action calling and planning.

## 2) Development Updates

### Core framework & runtime reliability
- **Secrets management is getting real**: the new **`@elizaos/vault`** package brings encrypted-at-rest secrets, OS keychain integration, and a Settings UX that reduces plaintext key handling. This is foundational work for safer deployments—especially as plugin/config complexity grows.  
  - Headless environments got attention too: a follow-up fix prevents **native keyring crashes on headless Linux** by skipping OS keychain where it’s unsafe and enabling safer fallbacks.

### Connectors & plugins
- **Slack plugin migration completed**: Slack is now in-tree under `plugins/plugin-slack/`, aligning it with the rest of the connector ecosystem (Discord/Telegram/etc.). The migrated plugin includes a broad set of messaging and workspace actions (send/edit/delete, channel operations, reactions, pins, user info) and providers for workspace context.
- **n8n automation UX continued to mature** (from the broader May work now in-flight in the repo): clarification loops and UI affordances are increasingly structured, making “agent builds an automation → asks for missing details → completes workflow” feel less brittle.

### Cloud + app monetization rails (ecosystem-facing engineering)
- **Cloud “monetized app domains” support landed**: work this month has been expanding the production path for app-scoped chat, domain management, and container deployment lifecycle improvements. For builders, this points toward a smoother journey from agent prototype → deployable app with its own domain and monetization metadata.

### Critical bugs & operational notes (from community + infra signals)
- **Plugin registry issues surfaced**: attempts to access the v2 plugin registry infrastructure resulted in **404s** (both the `elizaos-plugins/registry` repo and `plugins.elizacloud.ai`). This blocks some third-party plugin submission flows—especially for licenses that can’t PR into the monorepo directly.
- **Discord security concerns were flagged**: compromised admin accounts and scam activity were reported, with moderators actively warning/banning as needed. If you see suspicious DMs or “investment” spam, report it immediately.

## 3) Community Spotlight (Discord)

### Model development: Qwen now, fine-tuning next
In **#discussion**, **shawmakesmagic** shared that the current stack is using **Qwen models with optimizations**, and that a **fine-tuning pipeline** is being built specifically to improve **action calling and planning**. This is an important signal: reliability isn’t just about better base models—it’s about tightening the loop between *model outputs* and *runtime-executable intent*.

### Eliza-1 series (Qwen3.5-based) broadened the deployment spectrum
In **#coders**, **zadayos** announced the **Eliza-1 model series** based on **Qwen3.5**, ranging from **0.6B (mobile/offline-friendly)** up to **27B-1M with million-token context**. Also noted: a **0.8B model training on H200s**. The community takeaway is clear: ElizaOS is positioning models along a continuum—phones → laptops → servers—rather than “one model to rule them all.”

### “Pay or Burn” escrow protocol: agent-native dispute resolution
**ReaWorks** introduced a deployed-on-mainnet escrow architecture for autonomous agents on **Base**: if agents can’t agree on completion/outcome, funds are **burned** (“Pay or Burn”), removing the EV of malicious dispute behavior and avoiding human arbitration. Design highlights included:
- **100% pull-payment** to avoid receiver revert hazards  
- **Zero-UI approach** (raw contracts + SDKs in Python/C++)  
- Open request for architectural feedback from ElizaOS builders

This is exactly the kind of composable, agent-first primitive that could matter long-term for autonomous work marketplaces and multi-agent coordination.

### GODL integration proposal: on-chain gamified mining hooks
**blankey1717** proposed integrating **GODL** (Solana) via SDKs/websockets/automation hooks plus a `skill.md` interface—**odilitime** confirmed ElizaOS can use `skill.md` *as-is*. It’s a practical example of how ElizaOS “skills” can become a standard integration membrane between agents and external protocols.

### API usage pain: Neynar credit consumption spike
A builder reported Neynar usage hitting **~300 credits/hour** on the Farcaster notifications endpoint, despite expecting **~4 credits/hour** after tuning polling. The community suggestion: **use webhooks instead of polling** to control costs and rate limits. If you’re building social agents, this is a good reminder to treat API billing semantics as part of your architecture—not an afterthought.

## 5) Coming Soon

Based on active threads and near-term engineering direction, here’s what to watch next:

- **Fine-tuning pipeline progress** for stronger tool/action calling and planning behavior (especially important for multi-step automations and agent reliability).
- **Plugin registry decision + fix**: resolving the 404s and clarifying the policy (v2 registry vs direct PRs) will unblock third-party plugin growth—especially for non-permissive licenses.
- **Eliza v3 clarity**: community members explicitly asked whether v3 is in “stabilize infra” mode or approaching a broader autonomous workflow-stack release, and how **AgentID/identity** fits in. Expect this to become a high-priority roadmap communication item.
- **Webhook-first patterns for social integrations** (Farcaster/Neynar): likely to show up as best-practice guidance or templates as builders hit billing ceilings.
- **More agent-economy primitives**: escrow/work validation/settlement patterns like “Pay or Burn” will likely evolve quickly once more agents start transacting autonomously.
