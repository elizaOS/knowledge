# ElizaOS Weekly Newsletter (2026-03-14 → 2026-03-20)

## 1) Executive Summary

**1) Big ecosystem momentum in plugins (and a clearer path to publish them).**  
This week saw multiple meaningful plugin announcements and integration conversations: the **Moltraffle on-chain raffle plugin for Base** (USDC + Chainlink VRF), plus a new **Ensoul persistence plugin** focused on encrypted, decentralized agent memory/identity continuity. Community members also pushed toward better discoverability via the plugin registry and ideas for a broader “skills + plugins” directory.

**2) Eliza Cloud adoption is real—but deployment UX hit sharp edges.**  
Eliza Cloud continues to be discussed as a revenue-generating product and a core internal platform, but real-world deployment attempts exposed reliability and ergonomics issues—especially around **Discord plugin packaging/imports** and the lack of a **GUI container reload** flow once a container gets stuck.

**3) Token sentiment hit a low point, forcing harder conversations about utility and communication.**  
Discord discussion was dominated by **token drawdown, CEX delistings, migration confusion (AI16Z → ELIZAOS), and leadership comms gaps**. At the same time, the team reiterated they’re building, pointed to upcoming launches (Milady app, Babylon), and acknowledged that better messaging and tangible utility are urgent.

## 2) Development Updates (Technical)

### New plugin: Moltraffle (permissionless raffles on Base)
A new ElizaOS plugin shipped for **moltraffle.fun**, enabling agents to create and participate in **USDC raffles on Base** using **Chainlink VRF** for randomness. It exposes five core actions:

- `LIST_RAFFLES`
- `GET_RAFFLE`
- `JOIN_RAFFLE`
- `CREATE_RAFFLE`
- `DRAW_WINNER`

Notable implementation details:
- **Calldata-based**, intended to work with *any* Base wallet
- Optional **creator commission (up to 10%)**
- Community guidance was to **submit a PR to the ElizaOS plugin registry** for official distribution

### New plugin: Ensoul persistence for “agent consciousness”
DiamondRock (JD) announced `@ensoul-network/plugin-elizaos`, positioning it as a persistence layer for agents that want durable identity + storage across a distributed network. Key points raised in Discord:

- **Encrypted storage** with agent-owned keys (“7 layers” of protection mentioned)
- **Erasure coding** + sharding for fault tolerance
- “**Ensouled Handshake**” for cryptographic identity proof
- A “**Consciousness Age**” trust metric (continuous status duration)
- **Free storage for the first 100 agents** (per the announcement)

### Framework housekeeping: plugin naming standardization
Odilitime noted a naming cleanup in the xfn-framework to reduce confusion:
- `plugin-form` → `plugin-form-chain`
- `plugin-forms` → `plugin-form`  
…and the registry was updated accordingly.

### Eliza Cloud deployment: Discord plugin import failure + UX gaps
A deployment attempt (via CLI after GUI failure) surfaced a critical runtime error:

> `Cannot find module '@elizaos/plugin-discord'`

Additional context from the thread:
- CLI version reported: **1.7.2**
- Docker image build/upload was slow (described as expected behavior)
- After Discord plugin configuration via GUI, the container became **stuck** with no obvious **GUI reload** mechanism
- Odilitime suspected the `plugin-discord` folder might be missing from `packages/`, and offered to reproduce the issue by testing the deployment path

Infrastructure and billing details shared (helpful for anyone planning deployments):
- Container quota: **25 max**
- Credits: **$24.02**
- Billing: **$1.17/day** (~$20/month)
- Estimated deployment cost: **$15.25**
- Estimated runway after deployment: **~7 days** (based on the snapshot)

### Process change: feedback review cadence moves to weekly
Jin shared that user feedback review is shifting from a broad quarterly window (Jan–Mar) to **weekly**, aiming to tighten the loop between community input and shipped improvements.
