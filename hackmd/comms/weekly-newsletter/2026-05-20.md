# ElizaOS Weekly Newsletter (2026-05-14 → 2026-05-20)

## 1) Executive Summary

**1) ElizaOS v2.0.0 shipped (May 19).**  
The community celebrated the release in Discord, and the core team hinted that **v3 is “close”**. The big takeaway: the project is prioritizing *product delivery* (framework + Milady app + Eliza Cloud) while tokenomics details continue to mature.

**2) Major runtime + connector capabilities landed (merged May 18).**  
A substantial core upgrade added **connector capability detection** (typing indicators, message edits, threads, reactions) and hardened the **agent orchestrator** with self-healing, security gates, and cleaner sub-agent UX—especially on Discord where narration can now live in threads instead of polluting main chat.

**3) Platform work accelerated (Satellites + dynamic views).**  
A large WIP PR introduced **first-party “Satellites”** inside the Electrobun platform plus early **dynamic agent-created view** plumbing. It’s not merged yet, but it signals where the desktop app is heading: richer agent-driven UI surfaces and more structured renderer↔worker communication.

## 2) Development Updates

### Core runtime & orchestration
- **Connector capability detection + runtime API expansion** (PR **#7782**)  
  The runtime now exposes a consistent set of optional capabilities (e.g., `typing_indicator`, `edit_message`, `create_thread`, `post_to_thread`, `react_message`). This is foundational for making experiences consistent across Discord/Slack/other connectors without hardcoding “Discord-only behavior.”
- **Agent orchestrator hardening & security gates** (PR **#7782**)  
  The orchestrator gained:
  - self-healing primitives (stale lock cleanup, session reconciliation, crash detection)
  - **audit logging** for sensitive actions
  - tightened access control via explicit permission checks
- **Sub-agent UX redesign (Discord-first, capability-aware)** (PR **#7782**)  
  Sub-agents now post a single “running” message and stream detailed narration into a **thread** where supported—keeping main rooms readable and reducing “status text” contamination in LLM context windows.

### Desktop / Electrobun direction: Satellites & dynamic views (WIP)
- **First-party Satellites + dynamic view host bridge** (PR **#7779**, WIP)  
  Introduces a first-party Satellite set (`eliza.runtime`, `eliza.fs`, `eliza.git`, `eliza.pty`, `eliza.local-model`, etc.) and early dynamic-view infrastructure (register/list/open/push/close). If you’ve wanted “agent-generated UI panels” beyond static dashboards, this is the scaffolding.

### Local inference & performance notes (issues closed this week)
A cluster of local-inference issues closed around May 15–17, including speculative decoding edge cases (SWA + DFlash compatibility, M-RoPE position validation) and a Vulkan performance cliff report. Even when individual issues close quickly, they’ve been surfacing a clear theme: **modern model architectures need explicit gating + better runtime diagnostics** (so users know when speedups are actually active).

## 4) Token Economics (AI16z token + auto.fun)

This week’s most concrete token/econ updates came from Discord Q&A (not a formal spec drop):

- **Milady and ElizaOS are separate legal entities.** ElizaOS “built on their own stack” after taking fees; it did **not** “launch Milady.”
- **Buybacks are not yet formula-defined.** There’s **no fixed buyback percentage** today; the stated priority is **profitability first**, then tuning mechanisms.
- **Revenue streams are separated.** Milady has separate cloud infrastructure; profits there would drive **Milady** buybacks rather than being automatically attributed to ElizaOS.
- **No additional tokens planned** (with “Babylon” mentioned only as a distant possibility).
- **Governance is not formalized yet.** No established on-chain/off-chain governance mechanism exists today for buyback decisions.

On **auto.fun**: there weren’t specific feature announcements in the provided logs this week, but the continued presence of “auto.fun” community identity (and trading-bot chatter) suggests demand for tighter automation/trading integrations—expect future updates to connect product utility to token narratives more explicitly.
