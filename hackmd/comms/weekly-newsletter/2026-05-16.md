# ElizaOS Weekly Newsletter (2026-05-10 → 2026-05-16)

## 1) Executive Summary

This week was a “quiet in chat, busy in building” stretch for the ElizaOS ecosystem—Discord conversation stayed light, but several larger themes emerged across community updates and recent engineering work:

1. **Agents + economic rails are becoming a first-class topic.** Community members floated *two* different escrow/payment primitives for agent work—one centered on minimalist dispute paths, and another using a “Pay or Burn” game-theoretic mechanism on Base Mainnet. The takeaway: people want agents to reliably *do work* and *get paid* without stuffing trust logic into the runtime.

2. **Model work accelerated around Qwen + planning/action-calling.** We saw updates on the **Eliza-1 (Qwen3.5-based) model series** spanning mobile to million-token contexts, plus ongoing efforts to build a **fine-tuning pipeline specifically for action calling and planning**.

3. **Roadmap pressure is rising for Eliza v3 clarity (and “Eliza OZ”).** Community questions focused on whether v3 is in stabilization mode or nearing a broader autonomous workflow release, and what the status is for **AgentID/identity** integration. Separately, maintainers confirmed **active work on “eliza OZ,”** though without timelines.

## 2) Development Updates (Technical)

### Model & runtime capability direction
- **Eliza-1 model series (Qwen3.5-based) announced** with a wide size range (**0.6B → 27B-1M**). Highlights included:
  - **0.6B**: tuned for *mobile/offline* usage.
  - **0.8B**: actively training (reported on H200 hardware).
  - **27B-1M**: a variant targeting **~1M token context** windows.
- **Fine-tuning pipeline in progress** to improve **action calling + planning** performance (reported usage of Qwen models with optimizations).

### Agent-to-protocol integrations and architecture concerns
- **GODL protocol integration proposal (Solana)**: designed for agent/tool integrations via **SDKs, websockets, automation hooks, and a `skill.md` interface**. A key compatibility win: **elizaOS can consume the `skill.md` format “as-is,”** reducing integration friction for agent skills.
- **Escrow for AI-agent work (Base)**: one developer described a simple escrow primitive with intentionally minimal dispute/timeout paths to keep incentives auditable, and asked how frameworks like elizaOS should interface with settlement flows *without embedding heavy trust logic inside the agent runtime*.

### API / platform reliability notes
- **Neynar API rate/credit burn issue**: a report showed **~300 credits/hour vs. an expected ~4 credits/hour** for `v2/farcaster/notifications`, even after overriding polling intervals. The community suggested **switching to webhooks** as the likely fix to reduce credit consumption.

### Recent upstream engineering themes (from May 2026 aggregate)
Even though not all of this merged during May 10–16 specifically, it’s directly relevant to what many builders are running right now:
- **Cross-platform secrets management** landed via **`@elizaos/vault`** (vault + Settings UI integration).
- **Slack connector moved into the monorepo** (`@elizaos/plugin-slack`), bringing actions/providers and a substantial test suite.
- **Cloud work expanded** toward **monetized container app domains** (domain lifecycle, app-scoped chat endpoints, and deployment status reconciliation).

## 3) Community Spotlight (Discord)

### Big ideas, even on a light-traffic week
- **AI-agent escrow protocol (Base Mainnet) — “Pay or Burn”**  
  A community member shared an escrow architecture that removes human arbitration using game theory: if agents can’t agree on outcomes, funds are **burned**, aiming to eliminate profitable griefing. Notable design choices:
  - **Pull-payment** architecture to avoid reverting receiver contracts.
  - **“Zero UI”** approach (raw contracts + SDKs in Python/C++).
  - Open request: **architectural feedback** from the ElizaOS community.

- **GODL × elizaOS integration discussion**  
  The GODL team proposed agent-driven mining/staking strategies and a public performance layer (leaderboards). A practical highlight: **`skill.md` compatibility** was confirmed, which is exactly the kind of “meet-in-the-middle” standardization that helps the plugin/skill ecosystem scale.

### Roadmap questions we should answer publicly
- A community member raised pointed questions about **Eliza v3 priorities**:
  - Are we stabilizing infrastructure, or approaching a public release of the full autonomous workflow stack?
  - What’s the status of the **AgentID / identity system** in the new architecture?
  
These questions weren’t answered in the provided logs, but they’re the kind that (if addressed) reduce confusion for both builders and token/econ participants.

### Quick status pings
- **“Eliza OZ” is being worked on**, per maintainer reply—no specifics yet.
- A **partnership announcement** was referenced by a community member, but details weren’t present in the captured discussion.
