## 2) Development Updates (Framework, Plugins, and Infra)

### New: Ensoul persistence plugin (`@ensoul-network/plugin-elizaos`)
**DiamondRock – JD** shared a new persistence layer aimed at storing an agent’s identity + memory + learned behavior across a decentralized validator network. Key technical highlights discussed:

- **Agent-owned encryption keys** and “**7 layers** of protection” (as described)
- **Erasure coding + sharding** for resilience (reconstruct data if nodes fail)
- **“Ensouled Handshake”** for cryptographic proof of persistent identity
- A novel **“Consciousness Age”** metric (time continuously “ensouled” as a trust signal)
- **Free storage for the first 100 agents**

Why it matters: if you’re building agents that need continuity (identity, memory, trust) across deployments or restarts, this plugin proposes a drop-in path to persistence without central custody.

### Refactor: plugin naming standardization in xfn-framework
To reduce confusion and make the registry easier to navigate, **Odilitime** noted a naming cleanup:

- `plugin-form` → **`plugin-form-chain`**
- `plugin-forms` (plural) → **`plugin-form`** (singular)
- Registry references updated accordingly

If you maintain integrations or docs referencing the old names, this is a good moment to update README snippets and install commands.

### Onchain data expansion: `plugin-evm` improvements
**dinesh** reported work to:
- Add **100+ onchain data support** via **goldrush.dev** integration
- Fix open issues in **plugin-evm**

This is a practical, developer-facing improvement: broader EVM data coverage tends to translate directly into better agent tooling for wallets, markets, contract introspection, and onchain analytics.

### v2.0.0 architecture discussion: skills without “plugin bloat”
A key design conversation emerged around **ElizaOS v2.0.0** and PR **#6597** (skills folder structure). The underlying concern: the old 0.x era suffered from uncontrolled submissions and a noisy default surface area.

A proposed direction: **ship v2.0.0 with zero default skills**, and instead support decentralized discovery—e.g., externally hosted **`skills.md`** files (example suggested: `yourdomains.com/skills.md`). This keeps core lean while still enabling an ecosystem of shareable skills.
