## Weekly Newsletter (2026-06-02) — ElizaOS  
**Week of 2026-05-26 to 2026-06-01**

### 1) Executive Summary

This week was a big “systems week” for ElizaOS—meaning a lot of the wins make the platform feel more reliable, more live, and easier to build on:

1. **The `/orchestrator` coding workbench hit Claude-Code/Codex-style parity (for real).**  
   The workbench now supports *live streaming*, *structured tool-call cards*, *interleaved diffs*, and *Stop / Esc interrupt*—running end-to-end on **opencode + Cerebras**. This is a major milestone for anyone using ElizaOS as a serious coding agent platform.

2. **Live info is now available for non‑Anthropic models via a keyless `WEB_FETCH` action.**  
   Builders running Cerebras / gpt-oss setups no longer need to “delegate to a coding sub-agent” just to fetch something from the web. The runtime gained a **keyless, SSRF-guarded** fetch action to answer live queries inline.

3. **Cloud + deployment stability improved across the board.**  
   Routing and startup issues in cloud agent sandboxes were fixed, release and environment triggers were tightened, and the project promoted **develop → main** to sync production with recent Cloud V2 changes.

### 2) Development Updates

#### Agentic orchestration: workbench improvements
- **Live `/orchestrator` workbench shipped** with:
  - **SSE streaming** (`GET /tasks/:taskId/stream`) so the room updates sub-second (polling remains as fallback).
  - **Tool-call cards** and coalesced “agent chunks” into readable conversation turns.
  - **Interleaved LCS diffs** (instead of “all removals then all additions”).
  - **Stop button + Esc** to interrupt long runs quickly.
  - Fixes for cross-task transcript contamination and duplicate React key warnings.  
  *Impact:* the orchestrator now feels like a modern coding agent UI rather than a log viewer.

- **Reliability fix:** ACP failures are now diagnosable because JSON‑RPC `error.data` is preserved.  
  Previously, `-32602 Invalid params` errors lost the important schema details; now the error payload survives the transport layer.

#### Runtime: live information retrieval (beyond Anthropic)
- **New action:** `WEB_FETCH` (keyless) enables live info retrieval for **non‑Anthropic** providers.  
  This unblocks “what’s the current X?” workflows without requiring a Google/Gemini key and without spawning side-agents.

- **Planner routing fix:** follow-up changes ensured the v5 planner actually *surfaces* `WEB_FETCH` when appropriate (instead of filtering it out by context gates or picking a simile-colliding action).

#### Stability + UX hardening
- **Boot stability:** `plugin-google` and `plugin-lifeops` were moved to optional status to prevent default image boot regressions.
- **Vault resilience:** PGlite lock self-healing prevents “unclean shutdown → can’t boot” scenarios.
- **Windows release polish:** installer naming is now brand-agnostic for whitelabel builds.
- **Cloud sandbox routing + startup:** fixes landed to address native sandbox routing failures and local/prod parity issues.
- **Runtime config:** `run-node` now supports customizing the entry file via `ELIZA_ENTRY_FILE` (useful for custom bundles and alternate entrypoints).

### 3) Community Spotlight (Discord)

#### DeFi integration architecture: abstraction vs. protocol-first
Ben (Smeltor) sparked a high-signal discussion on a **multi-chain DeFi aggregation system** designed for AI agents. The pitch: a unified interface across **swap / earn / perps / bridge / borrow**, abstracting protocols like **Kamino, Aave, Morpho, Compound, Hyperliquid**, and returning **signable calldata** so agents remain **non-custodial**.

The community debate was constructive and very “builder-real”:
- **Abstraction layer advocates** want less boilerplate and faster iteration across venues.
- **Protocol-first advocates** (e.g., Kamino-focused) argued that deep protocol quirks matter, and shipping a first-class single integration can be the fastest way to earn ecosystem attention.

Notably, **N4vnt** shared that *plugin‑kamino* work is underway and that a meaningful portion of the implementation really is Kamino-specific.

#### New builder intros + testing calls
- **Ana‑Maria (NeverBell)** introduced infrastructure to let agents execute financial market actions via **natural language → controlled execution**, and asked for **early technical testers**. If you’re building agent trading flows or “intent → execution” pipelines, this is worth engaging with.

#### Project structure clarity (reducing confusion)
Questions surfaced about whether ElizaOS was “still active” and how waifu.fun relates. **odilitime** clarified:
- **waifu.fun does not have its own token**  
- **Solace on waifu has a token called `waifu`**  
- The team **has not left ElizaOS** (ongoing work continues)

Also: shoutout to **faouzi_122** for sharing *Arsenal Quant / Arsenal Decision Engine*—a DeFAI “decision oracle” integrating via **MCP** and using **L402 Lightning** for payments. Even as an early introduction, it’s the kind of composable primitive the ecosystem benefits from.
