## Weekly Newsletter (May 24–May 30, 2026)

### 1) Executive Summary

This week was a “plumbing + ecosystem” kind of sprint—less flashy, but hugely impactful for builders.

- **Solana plugin architecture got a meaningful refactor.** The former `plugin-solana` was **moved into the main `elizaos/eliza` monorepo** and is now surfaced as **`plugin-wallet`**, with Solana-specific implementation living under `plugins/plugin-wallet/src/chains/solana` (develop branch). This sets a clearer pattern for chain modules living inside shared wallet primitives.
- **Reliability + security hardening continued across core + cloud.** The project-wide push toward production-grade standards continued (SOC2 controls, release integrity, and cloud uptime improvements like Redis heartbeats replacing flaky probes). See the weekly GitHub summary for key merged items like SOC2 verification work and orchestrator stability improvements.
- **Community-led plugin momentum accelerated.** A **Kamino integration plugin** is now actively being developed by community members, with prior 1.x read-only work available as reference and a clear path to upstream contribution via PR.

### 2) Development Updates (Core, Cloud, Plugins)

Here are the most relevant engineering changes and themes from GitHub activity and Discord:

**Codebase organization & plugin consolidation**
- **Solana plugin relocation:** `plugin-solana` is no longer in its old standalone location. It has been integrated into the main repo as **`plugin-wallet`**, with the Solana code here:  
  `plugins/plugin-wallet/src/chains/solana` (develop)  
  This is a structural change that should reduce duplication and make cross-chain wallet behaviors easier to standardize.
- **Plugin contribution flow reminder:** new plugins (like Kamino) should be contributed via **standard pull requests** to `elizaos/eliza`.

**Agent quality, stability, and “don’t get weird” improvements**
- The weekly GitHub summary notes continued work on **planner behavior** to reduce repetition, meta-narration, and instruction-following edge cases—aimed at making agents more predictable and “clean output” by default.
- Infra reliability work continued: cloud systems increasingly rely on **heartbeats** and stronger release verification to reduce silent failures and deployment regressions.

**Local inference & runtime improvements (high-level)**
- The May monthly/weekly work shows ongoing investment in **cross-platform runtime stability** (headless environments, multi-arch builds, improved verification).
- If you’re building long-running or multi-agent workflows, the direction is clearly toward: fewer runtime surprises, better recovery paths, and clearer tool/evaluator contracts.
