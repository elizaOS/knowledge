## ElizaOS Weekly Newsletter (2026-05-31 → 2026-06-06)

### 1) Executive Summary

**A sturdier Cloud V2 landed in production.** This week’s biggest milestone was the **develop → main promotion** to keep production aligned with the latest **Cloud V2** stack changes, along with follow-on CI/deploy hygiene that makes cloud releases more predictable and less “hand-tuned.”

**Sign-in and onboarding got meaningfully smoother.** The team resolved a frustrating cluster of **staging magic-link auth failures** by tightening CSP/CORS rules and improving how the app handles missing auth context on callback. In parallel, the UI gained better first-run guidance via **prompt suggestions** in the Continuous Chat Overlay.

**The ecosystem expanded in two directions: safer keys + more DeFi capability.** On the security/ops side, the curated registry added **@1claw/plugin-elizaos** (HSM-backed secrets + multi-chain signing). On the DeFi side, Discord discussions crystallized around **Kamino integration** and the broader “abstraction layer vs protocol-specific plugins” debate.

---

### 2) Development Updates

#### Cloud & Infra
- **Production sync:** `release: promote develop → main (Cloud V2 sync)` shipped a large batch of Cloud V2 work and triggered the normal production auto-deploy chain (backend + CF deploy + provisioning worker + stable agent image).  
  - PR: https://github.com/elizaos/eliza/pull/8116
- **Magic-link auth stability (staging):** Multiple fixes together resolved the “Sign-in failed / Failed to fetch” class of errors by aligning **CSP `connect-src`**, **Cloud API CORS allowlists**, and handling **missing app-auth context** in the callback flow.  
  - PRs: https://github.com/elizaos/eliza/pull/8197 • https://github.com/elizaos/eliza/pull/8196 • https://github.com/elizaos/eliza/pull/8194
- **Reasoning-model billing correctness:** Cloud now **guarantees response-token budget** for reasoning models so users don’t get billed for “content: null” responses when `max_tokens` is too small.  
  - PR: https://github.com/elizaos/eliza/pull/8189
- **Native sandbox reliability:** Bootstrapping fixes improved **Cloud agent sandbox startup/routing**, reducing “it works locally but not in Cloud” mismatches.  
  - PR: https://github.com/elizaos/eliza/pull/8104

#### Core Agent Runtime & Orchestrator
- **Keyless web access across models:** A new **keyless `WEB_FETCH` action** means non‑Anthropic backends can perform live web retrieval without model-specific monkey patches.  
  - PR: https://github.com/elizaos/eliza/pull/8097
- **Orchestrator correctness fixes:** Several sharp-edge bugs were resolved (e.g., JSON-RPC error detail preservation; completion de-dup races; and URL verification freshness false positives). These are the kinds of fixes that don’t demo well—but dramatically improve trust in multi-agent flows.  
  - PRs: https://github.com/elizaos/eliza/pull/8098 • https://github.com/elizaos/eliza/pull/8161  
  - Issue/PR context: https://github.com/elizaos/eliza/issues/8127
- **Local inference resilience:** Local inference now throws **typed “unavailable” errors** (instead of generic errors) when native bindings are missing—helping downstream tooling degrade gracefully.  
  - PR: https://github.com/elizaos/eliza/pull/8191

#### UI / DX (Developer Experience)
- **Continuous Chat Overlay improvements:** Added a **prompt-suggestion strip** (five tailored suggestions) to reduce “blank page paralysis” and speed onboarding.  
  - PR: https://github.com/elizaos/eliza/pull/8201
- **Storybook coverage expanded:** Storybook documentation now includes **nine core UI primitives**, making it easier for plugin authors to build consistent UI surfaces.  
  - PR: https://github.com/elizaos/eliza/pull/8200
- **Work in progress worth tracking:** a **Cloud SSO pairing handler** (dashboard popup flow) and continued syncing of the **Odysseus UI** with upstream orchestrator changes.

### 3) Community Spotlight (Discord)

**Ecosystem architecture, clarified (again, but better).** A standout thread was stan0473’s clear breakdown of how the pieces fit:
- **ElizaOS** = core agent-building framework  
- **ElizaCloud** = hosting infrastructure for fully featured agents  
- **Milady** = consumer-facing “life management” UI  
- **Waifu** = side project focused on agent competition/economies running on ElizaCloud  
This helped newer builders map where to contribute and which layer they’re actually targeting.

**DeFi integrations: Kamino plugin proposal + architecture debate.**
- n4vnt proposed building a **comprehensive Kamino plugin** (lending/borrowing, liquidity vaults, farms, limit orders) to fill a real integration gap for Solana-native DeFi agents.
- A healthy debate emerged: should ElizaOS push **a unified multi-protocol abstraction layer** (Smeltor’s approach) or prioritize **protocol-specific plugins** to maximize control and reliability for risk-sensitive operations?

**Project relationships & org structure: fewer rumors, more facts.** odilitime clarified that **Hyperia is separate from ElizaOS** (not a joint venture like Babylon), and **Hyperforge** is Hyperia’s world-building component. Shaw founded both, but current involvement in Hyperia appears limited (likely advisory) as focus remains on ElizaOS + related initiatives.

**Neverbell: early testers wanted.** Neverbell introduced an early-stage system exploring **AI agents executing financial operations** via natural language, gated by **user-defined permissions/controls**—and invited technical testers (with funded accounts for experimentation).
