# ElizaOS Weekly Newsletter (2026-06-08)  
**Week:** 2026-06-01 → 2026-06-07

## 1) Executive Summary

This week was all about **hardening Eliza Cloud**, **making the UI feel “agent-native,”** and **modularizing big subsystems** so they’re easier to extend.

**Top milestones:**

1) **Cloud V2 shipped to production (develop → main promotion).**  
   The team promoted recent Cloud V2 stack changes to `main`, triggering production deploys for the cloud backend, frontend, provisioning worker, and refreshed agent images. This is the kind of “quiet” release that unlocks a lot of follow-on work because it aligns prod with the newest infra patterns.  
   - PR: https://github.com/elizaos/eliza/pull/8116

2) **LifeOps monolith split into focused plugins + new Personal Assistant core.**  
   The long-running LifeOps surface was decomposed into a slimmer orchestrator plugin (`plugin-personal-assistant`) plus a set of domain plugins (goals, inbox, finances, relationships, etc.). This is a big architectural bet: smaller plugins, clearer ownership, faster iteration.  
   - PR: https://github.com/elizaos/eliza/pull/8277

3) **Authentication + staging isolation got a serious upgrade (Steward / PKCE / session refresh / CORS).**  
   A sequence of fixes improved real-world login reliability (especially around Steward OAuth hardening), staging/prod separation, and browser compatibility. If you felt “Cloud login got less flaky,” it’s because several sharp edges were filed down at once.  
   - Example PRs: #8259, #8280, #8282, #8196, #8197 (links below)

## 2) Development Updates (Engineering Highlights)

### Cloud: Auth, staging isolation, and “it just works” reliability
- **PKCE support for Steward OAuth** landed to match Steward’s tightened security requirements, and the nonce-exchange calls are now signed where required. This directly addresses the class of failures where OAuth `/authorize` suddenly started rejecting clients.  
  - PR: https://github.com/elizaos/eliza/pull/8259  
  - Related open issue to watch (prod impact): https://github.com/elizaos/eliza/issues/8257

- **Session refresh hydration fixes** improved “bounce back to /login” loops by ensuring the SPA can rehydrate auth state even when tokens are refreshed via HttpOnly cookies.  
  - PR: https://github.com/elizaos/eliza/pull/8280

- **Staging tenant isolation improvements**: multiple patches ensured staging uses staging Workers/APIs, correct CSP `connect-src`, and correct CORS allowlists—eliminating subtle cross-environment calls (staging UI accidentally talking to prod API).  
  - PRs:  
    - https://github.com/elizaos/eliza/pull/8282  
    - https://github.com/elizaos/eliza/pull/8196  
    - https://github.com/elizaos/eliza/pull/8197

### Cloud: Containers + monetization plumbing is getting real
- **Container deploy/read routes** were added (`/api/v1/containers`) and then expanded with lifecycle verbs (PATCH/DELETE/scale). This is a core building block for “apps on Eliza Cloud,” not just agents.  
  - PRs:  
    - https://github.com/elizaos/eliza/pull/8258  
    - https://github.com/elizaos/eliza/pull/8270

- **Billing robustness**: container billing now proceeds even if earnings conversion fails, preventing stuck billing cycles and improving operational safety.  
  - PR: https://github.com/elizaos/eliza/pull/8269

- **CORS upgrades for third-party apps**: the public API is becoming friendlier to external builders (including headers like `Idempotency-Key` and affiliate attribution).  
  - PRs:  
    - https://github.com/elizaos/eliza/pull/8245  
    - https://github.com/elizaos/eliza/pull/8252

### App + UI: faster startup, better onboarding, and a more “agent workbench” feel
- **Desktop startup performance** continued to improve via background model warmup and tray-first startup options (work that reduces first-interaction latency).  
  - Context PRs referenced in weekly summary: #8148, #8151

- **Continuous Chat Overlay got polish + prompt suggestions**, making the “empty state” more helpful and lowering the barrier for new users.  
  - PRs:  
    - https://github.com/elizaos/eliza/pull/8201  
    - https://github.com/elizaos/eliza/pull/8224  
    - https://github.com/elizaos/eliza/pull/8238

- **Agent-controllable plugin views (“agentic view surface”)** shipped: a framework so the agent can “see” and interact with view elements more systematically, plus a view-action affinity layer to bias planners toward actions relevant to the current UI context.  
  - PR: https://github.com/elizaos/eliza/pull/8080

### Orchestrator: stability + UX improvements
- **Coding workbench parity** progressed (streaming, diffs, SSE, stop controls), plus several safety fixes and race-condition patches.  
  - Workbench PR: https://github.com/elizaos/eliza/pull/8095  
  - TOCTOU completion dedup race fix: https://github.com/elizaos/eliza/pull/8161

## 4) Token Economics (AI16z token + auto.fun)

No explicit on-chain metrics or token parameter changes were included in this week’s aggregated notes, so we won’t speculate on price or emissions.

What *did* move meaningfully on the “economics plumbing” side (and matters for AI16z/auto.fun-style ecosystems) is the continued build-out of **monetization primitives** in Eliza Cloud:

- **Metered hosting + container lifecycle APIs** are maturing (deploy, scale, delete, billing hardening).  
- **Affiliate attribution headers** and **idempotent billing-friendly requests** are being wired in.  
- The orchestrator’s **“monetized-app loop”** capabilities keep getting closer to “agents that can build and operate paid apps with spend caps.”

If you’re tracking AI16z and auto.fun, this is the kind of foundational work that usually precedes more visible ecosystem activity: smoother onboarding, reliable auth, predictable billing, and deployable app surfaces.
