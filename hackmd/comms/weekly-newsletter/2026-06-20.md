# ElizaOS Weekly Newsletter (Jun 14 – Jun 20, 2026)

## 1) Executive Summary

This week was all about **hardening Eliza Cloud for real-world load** while the community pushed on **clarity (branding + UX)** and **transparency**.

**Top milestones:**
- **Eliza Cloud moved from staging → production and stayed upright.** Core infra work (gateway, caching, provisioning durability, and mobile build parity) landed to support a more “commercially ready” platform surface.  
- **Major reliability win: managed agents now default to local, in-container state.** On Jun 20, the team shipped a root-cause fix for “too many clients” incidents by moving managed agents off shared Postgres pools and onto **local PGlite persistence** (**PR #8754**).  
- **Community alignment discussions intensified** around positioning Eliza as an “operating system for AI agents” and making the product feel approachable for everyday users—not just developers and traders.

## 2) Development Updates (Technical)

### Cloud infrastructure & reliability
- **Gateway consolidation + routing improvements:** BitRouter was decommissioned and the **Cloudflare Worker became the primary model gateway**, with improved reliability via caching and better provider routing behavior. A round-robin strategy for **multiple Cerebras API keys** was added to spread rate limits and reduce long-tail latency during spikes.
- **Cache hardening:** Cloudflare KV cache support landed to reduce reliance on brittle external connections and improve request consistency during transient network issues.
- **Provisioning durability upgrades:** Provision workflows were hardened with **atomic failure write-backs** and self-healing mechanisms, reducing the chance of “stuck” agents and making recovery more deterministic after incidents.

### Agent performance & orchestration
- **Faster cold boots:** Introduction of **`LEAN_CHAT_PLUGINS`** and better background task handling reduced boot time and latency for chat-focused agents.
- **Orchestrator upgrades:** Sub-agent nesting and **MCP auto-inheritance** improved multi-agent coordination (fewer manual config steps, better capability propagation).
- **Vision plugin modernization:** TensorFlow.js was replaced by a **native ggml YOLOv8n detector**, eliminating build warnings and tightening the runtime footprint.

### UI & mobile stability
- **Mobile build regressions fixed:** Android/iOS build failures were addressed, and web-only dashboards were correctly excluded from Capacitor builds.
- **Onboarding streamlined:** Cloud-only hosts can now skip irrelevant runtime selection steps, and the app added more direct **agent management controls** (including start/stop flows in Settings for cloud agents).

### The week’s headline fix (Jun 20)
- **“Too many clients” incidents resolved:** Managed agents were migrated to **local in-container PGlite databases** for state persistence—dramatically reducing load on shared Postgres pools.  
  - Link: **PR #8754** — https://github.com/elizaos/eliza/pull/8754

The Eliza mascot also got a shoutout as a strong, recognizable asset worth leaning into.

There was some pushback: **om1d_sa** noted that Eliza Cloud already supports **no-code agent creation, modification, and deployment**, suggesting the gap may be more about discoverability and onboarding than raw capability.
