# ElizaOS Weekly Newsletter (2026-06-07 → 2026-06-13)

## 1) Executive Summary

This week was all about **turning Eliza Cloud Apps from “it works on someone’s machine” into a repeatable, safer, staging-ready system**—while the community also had some important (and sometimes uncomfortable) conversations.

**Top milestones:**
- **Eliza Cloud Apps infrastructure hit a major staging milestone.** The team armed the Apps daemon, stabilized staging via modular infrastructure-as-code deployments, and enabled `APPS_DEPLOY_ENABLED` on staging—removing a lot of “SSH-and-pray” operations from the workflow.
- **Critical cloud resource leaks and state drift issues were addressed.** Deleted Apps now correctly tear down Postgres resources / release cluster slots, and Terraform lockfile/provider hash issues got cleaned up to keep CI and staging applies reliable.
- **Developer guidance sharpened around versioning + next-gen agent capabilities.** Discord discussions clarified that **2.x work lives on `develop`**, and we got a preview of **world model tooling** targeted for release in ~1 month.

## 2) Development Updates (Core + Cloud)

### Eliza Cloud Apps (Product 2): staging gets real
A bunch of the week’s work converged on one theme: **repeatability**.

**What changed (high-level):**
- **Apps daemon arming became idempotent and workflow-driven.** A dispatchable workflow now arms the daemon without manual SSH intervention, and `.env.local` operations were routed through `sudo` so arming works reliably in root-owned directories.
- **Staging environment stabilized with IaC-driven deployments.** Modular Terraform outputs are now used to fetch tenant DSNs and configure the daemon more safely (less secret-copy/paste drift).
- **`APPS_DEPLOY_ENABLED` was enabled on staging**, unlocking end-to-end staging validation for the Apps deploy path.

**Reliability & correctness fixes:**
- **Fixed a critical DB resource leak** where deleted apps failed to drop Postgres DBs/roles and did not release cluster slots—an issue that can quietly burn capacity and money over time.
- **Terraform “lockfile/provider hash” failures were resolved** (missing provider hashes are a classic cause of “works locally, fails in CI”).
- **Added 50 characterization tests for the warm-pool decision engine**, helping ensure container pre-warming logic remains stable as scaling behavior evolves.

**Still in progress (per engineering notes):**
- CI optimization to **gate expensive test lanes** more intelligently.
- Arming **build-from-repo** functionality on the daemon (so Apps deployments can be driven directly from a repo workflow, not just prebuilt artifacts).

### Authentication: confirmed stable in prod
Engineering verification confirmed **OAuth and magic-link login flows are functioning in production**. This matters because it’s foundational for Cloud onboarding and for third-party apps that rely on “Sign in with Eliza Cloud.”
