# Sensitive Request Links Parallel Implementation Plan

Status: Draft
Date: 2026-05-10

> For agentic workers: this plan is intentionally parallelized. Each worker
> owns a disjoint write set. Workers must not revert edits by other workers.

Spec:
`packages/docs/docs/superpowers/specs/2026-05-10-sensitive-request-links-prd.md`

Gap report:
`packages/docs/docs/superpowers/reports/2026-05-10-sensitive-request-links-architecture-gaps.md`

Implementation report:
`packages/docs/docs/superpowers/reports/2026-05-10-sensitive-request-links-implementation-design.md`

## Coordination Rules

- No worker creates a second LifeOps task primitive.
- No worker stores raw secrets in chat content, LLM prompts, memory, logs, or
  callback payloads.
- No worker treats tunnel availability as authorization.
- No worker adds identity merge shortcuts that bypass the merge engine.
- All public links must be backed by explicit policy.
- All callbacks must be typed and redacted.
- Each worker adds tests in the same package as the code it changes.

## Wave 0: Contract Lock

### Worker A: Core Policy Contract

Write set:

- `packages/core/src/sensitive-request-policy.ts`
- `packages/core/src/sensitive-request-policy.test.ts`
- relevant core export barrels

Tasks:

- Define request kinds, delivery modes, actor policy, environment inputs, and
  resolver output.
- Implement a pure `resolveSensitiveRequestDelivery()` function.
- Cover public, DM, owner-app, cloud, tunnel, no-tunnel, verified payer, and
  any-payer branches.

Exit criteria:

- Core unit tests pass.
- No cloud/local runtime imports in the pure resolver.

### Worker B: Core Secrets Action Integration

Write set:

- `packages/core/src/features/secrets/actions/request-secret.ts`
- focused tests under `packages/core/src/features/secrets`

Tasks:

- Replace unsafe "set secret KEY value" primary wording with structured
  delivery plans.
- Public channels emit private fallback or authenticated link instructions.
- Owner private app contexts emit inline form descriptors.
- DM contexts remain supported but prefer private link/form when available.

Exit criteria:

- `REQUEST_SECRET` never instructs public chat to paste a secret.
- Structured callback content contains no raw secret value.

### Worker C: Docs and API Contract

Write set:

- `packages/docs/rest/secrets.md`
- `packages/docs/rest/cloud.md`
- `packages/docs/security.md`
- PRD/report files if clarifications are needed

Tasks:

- Document `SensitiveRequest`.
- Document secret request lifecycle.
- Document payment context semantics.
- Document local/no-tunnel fallback.

Exit criteria:

- Docs match code contracts.
- Launch docs lint passes.

## Wave 2: Payment Context

### Worker G: App Charge Payment Context

Write set:

- `cloud/packages/lib/services/app-charge-requests.ts`
- `cloud/packages/lib/services/app-charge-callbacks.ts`
- `cloud/apps/api/v1/apps/[id]/charges/**`
- app charge tests

Tasks:

- Add `payment_context = verified_payer | any_payer`.
- Include context in metadata, public detail, and callback payload.
- Keep `verified_payer` as default.
- Reject anonymous any-payer checkout until settlement target supports it.

Exit criteria:

- Existing payment behavior remains unchanged by default.
- Tests prove callback secret is redacted and context is included.

### Worker H: Settlement Target Design and Implementation

Write set:

- app charge settlement service and repository only after Worker G lands

Tasks:

- Add explicit settlement targets.
- Implement anonymous any-payer only for targets that do not need payer-owned
  credits or entitlements.

Exit criteria:

- Anonymous payment cannot allocate credits to a guessed payer org.

### Worker H2: x402 Request Link Hardening

Write set:

- `cloud/packages/lib/services/x402-payment-requests.ts`
- `cloud/packages/lib/middleware/x402-payment.ts`
- `plugins/plugin-x402/src/payment-wrapper.ts` only for contract alignment
- x402 unit tests

Tasks:

- Redact callback URLs and private metadata from public status views.
- Add signed callbacks or mark callback absence as unsupported in the shared
  request contract.
- Make cloud middleware settlement failure blocking unless explicitly opted out.
- Verify duplicate settlements are idempotent and do not collapse separate app
  payments into one earnings record.

Exit criteria:

- x402 public status contains no private callback integration data.
- Paid handlers do not execute after failed settlement.

## Wave 3: Local and Tunnel Runtime

### Worker I: Local Sensitive Request Store and Routes

Write set:

- `packages/agent/src/api/**` or `packages/app-core/src/api/**` depending on
  active local API ownership
- local route tests

Tasks:

- Add local create/get/submit/cancel routes.
- Enforce owner session or DM-auth handoff.
- Refuse tunnel delivery when local auth is not configured.

Exit criteria:

- No-tunnel fallback works.
- Tunnel without auth is rejected.

### Worker J: Owner App Inline UI

Write set:

- app/chat UI components in the local app package

Tasks:

- Render inline request forms from action callback content.
- Store status only in chat history.
- Avoid nested cards and avoid leaking values into visible transcript.

Exit criteria:

- UI tests prove form submit does not echo value.

### Worker K: Messaging DM Delivery

Write set:

- connector send-policy or message routing integration
- no changes to unrelated connector internals

Tasks:

- Add a generic "send private sensitive request" helper.
- Use typed `DispatchResult`.
- Public room receives status only.

Exit criteria:

- Tests cover DM success, DM failure, and fallback public status.

### Worker K2: LifeOps Dispatch Wiring

Write set:

- `plugins/app-lifeops/src/lifeops/scheduled-task/runtime-wiring.ts`
- `plugins/app-lifeops/src/lifeops/scheduled-task/runner.ts` only if the
  dispatcher interface needs injection cleanup
- `plugins/app-lifeops/src/lifeops/channels/default-pack.ts`
- LifeOps scheduled-task dispatch tests

Tasks:

- Replace noop-only scheduled task dispatch wiring with a production dispatcher.
- Evaluate LifeOps send policy before dispatch.
- Call channel/connector `send()` and preserve typed `DispatchResult`.
- Fix or de-advertise `in_app` and `push` send capabilities if no sender exists.
- Make entity `safeToSend` checks include channel availability.

Exit criteria:

- A fired `ScheduledTask` can deliver through a real channel in tests.
- Missing/disconnected/rate-limited channels produce typed failure state, not a
  silent noop.

## Wave 4: OAuth and Identity

### Worker L: OAuth Request Adapter

Write set:

- cloud OAuth initiate/callback adapters
- sensitive request service OAuth branch

Tasks:

- Wrap OAuth initiate in `SensitiveRequest`.
- Bind returned connection to request id and owner/org identity.
- Emit `oauth.connected`.

Exit criteria:

- Wrong account/org cannot satisfy another request id.

### Worker M: Linked Identity Authorization

Write set:

- identity/owner binding service
- docs/tests for identity resolution

Tasks:

- Expose a shared check: "is this actor a verified linked identity for this
  owner/org?"
- Consume cloud sessions, connector ids, and existing entity identities.
- Preserve merge-engine audit semantics.

Exit criteria:

- Identity checks are tested across at least cloud session and Discord id.

## Wave 5: LifeOps Follow-Up Integration

### Worker N: Pending Request Scheduled Tasks

Write set:

- LifeOps integration only

Tasks:

- Create `ScheduledTask` records for pending request reminders/expiry when
  enabled.
- Use `subject.kind = "self"` or a stable request subject metadata shape.
- Runner behavior remains structural.

Exit criteria:

- No new task primitive.
- Pending reminders respect global pause where appropriate.

## Final Wave: Testing, Verification, Validation, E2E, Security

### Worker O: Contract and Unit Test Sweep

Tasks:

- Run core tests for policy and secrets actions.
- Run cloud app charge and sensitive request tests.
- Run local route tests.
- Run typecheck for touched packages.

### Worker P: Live LLM E2E

Tasks:

- Use a real configured LLM provider.
- Create an app-building scenario where the agent needs an API key.
- Assert the agent requests the key safely, receives it through the form/DM,
  resumes work, and never outputs the key.
- Repeat in public Discord-like room and private app chat.

### Worker Q: Payment E2E

Tasks:

- Create verified-payer charge, post link publicly, assert login required.
- Create any-payer charge with compatible settlement target, pay, assert
  callback.
- Assert wrong payer does not receive entitlements.

### Worker R: Security/Pentest Agents

Tasks:

- Attempt to submit another user's request id.
- Attempt replay after fulfillment.
- Attempt CSRF on browser submit.
- Attempt tunnel submit without auth.
- Attempt callback signature spoof.
- Search memory, conversation logs, action traces, HTTP callbacks, analytics,
  and audit logs for seeded canary secret value.
- Attempt prompt injection: user asks agent to repeat hidden secret.

Exit criteria:

- No seeded secret appears outside approved secret storage.
- All security findings are triaged before release.

## Suggested Commands

Use focused commands first, then package-level checks:

```bash
bun run --cwd packages/core test -- sensitive-request-policy
bun run --cwd packages/core test -- request-secret
bun run --cwd cloud test -- app-charge
bun run --cwd cloud typecheck
bun run test:launch-qa:docs
```

Live/e2e commands depend on credentials and connector setup:

```bash
TEST_LANE=post-merge bun run test:e2e:live
LIVE_LLM=1 bun run --cwd packages/core test:e2e:smoke
```
